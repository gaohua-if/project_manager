## 一、产品简介

AIDashboard 是 AI 部门内部的 Claude Code（及其他 AI Agent）使用追踪平台。它把 **需求 → 任务 → Session 上报 → Token 统计 → 日报** 这条数据链路打通，让每一条 AI 工作记录都可追溯、可统计、可汇报。

### 核心能力

| 模块 | 说明 |
|------|------|
| **需求管理** | 创建需求时由 AI 自动生成 3-8 条验收标准 (AC)，TL 把 AC 拆解为可执行任务并分配给员工 |
| **Session 上报** | 通过 CLI 工具扫描本地 `~/.claude/projects/*.jsonl`，一键上传当天的 Claude Code 会话；AI 自动匹配最相关的任务 |
| **Token 统计** | 按人员 / 团队 / 需求 / 模型 多维度聚合 Token 消耗，支持日 / 周 / 月切换 |
| **个人日报** | 员工在 Web 端点击一键生成 AI 草稿日报，可编辑后附加飞书链接 |
| **团队日报** | TL 一键汇总本组成员日报与自身 session，生成团队日报 |
| **组织视图** | 查看团队成员构成、活跃度、空闲告警（总监 / PM / TL 可见） |

### 系统架构

```
[员工本地机器]                              [服务器]

Claude Code → ~/.claude/projects/*.jsonl

aidashboard CLI ───HTTPS──→ Go API ──→ PostgreSQL
                                  ↕
                            Web Dashboard (Next.js)
                                  ↕
                            report-generator (consumer)
                                  ↓
                              claude -p 生成日报
```

## 二、登录与导航

### 2.1 登录

1. 打开 `http://<服务器地址>:3000`，进入登录页。
2. 在下拉列表中选择你的名字（如「张三」）。
3. 点击 **Login**，系统自动跳转到对应角色的 Dashboard。

> ⚠️ 系统为内部工具，登录仅按用户名识别身份，**无需密码**。请确保你的账号已由管理员在后台种入数据库。

### 2.2 主界面布局

登录后，左侧是**导航栏**，右侧是主内容区。

- **导航栏顶部**：显示你的名字与角色（如「张三 · 工程师」）。
- **导航栏底部**：
  - **复制 Token**：把 JWT 复制到剪贴板，CLI 工具登录时需要。
  - **退出登录**：清除本地凭据并返回登录页。
- 不同角色看到的导航菜单不同，详见下一节。

### 2.3 通用页面说明

| 页面 | 路径 | 作用 |
|------|------|------|
| Dashboard | `/dashboard` | 角色概览页（数据卡片 / 图表 / 告警） |
| 组织 | `/organization` | 团队成员、活跃度统计 |
| 需求 | `/requirements` | 创建和浏览需求 |
| 需求详情 | `/requirements/[id]` | 验收标准 (AC) 清单、关联任务、进度条 |
| 任务 | `/tasks` | 任务列表，可按状态/需求/负责人筛选 |
| 任务详情 | `/tasks/[id]` | 任务信息、依赖关系、状态更新 |
| 我的工作 | `/products` | 个人 Session 上报记录与匹配情况 |
| 报告 | `/reports` | 个人 / 团队日报 |

## 四、各角色操作指南

### 4.1 员工 (Employee)

#### 主流程

```
查看任务 → 执行 Claude Code 任务 → 标记任务完成
              ↓
        上报 Session (CLI)
              ↓
        在「我的工作」复核/修正任务关联
              ↓
        生成并编辑日报
```

#### 4.1.1 查看我的任务

1. 点击左侧 **任务**，可按 **状态 / 需求 / 负责人** 筛选。
2. 点击任务标题进入详情页：
   - 查看任务描述、关联的 AC、依赖关系。
   - 完成后点击 **Mark Done**（或修改状态为 `in_progress` / `blocked`）。

> 💡 任务状态变更后，系统自动重新计算所属需求的进度。需求进度达 100% 时，需求自动标记为 `completed`。

#### 4.1.2 上报 Claude Code Session

详见 [五、CLI 工具](#五cli-工具-aidashboard)。简要流程：

```bash
# 1. 首次使用：复制 Web 端的 Token，登录 CLI
aidashboard login --server http://<server>:8080/api/v1 --token eyJhbG...

# 2. 列出最近 48 小时的 session
aidashboard sessions

# 3. 上传指定编号的 session（会自动附带子 agent session）
aidashboard upload 1
```

#### 4.1.3 在「我的工作」复核 Session

1. 进入 **我的工作**，看到本人所有已上报的 Session 列表。
2. 每条记录显示：
   - 开始时间、模型、Token 用量、时长
   - AI 自动匹配到的 **任务** 和 **匹配置信度**
   - 摘要、Tool 调用统计、Git 提交
3. 如果 AI 匹配错误：
   - 通过下拉框手动选择正确的任务 → **保存**。
4. 如果上报了不该上报的 Session：
   - 点击 **Withdraw** → 物理删除（**不可恢复**，关联的 Token 记录也会一并删除）。

#### 4.1.4 生成日报

1. 进入 **报告** 页面。
2. 点击 **生成 AI 日报**：
   - 系统会拉取你**当天上报的所有 Session**，调用 `claude -p` 生成 Markdown 草稿。
   - 通常 10-60 秒，期间按钮显示「生成中...」。
3. 生成完成后：
   - 点击 **编辑** 修改内容。
   - 可选填入 **飞书文档链接**。
   - 点击 **保存**，日报标记为「已编辑」。
4. 历史日报按日期倒序展示在下方。

> ⚠️ **必须先上报 Session 才能生成日报**，否则日报内容为空。

### 4.2 团队负责人 (TL)

#### 主流程

```
创建需求 → AI 生成 AC → 复核 AC
              ↓
        拆解任务（关联 AC、指派成员）
              ↓
        跟踪成员 Session 上报情况
              ↓
        下班前一键生成团队日报
```

#### 4.2.1 创建需求

1. 进入 **需求** → 点击 **+ 新建需求**。
2. 填写：
   - **标题**：简短描述需求
   - **描述**：详细说明背景、目标、约束
   - **参与团队**：勾选需要协作的团队（可多选）
   - **优先级**：low / medium / high / urgent
   - **截止日期**：可选
3. 提交后：
   - 系统调用 AI 自动生成 **3-8 条验收标准 (AC)**。
   - AC 是「可观测、可检验」的一句话，例如「核心功能可以正常使用」。
4. 如果 AC 不满意，进入需求详情页 → 点击 **🤖 重新生成 AC**（**会覆盖当前 AC，可能影响已关联的任务**，需要二次确认）。

#### 4.2.2 拆解任务

1. 进入需求详情页 → **+ Add Task**。
2. 填写任务表单：
   - **任务标题**：动作 + 对象（如「实现登录接口」）
   - **关联验收标准**：勾选本任务要交付的 AC（可多选）
   - **指派给**：本团队成员
   - **优先级 / 截止日期**
3. 任务依赖（可选）：
   - 在任务详情页 → **Add Dependency**，建立 `finish_to_start` 依赖。
   - 被依赖的任务完成前，本任务会显示「被阻塞」。

> 💡 一条 AC 可以被多个任务关联，**只有所有关联任务都 `done`，该 AC 才算完成**。

#### 4.2.3 监控团队执行

1. **任务** 页面：按 **指派给** 筛选本团队成员，查看进度。
2. **我的工作** 页面：TL 可以看到本组**所有成员**的 Session 上报情况（员工只能看自己的）。
3. **组织** 页面：查看本组活跃度，包括：
   - 今日是否活跃（≥1 个已上报 session）
   - 最后活跃时间
   - 空闲天数（用于发现长时间未上报的成员）

#### 4.2.4 生成团队日报

进入 **报告** 页面（TL 视图为双 Tab）：

- **Team Daily Report Tab**
  - 点击 **Generate**：
    - 拉取 TL 自己当天的 Session。
    - 拉取本组成员当天的个人日报。
    - 调用 AI 生成包含「TL 工作 + 组员日报 + 团队总结」三段的 Markdown。
  - 生成结果可编辑内容和飞书链接。
  - 历史团队日报在下方折叠展示。

- **Member Reports Tab**
  - 按日期查看每位组员的日报。
  - **只读**，不能编辑组员日报（如需修改，请让组员本人编辑或联系总监）。
  - 用绿色 `Submitted` / 灰色 `Not submitted` 标识当日是否已提交。

> ⚠️ 团队日报只反映**生成时刻**的组员日报内容。如果组员在生成后修改了个人日报，TL 需要重新生成团队日报才能拿到最新内容。

### 4.3 产品经理 (PM)

PM 关注**需求交付质量与验收完成情况**，多数操作为只读。

#### 4.3.1 需求跟踪

1. **需求** 页面查看全部需求（含重点关注）。
2. 进入需求详情：
   - 查看 AC 清单和完成状态。
   - 查看关联任务的进度、负责人、截止日期。
3. PM 可创建需求和重新生成 AC，但**不能拆解任务、不能分配**（这是 TL 的职责）。

#### 4.3.2 跨团队阻塞监控

1. **Dashboard** 上查看：
   - 各需求进度条
   - 即将到期 / 已逾期需求告警
   - 跨团队依赖任务列表
2. **任务** 页面筛选 `status=blocked`，定位阻塞点。

#### 4.3.3 Token 与活跃度

- **报告** 页面查看全部门日报（按日期分组）。
- **组织** 页面查看团队活跃度，定位低活跃团队。

---

### 4.4 总监 (Director)

总监拥有最高权限，关注**部门全局成本、进度与跨团队协调**。

#### 4.4.1 Dashboard 概览

- **需求总览**：按状态分布的饼图、按月度趋势。
- **Token 消耗趋势**：日 / 周 / 月切换，按团队或模型分组。
- **团队活跃度**：每个团队的活跃人数 / 总人数。
- **空闲告警**：连续 N 天未上报 Session 的成员列表。

#### 4.4.2 跨团队协调

1. **需求** 页面：所有团队的需求，可按参与团队筛选。
2. **任务** 页面：跨团队依赖一目了然，定位阻塞。
3. **报告** 页面：
   - 按日期查看所有团队 / 成员的日报。
   - **总监可以编辑任何人的日报**（其他角色受限）。

#### 4.4.3 组织管理

进入 **组织** 页面：
- 查看全部团队和成员。
- 活跃度统计、空闲告警。
- （如未来开放）成员调动、新成员入组等管理操作。

## 五、CLI 工具 (aidashboard)

CLI 工具用于从员工本机扫描并上传 Claude Code Session 到服务器。

### 5.1 安装与登录

```bash
# 1. 编译（或使用发布版二进制）
cd daemon && go build -o aidashboard .

# 2. 复制到 PATH
sudo cp aidashboard /usr/local/bin/

# 3. 从 Web Dashboard 左下角「复制 Token」获取 JWT

# 4. 登录
aidashboard login --server http://<server>:8080/api/v1 --token eyJhbG...
```

登录成功后凭据保存在 `~/.aidashboard.yaml`。

### 5.2 子命令一览

| 子命令 | 别名 | 说明 |
|--------|------|------|
| `login` | - | 登录并保存 Token |
| `sessions` | `ls` | 列出本地 Session |
| `upload` | `push` | 上传 Session 到服务器 |
| `status` | - | 查看当前登录状态 |
| `serve` | - | （服务端用）启动 report-generator |
| `help` | `--help`, `-h` | 帮助 |

### 5.3 列出 Session

```bash
# 列出最近 48 小时的 session
aidashboard sessions

# 列出所有 session
aidashboard sessions --all

# 按项目目录筛选
aidashboard sessions --project project-manager
```

输出示例：

```
  #     Date                 Tokens     Duration   Model       Project                 Summary
  ------------------------------------------------------------------------------------------------------------
  1     2026-06-09 08:42     1.4M       1322m      glm-5.1     ..e/gh/project/manager  
                                               2 sub-agent(s)
  2     2026-06-08 12:11     473.3K     46m        glm-5.1     ..4b44f3/workspace/tmp  

  Total: 2 sessions
  Session logs: /home/gh/.claude/projects/
```

- `#` 列数字是上传时引用的编号。
- `Tokens` 包含子 agent。
- **列表只显示主 session**，子 agent 不单独显示。

### 5.4 上传 Session

```bash
# 上传指定编号
aidashboard upload 1
aidashboard upload 1 3 5

# 上传全部
aidashboard upload --all

# 交互式选择（不带参数）
aidashboard upload
```

**上传行为**：

- 上传主 session 时，**自动附带该主 session 下的所有子 agent session**。
- 已上传过的会显示 `[SKIP]`，不会重复上传。
- 上传后服务器**异步**调用 AI 匹配任务，几秒后刷新「我的工作」即可看到匹配结果。

### 5.5 查看登录状态

```bash
aidashboard status
```

输出：

```
Server:  http://localhost:8080/api/v1
Config:  /home/gh/.aidashboard.yaml
User:    张三 (employee)
Status:  logged in as 张三 (employee)
```

### 5.6 Session 文件位置

```
~/.claude/projects/
├── -home-gh-my-project/
│   ├── a48dd56f-...jsonl           ← 主 session
│   └── a48dd56f-.../
│       └── subagents/
│           ├── agent-ac179380a25e4d84b.jsonl    ← 子 agent session
│           └── agent-af7ac11502aba729d.jsonl
```

每个 `.jsonl` 是一次完整的 Claude Code 会话日志。

## 六、核心概念与数据规则

### 6.1 关键概念

| 概念 | 定义 |
|------|------|
| **Session** | 一次 Claude Code 启动 → 结束的完整交互。Resume 算新 session |
| **需求 (Requirement)** | 有明确验收标准的工作项 |
| **任务 (Task)** | TL 拆解的最小执行单元，状态：`todo` / `in_progress` / `done` / `blocked` |
| **验收标准 (AC)** | 需求创建时 AI 自动生成 3-8 条 |
| **Token** | 原始 Token 数（不折算为货币） |
| **活跃** | 当日 ≥1 个已上报 session |

### 6.2 自动计算规则

#### 需求进度

```
需求进度 = 已完成的 AC 数 / 总 AC 数 × 100%
AC 完成  = 该 AC 关联的所有任务均为 done 状态
```

当任何任务状态变化时，系统**自动**重算所属需求进度。进度达 100% → 需求状态自动变为 `completed`。

#### Session → 任务匹配

- 上传后服务器调用 AI 比对 Session 摘要与候选任务列表。
- 返回 `task_id` 和 `confidence`（0.0 - 1.0）。
- 用户可在「我的工作」手动覆盖匹配结果。

#### Token 聚合

- 按 `session_id` 维度存储，关联到 `user_id` / `task_id` / `requirement_id` / `model`。
- 同一 session 重复上传时，**替换**（非累加）Token 记录。
- 撤回 Session → 物理删除 Session + Token 记录。

### 6.3 数据保留

| 类型 | 保留策略 |
|------|---------|
| Session 元数据 | 永久 |
| Session 原始日志（MinIO） | 6 个月热存储，超期归档 |
| 日报 | 永久 |
| 撤回 | 物理删除，不可恢复 |

## 七、典型工作流

### 7.1 一个需求的完整生命周期

```
1. TL 创建需求 → AI 生成 AC
2. TL 复核 AC，必要时重新生成
3. TL 拆解任务：每个任务关联 AC、指派给员工
4. 员工看到任务 → 开始用 Claude Code 编码
5. 员工上报 Session → AI 自动匹配任务
6. 员工复核匹配，必要时手动修正
7. 员工完成任务 → Mark Done
8. 系统自动重算 AC 完成状态 → 需求进度更新
9. 所有 AC 完成 → 需求自动 completed
10. 员工下班前生成日报，TL 生成团队日报
```

### 7.2 上报 Session 的正确节奏

| 时间 | 动作 |
|------|------|
| 上午 | 开始用 Claude Code 工作 |
| 午休前 | `aidashboard sessions` 看一眼有数据 |
| 下班前 | `aidashboard upload --all` 上传全部当天 session |
| 上传后 | Web 端「我的工作」复核匹配 |
| 临走前 | Web 端「报告」生成日报 |

### 7.3 撤回错误 Session

如果不小心上传了私人项目 / 错误项目的 session：

1. 进入「我的工作」找到该 session。
2. 点击 **Withdraw**。
3. 确认删除（**不可恢复**）。
4. Token 统计、日报中相关数据会一并清除。

## 八、常见问题 (FAQ)

### Q1: CLI 提示 "Not logged in"

```bash
aidashboard login --server <url> --token <token>
```

Token 从 Web Dashboard 左下角「复制 Token」获取。

### Q2: `sessions` 命令看不到我的 session

- 确认 `~/.claude/projects/` 下有 `.jsonl` 文件。
- 默认只显示最近 48 小时，使用 `--all` 查看全部。
- 子 agent session 不单独显示，但上传时会自动附带。

### Q3: 上传后 Web 端看不到

- `aidashboard status` 确认登录的用户身份。
- Session 对**上报者本人、其 TL、总监**可见；其他 TL 看不到。
- 几秒后刷新页面（任务匹配是异步的）。

### Q4: 如何撤回已上报的 Session

「我的工作」→ 对应 session → **Withdraw**。撤回 = 物理删除。

### Q5: Token 数据为什么是 0

- Session 中可能没有 `assistant` 类型事件或缺少 `usage` 字段。
- 某些短 session 或仅使用内置工具的 session 可能不产生 Token。

### Q6: AI 匹配的任务不对

在「我的工作」中通过下拉框手动选择正确任务 → **保存**。后续 Token 统计和日报会按新任务归类。

### Q7: 如何修改日报

「报告」→ 找到日报 → **编辑** → 修改内容 / 附加飞书链接 → **保存**。

### Q8: TL 怎么汇总团队工作

「报告」→ **Generate** 团队日报。系统自动汇总 TL 自身 session + 本组成员日报。

### Q9: 忘记 CLI 配置文件在哪

```
~/.aidashboard.yaml
```

可以直接编辑或删除后重新 `aidashboard login`。

### Q10: 需求进度为什么不准确

- 检查任务是否都正确关联到 AC。
- 检查任务状态是否都为 `done`。
- AC 完成定义：**所有关联任务**都 `done`。

### Q11: 重新生成 AC 会有什么影响

会**覆盖**当前 AC，已关联的 AC 索引可能错位。建议仅在需求早期、尚未拆解任务时使用。系统会要求二次确认。

### Q12: 如何重置数据库（管理员）

```bash
docker compose down -v    # 停止并删除数据卷
docker compose up -d      # 重新启动（自动迁移 + 种子数据）
```
