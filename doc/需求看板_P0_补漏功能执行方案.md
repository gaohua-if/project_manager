> 本文件仅是「执行方案」，不动代码、不动 DB、不动接口、不改前端页面。

> 1. 需求删除：**支持** 在「无任何业务关联」时物理删除；有任何 task / session / token_usage / document 关联则强制走「取消需求」。

> 4. 同步操作：严格只写 `session.requirement_id` / `session.task_id`，不触碰 `task.status` / `task.progress`。
> 5. 实现优先级按用户给定 Phase 1→7。

## 〇 关键扫描结论

- **Sessions API 消费者全清**：`fetchSessions` 仅在 `SessionsPage.tsx`、`ProductsPage.tsx` 使用；`daemon/` 不调用 `GET /sessions`（仅 `POST /sessions/batch`）。 ⇒ 可直接换 response 形状。

## 三、需求取消 / 删除（按修正口径）

### 3.1 取消（必做）
- 复用 `PUT /requirements/{id} {status:"cancelled"}`，无新接口。
- 前端按钮文案 **「取消需求」**，确认弹窗文案：「取消后该需求将退出主看板，可在 `已取消` 列查看；历史 sessions / token 记录保留，关注关系保留」。
- Dashboard follows/risks SQL 已过滤 cancelled，无须改动。
- `user_follows` 保留。

### 3.2 物理删除（可选，低成本可做）
- **新增** `DELETE /api/v1/requirements/{id}?force=false`：
  1. 权限：`admin / director / pm`，`team_leader` 仅限创建人 == 自己或 requirement 与本团队关联。
  2. 事务内先做 EXISTS 检查（任一为 true 即拒绝）：
     ```sql
     EXISTS(SELECT 1 FROM tasks WHERE requirement_id=$1)
     OR EXISTS(SELECT 1 FROM sessions WHERE requirement_id=$1)
     OR EXISTS(SELECT 1 FROM token_usage WHERE requirement_id=$1)
     OR EXISTS(SELECT 1 FROM documents WHERE requirement_id=$1)
     ```
  3. 若有任何关联 → `409 Conflict { error:"requirement has associated tasks/sessions/usage; cancel instead", suggest:"cancel" }`；前端收到 409 自动弹「无法物理删除，是否改为取消？」。
  4. 若无关联 → 同事务：
     - `DELETE FROM user_follows WHERE target_type='requirement' AND target_id=$1`；
     - `DELETE FROM requirements WHERE id=$1`（`requirement_teams` 通过 CASCADE 清理）。
- 前端按钮：仅当 `tasks.length===0 && token_source_ids.length===0` 时显示「删除需求」；否则隐藏，只显示「取消需求」。
- 成本评估：后端 1 个新 handler + 4 个 EXISTS + 前端按钮可见性条件，**约 0.25 人日**，列为「P0 推荐做」。
- 若临实施仍嫌成本高，**回退方案**：本期只做取消，按钮文案严格使用「取消需求」。

## 五、任务删除（按修正口径，progress 重算入事务）

新增 `DELETE /api/v1/tasks/{id}`：

权限：`admin / director / pm / team_leader`（TL 走与 Create 相同的本团队校验）。

事务流程（**全部在同一 `db.Begin()` 内**）：

1. 取 `requirement_id`（FOR UPDATE 顺手锁需求行，避免并发聚合不一致）：
   ```sql
   SELECT requirement_id FROM tasks WHERE id=$1 FOR UPDATE
   ```
2. `UPDATE sessions SET task_id=NULL WHERE task_id=$1`；
3. `UPDATE token_usage SET task_id=NULL WHERE task_id=$1`；
4. `DELETE FROM user_follows WHERE target_type='task' AND target_id=$1`；
5. `DELETE FROM tasks WHERE id=$1`（依赖通过 `task_dependencies` 双侧 CASCADE 自动清）；
6. **同事务内**重算 requirement progress：
   ```sql
   UPDATE requirements
     SET progress = COALESCE((SELECT FLOOR(AVG(progress))::int FROM tasks WHERE requirement_id=$1),0),
         updated_at = now()
     WHERE id=$1
   ```
7. `tx.Commit()` —— 任一步失败 `tx.Rollback()`；progress 重算与删除强一致。
8. 提交后再返回 `{status:"deleted"}`。

前端：
- 抽屉任务卡 + `TaskDetailPage` 加「删除任务」按钮 + `modal.confirm`；
- 成功 invalidate `["requirements-board"]`、`["dashboard","follows"]`、`["dashboard","risks"]`。

孤儿验证（API 冒烟必须查）：
- `task_dependencies` 中无残留；
- `sessions.task_id` / `token_usage.task_id` 已 NULL；
- `user_follows` 无 task 项；
- `GET /dashboard/risks` 无 deleted task；
- requirement.progress 已重算。

## 十三、风险点

1. **类型迁移影响 ~50+ 处**：Phase 1 必须先行；用 `pnpm typecheck` 守门。
2. **任务删除事务中 progress 重算**：必须在事务内执行，避免事务提交后中断造成 progress 与 task 状态不一致。
3. **Session 分页 response 形状改变**：必须与前端同 PR 上线；否则前端 `.map(undefined)` 崩溃。
4. **物理删除 EXISTS 检查**：必须在事务里再检查一次（与首次预检之间可能有并发写入），否则可能误删。建议加 `FOR UPDATE` 锁。
5. **`team_leader` 删除权限**：必须复用 Create 时 team 校验（task）/ requirement_teams 校验（requirement），否则越权。
6. **Session picker 服务端分页多选保持**：分页切换时需以 id 集合保持已选，AntD `Table.rowSelection.preserveSelectedRowKeys` 务必开启。
7. **取消需求 vs 物理删除按钮可见性**：前端用 `tasks.length === 0 && token_source_ids.length === 0` 粗判，最终以后端 409 兜底；不要假设前端判定永远准确。
