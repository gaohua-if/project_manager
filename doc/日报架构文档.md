# AIcoding 管理平台日报架构设计文档

## 1. 文档目标

本文档用于梳理 AIcoding 管理平台日报模块的最终架构设计，包括：

1. 日报业务分层；
2. 角色权限与数据流；
3. 前端交互架构；
4. 后端接口设计；
5. 数据表设计；
6. Agent 生成链路；
7. 状态机设计；
8. Dashboard 与 `/reports/daily` 的数据一致性；
9. 测试与验收标准。

本设计只覆盖“日报”能力，不覆盖周报。

---

## 2. 核心产品口径

日报模块分为三个层级：

```text
个人日报
小组日报
部门日报
```

对应数据流：

```text
个人 session / AI 工作记录
→ 个人日报
→ TL 收集组员已发送的个人日报
→ 生成小组日报
→ TL 保存并发送给总监
→ 总监收集各小组已发送的小组日报
→ 生成部门日报
→ 总监保存部门日报
```

核心原则：

```text
原始日报是准确基准，Agent 生成内容只是基于原始数据的汇总草稿。
```

因此：

1. 个人日报由用户自己确认；
2. 小组日报必须先展示成员原始日报，再生成；
3. 部门日报必须先展示各小组日报，再生成；
4. 上级汇总只采纳“已发送”的日报内容；
5. 已保存但未发送的内容不能被上级汇总采纳。

## 3. 角色视角

### 3.1 Employee

Employee 只处理自己的个人日报。

能力：

```text
生成个人日报预览
保存个人日报
保存并发送给 TL
查看个人日报历史
```

Employee 不处理小组日报和部门日报。

---

### 3.2 TL / PM

TL / PM 同时有个人视角和管理视角。

个人视角：

```text
生成自己的个人日报
保存日报
保存并发送给总监
查看自己的个人日报历史
```

管理视角：

```text
查看成员已发送的个人日报
查看未发送成员
生成小组日报
保存小组日报
保存并发送给总监
查看小组日报历史
```

---

### 3.3 Director / Admin

Director / Admin 同时有个人视角和部门视角。

个人视角：

```text
生成自己的个人日报
保存日报
查看自己的个人日报历史
```

部门视角：

```text
查看各小组已发送的小组日报
查看未发送小组
生成部门日报
保存部门日报
查看部门日报历史
```

Director/Admin 的个人日报没有“发送给上级”动作。

## 5. 前端架构设计

## 5.1 Dashboard

Dashboard 只作为快捷入口，不作为完整日报管理页。

Dashboard 的职责：

```text
展示今日状态
打开日报弹窗
保存/发送成功后刷新状态
```

Dashboard 不应该：

```text
维护独立的假状态
跳转替代弹窗
实现另一套个人日报流程
使用 reportStateById 伪造状态
使用 ROLE_DATA 覆盖真实日报状态
```

正确交互：

```text
Dashboard 今日报告卡片
→ 点击“查看并生成日报”
→ 打开 DailyReportGenerateModal
→ 用户保存 / 保存并发送
→ invalidate/refetch 今日状态
→ Dashboard 卡片更新为已保存 / 已发送
```

---

## 5.3 个人日报弹窗

共享组件：

```text
web/src/features/aidashboard/reports/components/DailyReportGenerateModal.tsx
```

Dashboard 和 `/reports/daily` 必须共用该组件。

### 首次打开

如果今天没有已保存或已发送的个人日报：

```text
默认进入 Step 1：选择来源
```

Step 1：

```text
选择 session / AI 工作记录
点击生成日报
调用预览生成接口
进入 Step 2
```

Step 2：

```text
展示 Markdown 预览内容
用户编辑确认
底部展示保存 / 保存并发送按钮
```

Employee：

```text
上一步
保存日报
保存并发送给 TL
```

TL / PM：

```text
上一步
保存日报
保存并发送给总监
```

Director / Admin：

```text
上一步
保存日报
```

---

### 已保存 / 已发送后再次打开

如果今天已有已保存或已发送日报：

```text
默认进入 Step 2
编辑器展示 content
```

用户可以：

```text
直接修改后保存
直接修改后保存并发送
点上一步重新选择 session 并生成
```

如果重新生成会覆盖当前编辑区内容，需要提示：

```text
重新生成会覆盖当前编辑区内容，是否继续？
```

---

### 关闭弹窗

如果内容已修改但未保存 / 未发送：

```text
当前内容尚未保存，关闭后将丢失，是否关闭？
```

如果没有修改，直接关闭。

## 6. 个人日报状态设计

个人日报不需要草稿状态。

生成预览稿只是弹窗内临时内容，不是业务状态。

正式状态只保留：

```text
saved
submitted
```

### 状态含义

```text
saved:
日报已保存，但当前版本未发送给上级。

submitted:
日报已保存并发送给上级。
```

### 展示文案

Employee / TL / PM：

```text
无记录 / 无最终状态：待生成
status=saved：已保存
status=submitted：已发送
```

如果存在“已发送后又保存了新修改”的情况：

```text
status=saved 且 submitted_at 不为空：已保存，未发送最新修改
```

Director / Admin：

```text
无记录 / 无最终状态：待生成
status=saved：已保存
```

Director/Admin 不需要 submitted 状态。

## 7. 数据模型设计

## 7.1 `daily_reports`

建议字段：

```sql
id
user_id
report_date
content
submitted_content
session_ids
status
saved_at
submitted_at
submitted_to
feishu_doc_url
created_at
updated_at
```

### 字段说明

```text
content:
用户当前保存的日报内容。

submitted_content:
最近一次发送给上级的日报内容。

session_ids:
本次日报关联的 session。

status:
saved / submitted。

saved_at:
最近保存时间。

submitted_at:
最近发送时间。

submitted_to:
team_leader / director。

feishu_doc_url:
保留兼容字段，本期不驱动主状态。
```

### 为什么需要 `submitted_content`

原因：

```text
用户可能已经发送过日报，后来又修改并点击保存日报。
```

此时：

```text
content = 用户当前保存的新内容
submitted_content = 上一次已发送给上级的内容
```

上级汇总时必须使用：

```text
submitted_content
```

不能直接使用：

```text
content
```

否则用户本地保存但未发送的新修改会污染上级汇总来源。

---

## 7.2 `team_reports`

建议字段：

```sql
id
team_id
leader_id
report_date
content
submitted_content
source_daily_report_ids
status
saved_at
submitted_at
submitted_to
created_at
updated_at
```

说明：

```text
content:
TL 当前保存的小组日报内容。

submitted_content:
最近一次发送给总监的小组日报内容。

source_daily_report_ids:
本次生成小组日报使用的成员个人日报 ID。

status:
saved / submitted。

submitted_to:
director。
```

小组日报生成来源：

```text
status=submitted 的 daily_reports
优先读取 submitted_content
```

小组日报保存：

```text
更新 content
status=saved
saved_at=now()
不更新 submitted_content
```

小组日报保存并发送：

```text
更新 content
submitted_content=content
status=submitted
saved_at=now()
submitted_at=now()
submitted_to=director
```

---

## 7.3 `department_reports`

建议字段：

```sql
id
report_date
content
source_team_report_ids
status
saved_at
archived_at
created_at
updated_at
```

部门日报没有发送给上级动作。

说明：

```text

总监保存的部门日报内容。

本次生成部门日报使用的小组日报 ID。

saved。

保存时间。

archived_at:
如果后续区分归档，可使用该字段。
```

```text

```

## 8. 个人日报数据流

## 8.1 首次生成预览

```text
用户打开 DailyReportGenerateModal
→ 选择 session
→ POST /reports/today/draft
→ daemon 根据 session 生成 Markdown
→ API 返回预览内容
→ 前端展示在 Step 2 编辑器
```

注意：

```text
此时不写 status。
不产生 saved / submitted 状态。
用户关闭弹窗即视为未保存。
```

---

## 8.2 保存个人日报

```text
用户点击“保存日报”
→ PUT /reports/{id}
→ 保存 content / session_ids
→ status=saved
→ saved_at=now()
→ 返回最新 report
→ 前端刷新 Dashboard 和 /reports/daily
```

保存后：

```text
Dashboard 显示：已保存
/reports/daily Table 显示：已保存
上级不采纳该日报作为汇总来源
```

---

## 8.3 保存并发送个人日报

```text
用户点击“保存并发送给 TL / 总监”
→ POST /reports/{id}/submit
→ 后端事务内保存 content / session_ids
→ submitted_content=content
→ status=submitted
→ saved_at=now()
→ submitted_at=now()
→ submitted_to=team_leader / director
→ 返回最新 report
→ 前端刷新 Dashboard 和 /reports/daily
```

发送后：

```text
Dashboard 显示：已发送
/reports/daily Table 显示：已发送
上级汇总可以采纳 submitted_content
```

---

## 8.4 已发送后再次保存

```text
用户打开已发送日报
→ 默认进入 Step 2
→ 修改 content
→ 点击保存日报
→ content 更新
→ status=saved
→ saved_at 更新
→ submitted_content 不变
→ submitted_at 不变
```

结果：

```text
用户看到：已保存 / 未发送最新修改
上级仍然采纳上一次 submitted_content
```

---

## 8.5 已保存后再次发送

```text
用户打开已保存日报
→ 默认进入 Step 2
→ 点击保存并发送给上级
→ submitted_content=当前 content
→ status=submitted
→ submitted_at=now()
```

结果：

```text
用户看到：已发送
上级采纳最新 submitted_content
```

---

## 9. 小组日报数据流

## 9.1 TL 查看成员原始日报

```text
TL 打开小组日报
→ GET /reports/team/sources?date=YYYY-MM-DD
→ API 查询本组成员
→ 查询当天 status=submitted 的 daily_reports
→ 返回已发送成员日报和未发送成员列表
```

返回内容应包含：

```text
成员总数
已发送人数
未发送人数
已发送成员日报列表
未发送成员列表
```

已发送成员日报内容使用：

```text
daily_reports.submitted_content
```

而不是：

```text
daily_reports.content
```

---

## 9.2 生成小组日报

```text
TL 确认原始日报来源
→ POST /reports/team/today/generate
→ API 调 daemon /reports/team/generate
→ daemon 读取已发送成员日报 submitted_content
→ 生成小组日报 Markdown
→ upsert team_reports
→ 返回 team_report
```

生成后不代表已发送给总监。

推荐状态：

```text
status=saved
```

或如果不希望生成即保存，则只返回预览，由 TL 点击保存后落库。

P0 建议：

```text
生成小组日报可直接落 content，但状态为 saved。
```

---

## 9.3 保存小组日报

```text
TL 编辑小组日报
→ PUT /reports/team/{id}
→ 更新 content
→ status=saved
→ saved_at=now()
```

总监不会采纳该版本。

---

## 9.4 保存并发送小组日报

```text
TL 点击“保存并发送给总监”
→ POST /reports/team/{id}/submit
→ 保存 content
→ submitted_content=content

→ submitted_at=now()
→ submitted_to=director
```

```text

```

## 11. API 设计

## 11.1 个人日报接口

### 获取今天个人日报

```http
GET /api/v1/reports/today
```

用途：

```text
获取当前用户今天的个人日报。
如果当前实现会自动创建空记录，需要注意：空记录不能被展示成已生成。
```

返回字段：

```json
{
  "id": 1,
  "report_date": "2026-06-25",
  "content": "...",
  "submitted_content": "...",
  "session_ids": [],
  "status": "saved",
  "saved_at": "...",
  "submitted_at": "...",
  "submitted_to": "team_leader",
  "updated_at": "..."
}
```

---

### 生成个人日报预览

```http
POST /api/v1/reports/today/draft
```

用途：

```text
基于 session 生成 Markdown 预览。
```

要求：

```text
不写 status。
不写 saved。
不写 submitted。
不写 draft。
```

---

### 保存个人日报

```http
PUT /api/v1/reports/{id}
```

请求：

```json
{
  "content": "...",
  "session_ids": []
}
```

行为：

```text
content = 请求 content
session_ids = 请求 session_ids
status = saved
saved_at = now()
不更新 submitted_content
不更新 submitted_at
不更新 submitted_to
```

---

### 保存并发送个人日报

```http
POST /api/v1/reports/{id}/submit
```

推荐请求：

```json
{
  "content": "...",
  "session_ids": []
}
```

行为：

```text
content = 请求 content
session_ids = 请求 session_ids
submitted_content = 请求 content
status = submitted
saved_at = now()
submitted_at = now()
submitted_to = team_leader / director
```

权限规则：

```text
Employee → submitted_to=team_leader
TL / PM → submitted_to=director
Director/Admin → 不允许调用
```

---

### 我的日报历史

```http
GET /api/v1/reports/mine?from=YYYY-MM-DD&to=YYYY-MM-DD&page=1&page_size=20
```

用途：

```text
当前用户查看自己的个人日报历史。
```

注意：

```text
如果 GET /reports 对 TL/Director 返回下属或全量数据，则不能用作“我的日报”历史接口。
```

## 14. Dashboard 状态同步

Dashboard 今日报告卡片必须读取后端真实状态。

### Employee / TL / PM

```text
无记录 / 无最终状态 → 待生成
status=saved → 已保存
status=submitted → 已发送
status=saved 且 submitted_at 不为空 → 已保存，未发送最新修改
```

### Director / Admin

```text
无记录 / 无最终状态 → 待生成
status=saved → 已保存
```

保存 / 发送成功后必须：

```text
invalidate/refetch 当前用户今日日报
刷新 Dashboard 卡片状态
刷新 /reports/daily 我的日报 table
```

不能：

```text
使用 ROLE_DATA 覆盖真实状态
使用 reportStateById 伪造状态
保存后仍显示待生成
```

## 16. 异常与边界处理

### 16.1 用户生成预览后关闭

```text
不写 saved
不写 submitted
不更新 Dashboard 状态
```

如果内容已修改，应弹确认。

---

### 16.2 保存成功但发送失败

由于“保存并发送”建议在一个事务中完成，理论上不应出现半成功。

如果采用前端先 PUT 再 submit 的两步方案，可能出现：

```text
保存成功
submit 失败
```

这种情况会导致状态变为 saved。

推荐后端提供原子接口：

```http
POST /reports/{id}/submit
```

在一个事务中完成保存和提交。

---

### 16.3 已发送后再次保存

```text
status=saved
content 更新
submitted_content 保持不变
submitted_at 保持不变
```

上级仍看到旧的 submitted_content。

---

### 16.4 已保存后再次发送

```text
status=submitted
submitted_content 更新为当前 content
submitted_at 更新
```

上级看到新版本。

---

### 16.5 旧数据兼容

如果已有日报只有 content，没有 status：

处理建议：

```text
不强制迁移为 submitted。
可以迁移为 saved，或保持 status 为空并在报告中说明。
```

不要破坏旧 content。

## 18. 验收标准

### 18.1 个人日报

1. Dashboard 点击“查看并生成日报”仍然打开弹窗，不跳转。
2. `/reports/daily` 点击生成/编辑我的日报也打开同一个弹窗。
3. 两个入口共用 `DailyReportGenerateModal.tsx`。
4. 个人日报弹窗只有两步：

   * 选择来源
   * 确认日报
5. 没有“保存修改”按钮。
6. 没有第三步“保存日报”。
7. 不存在持久化 draft 状态。
8. Employee 按钮为：

   * 保存日报
   * 保存并发送给 TL
9. TL / PM 按钮为：

   * 保存日报
   * 保存并发送给总监
10. Director/Admin 只有：

* 保存日报

11. 保存日报后状态为“已保存”。
12. 保存并发送后状态为“已发送”。
13. 已保存/已发送日报再次打开，默认进入 Step 2。
14. 已发送后再次保存，不覆盖 submitted_content。
15. 再次保存并发送后，submitted_content 更新。
16. 关闭未保存预览，不产生 saved/submitted 状态。
17. Dashboard 和 `/reports/daily` 状态一致。
18. 刷新页面后状态仍然正确。

---

### 18.2 小组日报

1. TL 生成小组日报前，能看到成员原始日报来源。
2. 成员来源只统计已发送个人日报。
3. 小组日报生成使用 `daily_reports.submitted_content`。
4. 已保存但未发送的个人日报不进入小组日报来源。
5. 小组日报保存后不进入部门来源。
6. 小组日报保存并发送后，总监可在部门日报来源中看到。

---

### 18.3 部门日报

1. 总监生成部门日报前，能看到各小组日报来源。
2. 部门日报来源只统计已发送小组日报。
3. 部门日报生成使用 `team_reports.submitted_content`。
4. 未发送小组日报不进入部门日报来源。
5. 部门日报保存后刷新仍存在。

---

## 19. 最终数据流总结

```text
个人 session
→ 个人日报预览
→ 保存日报：daily_reports.content + status=saved
→ 保存并发送：daily_reports.submitted_content + status=submitted

TL 来源确认
→ 读取 status=submitted 的 daily_reports.submitted_content
→ 生成小组日报
→ 保存小组日报：team_reports.content + status=saved
→ 保存并发送：team_reports.submitted_content + status=submitted

总监来源确认
→ 读取 status=submitted 的 team_reports.submitted_content
→ 生成部门日报
→ 保存部门日报：department_reports.content + status=saved
```

一句话总结：

```text
content 是当前编辑版本，submitted_content 是上级可采纳版本。
保存只更新 content，保存并发送才更新 submitted_content。
```
