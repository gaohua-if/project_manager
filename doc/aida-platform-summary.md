## 0. 文档说明

本文档是 Aida 平台的**完整业务需求文档**，描述"做什么、给谁用、长什么样、规则是什么、怎样验收、怎样分阶段建设"。

不包含具体接口、表结构、技术选型——这些归技术方案与详细设计文档。本文档是产品层的唯一真相源。

## 1. 项目背景

### 1.1 现状

AI 部门当前 40 名工程师 + 5 名管理者，分布在 3 个团队（AI 工程 / 推理加速 / 模型训练）。日常使用 Claude Code 等 AI 编码助手已成常态，但出现以下问题：

1. **AI 使用情况不透明**：没人说得清部门一天烧了多少 Token、哪个团队/项目/模型为主。
2. **工作进展难以汇总**：员工每周写日报/周报靠手工拼凑，数据散落在 IM、文档、个人记忆里。
3. **需求与执行脱节**：需求文档写完即止，AC 没有被持续追踪，"做完了没"靠口头确认。
4. **跨团队协作阻塞难以暴露**：阻塞通常在 deadline 前一两天才被 PM/总监感知。
5. **AI 使用合规风险**：谁在用、用什么模型、跑了什么内容，没有审计抓手。

### 1.2 目标

打造一个 AI 部门内部的 **AI 编码工作流与效能洞察平台**，把员工的 AI 编码行为，按以下链路串成闭环：

```
需求 → 验收标准（AC）→ 任务 → Session → Token 消耗 → 个人/团队/部门日报与周报
```

让总监、PM、TL、员工每个角色在自己关心的视角下看到真实数据，并通过 AI 自动生成日报/周报（写入飞书文档），最终形成可量化、可追溯、可上飞书的部门 AI 效能闭环。

### 1.3 非目标（本期不做）

- 不替代飞书做即时通讯。
- 不替代 Jira / 禅道做精细工时与缺陷管理。
- 不做对外计费/对外报表。
- 不做 Token 折现成钱（会议明确：只看原始量）。
- 不做代码内容审计（仅元数据 + 摘要 + 用户主动上报的内容）。

## 3. 业务术语词典

| 术语 | 定义 | 关键约束 |
| --- | --- | --- |
| **需求（Requirement）** | 一段有明确验收标准的工作 | 有 AC 即可，不限大小 |
| **验收标准（AC，Acceptance Criteria）** | 衡量需求是否完成的可勾选条目 | AI 自动生成 3~8 条；TL/PM 可改 |
| **任务（Task）** | 需求拆出的可执行单元 | 任务**不带 AC**，只有完成/未完成；由 TL 关联到具体 AC 编号 |
| **任务依赖** | 任务 A 必须在任务 B 之后开始 | 形成 DAG；下游任务在上游完成后自动通知接手人 |
| **Session** | 一次 `claude` 启动到结束 | `/resume` 计为新 Session |
| **Token Usage** | 一个 Session 的输入/输出 Token 数 | 按模型分别记录；只看原始量，不折现 |
| **匹配置信度** | AI 把 Session 归到某个任务的把握 | 高/中/低 三档；员工可手动覆盖 |
| **活跃** | 当日 ≥ 1 个已上报 Session 的人 | 部门活跃度 = 活跃人数 / 团队总人数 |
| **沉寂预警** | 连续 3 天 0 上报 Session | 出现在总监 / TL 看板，不私聊员工 |
| **撤回** | 员工把已上报的 Session 物理删除 | 立即生效；同 Session 可再次上报 |
| **日报** | 每天 20:00 自动生成的 Markdown | 个人/团队/部门 三层 |
| **周报** | 每周五 17:00 自动生成 | 个人/团队/PM/部门 四层 |
| **数字分身** | 员工本地的 agent，未来代表本人接派单 | 第三阶段引入 |
| **Spec** | 用 AC 列表替代长篇需求文档的开发模式 | 平台是 spec 的"操作系统" |

## 4. 端到端业务流程

```
 ┌─ ① 创建需求 ─┐  ┌─ ② TL 拆任务 ──┐  ┌─ ③ 执行+上报 ──┐  ┌─ ④ 自动汇总 ──┐
 │ 总监/PM/TL    │→│ 任务+依赖       │→│ Claude Code     │→│ AC 自动确认   │
 │ AI 生成 3-8 AC │  │ 关联 AC          │  │ Session 自主勾选 │  │ 进度更新     │
 │                 │  │                  │  │ AI 匹配任务      │  │ 日/周报      │
 └─────────────────┘  └──────────────────┘  └──────────────────┘  └──────────────┘
```

### 4.1 创建需求

1. 入口：总监 / PM / TL 在「需求列表」点"新建"。
2. 必填：标题、描述、关联团队（多选）、deadline；可选：飞书文档链接、关联项目。
3. 提交后系统调用 AI 生成 3~8 条 AC（可重新生成，可手工编辑）。
4. **三层 review**（会议决策）：员工提的需求 → TL 审 → PM 审 → 总监审；下钻只看与自己相关的。
5. AC 锁定后进入"待拆解"。

### 4.2 TL 拆解任务

1. TL 进入需求详情，按 AC 拆任务。
2. 每个任务必填：标题、负责人、关联 AC 编号；可选：依赖（其它任务 ID）、预计工时。
3. 跨团队任务：可指定其它团队成员；该任务出现在双方 TL 视图。
4. 拆解完成后状态变为"执行中"。

### 4.3 执行 + Session 上报

1. 员工在本地用 Claude Code 工作，所有 Session 默认**留在本机**。
2. 每天若干次（员工自定）打开 Aida 客户端 → 看到本机当天 Session 列表 → 勾选要上报的 → 提交。
3. 服务端：
   - 接收 Session 元数据 + Token + 摘要；
   - AI 自动匹配到最可能的任务，给出置信度；
   - 员工可在 Web 端覆盖匹配结果。
4. **撤回**：员工对任意已上报 Session 点撤回 → 物理删除 + 关联 token 行 CASCADE 删除。

### 4.4 自动汇总

1. 任务状态变为完成 → 系统检查关联 AC 的所有任务是否都完成 → 是则该 AC 自动勾选。
2. 需求进度 = 完成 AC 数 / 全部 AC 数；100% 时需求自动转"已完成"。
3. 每天 20:00 / 每周五 17:00 触发报告生成（详见 §7）。

### 4.5 跨团队阻塞升级

| 时点 | 处理人 | 动作 |
| --- | --- | --- |
| 出现阻塞当天 | 双方 TL | 自行协商 |
| 超期 1 天未解决 | PM | 介入推动 |
| 超期 3 天未解决 | 总监 | 强制裁决 |

平台职责：把阻塞高亮在对应角色看板上；不主动派 IM（IM 由飞书机器人在第二阶段补）。

  - 红色告警条：异常成员 + 跨团队阻塞详情。

| 列 | 说明 |
| --- | --- |
| ☐ | 勾选要上报 |

### 5.5 需求列表 / 需求详情

- 列表：表头同总监看板的需求总览；按角色过滤数据范围（员工只看到与自己任务相关的需求）。
- 详情：基本信息、AC 列表（已完成画删除线）、任务列表（按 AC 编号分组）、Token 趋势（本需求维度）、关联飞书文档、操作区（编辑、重新生成 AC、关闭需求）。

### 5.6 任务列表 / 任务详情

- 列表：可按需求 / 负责人 / 状态过滤。
- 详情：任务信息、所属需求与 AC、上下游依赖图、关联 Session 列表（含 Token）、状态变更记录。

### 5.7 Session 中心

- 我的 Session：跨任务全量；可勾选/撤回。
- 团队 Session（仅 TL+）：本队所有人，仅元数据。

### 5.8 Token 中心

- 多维度切换：个人 / 任务 / 需求 / 团队 / 模型 / 项目。
- 时间粒度：日 / 周 / 月。

### 5.9 报告中心

- 我的日/周报历史。
- 团队 / 部门日/周报（按角色可见）。
- 每条报告显示：生成时间、飞书链接、Markdown 预览、重新生成按钮（仅当天）。

### 5.10 组织 / 账号

- 仅 Admin 可见的"用户管理"：列表、改角色、改团队、重置密码。
- 普通用户的"个人设置"：改密码、绑定飞书、自定义日报 prompt（千人千面）。

### 5.11 项目维度（会议追加）

- 在需求上挂"项目"字段；
- Token、报表、看板都加 `group_by=project` 维度；
- 项目列表页：项目名、负责人、关联需求数、累计 Token、活跃成员、近 30 天进展。

## 6. 数据规则（业务侧）

> 这是业务规则，不是表结构。

### 6.1 Session

- 唯一性：`(session_ref, user_id)` 不重复——同一 Session 多次上报视为更新。
- Token 行：每次上报都先**删除**该 Session 的全部 token 行，再重新插入；永不 append。
- 撤回：物理删除 Session + Token 行；可再次上报。
- `/resume` 视为新 Session（不同 `session_ref`）。

### 6.2 需求 / AC / 进度

- 需求进度 100% 派生：`progress = 完成 AC 数 / 总 AC 数`，禁止任何外部直写。
- AC 自动完成：当一个 AC 关联的所有任务都标记完成时，AC 自动勾选。
- 需求 100% → 自动状态转"已完成"。

### 6.3 Token

- 不做加权折现；不同模型只在维度里区分。
- "本周/本月"按部门时区计算（默认上海时区）。
- 可切换维度：用户 / 任务 / 需求 / 团队 / 模型 / 项目。

### 6.4 活跃 & 沉寂

- 活跃：当日有 ≥ 1 个**已上报**且**未撤回**的 Session 的人。
- 沉寂：连续 3 个工作日 0 上报；预警出现在总监/TL 视图，不打扰本人。

### 6.5 数据保留

- 热存储：6 个月。
- 冷归档：6 个月以上。
- 撤回：立即物理删除（不进归档）。

### 6.6 工作记录标签（会议新增）

- 每条 Session 可携带多个 tag：业务/非业务、需求 ID、项目 ID、自定义。
- AI 在匹配任务时同步给出 tag 候选；员工勾选确认。
- PM/TL 报表按 tag 维度抽取。

## 7. 报告规则

### 7.1 触发时点

| 报告 | 触发时点 | 投递目标 |
| --- | --- | --- |
| 个人日报 | 每天 20:00 | 飞书个人空间 |
| 个人周报 | 周五 17:00 | 飞书个人空间 |
| 团队日报 | 每天 20:00（个人完成后） | 团队飞书群 |
| 团队周报 | 周五 17:00（个人完成后） | 团队飞书群 |
| PM 周报 | 周五 17:30 | PM 个人 + 部门群 |
| 部门每日进展 | 每天 20:30（团队完成后） | 部门飞书群 |
| 部门周报 | 周五 18:00 | 部门飞书群 |

### 7.2 个人日报内容（千人千面 · 会议决策）

- 默认模板：今日完成任务、今日 Token、关联 Session 摘要、明日计划。
- 用户可在「个人设置」编辑 `report_prompt`（自定义风格、关注点、字数）。
- 同一份原始数据 + 不同 prompt → 不同风格的日报。

### 7.3 团队/部门报告内容

- 团队：成员逐项进度、本队 Token 分布、跨团队任务、阻塞、活跃度。
- 部门：3 团队对比、Token 趋势、需求总览、跨团队阻塞、升级建议。

### 7.4 投递载体

- **强约束**：所有正式报告写入飞书文档，不在站内单独维护富文本。
- 站内仅保留 Markdown 原文与飞书链接。

## 8. 权限与数据可见性矩阵

### 8.1 视图可见性

| 视图 | 员工 | TL | PM | 总监 | Admin |
| --- | :-: | :-: | :-: | :-: | :-: |
| 自己的 Session/Token/任务/日报 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 本团队成员所有数据 | — | ✓ | ✓ | ✓ | — |
| 全部门成员数据 | — | — | ✓ | ✓ | — |
| 用户管理（改角色/重置密码） | — | — | — | — | ✓ |
| 部门战略需求创建 | — | — | ✓ | ✓ | — |
| 跨团队阻塞看板 | — | 仅本队相关 | ✓ | ✓ | — |
| 项目维度汇总 | 仅自己参与的 | 本队相关 | ✓ | ✓ | — |

### 8.2 操作权限

| 操作 | 员工 | TL | PM | 总监 |
| --- | :-: | :-: | :-: | :-: |
| 创建需求 | 可（须 TL→PM→总监 review） | ✓ | ✓ | ✓ |
| 修改 AC | — | ✓ | ✓ | ✓ |
| 拆解任务 | — | ✓ | — | — |
| 改任务负责人 | — | ✓（本队） | ✓ | ✓ |
| 改 Session→任务 匹配 | 自己的 | 本队 | ✓ | ✓ |
| 撤回 Session | 自己的 | — | — | — |
| 关闭需求 | — | — | ✓ | ✓ |
| 标记重点关注 | — | — | ✓ | ✓ |
| 触发报告重新生成 | 自己的 | 本队 | ✓ | ✓ |

### 8.3 不可越权红线

- 员工**永远**看不到他人 Session 内容；
- TL 看不到外团队员工的明细；
- PM/总监看不到 Session 摘要正文（只看元数据 + Token），除非员工主动分享；
- Admin 不参与业务，不显示业务数据。

## 9. 信息安全与隐私

### 9.1 数据不离机原则

- 本地 Claude Code Session 默认**不上传**，存在 `~/.claude/projects/*/*.jsonl`。
- 员工每次主动勾选才上传；未勾选的永不离机。
- 已上报数据可随时撤回，撤回 = 物理删除（无软删除、无回收站）。

### 9.2 网络与部署

- 全部署在公司内网；不出公司网络。
- 对接 Claude / GPT 等外部 AI 接口走统一代理，留审计日志。

### 9.3 审计

- 所有 AI 操作（谁在何时用了哪个模型、Token、目的任务）必须可追溯。
- 管理员可导出审计日志。

### 9.4 鉴权

- 第一阶段：工号 + 密码（bcrypt）。
- 第二阶段起：飞书 SSO（强约束，会议决策）。

### 9.5 合规

- 数据保留 6 个月热存储；超期冷归档；冷归档保留 1 年后清理（待法务确认）。

## 10. 三阶段建设规划

会议决策：必须分阶段，不可一步到位。

### 10.1 第一阶段 · 统一 UI（=当前 demo 阶段）

**目标**：把"看板 + 上报 + 报告"基本形态打通，可在部门内 demo。

**交付**：

1. 4 角色 Web 看板（director / pm / tl / employee）。
2. 需求 → AC → 任务 → Session → Token 数据闭环。
3. 本地 CLI 上报 + 撤回。
4. AI 自动匹配 Session → 任务。
5. 个人/团队/部门日报与周报（生成 Markdown，飞书投递可后置）。

**收尾项**（demo 转 v1 前必做）：

- 创建需求/任务统一改为独立路由页面（符合模板规范）。
- PagePanel/HeaderNavProvider 全局接入。
- 角色权限二次 review，确认 admin 不污染统计。
- 工号+密码登录已可用。

### 10.2 第二阶段 · 小范围试点

**目标**：选 1~2 个团队（建议 AI 工程 + 推理加速）跑通真实业务，验证流程合理性。

**交付**：

1. **飞书 SSO** 替代密码登录。
2. **飞书文档投递**：所有报告自动写飞书。
3. **千人千面日报**：每个员工挂 `report_prompt`，daemon 用本人 prompt 渲染。
4. **工作记录标签体系**：业务/非业务、需求、项目 tag。
5. **移动端兼容**：日报查看 / Session 勾选/上报 / 需求审批 三条主路径响应式。
6. **项目维度汇总**：需求挂 `project_id`，全部报表加 group by。
7. **三层 review 工作流**：员工→TL→PM→总监审批链。
8. **告警/沉寂预警自动推送**：飞书机器人 @PM @总监。

### 10.3 第三阶段 · 全量自动化

**目标**：扩到全部门，实现"数字分身"承接派单 + 真正的 spec 驱动开发。

**交付**：

1. **数字分身**（员工本地 agent）：监听 Claude Code 启动 → 自动拉派给自己的任务上下文 → 完成后回写。
2. **自动派单**：TL 拆任务后由 agent 推到目标员工的分身（飞书消息 + 本地 daemon）。
3. **AI 操作审计看板**：飞书审计页 + 部门 dashboard。
4. **跨团队阻塞自动升级**：达到阈值自动飞书 @ 升级到下一级。
5. **Spec 驱动开发深度**：AC 编辑器、AC → 任务一键拆、AC 修改回流到任务。
6. **车间/外出移动端**：完整移动端覆盖。

### 10.4 阶段不可并行

- 第二阶段未跑通真实业务前**不上**数字分身；
- 第三阶段不并行启动，避免试点结论尚未沉淀就改方向。

## 12. 风险与依赖

### 12.1 业务风险

| 风险 | 等级 | 缓解 |
| --- | :-: | --- |
| 员工抵触上报（怕被监控） | 高 | "数据不离机 + 自主勾选 + 撤回物理删除" 三件套；管理者承诺不用 Token 数据做绩效评判 |
| AC 自动生成质量差 | 中 | 兜底 fallback AC + 人工修改 + 重新生成按钮 |
| 跨团队任务责任不清 | 中 | 阻塞自动可视化 + 三级升级路径 |
| 报告内容空洞 | 中 | 千人千面 prompt + 团队/部门报告人工 review 一周 |

### 12.2 技术依赖

- **Claude CLI（`claude -p`）必须可用**：AC 生成、Session 匹配、报告生成都依赖。
- **飞书开放平台**：第二阶段起 SSO + 文档写入接口必须打通。
- **公司内网**：内部部署、内网代理审计 Claude/GPT 流量。

### 12.3 组织依赖

- 需要总监 / PM / 3 个 TL 各 1 人参与首次试点 review。
- 移动端验证需要找 2~3 个外勤工程师配合。

## 13. 关键约定（写给后续维护者）

- **Token 不折现**——任何场景都不要把 Token 换算成钱。
- **撤回 = 物理删除**——不要做软删除/回收站，符合"数据不离机"心智。
- **AC 是真相**——`requirement.progress` 永远从 AC 派生，禁止任何接口直写。
- **AI 兜底必有**——`claude -p` 失败时走 fallback，不让用户卡住。
- **三阶段不并行**——上一阶段未收口前不启动下一阶段。
- **飞书是唯一报告载体**——站内不维护富文本编辑器。
- **不做绩效评判**——平台只做可视化，不直接打分。
