# 日报 / 周报页面重构需求方案

## 1. 背景

当前 AIcoding 管理平台的 `/reports` 页面同时承载“日报 / 周报 / 我的报告 / 小组报告 / 部门报告”等概念，但页面结构按角色直接分流，导致信息架构混乱。

主要问题包括：

1. TL 和总监进入报告页后，主要看到小组或部门视角，自己的个人日报入口不明显，甚至不可见。
2. 日报和周报混在同一个页面中，用户无法明确区分当前处理的是日报还是周报。
3. 小组日报、部门日报已经有部分真实接口和落库能力，但前端页面没有形成完整的“来源确认 → 生成草稿 → 编辑保存 → 提交/归档 → 历史记录”闭环。
4. 周报目前没有真实后端表、接口和生成链路，但 Dashboard 中存在前端本地状态伪造的周报能力，容易造成“看起来可用，实际不落库”的问题。
5. 不管是个人日报、小组日报、部门日报，还是小组周报、部门周报，都需要历史记录能力。

因此，本次需要重新梳理报告模块的信息架构。

---

## 2. 产品目标

报告模块拆成两个独立页面：

```text
/reports/daily   日报
/reports/weekly  周报
```

日报页面解决个人、小组、部门三个层级的日报处理与历史记录。

周报页面解决小组、部门两个层级的周报处理与历史记录。

当前产品口径暂时没有“个人周报”，因此本期不实现个人周报，也不新增 `personal_weekly_reports`。

---

## 3. 核心产品口径

### 3.1 日报

日报包含三个层级：

1. 我的日报
   所有角色都应该能看到自己的个人日报。

2. 小组日报
   TL 基于成员原始日报生成小组日报，并提交给总监。

3. 部门日报
   总监基于各 TL 提交的小组日报生成部门日报，并保存归档。

日报的核心流程是：

```text
成员个人日报
→ TL 查看成员原始日报
→ TL 生成小组日报草稿
→ TL 编辑并提交给总监
→ 总监查看各组小组日报
→ 总监生成部门日报草稿
→ 总监编辑并归档
```

关键原则：

```text
原始日报是准确基准，Agent 生成内容只是基于原始数据的汇总草稿。
```

所以无论是 TL 还是总监，都必须先看到原始来源，再触发 Agent 生成汇总稿。

---

### 3.2 周报

周报不能从菜单移除，也不能继续使用 Dashboard 本地假状态。

本期周报只做两个层级：

1. 小组周报
   TL 基于本周成员日报、本周小组日报、任务/风险摘要生成小组周报，并提交给总监。

2. 部门周报
   总监基于各 TL 已提交的小组周报生成部门周报，并保存归档。

当前不做：

```text
个人周报
```

因此员工角色可以不显示周报入口，或显示“个人周报暂未开放”。

## 4. 页面结构设计

## 4.1 日报页面 `/reports/daily`

日报页不再按角色完全分流成不同页面，而是在同一个日报页面中按角色展示不同视角。

### Employee

展示：

```text
我的日报
```

能力：

* 今日日报查看
* 生成日报草稿
* 编辑保存
* 我的日报历史记录

---

### TL

展示：

```text
我的日报
小组日报
小组日报历史
```

#### 我的日报

TL 也必须能看到自己的个人日报，不能因为有管理角色就丢失个人日报入口。

能力：

* 今日日报查看
* 生成日报草稿
* 编辑保存
* 我的日报历史记录

#### 小组日报

小组日报分为两个步骤。

Step 1：原始日报收集

展示内容：

* 成员总数
* 今日已交人数
* 今日未交人数
* 已编辑人数
* 成员日报列表
* 未提交成员列表
* 已提交成员日报原文查看

Step 2：小组日报草稿

操作：

* 基于已提交成员日报生成小组日报草稿
* 编辑保存
* 提交给总监

#### 小组日报历史

展示历史小组日报：

* 日期
* 提交人数
* 状态
* 提交时间
* 操作：查看 / 编辑

---

### Director / Admin

展示：

```text
我的日报
部门日报
小组日报记录
部门日报历史
```

#### 我的日报

总监也必须能看到自己的个人日报。

能力：

* 今日日报查看
* 生成日报草稿
* 编辑保存
* 我的日报历史记录

#### 部门日报

部门日报也必须先确认来源，再生成草稿。

Step 1：小组日报收集

展示内容：

* 已提交小组数
* 未提交小组数
* 已提交小组日报列表
* 未提交小组列表
* 每个小组日报原文查看

Step 2：部门日报草稿

操作：

* 基于已提交小组日报生成部门日报草稿
* 编辑保存
* 保存归档

#### 小组日报记录

总监可以查看各小组历史日报，作为部门日报来源追溯。

#### 部门日报历史

总监可以查看历史部门日报，包括已归档和未归档记录。

## 7. 周报新增数据模型

当前没有真实周报表。本期需要新增两个表。

### 7.1 `team_weekly_reports`

用途：

```text
小组周报真实留存、编辑保存、提交给总监、历史记录。
```

建议字段：

```text
id
team_id
leader_id
week_start
content
source_daily_report_ids
source_team_report_ids
source_task_ids
submitted_at
created_at
updated_at
```

唯一约束：

```sql
UNIQUE (team_id, week_start)
```

说明：

* `submitted_at` 为空表示未提交。
* `submitted_at` 非空表示已提交给总监。
* `leader_id` 表示负责生成和提交的小组负责人。

---

### 7.2 `department_weekly_reports`

用途：

```text
部门周报真实留存、编辑保存、归档、历史记录。
```

建议字段：

```text
id
week_start
content
source_team_weekly_report_ids
archived_at
created_at
updated_at
```

唯一约束：

```sql
UNIQUE (week_start)
```

说明：

* `archived_at` 为空表示未归档。
* `archived_at` 非空表示已归档。

---

### 7.3 本期不新增

```text
personal_weekly_reports
weekly_reports 通用表
审批流状态表
```

## 8. 周报新增接口

## 8.1 小组周报接口

### 本周来源确认

```http
GET /api/v1/reports/team/weekly/sources?week_start=YYYY-MM-DD
```

用途：

* 查询本周小组周报来源。
* 来源包括本周成员个人日报、本周小组日报、任务/风险摘要。

权限：

```text
team_leader 查看本组
director/admin 可查看全部或指定 team_id
```

---

### 本周小组周报详情

```http
GET /api/v1/reports/team/weekly/current?week_start=YYYY-MM-DD
```

用途：

* 查询本周小组周报详情。

---

### 生成小组周报

```http
POST /api/v1/reports/team/weekly/current/generate
```

用途：

* 基于本周来源生成小组周报草稿。
* 生成后 upsert 到 `team_weekly_reports`。

---

### 保存小组周报

```http
PUT /api/v1/reports/team/weekly/{id}
```

用途：

* 编辑保存小组周报正文。

---

### 提交小组周报给总监

```http
POST /api/v1/reports/team/weekly/{id}/submit
```

用途：

* 设置 `submitted_at`。
* 提交后总监可用于生成部门周报。

---

### 小组周报历史

```http
GET /api/v1/reports/team/weekly?from_week=YYYY-MM-DD&to_week=YYYY-MM-DD
```

用途：

* TL 查看本组历史小组周报。
* 总监查看各组小组周报记录。

---

## 8.2 部门周报接口

### 部门周报来源确认

```http
GET /api/v1/reports/department/weekly/sources?week_start=YYYY-MM-DD
```

用途：

* 查询各 TL 已提交的小组周报。

要求：

* 总监生成部门周报前，必须先看到按组提交的小组周报原文。
* 部门周报主来源是各 TL 已提交的小组周报。
* 不要直接使用总监个人 session、个人任务作为部门周报主来源。

---

### 本周部门周报详情

```http
GET /api/v1/reports/department/weekly/current?week_start=YYYY-MM-DD
```

用途：

* 查询本周部门周报详情。

---

### 生成部门周报

```http
POST /api/v1/reports/department/weekly/current/generate
```

用途：

* 基于已提交的小组周报生成部门周报草稿。
* 生成后 upsert 到 `department_weekly_reports`。

---

### 部门周报历史

## 10. Dashboard 处理原则

本期实施范围明确不修改 Dashboard 页面，也不修改 `web/src/features/aidashboard/dashboard/**` 下的文件。

处理原则：

1. 真实日报 / 周报闭环只在 `/reports/daily` 和 `/reports/weekly` 落地。
2. Dashboard 现有本地假周报状态不作为本次验收范围。
3. 报告页不得依赖 Dashboard 本地状态，必须走真实接口和真实落库。
4. 后续如要治理 Dashboard 中的假周报状态，应单独立项，避免和本次报告页重构混在一起。
5. 本期不出现“发送失败 / 重试发送 / 发送成功”等飞书相关状态。

## 12. 验收标准

### 日报

1. `/reports/daily` 独立存在。
2. Employee 只能看到“我的日报”。
3. TL 能看到“我的日报 + 小组日报 + 小组日报历史”。
4. Director/Admin 能看到“我的日报 + 部门日报 + 小组日报记录 + 部门日报历史”。
5. TL 生成小组日报前，必须能看到成员原始日报收集情况。
6. 总监生成部门日报前，必须能看到各组已提交的小组日报原文。
7. TL 和总监都能查看自己的个人日报历史。
8. 小组日报支持生成、编辑保存、提交给总监和历史记录。
9. 部门日报支持生成、编辑保存、归档和历史记录。
10. 不再出现假的“发送失败 / 重试发送 / 发送成功”。

### 周报

1. `/reports/weekly` 独立存在。
3. 本期不展示或不实现个人周报。
4. TL 能处理小组周报，包括来源确认、生成草稿、编辑保存、提交给总监、历史记录。
5. Director/Admin 能处理部门周报，包括小组周报来源确认、生成草稿、编辑保存、归档、历史记录。
6. 总监生成部门周报前，必须能看到各组提交的小组周报原文。
7. 小组周报和部门周报都必须真实落库。
8. 刷新页面后，周报内容和状态仍然存在。
9. 历史记录能按周查询。
10. 不新增个人周报表和个人周报生成接口。
