# 日报周报页面需求方案

## 背景与目标

本文件只做当前代码盘点和最小新增接口建议，不包含实现方案和代码改动。

目标是为报告页面重构提供事实基线：将“日报 / 周报 / 我的报告 / 小组报告 / 部门报告”梳理清楚，并支撑后续拆分为：

- `/reports/daily`
- `/reports/weekly`

当前代码中，日报能力已经覆盖个人、小组、部门三个层级的一部分；周报主要停留在 Dashboard 前端状态和文案，占位成分较重，没有真实后端 API 和落库模型。

---

## 一、当前前端页面现状

### 1. `/reports` 当前如何按角色分流

路由入口在 `web/src/router/routes.tsx`，`/reports` 菜单标题为“日报 / 周报”，但实际页面组件是 `ReportsPage`。

`ReportsPage` 在 `web/src/features/aidashboard/reports/pages/ReportsPage.tsx` 内部按 `user.role` 分流：

| 角色 | 渲染组件 | 当前页面口径 |
| --- | --- | --- |
| `employee` | `EmployeeReportsView` | 个人日报 |
| `pm` | `PMReportsView` | 团队成员日报 + 生成我的日报 |
| `team_leader` | `TLReportsView` | 团队日报 |
| `director` | `DirectorReportsView` | 部门报告 |
| `admin` | `DirectorReportsView` | 部门报告 |

路由层未对 `/reports` 配置角色白名单，所有已登录用户都可以进入，具体内容由页面内部决定。

### 2. TL 当前看到哪些内容

TL 进入 `/reports` 后看到 `TLReportsView`，页面标题为“团队日报”，当前不是“我的日报 + 小组日报”的组合页。

TL 页面有两个视图，由 `Segmented` 切换：

1. 原始日报收集
   - 日期可选。
   - 调用 `fetchTeamReportSources(date)`。
   - 后端对应 `GET /api/v1/reports/team/sources?date=YYYY-MM-DD`。
   - 展示组员原始个人日报、已提交/未提交数量、成员内容。

2. 小组日报草稿
   - 调用 `fetchTeamReportTodayOrNull()`。
   - 后端对应 `GET /api/v1/reports/team/today`。
   - 只读取“今日小组日报”，不是按所选日期读取草稿。
   - 可调用 `generateTeamReport()` 生成小组日报草稿。
   - 可调用 `updateTeamReport(id, { content })` 保存。
   - 可调用 `submitTeamReport(id)` 提交给总监。

TL 当前问题：

- `/reports` 页面没有展示 TL 自己的个人日报/周报。
- TL 的小组日报草稿只查今日，缺少前端历史列表。
- 后端有 `GET /reports/team?from=&to=` 可查小组日报历史，但当前 TL 页面没有使用。

### 3. 总监当前看到哪些内容

总监/Admin 进入 `/reports` 后看到 `DirectorReportsView`，页面标题为“部门报告”。当前不是“我的日报 + 部门日报 + 各小组记录”的组合页。

总监页面有两个视图：

1. 小组日报收集
   - 日期可选。
   - 调用 `fetchDepartmentReportSources(date)`。
   - 后端对应 `GET /api/v1/reports/department/sources?date=YYYY-MM-DD`。
   - 只展示当天已提交给总监的小组日报，以及未提交小组。
   - 数据来源是 `team_reports.submitted_at IS NOT NULL`。

2. 部门日报草稿
   - 调用 `fetchDepartmentReportTodayOrNull()`。
   - 后端对应 `GET /api/v1/reports/department/today`。
   - 只读取今日部门日报。
   - 可调用 `generateDepartmentReport()` 生成部门日报草稿。
   - 可调用 `updateDepartmentReport(id, { content })` 保存。
   - 可调用 `updateDepartmentReport(id, { archive: true })` 归档。

总监当前问题：

- `/reports` 页面没有展示总监自己的个人日报/周报。
- `/reports` 页面没有直接展示“各小组日报历史记录”列表。
- 后端有 `GET /reports/team?from=&to=` 可查小组日报历史，但当前总监报告页未使用。
- 部门日报只有今日详情和按日期来源，没有部门日报历史列表接口。

### 4. 个人日报入口在哪里

当前个人日报入口分散在两个地方：

1. `/reports`
   - `employee`：进入后就是个人日报页。
   - `pm`：页面右上角有“生成我的日报”，但主体是团队成员日报列表。
   - `team_leader`：没有个人日报入口。
   - `director/admin`：没有个人日报入口。

2. Dashboard 页面 `web/src/features/aidashboard/dashboard/DashboardPage.tsx`
   - `ROLE_DATA` 中为 employee、TL、director、PM 都配置了个人日报和个人周报卡片。
   - 个人日报卡片会读取当天真实日报列表：`fetchReports({ from: reportDate, to: reportDate })`。
   - 个人日报弹窗可走真实接口：选择 session → `POST /reports/today/draft` 生成草稿 → `PUT /reports/{id}` 保存。
   - TL/总监“必须能看到自己的个人日报/周报”目前主要在 Dashboard 有入口，不在 `/reports` 页面内闭环。

### 5. 周报当前是真实接口还是前端 mock / 本地状态

结论：周报当前不是后端真实接口能力，主要是 Dashboard 前端本地状态/占位文案。

证据：

- `api/main.go` 没有注册 weekly/周报相关路由。
- `api/db/migrations/` 没有 `weekly_reports` 或 `week_start` 字段。
- `daemon/server_reports.go` 没有 weekly 生成 endpoint。
- `web/src/features/aidashboard/api/client.ts` 没有 weekly API client。
- `DashboardPage.tsx` 中存在 `personal_weekly`、`team_weekly`、`department_weekly` 类型和按钮文案，但非 `personal_daily` / `department_daily` 的生成、保存、归档主要通过 `reportStateById` 本地状态更新完成。

因此：

| 周报类型 | 当前状态 |
| --- | --- |
| 个人周报 | Dashboard 有卡片和弹窗文案，但无真实 API / DB |
| 小组周报 | Dashboard 有卡片和弹窗文案，但无真实 API / DB |
| 部门周报 | Dashboard 有卡片和弹窗文案，但无真实 API / DB |

### 6. 历史记录当前是否存在，分别覆盖哪些报告类型

| 报告类型 | 后端历史能力 | 前端当前使用情况 |
| --- | --- | --- |
| 个人日报 | 有，`GET /reports?from=&to=` | Employee/PM `/reports` 使用近 7 天；Dashboard 只查当天 |
| 小组日报 | 有，`GET /reports/team?from=&to=` | 当前 `/reports` TL/总监页面未形成历史列表；总监部门来源只按日期看已提交小组日报 |
| 部门日报 | 无历史列表接口；只有今日详情和按日期来源 | `/reports` 只读今日部门日报草稿；可按日期看来源 |
| 个人周报 | 无 | Dashboard 本地状态 |
| 小组周报 | 无 | Dashboard 本地状态 |
| 部门周报 | 无 | Dashboard 本地状态 |

### 4. 外部 API：周报

当前没有以下真实接口：

- 个人周报接口
- 小组周报接口
- 部门周报接口
- 本周详情接口
- 周报来源接口
- 周报生成接口
- 周报保存接口
- 周报提交/归档接口
- 周报历史接口

Dashboard 中的周报卡片和弹窗为前端本地状态，不是真实 API。

### 5. daemon 内部 report-generator 接口

这些接口不是前端直接调用，而是 API 通过 `REPORT_GENERATOR_URL` 调用的内部服务接口。

| Method + Path | 用途 | 输入来源 | 是否调用 Claude | 是否落库 |
| --- | --- | --- | --- | --- |
| `POST /reports/generate` | 生成个人日报并 upsert | DB 中用户当天 sessions、token、tasks、requirements | 是；失败返回错误 | `daily_reports` |
| `POST /reports/draft` | 基于 API 传入的 session/task 候选生成个人日报草稿 | API 传入的 selected sessions 和 task candidates | 是；返回草稿 | 不直接落库 |
| `POST /reports/team/generate` | 生成小组日报 | TL 当天 sessions + 组员个人日报 | 是；失败 fallback | `team_reports` |
| `POST /reports/department/generate` | 生成部门日报 | 已提交小组日报 | 是；失败 fallback | `department_reports` |

## 三、当前数据表清单

### 1. `daily_reports`

定义在 `api/db/migrations/001_init.sql`。

| 字段能力 | 是否支持 | 说明 |
| --- | --- | --- |
| `report_date` | 支持 | 日报日期；唯一约束 `(user_id, report_date)` |
| `week_start` | 不支持 | 无周报字段 |
| `content` | 支持 | 日报正文 |
| `status` | 不支持 | 无草稿/已提交/已归档状态列 |
| source ids | 部分支持 | `session_ids UUID[]`，记录来源 session；无 task/requirement source ids |
| `submitted_at` | 不支持 | 无提交时间 |
| `archived_at` | 不支持 | 无归档时间 |
| `generated_by` | 不支持 | 无生成者字段 |
| 其他 | 支持 | `edited`、`feishu_doc_url`、`created_at`、`updated_at` |

### 2. `team_reports`

定义在 `api/db/migrations/004_team_reports.sql`，并由 `010_report_sources.sql` 增补来源和提交字段。

| 字段能力 | 是否支持 | 说明 |
| --- | --- | --- |
| `report_date` | 支持 | 小组日报日期；唯一约束 `(team_id, report_date)` |
| `week_start` | 不支持 | 无周报字段 |
| `content` | 支持 | 小组日报正文 |
| `status` | 不支持 | 无统一 status；提交状态由 `submitted_at IS NOT NULL` 表示 |
| source ids | 支持 | `member_report_ids`、`source_daily_report_ids`、`session_ids` |
| `submitted_at` | 支持 | TL 提交给总监的时间 |
| `archived_at` | 不支持 | 小组日报无归档字段 |
| `generated_by` | 不支持 | 无通用字段；`leader_id` 可视为小组日报负责人 |
| 其他 | 支持 | `feishu_doc_url`、`created_at`、`updated_at` |

### 3. `department_reports`

定义在 `api/db/migrations/010_report_sources.sql`。

| 字段能力 | 是否支持 | 说明 |
| --- | --- | --- |
| `report_date` | 支持 | 部门日报日期；唯一约束 `(report_date)` |
| `week_start` | 不支持 | 无周报字段 |
| `content` | 支持 | 部门日报正文 |
| `status` | 不支持 | 无统一 status；归档状态由 `archived_at IS NOT NULL` 表示 |
| source ids | 支持 | `source_team_report_ids UUID[]` |
| `submitted_at` | 不支持 | 部门日报不提交，只归档 |
| `archived_at` | 支持 | 总监归档时间 |
| `generated_by` | 不支持 | 无生成者字段 |
| 其他 | 支持 | `created_at`、`updated_at` |

### 4. weekly / department weekly 相关表

当前没有发现：

- `weekly_reports`
- `personal_weekly_reports`
- `team_weekly_reports`
- `department_weekly_reports`
- `week_start`

因此所有周报能力没有真实落库模型。

## 四、缺口分析

### 1. 个人日报

| 能力 | 当前状态 | 是否缺口 |
| --- | --- | --- |
| 今日详情 | `GET /reports/today` 支持 | 不缺 |
| 来源数据 | Dashboard 通过 `/sessions` 选 session，非专用 source 接口 | 部分缺 |
| 生成草稿 | `POST /reports/today/draft` 支持 | 不缺 |
| 生成并落库 | `POST /reports/today/generate` 支持 | 不缺 |
| 保存 | `PUT /reports/{id}` 支持 | 不缺 |
| 历史列表 | `GET /reports?from=&to=` 支持 | 不缺 |
| 提交/归档 | 无字段、无接口 | 如果产品要求个人日报归档，则缺；当前口径可不新增 |
| TL/总监个人日报入口 | Dashboard 有，`/reports` 无 | 页面缺口 |

### 2. 小组日报

| 能力 | 当前状态 | 是否缺口 |
| --- | --- | --- |
| 原始成员日报来源 | `GET /reports/team/sources?date=` 支持 | 不缺 |
| 今日小组日报详情 | `GET /reports/team/today` 支持 | 不缺 |
| 生成草稿 | `POST /reports/team/today/generate` 支持，但生成后即 upsert `team_reports` | 不缺；但不是纯 draft 表 |
| 保存 | `PUT /reports/team/{id}` 支持 | 不缺 |
| 提交给总监 | `POST /reports/team/{id}/submit` 支持 | 不缺 |
| 历史列表 | `GET /reports/team?from=&to=` 支持 | 后端不缺；TL 前端未使用 |
| 总监查看各小组日报记录 | 可复用 `GET /reports/team?from=&to=` | 前端页面缺口 |
| 按日期查看小组日报详情 | 只有 today 专用；历史可从 list 取 | 轻微缺口，可先复用列表 |

### 3. 部门日报

| 能力 | 当前状态 | 是否缺口 |
| --- | --- | --- |
| 原始小组日报来源 | `GET /reports/department/sources?date=` 支持 | 不缺 |
| 今日部门日报详情 | `GET /reports/department/today` 支持 | 不缺 |
| 生成草稿 | `POST /reports/department/today/generate` 支持，并 upsert `department_reports` | 不缺；但不是纯 draft 表 |

| 历史列表 | 无 `GET /reports/department?from=&to=` | 缺 |
| 按日期部门日报详情 | 外部只有 today；内部有按 date 查询函数 | 缺外部接口 |

### 4. 个人周报

| 能力 | 当前状态 | 是否缺口 |

## 五、最小新增接口建议

约束：

- 不接飞书。
- 不引入审批流。
- 不改日报之外模块。
- 不设计复杂权限系统，只延续现有角色权限。
- 优先复用已有 `daily_reports` / `team_reports` / `department_reports` 逻辑。

### 2. 日报必须新增或补齐的接口

#### 新增：部门日报历史列表

```http

```

用途：

- 支撑 `/reports/daily` 中“部门日报历史记录”。
- 支撑总监查看已归档/未归档部门日报记录。

权限：

返回字段建议沿用 `DepartmentReport`：

- `id`
- `report_date`


### 3. 周报必须新增的最小接口

周报目前无表、无 API、无 daemon 生成入口。只靠复用日报表无法满足“保存、历史、提交/归档”的产品要求，因此必须新增周报持久化模型。接口层建议按现有日报层级镜像，权限沿用现有角色。

#### 个人周报

| Method + Path | 用途 | 权限 | 来源建议 |
| --- | --- | --- | --- |
| `GET /api/v1/reports/weekly/today` 或 `GET /api/v1/reports/weekly/current?week_start=` | 本周个人周报详情 | 登录用户，仅本人 | 周报表 |
| `GET /api/v1/reports/weekly/sources?week_start=` | 本周来源数据 | 登录用户，仅本人 | `daily_reports` + sessions/tasks/token |
| `POST /api/v1/reports/weekly/current/generate` | 生成个人周报草稿 | 登录用户，仅本人 | daemon 新增 weekly generate |
| `PUT /api/v1/reports/weekly/{id}` | 保存个人周报 | 登录用户，仅本人 | 周报表 |
| `GET /api/v1/reports/weekly?from_week=&to_week=` | 个人周报历史 | 登录用户；角色范围可沿用 `GET /reports` | 周报表 |

最小说明：

- 个人周报没有提交/审批要求时，不建议新增 submit。
- 如果产品需要“归档周报”，可在 `PUT` 中支持 `archive: true`，但不要引入审批流。

### 4. 周报数据模型最小建议

为避免复杂权限系统，建议沿用现有日报的三层模型，而不是设计通用审批流。

最小可选方案：

1. 方案 A：三张表，最贴近现有日报结构
   - `weekly_reports`：个人周报。
   - `team_weekly_reports`：小组周报。
   - `department_weekly_reports`：部门周报。

2. 方案 B：一张统一表，接口仍按层级拆分
   - `weekly_reports` 增加 `scope`、`user_id`、`team_id` 等字段。
   - 好处是表少；坏处是权限和唯一约束更复杂。

建议采用方案 A，因为当前日报已经是 `daily_reports` + `team_reports` + `department_reports` 的分层模型，周报镜像实现最小、最稳定。

周报表必须支持字段：

- `week_start`
- `content`
- source ids
- `submitted_at`：小组周报需要。
- `archived_at`：部门周报需要；个人周报如要归档也可支持。
- `created_at`
- `updated_at`

`status` 和 `generated_by` 可先不做：

- 草稿/提交/归档可以继续由 `submitted_at`、`archived_at` 推导。
- 生成者可先沿用 `user_id` / `leader_id`，避免引入复杂审计模型。

### 5. 最小页面落地建议

#### `/reports/daily`

页面内按视角分块，而不是按角色覆盖掉个人视角：

- 我的日报：所有角色都显示。
- 小组日报：TL/PM/director/admin 显示；TL 可生成/保存/提交，director/admin 可看记录。
- 部门日报：director/admin 显示；可看来源、生成、保存、归档、历史。

#### `/reports/weekly`

页面内按视角分块：

- 我的周报：所有角色都显示。
- 小组周报：TL/PM/director/admin 显示。
- 部门周报：director/admin 显示。

周报上线前必须先补真实 API 和表，否则 Dashboard 当前周报卡片只能算前端占位。
