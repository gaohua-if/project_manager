# 决策 001：首页一期方向

| 项 | 值 |
| --- | --- |
| 状态 | 当前有效决策 |
| 日期 | 2026-06-21 |
| 范围 | Aida 一期首页方向 |
| 类型 | 产品决策，不是实现规格 |

## 1. 结论

一期先冻结角色化首页。

首页定位为员工、TL、总监、PM 的角色化工作台。它不是 Token 大屏，不是通用 BI Dashboard，也不是完整项目管理系统的首页。

当前 demo 的首页应优先向固定角色模板调整。需求、任务、AC、Session、报告和 Token 页面可以继续作为首页动作的承接页，但一期不从重做整套任务管理链路开始。

## 2. 背景判断

首页是当前 demo 和一期方向差距最大的部分，也决定用户如何理解 Aida：

- 如果首页以 Token 为中心，Aida 会被理解成用量统计平台。
- 如果首页以任务列表为中心，Aida 会被理解成项目管理工具。
- 如果首页以报告为中心，Aida 会被理解成日报/周报入口。
- 如果首页以角色工作台为中心，Aida 才能验证核心价值：让不同角色快速知道当前最该关注和处理什么。

需求、任务、AC 这套链路在一期不需要先大改。它们目前更适合作为首页事项的承接页和上下文来源。

## 3. 决策依据与优先级来源

本决策不是从当前 demo 或当前代码实现反推出来的。

2026-06-16 会议纪要中的明确方向优先于旧 PRD 和当前 demo。旧 PRD 提供一期功能骨架和角色模块来源，会议纪要提供首页信息权重和产品表达修正，决策 001 是当前首页方向的冻结结果。

| 来源 | 作用 | 优先级 |
| --- | --- | --- |
| 2026-06-16 会议纪要中的明确决策 | 修正方向、信息权重、产品表达 | 最高 |
| 决策 001 | 当前首页方向冻结口径 | 当前有效 |

关键口径：

后续首页设计应以“旧 PRD 角色模块 + 会议纪要方向修正”为准。demo 和现有代码只能作为实现现状参考，不能反向决定产品方向。

## 4. 首页原则

每个角色首页首屏必须回答一个问题：

> 我现在最应该看什么、处理什么？

Token、Session 和报告数据都是辅助证据。它们可以解释工作量、活跃度和异常信号，但不能成为首页主叙事。PM 首页不展示 Token 模块，避免把 PM 视角带偏到用量分析。

## 5. 固定角色模块

| 角色 | 首页主模块 |
| --- | --- |
| 员工 | Session 上报、飞书日报/周报、个人 Token、被关注任务 |
| TL | 任务拆解 + AC 关联、团队报告、成员面板、本队 Token、上级关注事项 |
| 总监 | 需求总览、部门报告、团队活跃度、Token 趋势、重点关注需求 |
| PM | 重点关注需求、AC 追踪、PM 报告、跨团队阻塞 / 推进异常 |

## 6. 员工首页

员工首页服务个人执行和低负担上报。

主要目标：

- 上传或绑定当天 Session。
- 确认个人日报 / 周报状态，并展示飞书日报 / 周报链接入口。
- 查看被 TL、PM 或总监关注的个人任务。
- 理解是否有下游任务正在等待自己的输出。
- 查看个人 Token，但只作为工作证据和辅助信息。

首屏优先级：

1. 个人待处理动作：Session 上报、Session 绑定、报告确认。
2. 被关注任务：谁关注、为什么关注、deadline、是否缺少工作证据。
3. 个人证据小组件：今日 Session 和 Token 摘要。

文案原则：

员工侧要避免“被监控”的感受。优先使用“补齐工作记录”“确认报告草稿”“支撑任务进展”等表达，谨慎使用“异常”“无记录”等管理视角文案。

## 7. TL 首页

TL 首页服务团队执行和上级关注事项处理。

主要目标：

- 查看待拆解需求。
- 查看缺少 AC 关联的任务。
- Review 团队日报 / 周报，并展示团队报告链接入口。
- 扫描成员状态、阻塞和缺少工作证据的事项。
- 查看 PM 或总监关注到本队的事项。
- 查看本队 Token，但只作为团队活跃和工作证据辅助信号。

首屏优先级：

1. 上级关注事项和团队阻塞任务。
2. 任务拆解与 AC 关联缺口。
3. 团队报告 Review 状态。
4. 成员面板。

## 8. 总监首页

总监首页服务部门级可见性和重点风险判断。

主要目标：

- 查看部门需求总览、整体进度和风险。
- 查看部门报告状态，并展示部门报告链接入口。
- 对比团队活跃度。
- 查看 Token 趋势作为背景证据。
- 查看高关注需求和跨团队风险。

首屏优先级：

1. 部门需求总览和整体风险。
2. 重点关注需求。
3. 部门报告和团队活跃度。
4. Token 趋势作为右侧或次级小组件。

隐私边界：

总监首页应优先展示部门和团队级信号，避免把个人 Token 排名作为管理视角。

## 9. PM 首页

PM 首页服务需求健康度、AC 进展和跨团队风险。

主要目标：

- 查看重点关注需求。
- 追踪缺 AC、AC 进度、AC 与任务关联缺口。
- 查看 PM 报告状态，并展示 PM 报告链接入口。
- 识别需要协调的跨团队阻塞。
- 查看需求推进异常，例如缺 AC、跨团队依赖未处理、范围待确认。

首屏优先级：

1. 重点关注需求。
2. AC 追踪。
3. 跨团队阻塞。
4. 需求推进异常。

## 10. Demo 调整方向

当前 demo 首页应按以下方向调整：

- 从通用 Dashboard 组合改为固定角色首页模板。
- 四个角色首屏必须明显不同。
- 降低 Token 和图表的视觉优先级。
- 员工、TL、PM 首页优先展示可处理动作；总监首页优先展示部门总体态势、重点需求、团队活跃和异常信号，避免做成一线待办处理台。
- 保留需求、任务、AC、Session、报告和 Token 页面作为承接页。
- 首页点击可用轻量抽屉或详情面板承接上下文。

## 11. 重点关注边界

重点关注只是首页和报告中的轻量增强机制，不是新的产品主线。

当前口径：

- 重点关注对象限定为项目 / 需求 / 任务。
- 重点关注影响：首页更显眼、排序更靠前、列表可筛选、报告中优先出现。
- 重点关注不等于正式业务优先级。
- 不做复杂关注算法。
- 不做自动提醒升级。
- 不做自动调整优先级。

## 12. 明确不做

本决策不包含以下范围：

- 自由配置 Dashboard。
- Token 优先的数据大屏首页。
- 完整任务管理重构。
- 完整需求管理重构。
- 完整日报 / 周报系统重构。
- 首页一期原型不实现真实飞书集成，但需要在首页展示飞书日报 / 周报链接入口和登录态占位。真实飞书 SSO、报告写入和链接生成是否进入一期交付，以正式 PRD 为准。
- 真实外部消息推送闭环。
- 自动派单、自动排期、自动调整优先级。
- 甘特图或资源计划。
- 个人 Token 排名管理视图。

## 13. 已冻结但仍待细化的问题

以下问题尚未完全定稿，但不阻塞首页方向冻结：

- 每个首页模块的具体字段。
- Token 趋势的具体图表形式。
- 首页点击后打开抽屉还是跳转现有详情页。
- 报告相关模块的具体状态文案。
- 缺 Session、阻塞、无进展等风险阈值。

后续如果这些问题被确认，应更新本决策或新增编号决策文档。不要从旧历史文档中反推答案。
