# Dashboard 风险列表产品需求文档

## 1. 产品定位

Dashboard 风险列表是首页上的“我的风险提示”模块。

它的目标是帮助当前登录用户快速看到自己需要关注、推进或处理的风险事项。它不是全局风险池，也不是团队风险池，更不是所有任务风险的汇总列表。

用户进入 Dashboard 时，风险列表应该回答三个问题：

1. 我现在有哪些需要处理的风险？
2. 这些风险属于哪个需求或任务？
3. 我应该点到哪里继续处理？

## 2. 使用场景

### 2.1 日常处理风险

员工、组长、PM、总监打开 Dashboard 后，可以看到自己负责、创建或关注范围内的风险提示，并进入对应需求或任务处理。

### 2.2 关注需求进展

用户主动关注某个需求后，如果该需求自身超期，或该需求下任务出现超期、依赖阻塞，Dashboard 需要提醒用户。

### 2.3 避免被无关风险打扰

即使用户是 PM、总监或组长，Dashboard 首页也不应该默认把全局、部门或团队里的所有风险都推给他。管理视角的风险池应作为后续独立入口，不放在“我的风险提示”里。

## 3. 模块名称与空状态

模块标题：

```text
我的风险提示
```

空状态：

```text
暂无需要处理的风险
```

## 4. P0 风险类型

P0 只支持 3 类明确、可判断、可行动的风险。

| 风险类型 | 展示文案 | 说明 |
| --- | --- | --- |
| `requirement_overdue` | 需求超期 | 需求本身超过截止日期且还未完成 |
| `deadline` | 任务超期 | 任务超过截止日期且还未完成 |
| `dependency_blocker` | 依赖阻塞 | 任务存在未完成的上游依赖，导致当前任务无法推进 |

P0 不做以下风险：

- 需求缺 AC；
- 需求未拆任务；
- 需求停滞；
- 进展异常；
- 高关注需求异常；
- 团队风险池；
- 全局风险池；
- 智能风险判断。

这些可以作为后续版本扩展，不混入本次 Dashboard 首页风险列表。

## 5. 谁能看到哪些风险

Dashboard 风险范围统一叫“我的风险范围”。

所有角色使用同一套规则，不因角色更高就默认扩大范围。

### 5.1 需求风险可见范围

用户可以看到以下需求本身的风险：

1. 我创建的需求；
2. 我关注的需求。

不包含：

- 我只是负责了该需求下的某个任务；
- 我只是该需求所属团队成员；
- 我是 PM、总监或组长；
- 我有权限在管理页面查看该需求。

原因：需求级风险是整个需求的推进风险，只应该提醒需求创建者或主动关注者。

### 5.2 任务风险可见范围

用户可以看到以下任务的风险：

1. 我负责的任务；
2. 我创建的任务；
3. 我关注的任务；
4. 我创建的需求下的任务；
5. 我关注的需求下的任务。

关键规则：

- 如果我关注了一个需求，该需求下的任务风险也应该提醒我；
- 如果我只负责一个任务，只提醒这个任务自己的风险；
- 不应该因为我负责了同需求下某个任务，就把该需求下所有其他任务风险都推给我。

## 6. 角色口径

Dashboard 首页风险提示不按角色默认扩大范围。

| 角色 | Dashboard 风险范围 |
| --- | --- |
| director | 我的需求风险 + 我的任务风险 |
| pm | 我的需求风险 + 我的任务风险 |
| team_leader | 我的需求风险 + 我的任务风险 |
| employee | 我的需求风险 + 我的任务风险 |

明确不做：

- director 不默认看全局风险；
- pm 不默认看全局风险；
- team_leader 不默认看团队风险；
- employee 不看无关需求或无关任务风险。

如果后续需要团队风险池或全局风险池，应新增独立入口，不复用 Dashboard 首页“我的风险提示”。

## 7. 风险展示方式

Dashboard 风险列表以“风险组”的方式展示。

同一个需求下，如果有多个当前用户可见的风险，首页应合并成一个风险组，避免大量任务风险刷屏。

重要规则：

```text
先判断哪些风险属于当前用户
再把这些风险按需求合并展示
```

合并展示不能扩大可见范围。也就是说，如果用户只负责某个需求下的一个任务，风险组里只能展示这个任务的风险，不能顺带展示同需求下其他无关任务风险。

## 8. 两种展示形态

### 8.1 需求风险组

适用于以下情况：

- 需求本身超期；
- 同一需求下有多个当前用户可见的任务风险；
- 需求本身风险和任务风险同时存在。

展示建议：

```text
[需求] 需求标题
风险标签：需求超期 / N 个任务超期 / N 个依赖阻塞
摘要：重点任务或最严重风险
操作：查看需求
```

点击后进入需求：

```text
/requirements?requirementId=xxx
```

按钮文案：

```text
查看需求
```

### 8.2 单任务风险

适用于以下情况：

- 该需求下当前用户只可见一个任务风险；
- 需求本身没有超期风险。

展示建议：

```text
[任务] 任务标题
所属需求：需求标题
风险标签：任务超期 / 依赖阻塞
操作：查看任务
```

点击后进入任务：

```text
/requirements?requirementId=xxx&taskId=yyy
```

按钮文案：

```text
查看任务
```

如果只有依赖阻塞风险，也可以显示：

```text
处理依赖
```

## 9. 风险判断规则

### 9.1 需求超期

当需求超过截止日期，且需求还未完成或取消时，展示“需求超期”。

说明：

- 需求超期只看需求自己的截止日期；
- 不因为需求下有任务超期，就把需求标记为“需求超期”；
- 需求今天到期不算超期；
- 没有截止日期的需求不算超期；
- 已完成或已取消的需求不再提示风险。

### 9.2 任务超期

当任务超过截止日期，且任务还未完成时，展示“任务超期”。

说明：

- 任务今天到期不算超期；
- 没有截止日期的任务不算超期；
- 已完成任务不再提示任务超期；
- 已完成或已取消需求下的任务不再提示风险。

### 9.3 依赖阻塞

当一个任务依赖的上游任务还未完成时，该任务展示“依赖阻塞”。

说明：

- 风险归属到被阻塞的下游任务；
- 不归属到未完成的上游任务；
- 一个任务即使有多个未完成依赖，也只展示一个“依赖阻塞”标签；
- 可以在摘要中展示“存在 N 个未完成依赖”。

## 10. 多风险合并规则

同一个任务可以同时出现多个风险，例如：

```text
任务超期 + 依赖阻塞
```

Dashboard 展示时应合并到同一个任务摘要里，不拆成两张卡片。

示例：

```text
[任务] 支付回调联调
所属需求：支付链路优化
风险标签：任务超期 / 依赖阻塞
操作：查看任务
```

这只是展示合并，不代表风险本身被删除。后续统计或详情页仍可以区分它同时存在两个风险。

## 11. 示例

### 示例 1：我关注了需求 A

需求 A 当前状态：

```text
需求已超期
T1 已超期
T2 依赖阻塞
T3 正常
```

因为我关注了需求 A，Dashboard 展示：

```text
[需求] 需求 A
需求超期 · 1 个任务超期 · 1 个依赖阻塞
操作：查看需求
```

### 示例 2：我只负责需求 A 下的 T1

需求 A 当前状态：

```text
T1 已超期，我负责
T2 已超期，别人负责
T3 依赖阻塞，别人负责
```

Dashboard 只展示：

```text
[任务] T1
所属需求：需求 A
任务超期
操作：查看任务
```

不展示 T2、T3。

### 示例 3：我创建了需求 B

需求 B 当前状态：

```text
需求未超期
T1 已超期
T2 已超期 + 依赖阻塞
```

因为我创建了需求 B，Dashboard 展示：

```text
[需求] 需求 B
2 个任务超期 · 1 个依赖阻塞
重点任务：T2 已超期，且存在依赖阻塞
操作：查看需求
```

### 示例 4：无关需求 C 风险很多

需求 C 当前状态：

```text
多人关注
多个任务超期
多个依赖阻塞
```

但我没有创建需求 C，没有关注需求 C，也没有负责、创建或关注其中任何任务。

Dashboard 不展示需求 C。

## 12. 排序规则

P0 排序规则保持简单：

1. 有超期风险的优先；
2. 超期更早的优先；
3. 关注度更高的优先；
4. 最近更新的优先。

说明：

- 关注度只影响排序，不影响风险是否出现；
- 无关风险即使关注度很高，也不能出现在我的 Dashboard；
- 排序只基于当前用户可见的风险内容，不能引用不可见任务的信息。

## 13. 与关注列表的关系

`/dashboard/follows` 展示“我的关注事项”。

`/dashboard/risks` 展示“我的风险提示”。

两者不是替代关系。

规则：

- 关注了一个需求，不代表它一定出现在风险列表；
- 只有关注对象实际触发风险时，才出现在风险列表；
- 我负责的任务即使未关注，只要触发风险，也应该出现在风险列表；
- 无关需求或任务即使风险很高，也不应该出现在风险列表。

## 14. P0 范围

### 14.1 P0 做

1. 展示我的需求级风险；
2. 展示我的任务级风险；
3. 支持需求超期；
4. 支持任务超期；
5. 支持依赖阻塞；
6. 按需求合并展示当前用户可见风险；
7. 同一任务多风险合并展示；
8. director、pm、team_leader、employee 都不默认扩大风险范围。

### 14.2 P0 不做

1. 全局风险池；
2. 团队风险池；
3. 需求缺 AC；
4. 需求未拆任务；
5. 需求停滞；
6. 进展异常；
7. 智能风险判断；
8. 跨团队风险看板。

## 15. 验收标准

### 15.1 范围验收

1. director 登录后，不默认看到全局风险。
2. pm 登录后，不默认看到全局风险。
3. team_leader 登录后，不默认看到团队全部风险。
4. employee 不看到无关需求或无关任务风险。
5. 用户能看到自己创建的需求风险。
6. 用户能看到自己关注的需求风险。
7. 用户能看到自己负责、创建、关注的任务风险。
8. 用户关注某个需求后，能看到该需求本身风险和该需求下任务风险。
9. 用户只负责某个任务时，不能看到同需求下其他无关任务风险。
10. 风险组只能包含当前用户可见的风险。

### 15.2 风险类型验收

1. 未完成、未取消的需求超过截止日期时，展示需求超期。
2. 已完成或已取消需求不展示需求超期。
3. 未完成任务超过截止日期时，展示任务超期。
4. 已完成任务不展示任务超期。
5. 已完成或已取消需求下的任务不展示任务风险。
6. 任务存在未完成上游依赖时，展示依赖阻塞。
7. 一个任务有多个未完成上游依赖时，只展示一个依赖阻塞标签。

### 15.3 展示验收

1. 同一需求下多个可见风险应合并成一个需求风险组。
2. 同一任务同时出现任务超期和依赖阻塞时，应合并在同一个任务摘要中。
3. 单任务风险可以直接展示为任务项。
4. 需求风险组点击进入需求。
5. 单任务风险点击进入任务。
6. 风险组应展示清晰摘要，例如“需求超期”“2 个任务超期”“1 个依赖阻塞”。
7. 空状态文案为“暂无需要处理的风险”。
8. 模块标题为“我的风险提示”。

### 15.4 关注度验收

1. 关注度不扩大风险来源。
2. 无关风险即使关注度高，也不出现在当前用户 Dashboard。
3. 当前用户可见范围内的风险，可以按关注度影响排序。

## 16. 最终口径

Dashboard 风险提示的 P0 产品口径是：

```text
我的风险提示 = 我的需求风险 + 我的任务风险
```

其中：

```text
需求风险：需求超期
任务风险：任务超期、依赖阻塞
```

展示规则是：

```text
先过滤我的风险
再按需求合并展示
单任务风险可轻量展示为任务项
```

范围规则是：

```text
我创建/关注的需求 => 看该需求本身风险和该需求下任务风险
我只关联某个任务 => 只看该任务自己的风险
```

这保证 Dashboard 首页既能提醒用户真正需要处理的风险，又不会被全局风险、团队风险或无关任务刷屏。
