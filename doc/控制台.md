# 控制台页面独立需求基线说明

版本：v0.1
模块：控制台
定位：AIcoding 管理平台登录后的首屏页面，负责聚合报告入口、我关注的、风险项，以及 TL / 总监的组内或部门汇总报告入口。

---

## 1. 模块背景

当前 AIcoding 管理平台重新收敛为两个核心页面：

```text
AIcoding 管理平台
├─ 控制台
└─ 需求看板
```

其中：

* 控制台负责聚合提醒、报告入口、我关注的、风险项；
* 需求看板负责真实工作承载，包括需求、任务、任务树、依赖、阻塞、deadline、验收标准、关注与风险处理。

本需求文档只讨论「控制台页面」，不展开需求看板的具体原型。

---

## 2. 控制台产品定位

控制台是用户登录后的第一屏，核心不是处理复杂工作，而是回答三个问题：

```text
1. 我的日报 / 周报是否需要生成、确认或发送？
2. 我关注的需求或任务现在怎么样？
3. 当前有哪些风险需要我处理？
```

对总监和 TL，还需要额外回答：

```text
4. 我负责范围内的组 / 部门报告是否需要确认或补救？
```

一句话定义：

> 控制台是围绕“报告状态、关注对象、风险提醒”的轻量聚合入口，不是需求处理中心，也不是项目管理大盘。

---

## 3. 页面边界

### 3.1 控制台负责

```text
控制台负责
├─ 我的报告入口
├─ 个人日报 / 个人周报状态展示
├─ 报告草稿编辑入口
├─ 报告发送入口
├─ 我关注的需求 / 任务聚合
├─ 风险项聚合
├─ TL 的组日报 / 组周报入口
├─ 总监的部门日报 / 部门周报入口
└─ 跳转到需求看板定位处理
```

### 3.2 控制台不负责

```text
控制台不负责
├─ 需求完整编辑
├─ 任务树编辑
├─ 任务依赖配置
├─ 阻塞原因详细维护
├─ deadline 修改
├─ 验收标准 AC 维护
├─ 复杂筛选器
├─ 甘特图
├─ 复杂统计图表
├─ 完整项目管理看板
└─ 独立风险处理工作流
```

控制台可以展示这些信息的摘要，但具体处理应跳转到需求看板或报告弹窗流程。

---

## 4. 控制台核心模块

控制台建议固定为四个核心模块：

```text
控制台
├─ 我的报告
├─ 组报告 / 部门报告（仅 TL / 总监）
├─ 我关注的
└─ 风险项
```

其中：

* 我的报告：所有角色都有，包含个人日报和个人周报；
* 组报告：仅 TL 可见，包含组日报和组周报；
* 部门报告：仅总监可见，包含部门日报和部门周报；
* 我关注的：所有角色都有；
* 风险项：所有角色都有。

---

## 5. 模块一：报告入口

### 5.1 模块定位

控制台上的报告区域是轻量状态入口，不是报告管理后台，也不是文档编辑产品。

它解决：

```text
用户看到当前报告状态
        ↓
判断是否需要确认、发送或补救
        ↓
点击进入对应报告弹窗
        ↓
完成生成、编辑、查看或发送
```

### 5.2 报告类型

当前报告能力分为：

```text
个人报告
├─ 个人日报
└─ 个人周报

汇总报告
├─ 组日报
├─ 组周报
├─ 部门日报
└─ 部门周报
```

角色可见范围：

```text
个人 / 普通成员
├─ 个人日报
└─ 个人周报

PM
├─ 个人日报
└─ 个人周报

TL
├─ 个人日报
├─ 个人周报
├─ 组日报
└─ 组周报

总监
├─ 个人日报
├─ 个人周报
├─ 部门日报
└─ 部门周报
```

PM 先不做复杂需求维度汇总，避免控制台跑偏。

### 5.3 控制台展示规则

个人 / PM 视图只展示：

```text
我的报告
├─ 今日日报：主内容区
└─ 本周周报：次级提醒行
```

今日日报是每日主动作，展示报告名称、状态、一句说明和一个主操作按钮。

本周周报是周期提醒，不和今日日报同权重展开，只展示名称、状态、简短来源说明和一个入口。如果未到生成时间，展示弱提醒：

```text
本周周报将于周五自动生成。
```

TL 视图展示：

```text
我的报告
├─ 今日日报
└─ 本周周报

组报告
├─ 今日组日报
└─ 本周组周报
```

总监视图展示：

```text
我的报告
├─ 今日日报
└─ 本周周报

部门报告
├─ 今日部门日报
└─ 本周部门周报
```

我的报告中的今日日报主区域只展示：

```text
报告名称
当前状态
一句状态说明
主操作按钮
```

我的报告中的本周周报提醒行只展示：

```text
名称：本周周报
状态
简短来源说明：基于本周日报和任务记录生成
一个主操作：查看周报 / 查看草稿 / 生成周报
```

控制台外层不展示 session 数量、Skill、生成方式、复杂提交进度条等内部字段。

### 5.4 统一报告状态

日报和周报统一使用同一套状态：

```text
待生成
生成中
草稿待确认
已发送
生成失败
发送失败
```

按钮规则：

```text
待生成
└─ 生成报告

生成中
└─ 生成中（不可操作或弱操作）

草稿待确认
├─ 查看草稿
└─ 发送

已发送
└─ 查看报告

生成失败
└─ 重新生成

发送失败
├─ 重试发送
└─ 查看草稿
```

已发送状态打开只读查看态，不默认进入编辑态。草稿待确认状态点击“查看草稿”才进入编辑态。

### 5.5 个人日报生成逻辑

个人日报来源：

```text
个人当日 session
+
用户当天相关任务 / 需求状态
```

个人日报主流程：

```text
系统自动收集今日 session
  ↓
系统自动生成个人日报草稿
  ↓
用户查看 / 编辑
  ↓
用户发送
```

手动选择 session 只是兜底流程：

```text
自动生成失败
或用户需要补充 session
  ↓
选择 session
  ↓
默认日报 Skill 生成草稿
  ↓
编辑确认
  ↓
发送
```

个人日报弹窗可以保留：

```text
选择 session
→ 选择生成方式
→ 编辑草稿
```

该流程只适用于个人日报，不适用于 TL / 总监的组或部门汇总报告主流程。

### 5.6 个人周报生成逻辑

个人周报不是直接上传 session 生成。

个人周报来源：

```text
本周个人日报
+
本周个人 session 摘要
+
我负责的任务
+
我关注的任务
+
我相关的风险与阻塞
```

个人周报流程：

```text
到周报生成时间
  ↓
系统读取本周个人日报和任务数据
  ↓
AI 生成个人周报草稿
  ↓
用户编辑确认
  ↓
发送个人周报
```

### 5.7 TL 组报告逻辑

TL 的组报告不是通过上传 session 生成。

组日报来源：

```text
组内成员今日个人日报
+
组内需求 / 任务状态
+
组内风险项
+
阻塞任务
+
临近 deadline 事项
```

组周报来源：

```text
组内成员本周个人日报
+
组内成员个人周报
+
组内需求看板数据
+
本周风险与阻塞
+
本周完成 / 延期 / 下周计划
```

TL 操作流程：

```text
系统生成组日报 / 组周报草稿
  ↓
TL 查看汇总草稿
  ↓
TL 编辑
  ↓
TL 发送
```

如果需要查看来源，可以提供“查看来源日报”或“查看成员提交情况”，但不要作为主流程。

### 5.8 总监部门报告逻辑

总监的部门报告也不是通过上传 session 生成。

部门日报来源：

```text
各组日报
+
各组提交情况
+
部门重点需求
+
高优先级风险
+
跨组依赖和阻塞
```

部门周报来源：

```text
各组周报
+
各组日报摘要
+
部门重点需求状态
+
高风险事项
+
资源 / 依赖 / 交付风险
```

总监操作流程：

```text
系统生成部门日报 / 部门周报草稿
  ↓
总监查看汇总草稿
  ↓
总监编辑
  ↓
总监发送
```

### 5.9 编辑器口径

报告编辑只使用纯文字 Markdown textarea。

允许：

```text
标题
小标题
无序列表
有序列表
换行
```

不做：

```text
富文本工具栏
图片
表格
链接
代码块
默认左右分屏预览
复杂文档编辑器
```

## 6. 模块二：我关注的

### 6.1 模块定位

“我关注的”展示用户主动关注的需求或任务。

它不是权限列表，也不是全部任务列表，而是用户自己的关注订阅。

### 6.2 关注对象

```text
关注对象
├─ 需求
└─ 任务
```

任务必须展示所属需求，否则上下文不完整。

### 6.3 默认关注对象

不同角色默认展示对象不同：

| 角色 | 默认关注对象  | 说明                   |
| -- | ------- | -------------------- |
| 总监 | 需求      | 看整体进展、重点需求、风险        |
| TL | 需求 + 任务 | 既看组内需求，也看关键任务阻塞      |
| PM | 需求      | 看需求推进、AC、依赖、deadline |
| 个人 | 任务      | 看自己负责或关注的任务          |

但所有角色都可以关注需求或任务。

### 6.4 展示内容

“我关注的”建议采用轻量列表，不建议做复杂表格。

需求关注项展示：

```text
需求关注项
├─ 需求名称
├─ 状态
├─ 负责人
├─ deadline
├─ 风险数量
├─ 阻塞任务数量
└─ 点击进入需求详情
```

任务关注项展示：

```text
任务关注项
├─ 任务名称
├─ 所属需求
├─ 状态
├─ 负责人
├─ deadline
├─ 阻塞状态
├─ 依赖状态
└─ 点击进入需求详情并定位任务
```

### 6.5 点击跳转

```text
点击关注的需求
  ↓
进入需求看板
  ↓
定位到对应需求详情

点击关注的任务
  ↓
进入需求看板
  ↓
定位到所属需求
  ↓
展开任务树并定位该任务
```

## 7. 模块三：风险项

### 7.1 模块定位

风险项是控制台最重要的提醒模块之一。

它展示系统根据需求、任务、日报状态聚合出来的异常、阻塞、缺失、超时信息。

风险项不是独立工作流，不建议在控制台上做复杂处理按钮。

### 7.2 风险来源

风险项来源包括：

```text
风险来源
├─ 需求风险
│  ├─ 高关注需求缺少 AC
│  ├─ 需求临近 deadline
│  ├─ 需求已超期
│  └─ 需求下存在阻塞任务
│
├─ 任务风险
│  ├─ 任务被手动标记阻塞
│  ├─ 任务阻塞超过阈值
│  ├─ 任务临近 deadline
│  ├─ 任务已超期
│  ├─ 任务缺少 AC
│  ├─ 上游依赖未完成
│  └─ 上游依赖已阻塞
│
└─ 日报风险
   ├─ 今日日报未生成
   ├─ 日报生成失败
   ├─ 日报草稿未发送
   ├─ 日报发送失败
   └─ 管理范围内成员日报缺失
```

### 7.3 风险展示字段

风险项建议展示：

```text
风险项
├─ 风险标题
├─ 风险类型
├─ 关联对象：需求 / 任务 / 日报
├─ 所属需求
├─ 负责人
├─ deadline
├─ 风险原因
├─ 风险等级
└─ 操作入口
```

示例：

```text
风险：高关注需求下有 1 个阻塞任务
对象：需求「日报生成能力」
阻塞任务：飞书发送联调
负责人：张三
原因：依赖接口未完成
操作：查看详情
```

### 7.4 风险优先级

P0 可以先采用简单排序：

```text
风险排序优先级
1. 已超期
2. 阻塞
3. 依赖阻塞
4. 临近 deadline
5. 缺 AC
6. 日报未发送
```

后续可根据角色调整排序权重。

### 7.5 风险点击跳转

```text
任务阻塞风险
  ↓
跳转需求看板
  ↓
定位到所属需求
  ↓
展开阻塞任务

依赖阻塞风险
  ↓
跳转需求看板
  ↓
定位当前任务和上游依赖任务

deadline 风险
  ↓
跳转需求看板
  ↓
定位到对应需求或任务

日报风险
  ↓
进入日报生成 / 编辑 / 查看视图
```

## 8. 模块四：汇总报告入口

### 8.1 模块定位

TL 和总监需要查看并发送自己管理范围内的汇总报告。

这部分是管理角色的额外能力，不应影响控制台主结构。

### 8.2 查看范围

| 角色 | 查看范围 |
| -- | -- |
| 总监 | 部门日报、部门周报 |
| TL | 组日报、组周报 |
| PM | 个人日报、个人周报 |
| 个人 | 个人日报、个人周报 |

这里的查看范围只影响汇总报告入口。个人日报仍然只处理当前用户自己的 session 和个人日报；个人周报基于个人日报、个人任务、风险与阻塞生成。

### 8.3 P0 展示内容

P0 建议先展示汇总报告状态，不做复杂分析。

```text
组报告 / 部门报告
├─ 今日组日报 / 今日部门日报
├─ 本周组周报 / 本周部门周报
├─ 当前状态
├─ 一句来源说明
└─ 生成 / 查看草稿 / 发送 / 重新生成入口
```

成员提交情况可以在弹窗内展示，不作为控制台外层的主信息。

### 8.4 P1 展示内容

```text
P1
├─ 周报自动汇总
├─ 部门日报趋势
├─ 飞书文档同步状态
├─ 组内风险提取
└─ 按需求聚合日报内容
```

P0 不建议做“日报质量评分”或复杂内容分析，容易把控制台带偏。

## 9. 角色差异

控制台不做四套独立页面。

角色差异只体现在：

```text
角色差异
├─ 默认关注对象不同
├─ 数据范围不同
├─ 风险项排序权重不同
├─ 报告查看和发送范围不同
└─ 可见操作不同
```

### 9.1 总监控制台

默认重点：

```text
总监
├─ 我的报告：个人日报 / 个人周报
├─ 部门报告：部门日报 / 部门周报
├─ 重点需求
├─ 高风险需求
└─ 阻塞超过阈值的需求 / 任务
```

不建议总监首页展示太多具体任务细节，只展示关键任务风险。

### 9.2 TL 控制台

默认重点：

```text
TL
├─ 我的报告：个人日报 / 个人周报
├─ 组报告：组日报 / 组周报
├─ 我关注的需求
├─ 我关注的任务
├─ 组内阻塞任务
└─ 依赖未处理任务
```

TL 是最需要看任务风险的角色。

### 9.3 PM 控制台

默认重点：

```text
PM
├─ 我的报告：个人日报 / 个人周报
├─ 我关注的需求
├─ 缺 AC 的需求
├─ 有阻塞任务的需求
├─ 临近 deadline 的需求
└─ 跨任务依赖风险
```

PM 不应被大量个人任务淹没，核心仍是需求推进。

### 9.4 个人控制台

默认重点：

```text
个人
├─ 我的报告：个人日报 / 个人周报
├─ 我关注的任务
├─ 我负责的任务
├─ 我的阻塞任务
├─ 即将到期任务
└─ 依赖我完成的任务
```

个人控制台最重要的是“我今天该处理什么”。

## 10. 控制台数据来源

控制台本身不应维护独立业务数据。

它的数据来源应包括：

```text
数据来源
├─ 需求看板
│  ├─ 需求
│  ├─ 任务
│  ├─ 任务树
│  ├─ deadline
│  ├─ AC
│  ├─ 阻塞
│  ├─ 依赖
│  └─ 关注关系
│
├─ 报告系统
│  ├─ 报告状态
│  ├─ session
│  ├─ 报告草稿
│  ├─ 发送状态
│  └─ 汇总报告
│
└─ 用户与角色
   ├─ 当前用户
   ├─ 当前角色
   ├─ 所属组
   └─ 管理范围
```

关键口径：

> 需求、任务、风险、关注状态以需求看板为事实源；报告状态以报告系统为事实源；控制台只做聚合展示。

## 11. 页面结构建议

### 11.1 推荐结构

控制台建议采用：

```text
控制台
├─ 顶部摘要区
│  ├─ 当前角色 / 管理范围
│  ├─ 今日日报状态
│  ├─ 当前风险数
│  └─ 我关注的数量
│
├─ 报告入口区
│  ├─ 我的报告
│  └─ 组报告 / 部门报告（仅 TL / 总监）
│
├─ 主内容区
│  ├─ 我关注的
│  └─ 风险项
```

### 11.2 页面布局建议

P0 推荐布局：

```text
顶部：轻量摘要 + 报告状态入口

中部左侧：我关注的
中部右侧：风险项
```

但要注意：

> 风险项优先级通常高于“我关注的”。当存在高风险时，风险项应在视觉上更突出；没有风险时，页面不应空出大块区域。

### 11.3 空状态

控制台必须设计空状态，否则没有风险时页面会显得空。

```text
无风险状态
├─ 展示“当前无高优先级风险”
├─ 引导查看需求看板
└─ 不要强行塞图表

无关注状态
├─ 展示“暂无关注的需求或任务”
├─ 引导去需求看板关注
└─ 不要默认展示全部需求

无报告状态
├─ 展示“报告尚未生成”
└─ 提供生成报告按钮
```

## 12. 核心用户流程

### 12.1 登录查看控制台

```text
用户登录
  ↓
进入控制台
  ↓
查看今日日报状态
  ↓
查看风险项
  ↓
查看我关注的
  ↓
点击进入需求看板或报告弹窗
```

### 12.2 生成个人日报

```text
用户进入控制台
  ↓
点击生成日报
  ↓
选择 / 上传 session
  ↓
系统使用默认日报 Skill 生成草稿
  ↓
用户编辑草稿
  ↓
用户发送日报
  ↓
控制台更新日报状态
```

### 12.3 生成个人周报

```text
用户进入控制台
  ↓
点击生成本周周报
  ↓
系统读取本周个人日报、任务、风险与阻塞
  ↓
AI 生成个人周报草稿
  ↓
用户编辑确认
  ↓
用户发送周报
```

### 12.4 生成组 / 部门报告

```text
TL / 总监进入控制台
  ↓
点击生成组报告或部门报告
  ↓
系统读取成员报告、需求看板数据和风险项
  ↓
AI 生成汇总报告草稿
  ↓
TL / 总监编辑确认
  ↓
发送汇总报告
```

### 12.5 处理风险

```text
控制台展示风险项
  ↓
用户点击风险项
  ↓
跳转到需求看板
  ↓
定位到需求 / 任务
  ↓
用户处理阻塞、deadline、AC 或依赖问题
  ↓
风险重新计算
  ↓
控制台风险项更新
```

## 16. 原型设计重点

控制台原型设计优先关注信息层级，不要先做视觉发散。

### 16.1 第一优先级

```text
第一优先级
├─ 报告入口怎么放
├─ 我关注的和风险项谁更突出
├─ 风险为空时页面如何布局
├─ 不同角色默认数据如何变化
├─ 点击风险后如何跳转需求看板
├─ 点击任务后如何定位所属需求
└─ TL / 总监的组或部门报告入口怎么放
```

### 16.2 第二优先级

```text
第二优先级
├─ 顶部摘要数字
├─ 简单筛选
├─ 风险标签颜色
├─ 列表密度
├─ 卡片还是表格
└─ 空状态文案
```

### 16.3 不建议一开始做

```text
不要一开始做
├─ 大型 Dashboard
├─ 复杂统计图表
├─ 花哨可视化
├─ 多角色多套页面
├─ 复杂表格筛选
├─ 控制台内完整处理流程
└─ 飞书深度交互
```

## 20. 一句话基线

控制台不是需求看板，也不是数据大盘。

它的核心主线是：

```text
报告状态
  ↓
生成 / 编辑 / 查看 / 发送报告

我关注的
  ↓
查看需求 / 任务当前状态
  ↓
跳转需求看板

风险项
  ↓
识别阻塞 / 超期 / 缺 AC / 日报异常
  ↓
跳转到对应处理位置
```

P0 的目标不是功能多，而是让用户登录后立刻知道：

```text
今天日报 / 本周周报要不要处理
我关注的对象有没有变化
当前有哪些风险
点哪里可以处理
```
