## 1. 当前实现盘点

### 1.4 “下一步 / 保存修改 / 保存日报”当前分别调用什么

#### “下一步”

个人日报在非编辑步骤点击“下一步”会执行 `handleGenerate`：

- `web/src/features/aidashboard/reports/components/DailyReportGenerateModal.tsx:277`

它调用：

- `generateTodayReportDraft(payload)`
- 前端 client：`web/src/features/aidashboard/api/client.ts:221`
- 后端 API：`POST /reports/today/draft`
- 后端 handler：`api/handler/report.go:297`
- daemon 生成服务：`daemon/server_reports.go:212`

这个接口只生成草稿，不落库个人日报最终状态。

#### “保存修改”

编辑页中的“保存修改”按钮调用：

```ts
saveMutation.mutate({ finalAction: false })
```

位置：`web/src/features/aidashboard/reports/components/DailyReportGenerateModal.tsx:301`

个人日报分支内部会：

1. 调用 `fetchTodayReport()`。
2. 调用 `updateReport(report.id, { content, session_ids })`。

位置：`web/src/features/aidashboard/reports/components/DailyReportGenerateModal.tsx:224`

对应接口：

- `GET /reports/today`
- `PUT /reports/{id}`

#### “保存日报”

编辑页中的主按钮对个人日报显示为“保存日报”：

- `web/src/features/aidashboard/reports/components/DailyReportGenerateModal.tsx:304`
- 文案分支：`web/src/features/aidashboard/reports/components/DailyReportGenerateModal.tsx:310`

它调用：

```ts

```

但个人日报后端调用路径和“保存修改”完全一样，仍然只是：

1. `fetchTodayReport()`
2. `updateReport(report.id, { content, session_ids })`

`finalAction` 对个人日报只影响成功提示文案，不会写入提交状态，也不会区分 employee/TL/PM/director/admin。

### 1.6 `GET /reports/today` 行为

`GET /reports/today` 当前是 `GetOrCreateToday`：

- 路由：`api/main.go:121`
- Handler：`api/handler/report.go:211`

如果当天没有个人日报，它会：

1. 调用 `generateReportContent(u.ID, today)` 生成默认内容。
2. 插入 `daily_reports (user_id, report_date, content)`。

## 3. 当前问题列表

1. **个人日报没有最终状态字段。** 只有 `edited`，无法区分草稿、已提交、已保存。
2. **个人日报没有提交接口。** employee/TL/PM 的“提交给 TL / 总监”无法落库。
3. **“保存修改”和“保存日报”当前调用同一套接口。** 两者都只是 `PUT /reports/{id}` 更新正文和 session，业务效果相同。
4. **个人日报弹窗视觉上是三步，但实际只有选择来源和编辑确认两段。** 第三步“保存日报”没有独立步骤，造成用户误解。
5. **Dashboard 状态映射过粗。** 有日报时一律显示“草稿待确认”，无日报时显示“待生成”，无法显示“已提交 / 已保存”。
6. **Dashboard 仍有静态状态兜底。** 查询未成功时会保留 `ROLE_DATA` 中的默认状态，可能出现“待生成”等假状态覆盖真实业务预期。
7. **`GET /reports/today` 自动创建日报。** 它不能作为判断是否缺失日报的纯读接口，否则会把 `missing` 变成默认草稿。
8. **Feishu 字段容易造成语义混淆。** 当前没有 Feishu 发送，本阶段所有“提交”都应是内部状态流转。
9. **Dashboard 文件保留旧弹窗实现。** 当前个人日报使用共享弹窗，但旧 Modal、旧状态和旧按钮仍在文件中，维护上容易误判。

## 5. 状态机设计

推荐使用以下个人日报状态：

| 状态 | 含义 | 是否落库 |
| --- | --- | --- |
| `missing` | 当天没有日报记录 | 不落库，由查询结果为空推导 |
| `draft` | 已生成或已保存草稿，尚未最终提交/保存 | 落库 |
| `submitted` | employee/TL/PM 已提交给上级 | 落库 |
| `saved` | director/admin 已完成保存 | 落库 |
| `generate_failed` | 草稿生成失败 | 可先作为前端瞬时状态；如需要历史追踪再落库 |

### 状态流转

```text
missing
  └─ 生成草稿成功但未最终保存：仍可不落库，或落库为 draft

draft
  ├─ employee 最终动作：submitted + submitted_to=team_leader + submitted_at=now()
  ├─ team_leader/pm 最终动作：submitted + submitted_to=director + submitted_at=now()
  └─ director/admin 最终动作：saved

submitted
  └─ 再次编辑后如只保存草稿：回到 draft；如直接最终动作：保持 submitted 并刷新 submitted_at

saved
  └─ director/admin 再次编辑并保存：保持 saved，刷新 updated_at
```

建议最小实现中：

- `missing` 不入库。
- `draft/submitted/saved` 入库。
- `generate_failed` 先留在前端，不必第一阶段入库，除非产品要求失败历史。

## 6. 前端弹窗修复方案

### 6.1 个人日报改成两步

个人日报弹窗只保留两步：

1. **选择来源**
   - 选择当天 session。
   - 选择/上传 Skill。
   - 点击“生成草稿”。
2. **确认日报**
   - 展示 Markdown 编辑器。
   - 只保留一个最终主按钮。

移除或合并：

- “保存修改”按钮。
- 个人日报“保存日报”与“保存修改”的双按钮并存。
- Step 3 “保存日报”可视步骤。

### 6.2 最终按钮按角色显示

在 `DailyReportGenerateModal` 中根据 `user.role` 计算个人日报最终按钮文案：

- employee：`保存并提交给 TL`
- team_leader / pm：`保存并提交给总监`
- director / admin：`保存日报`

### 6.3 前端 API 调用拆分

建议新增 client 方法：

```ts
export const submitReport = (id: string, data?: { content?: string; session_ids?: string[] }) =>
  unwrap(api.post<DailyReport>(`/reports/${id}/submit`, data));
```

个人日报最终动作：

- employee/TL/PM：调用 `submitReport(id, { content, session_ids })`。
- director/admin：调用 `updateReport(id, { content, session_ids, status: "saved" })`，或调用统一的 `finalizeReport` 接口由后端按角色落 `saved`。

为了避免 `GET /reports/today` 自动创建造成语义混淆，建议保存流程使用以下方式之一：

- 方案 A：新增 `POST /reports/today` 或 `PUT /reports/today`，按 `report_date + 当前用户` upsert 草稿/最终状态。
- 方案 B：保留 `GET /reports/today` 仅作兼容，但新增明确的 `POST /reports/today/save` 和 `POST /reports/today/submit`。
- 方案 C：前端先通过 `/reports/mine?from=&to=` 查今日是否已有记录；没有则调用 `POST /reports/today/generate` 或新增 create/upsert 接口。

最小改动推荐：新增 `POST /reports/today/save` 和 `POST /reports/{id}/submit`，避免继续把 GET 当创建接口使用。

## 7. Dashboard 状态修复方案

### 7.1 状态必须来自真实接口

Dashboard 今日个人日报卡片应以真实个人日报状态为准：

- 查询 `/reports/mine?from=today&to=today&page=1&page_size=1`。
- 无记录：显示 `missing` / “待生成”。
- 有记录：按 `daily_reports.status` 显示。

`ROLE_DATA` 只能保留非业务文案兜底，不能覆盖个人日报真实状态。

### 7.2 增加状态映射

前端类型中补充：

```ts
type DailyReportStatus = "draft" | "submitted" | "saved" | "generate_failed";
```

UI 映射建议：

| 后端状态 | Dashboard 文案 | Tag |
| --- | --- | --- |
| 无记录 | 待生成 | gold |
| `draft` | 草稿待确认 | blue |
| `submitted` | 已提交 | green |
| `saved` | 已保存 | green |
| `generate_failed` | 生成失败 | red |

同时把旧的 `ReportStatus` 扩展或替换为能表达“已提交 / 已保存”的状态。

### 7.3 最终动作后刷新

共享弹窗 `onDone` 当前会 invalidate `queryKey: ["reports"]`：

- `web/src/features/aidashboard/dashboard/DashboardPage.tsx:1468`

建议改为同时明确刷新：

- `['reports', 'dashboard-today', reportDate]`
- `['reports', 'daily', 'personal-list']`
- 详情页相关 query key（如果当前在详情页调用）

避免只依赖宽泛 key 或旧 key 前缀导致卡片短时间显示静态兜底状态。

### 7.4 不再使用 Feishu 文案

Dashboard 文案中与“发送”“飞书发送”相关的历史文案应替换为内部状态文案：

- employee/TL/PM：提交给 TL / 总监。
- director/admin：保存日报。

不出现：

- 发送成功
- 发送失败
- 重试发送
- 飞书发送

## 9. 后端接口缺口

### 9.1 新增/调整模型字段

`model.DailyReport` 和 `model.DailyReportListItem` 增加：

```go
Status      string     `json:"status"`
SubmittedAt *time.Time `json:"submitted_at,omitempty"`
SubmittedTo *string    `json:"submitted_to,omitempty"`
```

前端 `DailyReport` / `DailyReportListItem` 同步增加字段。

### 9.2 新增 migration

新增 forward migration，例如：

```sql
ALTER TABLE daily_reports
  ADD COLUMN IF NOT EXISTS status TEXT NOT NULL DEFAULT 'draft'
    CHECK (status IN ('draft', 'submitted', 'saved', 'generate_failed')),
  ADD COLUMN IF NOT EXISTS submitted_at TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS submitted_to TEXT
    CHECK (submitted_to IS NULL OR submitted_to IN ('team_leader', 'director'));

CREATE INDEX IF NOT EXISTS idx_daily_reports_status ON daily_reports(status);
```

历史已有记录默认作为 `draft`。如果需要迁移 director/admin 旧记录为 `saved`，需要额外 SQL 结合 `users.role` 判断；最小实现可不做，避免误判旧数据。

### 9.3 新增个人日报提交接口

新增：

```http
POST /api/v1/reports/{id}/submit
```

请求体可支持：

```json
{
  "content": "...",
  "session_ids": ["..."]
}
```

后端规则：

- 只能提交自己的个人日报。
- employee：`status='submitted'`，`submitted_to='team_leader'`，`submitted_at=now()`。
- team_leader / pm：`status='submitted'`，`submitted_to='director'`，`submitted_at=now()`。
- director / admin：不走 submit；如调用则可返回 400，提示使用保存。
- 不写 `feishu_doc_url`，不调用外部系统。

### 9.4 调整保存接口

`PUT /reports/{id}` 继续用于保存草稿正文和来源 session。

建议规则：

- employee/TL/PM 普通保存：`status='draft'`。
- director/admin 点击最终“保存日报”：允许设置 `status='saved'`，或者新增 `POST /reports/{id}/save` 由后端按角色落 `saved`。
- 任何保存都刷新 `updated_at`。

### 9.5 `GET /reports/today` 处理

当前 `GET /reports/today` 会自动创建。建议保留兼容但不要再作为 Dashboard 状态来源。

后续可考虑新增纯读接口：

```http
GET /reports/today/current
```

无记录返回 404 或 `null`，不自动创建。

最小实现中，Dashboard 直接使用 `/reports/mine` 查询今日记录即可。

## 11. 验收标准

### 11.1 employee

- Dashboard 初始无日报：显示“待生成”。
- 打开个人日报弹窗只有两步：选择来源、确认日报。
- 编辑页只有一个主按钮：“保存并提交给 TL”。
- 点击后落库：`status='submitted'`，`submitted_to='team_leader'`，`submitted_at` 非空。
- Dashboard 刷新后显示“已提交”。
- `/reports/daily` 列表该记录显示“已提交”。

### 11.2 team_leader / pm

- 个人日报最终按钮为“保存并提交给总监”。
- 点击后落库：`status='submitted'`，`submitted_to='director'`，`submitted_at` 非空。
- Dashboard 显示“已提交”。
- 不调用 Feishu。

### 11.3 director / admin

- 个人日报最终按钮为“保存日报”。
- 点击后落库：`status='saved'`。
- `submitted_to` 为空，`submitted_at` 为空或不作为保存状态依据。
- Dashboard 显示“已保存”。
- 不出现“提交给上级”文案。

### 11.4 通用

- 不出现“发送失败 / 重试发送 / 发送成功”。
- `feishu_doc_url` 不影响个人日报状态。
- `/reports/daily` 仍是记录列表页，不展示全文和编辑器。
- 个人日报详情页才展示正文、来源和编辑能力。
- 不新增个人周报。
- 不引入审批流。
- 不接 Feishu。
- 不用前端假状态代替数据库状态。
