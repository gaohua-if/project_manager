# AI Assets 定时任务现状与改造方案

## 0. 本文档核心结论

本次改造必须按下面这句话理解：

> **定时任务 = 定时器配置 + Agent 运行配置。**

两部分必须分离：

- **定时器配置**只负责“什么时候触发”。
- **Agent 运行配置**只负责“触发后给 Agent 传什么配置”。

不要再把 Report Agent 的 `report_type` 和定时器的 `schedule_type` 绑定在一起。

错误理解：

```text
日报 = 每天触发
周报 = 每周触发
```

正确理解：

```text
定时器每天/每周只是触发频率。
Report Agent 的报告类型是 Agent 入参。
二者是并列配置，不是谁推导谁。
```

例如：

- 每天 19:00 触发一次个人日报：合理。
- 每周五 19:00 触发一次个人日报：也合理，表示每周五生成周五当天日报。
- 每天 19:00 触发一次个人周报：也可以成立，表示每天刷新当前自然周周报。
- 每周五 19:00 触发一次个人周报：常见配置，表示每周五生成当前自然周周报。

产品上可以推荐常用组合，但实现上不能把 `daily schedule` 等同于 `personal_daily`，也不能把 `weekly schedule` 等同于 `personal_weekly`。

---

## 1. 背景

当前用户希望在 AI Assets 中创建可自动执行 Agent 的定时任务，例如：

- 每天 19:00 运行 Report Agent，生成个人日报。
- 每周五 18:30 运行 Report Agent，生成个人周报。
- 每天 09:00 运行普通 Agent，执行固定任务。

当前 AI Assets 的“新建定时任务”是通用弹窗，字段偏底层，包括：

- 任务名称
- Agent
- 模型
- 重复
- 星期
- 触发时间
- 时区
- 触发消息
- 参数
- 状态

它的问题不是“少一个字段”，而是**没有把定时器配置和 Agent 运行配置分开**。

Report Agent 手动运行页已经有业务化配置，例如：

- 报告类型
- 报告日期 / 报告周期
- Initial Message

定时任务不是直接照搬手动运行页，但必须完整配置 Report Agent 运行所需要的业务入参。

定时任务中的 Report Agent 不让用户固定填写某一天的报告日期，而是由系统在每次触发时根据计划触发时间动态生成报告周期。

## 3. 当前问题

### 3.1 创建体验不可理解

当前弹窗字段密集，用户无法判断：

- 哪些是定时器配置。
- 哪些是 Agent 运行配置。
- Report Agent 的报告类型在哪里配置。
- Report Agent 的 Initial Message 和触发消息是什么关系。
- Report Agent 的 Start Prompt Values 如何配置。
- 参数 textarea 中应该写哪些 key。

弹窗不适合承载定时任务创建这种业务配置，应改为独立创建 / 编辑页面。

### 3.2 `params key=value` 是错误产品形态

当前 `params` 是 `key=value` textarea，属于接口调试形态，不是用户产品形态。

对普通 Agent：

- 应根据 Agent 的 Start Prompt Template 生成结构化表单。

对 Report Agent：

- 应展示 Report Agent 运行所需的业务字段，例如报告类型、模型、补充要求、Start Prompt Values。
- 不应让用户手写 `report_type=xxx`。
- 不应让用户手写 `period_json`、`target_json`、`run_id`、`mcp_url`、credential。

### 3.3 Report Agent 不能走普通 SubmitTask 参数拼接

Report Agent 手动运行链路不是普通 SubmitTask。

Report Agent 运行需要：

- 创建 Aida ai_run。
- 生成 Report MCP credential。
- 注入 Report MCP 相关系统变量。
- 注入 `run_id`。
- 注入 `report_type`。
- 注入 `period_json`。
- 注入 `target_json`。
- Agent 通过 Report MCP 取数并回写报告事实源。

因此定时 Report Agent 不能直接走：

```go
SubmitTask(agent_id, model_id, params)
```

必须复用手动 Report Agent Run 的核心服务。

### 3.4 定时器配置与 Agent 运行配置混在一起

当前文档和实现风险在于容易把两类配置混成一类：

```text
schedule_type = daily / weekly
report_type = personal_daily / personal_weekly
```

这两个字段含义不同：

- `schedule_type`：定时器触发频率。
- `report_type`：Report Agent 入参。

必须分开保存、分开展示、分开处理。

### 3.5 时区不应展示给普通用户

时区是后端计算字段，默认使用系统统一时区，例如 `Asia/Shanghai`。

定时任务页面不要展示时区选择器，避免用户误解。

后端仍应保存和使用 timezone，但前端不让用户配置。

## 4. 产品口径定稿

### 4.1 定时任务定义

```text
定时任务 = 定时器配置 + Agent 运行配置
```

其中：

```text
定时器配置 = 什么时候触发
Agent 运行配置 = 触发后给 Agent 传什么
```

### 4.2 定时器配置

定时器只配置触发规则。

用户可配置：

- 触发频率：每天 / 每周
- 每天：几点执行
- 每周：周几 + 几点执行
- 启用状态

用户不可配置：

- 时区
- cron 表达式
- 多时段
- 节假日跳过
- 重试次数
- 队列并发

时区由后端默认处理。

### 4.3 Agent 运行配置

Agent 运行配置根据 Agent 类型不同而不同。

普通 Agent：

- 模型
- Initial Message
- Start Prompt Values

Report Agent：

- 报告类型
- 模型
- Initial Message / 补充要求
- Start Prompt Values

Report Agent 中以下内容不让用户填写：

- 报告对象
- 报告周期取值规则
- report_date
- period_start
- period_end
- period_json
- target_json
- run_id
- mcp_url
- credential
- AIDA_REPORT_MCP_AUTH

### 4.4 Report Agent 报告对象

Report Agent 手动运行时如果报告对象不是用户填写的，定时任务也不能让用户填写。

本期默认：

```text
个人日报 -> 报告对象 = 我自己
个人周报 -> 报告对象 = 我自己
```

运行时后端根据：

- schedule.user_id
- report_type

推导 target。

页面最多展示只读文案：

```text
报告对象：我自己
```

也可以只在预览中展示，不作为可编辑表单项。

### 4.5 Report Agent 报告周期

报告周期不是用户填写的固定值。

它每次运行时由后端动态生成。

动态生成依据：

- `report_type`
- `scheduled_trigger_at`
- 后端默认 timezone

规则：

```text
personal_daily:
  报告周期 = scheduled_trigger_at 所在日期的全天

personal_weekly:
  报告周期 = scheduled_trigger_at 所在自然周，周一到周日
```

示例：

```text
report_type = personal_daily
scheduled_trigger_at = 2026-07-03 19:00

period_start = 2026-07-03 00:00:00
period_end   = 2026-07-04 00:00:00
period_display = 2026-07-03 全天
```

```text
report_type = personal_weekly
scheduled_trigger_at = 2026-07-03 19:00

period_start = 2026-06-29 00:00:00
period_end   = 2026-07-06 00:00:00
period_display = 2026-06-29 ~ 2026-07-05
```

技术查询必须使用半开区间：

```text
[period_start, period_end)
```

不要使用闭区间 `23:59:59` 作为查询条件。

展示给用户可以写：

- `2026-07-03 全天`
- `2026-06-29 ~ 2026-07-05`

## 5. 最终页面设计

### 5.1 路由

建议使用独立页面：

```text
/ai-assets/agent-schedules/new
/ai-assets/agent-schedules/:scheduleId/edit
```

不要继续使用弹窗。

### 5.2 新建页面结构

页面分为 5 块：

```text
1. 基础信息
2. 选择 Agent
3. 定时器配置
4. Agent 运行配置
5. 预览
```

### 5.3 基础信息

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| 任务名称 | Input | 必填 |
| 启用状态 | Switch | 默认启用 |

不做：

- 优先级
- 队列配置
- 重试策略
- 负责人
- 复杂描述

### 5.4 选择 Agent

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| Agent | Select | 必填 |
| Agent 类型 | 只读 Tag | 普通 Agent / Report Agent |

Agent 类型由选中的 Agent 元数据决定，不让用户手选。

如果是系统预置 Agent，只显示轻量标签即可，例如：

```text

```

不要展示冗长说明。

### 5.5 定时器配置

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| 触发频率 | Radio / Select | 每天 / 每周 |
| 执行时间 | TimePicker | HH:mm |

页面文案：

```text
每天 19:00 执行
每周五 18:30 执行
```

不要展示时区。后端默认使用 `Asia/Shanghai` 或系统统一时区。

注意：这里的每天 / 每周只是定时器触发频率，不等于 Report Agent 的日报 / 周报。

### 5.6 普通 Agent 运行配置

当选择普通 Agent 时展示：

| 字段 | 类型 | 说明 |
|---|---|---|
| 模型 | Select | 默认使用 Agent 默认模型 |
| Initial Message | Textarea | 可选 |
| Start Prompt Values | 动态表单 | 根据 Agent 非系统变量生成 |

不展示 Report Agent 相关字段。

### 5.7 Report Agent 运行配置

当选择 Report Agent 时展示：

| 字段 | 类型 | 说明 |
|---|---|---|
| 报告类型 | Select | 必填，本期默认个人日报 / 个人周报 |
| 模型 | Select | 默认使用 Agent 默认模型 |
| Initial Message / 补充要求 | Textarea | 可选 |
| Start Prompt Values | 动态表单 | 根据 Agent 非系统变量生成 |

报告类型可选项：

```text
个人日报
个人周报
```

如果后端已经完整支持小组 / 部门报告的目标推导，可以再开放：

```text
小组日报
小组周报
部门日报
部门周报
```

但本期默认不要为了定时任务新增团队 / 部门选择器。

Report Agent 页面不展示：

- 报告对象编辑器
- 报告周期取值规则
- 时区
- 报告日期
- period_json
- target_json
- run_id
- mcp_url
- credential

### 5.8 预览

预览用于解释：

- 下一次什么时候触发。
- 触发后会以什么配置运行 Agent。
- 如果是 Report Agent，预计生成哪份报告。

普通 Agent 预览：

```text
下次触发时间：2026-07-01 09:00
将运行 Agent：xxx
模型：xxx
```

Report Agent 预览：

```text
下次触发时间：2026-07-01 19:00
报告类型：个人日报
报告对象：我自己
预计报告周期：2026-07-01 全天
```

```text
下次触发时间：2026-07-03 18:30
报告类型：个人周报
报告对象：我自己
预计报告周期：2026-06-29 ~ 2026-07-05
```

预览以后端计算为准。

如果本期能加 preview 接口，建议：

```text
POST /api/v1/ai-assets/agent-schedules/preview
```

入参：

```json
{
  "trigger_config": {
    "schedule_type": "weekly",
    "weekdays": [5],
    "time_of_day": "18:30"
  },
  "run_config": {
    "report_config": {
      "report_type": "personal_weekly"
    }
  }
}
```

返回：

```json
{
  "next_run_at": "2026-07-03T18:30:00+08:00",
  "period_start": "2026-06-29T00:00:00+08:00",
  "period_end": "2026-07-06T00:00:00+08:00",
  "period_display": "2026-06-29 ~ 2026-07-05"
}
```

如果本期不做 preview 接口，前端可以做辅助展示，但保存和执行必须以后端计算为准。

### 5.9 页面底部按钮

按钮：

```text
取消
保存
保存并立即运行
```

含义：

- 保存：保存定时任务，等待下一次定时触发。
- 保存并立即运行：保存配置后立即创建一次运行。
- 取消：返回列表。

保存并立即运行规则：

- 保存后 `next_run_at` 仍按定时器规则正常计算。
- 立即运行单独创建一次 run。
- 立即运行的 `scheduled_trigger_at = 用户点击保存并立即运行时的当前时间`。
- 不使用 `next_run_at` 作为立即运行的报告归属时间。

## 8. 后端执行模型

### 8.1 到点触发

调度器扫描到 `next_run_at <= now` 后：

1. 固化本次 `scheduled_trigger_at = schedule.next_run_at`。
2. 创建本次 `ai_run`。
3. 推进下一次 `next_run_at`。
4. 事务提交。
5. 事务外调用 Agent 平台。

注意：

- `next_run_at` 推进不等待 Agent 执行完成。
- `scheduled_trigger_at` 是报告周期计算依据。
- `actual_started_at` 只用于审计，不用于报告周期计算。

### 8.2 普通 Agent 执行

普通 Agent：

```text
读取 run_config
调用普通 Agent 运行链路
传入 model_id / initial_message / start_prompt_values
```

### 8.3 Report Agent 执行

Report Agent：

```text
读取 run_config.report_config.report_type
根据 schedule.user_id 推导 target
根据 report_type + scheduled_trigger_at 动态推导 period
调用手动 Report Agent Run 的核心服务
由服务创建 Report MCP credential 并注入系统变量
Agent 通过 Report MCP 取数和回写
```

必须复用手动 Report Agent Run 的核心逻辑，不允许定时器自己拼底层参数。

运行时后端生成：

- `target`
- `period_start`
- `period_end`
- `period_display`
- `period_json`
- `target_json`
- `run_id`
- `mcp_url`
- `credential`
- `AIDA_REPORT_MCP_AUTH`

这些都不是前端字段。

### 8.4 Agent 平台并发

Agent 平台已提供并发队列和执行排队能力。

本期 Aida 不额外设计复杂执行队列。

Aida 只负责：

- 定时触发幂等
- 短锁领取
- ai_run 状态承接
- next_run_at 推进
- timeout 熔断
- Report MCP 回写保护

## 9. next_run_at 计算规则

### 9.1 创建定时任务

后端根据当前时间、默认 timezone、trigger_config 计算首个 `next_run_at`。

daily：

- 今天配置时间未到：next_run_at = 今天该时间。
- 今天配置时间已过：next_run_at = 明天该时间。

weekly：

- 本周目标 weekday 的配置时间未到：next_run_at = 本周该 weekday 的该时间。
- 已过：next_run_at = 下一周该 weekday 的该时间。

### 9.2 编辑定时任务

修改以下字段后，后端重新计算 `next_run_at`：

- schedule_type
- weekdays
- time_of_day
- enabled 从 false 改 true

不沿用旧 `next_run_at`。

### 9.3 停用定时任务

`enabled=false` 后 scanner 必须忽略。

可以保留 `next_run_at`，也可以置空，但扫描逻辑必须以后端 enabled 判断为准。

### 9.4 重新启用

重新启用时，后端基于当前时间重新计算下一次 `next_run_at`。

不补跑停用期间错过的触发。

### 9.5 到点触发后

本次触发：

```text
scheduled_trigger_at = 旧 next_run_at
```

创建本次 run 后立即推进新的 `next_run_at`。

即使 scanner 延迟，例如 19:00 的任务 19:03 才被扫描到，本次 `scheduled_trigger_at` 仍然是 19:00。

### 9.6 保存并立即运行

保存并立即运行是一次手动触发来源：

- `trigger_source = save_and_run`
- `scheduled_trigger_at = 用户点击时的当前时间`

不使用 `next_run_at` 作为立即运行的报告归属时间。

## 10. 状态模型

优先兼容当前 `ai_runs.status`：

- pending
- running
- succeeded
- failed
- timeout

不强制新增大量状态枚举。

### 10.1 状态映射

| 语义 | ai_runs.status |
|---|---|
| 已创建 run，尚未提交 Agent 平台 | pending |
| 已提交到 Agent 平台，平台 queued / running | running |
| 成功 | succeeded |
| 明确失败 | failed |
| 超时 / 疑似卡住 / stale | timeout |

### 10.2 skipped 不作为 ai_runs.status

同一个 schedule 上一轮未结束导致本次跳过时，不建议创建 ai_run 并设置 skipped。

本期记录到 schedule：

- `last_skip_reason`
- `last_skip_at`
- `last_skipped_trigger_at`

如果后续新增 `schedule_runs` 表，再考虑 `skipped` 状态。

### 10.3 pending 超时

如果 Aida 已创建 ai_run，但因为进程崩溃、网络中断或未知原因，pending 超过 10 分钟仍未拿到 external task/session id：

```text
ai_runs.status = timeout
```

### 10.4 running 超时

如果 running 超过阈值仍未成功或失败：

```text
ai_runs.status = timeout
```

建议初始阈值：

- pending 超时：10 分钟
- running 超时：2 小时

不做自动重试。

## 11. Report MCP 回写保护

Report MCP `write_report_result` 必须校验 run 状态。

允许正常回写：

```text
running
```

`succeeded` 状态只允许幂等确认：

- 同一 run_id
- 同一 report_type
- 同一 period_start / period_end
- 同一 target
- payload hash 或关键参数一致

返回成功，但不重复覆盖报告内容。

必须拒绝：

- failed
- timeout
- 不存在的 run_id
- user_id 不一致
- report_type 不一致
- period 不一致
- target 不一致

目的：

- 防止 timeout 的旧 run 迟到回写污染报告事实源。
- 允许 Agent 因网络抖动重试 write_report_result 时得到幂等成功。

Credential TTL 是第二道防线。

如果当前 credential 支持 expires_at，本期应加上 TTL 校验。

如果暂不支持 TTL，标记为后续增强，但本期必须实现 run 状态校验。

## 12. 多用户与长任务安全

### 12.1 Agent 平台负责执行并发

Agent 平台已具备并发控制和排队能力。

几百个任务提交到 Agent 平台后，由 Agent 平台排队执行。

Aida 本期不再额外设计复杂执行队列。

### 12.2 Aida 负责调度正确性

Aida 必须保证：

- 同一 schedule 同一 scheduled_trigger_at 只触发一次。
- 一个用户的 run 死掉，不影响其他用户。
- 一个用户的旧 run timeout 后，不永久阻塞后续触发。
- timeout 的旧 run 后续不能回写报告。

### 12.3 同一 schedule active run 规则

同一个 schedule 如果上一轮 run 仍是 active：

- pending 未超时：本次跳过，记录 last_skip_reason。
- running 未超时：本次跳过，记录 last_skip_reason。
- pending / running 已超时：旧 run 标记 timeout，新触发可以继续。

不同 schedule 大量同时到期时，不跳过，由 Agent 平台排队执行。

## 16. 实施步骤

### 阶段 1：后端模型收敛

1. 梳理当前 `managed_agent_schedules` 字段和请求体。
2. 增加或兼容 `run_kind`。
3. 增加或兼容 `initial_message`。
4. 增加或兼容 `start_prompt_values_json`。
5. 增加或兼容 `report_config_json`。
6. 增加 `next_run_at`。
7. 后端默认 timezone，不要求前端传。

### 阶段 2：Report Agent 定时运行链路

1. 抽取手动 Report Agent Run 核心服务。
2. 定时任务到点时，使用 schedule 的 run_config 调用该服务。
3. 运行时根据 report_type + scheduled_trigger_at 动态生成 period。
4. 运行时根据 schedule.user_id 推导 target。
5. 由服务注入 Report MCP 系统变量。
6. 禁止定时任务直接拼 Report MCP 参数。

### 阶段 3：调度执行安全

1. Scanner 使用 next_run_at 判断到期。
2. 到期后固化 scheduled_trigger_at。
3. 创建 ai_run 后立即推进 next_run_at。
4. 调用 Agent 平台失败时标记 failed。
5. pending / running 超时标记 timeout。
6. 同一个 schedule active run 未超时时跳过本次。
7. timeout 的旧 run 不再阻塞后续触发。
8. Report MCP 回写时校验 run 状态。

### 阶段 4：前端页面改造

1. 将“新建定时任务”从弹窗改为独立页面。
2. 页面拆分为：基础信息、选择 Agent、定时器配置、Agent 运行配置、预览。
3. 删除 params textarea。
4. 删除时区展示。
5. 普通 Agent 展示模型、Initial Message、Start Prompt Values。
6. Report Agent 展示报告类型、模型、Initial Message / 补充要求、Start Prompt Values。
7. Report Agent 不展示报告对象编辑器和报告周期规则。
8. 增加预览。
9. 列表展示下次运行和最近结果。

### 阶段 5：测试

## 17. 给 Codex 的执行约束

实现时必须遵守：

1. 不要把日报绑定每天触发。
2. 不要把周报绑定每周触发。
3. 不要让定时器决定 report_type。
4. 不要让用户填写报告对象。
5. 不要让用户填写报告周期规则。
6. 不要展示时区。
7. 不要展示 period_json / target_json / run_id / mcp_url / credential。
8. 不要使用 params key=value textarea。
9. Report Agent 定时运行必须复用手动 Report Agent Run 核心服务。
10. Agent 平台负责执行并发，Aida 不做复杂执行队列。
11. Aida 必须做调度幂等、timeout 熔断、Report MCP 回写保护。
12. 前端页面只配置：定时器 + Agent 运行配置。

---

## 18. 最终一句话

最终方案是：

> 用户创建定时任务时，只配置“什么时候触发”和“触发后 Agent 需要的业务入参”。Report Agent 的报告对象、报告周期、MCP 参数、run_id、credential 都由后端在每次运行时动态生成。定时器的每天/每周只是触发频率，不等于 Report Agent 的日报/周报类型。
