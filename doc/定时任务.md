# AI Assets 定时任务现状与改造方案（最终执行版）

## 0. 核心结论

本次改造必须按下面这句话理解：

> **定时任务 = 定时器配置 + Agent 运行配置。**

两部分必须分离：

- **定时器配置**：只负责“什么时候触发”。
- **Agent 运行配置**：只负责“触发后给 Agent 传什么”。

不要把 Report Agent 的 `report_type` 和定时器的 `schedule_type` 绑定。

错误理解：

```text
日报 = 每天触发
周报 = 每周触发
```

正确理解：

```text
schedule_type=daily / weekly 是定时器触发频率。
report_type 是 Report Agent 的运行入参。
二者是并列配置，不是谁推导谁。
```

例如：

- 每天 19:00 触发个人日报：合理。
- 每周五 19:00 触发个人日报：合理，表示每周五生成周五当天日报。
- 每天 19:00 触发个人周报：也可以成立，表示每天刷新当前自然周周报。
- 每周五 19:00 触发个人周报：常见配置，表示每周五生成当前自然周周报。

产品上可以推荐常用组合，但实现上不能把 `daily schedule` 等同于日报，也不能把 `weekly schedule` 等同于周报。

---

## 1. 背景

当前 AI Assets 已有 Agent 定时任务能力，但现状偏底层，容易把定时器字段、Agent 入参、Report MCP 系统参数混在一起。

目标不是新增一个“Report Agent 调度中心”，而是在 AI Assets 里把“定时运行 Agent”做成稳定、可理解、可维护的能力。

典型场景：

- 每天 19:00 运行 Report Agent，生成个人日报。
- 每周五 18:30 运行 Report Agent，生成部门周报。
- 每天 09:00 运行普通 Agent，执行固定任务。

本次页面和接口必须围绕两个配置块展开：

```text
trigger_config：定时器配置
run_config：Agent 运行配置
```

## 2. 当前问题

### 2.1 前端表单偏底层

当前定时任务表单容易出现以下底层配置：

- params key=value textarea
- timezone
- period_json
- target_json
- run_id
- mcp_url
- credential
- AIDA_REPORT_MCP_AUTH

这些都不应该让用户填写。

### 2.2 Report Agent 运行配置没有业务化

Report Agent 运行必须有报告类型、模型、Initial Message / 补充要求、Start Prompt Values。

定时任务里必须配置 Report Agent 运行所需的业务入参，但不能让用户填写报告对象、报告周期或 MCP 系统参数。

### 2.3 定时器和报告类型容易被错误绑定

定时器的 daily / weekly 只是触发频率。

Report Agent 的 personal_daily / personal_weekly / group_daily / group_weekly / department_daily / department_weekly 是报告类型。

二者不能互相推导。

### 2.4 后台执行身份容易错位

定时任务是后台执行，必须明确：

- 报告上下文权限基于 `schedule.user_id`。
- Report MCP credential 创建基于 `schedule.user_id`。
- 报告对象推导基于 `schedule.user_id + report_type`。
- 不得使用全局 `MANAGED_AGENT_TOKEN` 代替用户身份读取报告上下文或写回报告。

`MANAGED_AGENT_TOKEN` 如仍存在，只能用于 Aida 调 Agent 平台的服务级鉴权，不能代表报告数据权限。

## 3. 产品口径

### 3.1 定时任务定义

```text
定时任务 = 定时器配置 + Agent 运行配置
```

定时任务不是新的 Agent 类型，也不是新的报告模块。

### 3.2 定时器配置

定时器只配置“什么时候触发”：

- 每天几点执行。
- 每周几、几点执行。

不展示时区。后端默认使用系统时区，建议默认 `Asia/Shanghai`。

用户不配置：

- 报告日期
- 报告周期
- 报告对象
- period_json
- target_json
- cron
- retry
- queue
- concurrency

### 3.3 Agent 运行配置

Agent 运行配置负责“触发后给 Agent 传什么”。

普通 Agent：

- 模型
- Initial Message
- Start Prompt Values

Report Agent：

- 报告类型
- 模型
- Initial Message / 补充要求
- Start Prompt Values

Report Agent 的报告对象、报告周期、MCP 参数均由后端运行时生成。

#### 4.2.1 基础信息

- 启用状态，默认启用

#### 4.2.2 选择 Agent

字段：

- Agent Select
- Agent 类型只读展示：普通 Agent / Report Agent

Agent 类型不允许用户手选。

#### 4.2.3 定时器配置

字段：

- 触发频率：每天 / 每周
- 如果每天：几点执行
- 如果每周：周几 + 几点执行

不展示时区。

```text

```

#### 4.2.4 普通 Agent 运行配置

如果选择普通 Agent，展示：

- 模型
- Initial Message
- Start Prompt Values

不展示 Report 相关字段。

#### 4.2.5 Report Agent 运行配置

如果选择 Report Agent，展示：

- 报告类型
- 模型
- Initial Message / 补充要求
- Start Prompt Values

报告类型为当前系统支持的 6 种：

- 个人日报
- 个人周报
- 小组日报
- 小组周报
- 部门日报
- 部门周报

报告类型字段保存为现有后端 report_type 枚举。若当前枚举值为英文，可按现有实现，例如：

- `personal_daily`
- `personal_weekly`
- `group_daily`
- `group_weekly`
- `department_daily`
- `department_weekly`

最终枚举名以当前后端已有定义为准，不要在前端另造一套命名。

Report Agent 运行配置不展示：

- 报告对象选择器
- 报告周期取值规则
- 报告日期
- 时区
- period_json
- target_json
- run_id
- mcp_url
- credential
- AIDA_REPORT_MCP_AUTH

#### 4.2.6 预览

预览展示“定时器结果 + Agent 入参结果”。

普通 Agent 预览：

- 下次触发时间
- 将运行的 Agent

Report Agent 预览：

- 下次触发时间
- 报告类型
- 报告对象，只读展示
- 预计报告周期

报告对象只读展示规则：

| 报告类型 | 报告对象展示 |
|---|---|
| 个人日报 / 个人周报 | 我自己 |
| 小组日报 / 小组周报 | 我所在小组 |
| 部门日报 / 部门周报 | 我所在部门 |

报告对象不是用户输入项。后端根据 `schedule.user_id + report_type` 推导。

如果后端无法根据当前用户推导小组或部门，则对应报告类型不可创建，或保存时返回明确错误。

### 4.3 编辑定时任务页

路径：

```text
/ai-assets/agent-schedules/:id/edit
```

与新建页结构一致，带已有数据回显。

可编辑：

- 任务名称
- 启用状态
- Agent
- 定时器配置
- 模型
- Initial Message / 补充要求
- Start Prompt Values
- Report Agent 的报告类型

编辑触发规则后，后端重新计算 `next_run_at`。

### 4.4 页面底部按钮

新建 / 编辑页底部按钮：

- 取消
- 保存
- 保存并立即运行

保存并立即运行规则：

- 先保存定时任务配置。
- `next_run_at` 仍按定时器配置正常计算。
- 额外创建一次 `save_and_run` 运行。
- `save_and_run` 的 `scheduled_trigger_at = 用户点击保存并立即运行时的当前时间`。
- 不使用 `next_run_at` 作为立即运行的报告归属时间。

    "schedule_type": "daily",

  },
  "run_config": {
    "model_id": "model_xxx",
    "initial_message": "请重点关注本部门本周完成事项、风险和下周计划。",
    "start_prompt_values": {},
    "report_config": {
      "report_type": "department_weekly"
    }
  }

如果当前后端接口仍使用平铺字段，可以兼容为：

```json
{
  "name": "每日部门周报",
  "agent_id": "agent_xxx",
  "run_kind": "report_agent",
  "enabled": true,
  "schedule_type": "daily",
  "time_of_day": "19:00",
  "model_id": "model_xxx",
  "initial_message": "请重点关注本部门本周完成事项、风险和下周计划。",
  "start_prompt_values": {},
  "report_config": {
    "report_type": "department_weekly"
  }
}
```

后端补默认值：

```text
timezone = Asia/Shanghai
```

前端不传：

- target
- period_rule
- period_start
- period_end
- period_json
- target_json
- credential

### 6.2 trigger_config

定时器配置只包含：

```text
schedule_type: daily / weekly
weekdays: weekly 时使用
time_of_day: HH:mm
timezone: 后端默认 Asia/Shanghai，前端不展示
```

### 6.3 run_config

普通 Agent：

```text
model_id
initial_message
start_prompt_values
```

Report Agent：

```text
model_id
initial_message
start_prompt_values
report_config.report_type
```

### 6.4 run_kind 与 Agent profile 类型映射

`run_kind` 是定时任务层字段，用于区分执行路径：

```text
generic_agent

```

如果当前 Agent profile 使用 `business_type=generic/report`，必须固定映射：

```text
run_kind=generic_agent -> business_type=generic
run_kind=report_agent  -> business_type=report
```

不要在实现里同时使用两套互不映射的类型判断。

## 8. ai_runs 运行字段承接

本期优先复用 `ai_runs`，不强制新增 `schedule_runs` 表。

定时任务每次触发必须创建或关联一条 `ai_runs` 记录，用于承接本次运行状态和审计信息。

以下字段统一建议放到 `ai_runs.input_ref_json` / `metadata` 中，不要散落在临时字段里：

```text
schedule_id
trigger_source
scheduled_trigger_at
actual_started_at
report_type
target
period_start
period_end
period_display
model_id
start_prompt_values
initial_message
external_session_id
external_task_id
external_status
```

如果当前 `ai_runs` 已有明确字段可承接某些值，可以复用现有字段；否则统一进入 JSON metadata。

列表页、详情页、Report MCP 回写校验都应从同一处读取这些字段，避免多处拼接。

### 8.1 trigger_source

统一取值：

```text
scheduled：系统到点自动触发
manual：用户在定时任务列表点击“运行”
save_and_run：用户创建/编辑后点击“保存并立即运行”
```

三者复用同一套执行逻辑，但必须记录来源，便于审计。

## 9. 调度时间计算规则

### 9.1 创建定时任务

后端根据 `schedule_type`、`weekdays`、`time_of_day`、`timezone` 计算首个 `next_run_at`。

日报/周报报告类型不参与 `next_run_at` 计算。

### 9.2 编辑触发规则

修改以下任一字段后，后端基于当前时间重新计算 `next_run_at`：

- schedule_type
- weekdays
- time_of_day
- timezone

不沿用旧 `next_run_at`。

### 9.3 停用

`enabled=false` 后 scanner 不扫描。

可以保留 `next_run_at`，但 scanner 必须忽略 disabled schedule。

### 9.4 重新启用

`enabled` 从 false 改为 true 时，后端基于当前时间重新计算下一次 `next_run_at`。

不补跑停用期间错过的触发。

### 9.5 到点触发

到点触发时：

```text
本次 scheduled_trigger_at = 当前旧的 next_run_at
```

创建本次 `ai_run` 后，立即推进下一次 `next_run_at`。

不要等待 Agent 执行完成。

如果 scanner 延迟，例如 19:00 的 `next_run_at` 到 19:03 才被扫描到，本次 `scheduled_trigger_at` 仍然是 19:00。

## 10. Report Agent 运行时推导规则

Report Agent 到点运行时，后端根据：

```text
schedule.user_id
report_config.report_type
scheduled_trigger_at
```

动态生成：

```text
target
period_start
period_end
period_display
period_json
target_json
run_id
mcp_url
credential
AIDA_REPORT_MCP_AUTH
```

这些字段不由前端保存，也不由用户填写。

### 10.1 报告对象推导

| report_type | target 推导 |
|---|---|
| 个人日报 / 个人周报 | schedule.user_id 对应用户本人 |
| 小组日报 / 小组周报 | schedule.user_id 所在小组 |
| 部门日报 / 部门周报 | schedule.user_id 所在部门 |

如果无法推导小组或部门，必须返回明确错误，不要让前端补选择器。

### 10.2 报告周期推导

| report_type | 周期规则 |
|---|---|
| 个人日报 / 小组日报 / 部门日报 | `scheduled_trigger_at` 所在日期全天 |
| 个人周报 / 小组周报 / 部门周报 | `scheduled_trigger_at` 所在自然周，周一到周日 |

技术查询使用半开区间：

日报类：

```text
[当天 00:00:00, 次日 00:00:00)
```

周报类：

```text
[本周一 00:00:00, 下周一 00:00:00)
```

自然周统一按周一开始、周日结束。

展示文案可以是：

```text
2026-07-01 全天
2026-06-29 ~ 2026-07-05
```

## 11. Report Agent 定时执行链路

Report Agent 定时任务不能直接走普通 Agent 的 `SubmitTask(agent_id, params)`。

必须复用手动 Report Agent Run 的核心服务。

流程：

```text
Schedule scanner 到点
  -> 固化 scheduled_trigger_at
  -> 创建 ai_run
  -> 推进 next_run_at
  -> 根据 schedule.user_id + report_type 推导 target
  -> 根据 report_type + scheduled_trigger_at 推导 period
  -> 创建 Report MCP credential
  -> 注入 report_type / period_json / target_json / run_id / mcp_url / credential
  -> 调用 Agent 平台创建 session / task
  -> Agent 通过 Report MCP get_report_context 取数
  -> Agent 通过 Report MCP write_report_result 回写
  -> Aida 校验 run 状态与幂等信息
  -> 写报告事实源
  -> ai_run succeeded
```

### 11.1 后台身份硬规则

定时任务后台执行时：

- 报告上下文必须基于 `schedule.user_id`。
- 报告对象推导必须基于 `schedule.user_id`。
- Report MCP credential 创建必须绑定 `schedule.user_id` / `ai_run.user_id`。
- Report MCP `get_report_context` / `write_report_result` 权限判断必须回到 `schedule.user_id` / `ai_run.user_id`。
- 不得使用全局 `MANAGED_AGENT_TOKEN` 代替用户身份访问 Report MCP 或报告上下文。

`MANAGED_AGENT_TOKEN` 如仍存在，只能用于 Aida 调用 Agent 平台的服务级鉴权。

## 12. 大规模并发与长任务安全方案

Agent 平台已确认自带并发排队能力，因此 Aida 本期不额外设计复杂执行队列。

职责边界：

```text
Agent 平台负责：
- Agent 任务排队
- Agent 执行并发
- 模型调用并发

Aida 负责：
- 到期 schedule 幂等触发
- ai_run 创建
- schedule.user_id 身份绑定
- scheduled_trigger_at 固化
- next_run_at 推进
- active run 防重
- timeout 熔断
- Report MCP 回写保护
```

Aida 不做：

- 复杂执行队列
- 自动重试
- 失败自动补跑
- 指数退避
- Report Agent 专用 worker pool
- 队列监控页

### 12.1 短锁原则

数据库锁只用于短时间领取 schedule 和创建 ai_run。

绝不能持锁等待 Agent 平台执行完成。

### 12.2 幂等触发

建议保证：

```text
同一个 schedule_id + scheduled_trigger_at 只能创建一条 ai_run
```

可使用唯一约束或原子领取机制实现。

### 12.3 同一个 schedule active run 规则

同一个 schedule 不允许多个 active run 长时间并发。

active 可按现有状态理解为：

```text
pending
running
```

规则：

- 如果无 active run：创建本次 run。
- 如果有 active run 且未超时：本次跳过，写 `last_skip_reason`。
- 如果有 active run 且已超时：旧 run 标记 `timeout`，新触发继续创建 run。

## 13. ai_runs 状态模型

本期优先复用现有 `ai_runs.status`：

```text
pending
running
succeeded
failed
timeout
```

不强制新增大量状态枚举。

映射规则：

| 状态 | 含义 |
|---|---|
| pending | Aida 已创建 ai_run，尚未完成 Agent 平台提交，或正在提交中 |
| running | 已提交到 Agent 平台；Agent 平台内部 queued / submitted / running 可统一映射为 running |
| succeeded | 运行成功；Report Agent 以 Report MCP 成功回写为主要依据 |
| failed | 明确失败，例如 Agent 平台提交失败、运行明确失败、Report MCP 明确失败 |
| timeout | Aida 判断运行超时 / stale / 疑似卡住 |

`skipped` 不作为 `ai_runs.status`。

同一个 schedule 上一轮未结束导致本次跳过时，记录到：

```text
schedule.last_skip_reason
schedule.last_skip_at
schedule.last_skipped_trigger_at
```

### 13.1 提交失败与超时

明确失败：

```text
Agent 平台 CreateSession / SubmitTask 明确返回错误
  -> ai_run.status = failed
  -> 写 error_message
  -> 写 schedule.last_error
  -> 不回滚 next_run_at
  -> 不自动重试
```

未知卡住：

```text
Aida 已创建 ai_run，但 10 分钟后仍未拿到 external_session_id / external_task_id
  -> ai_run.status = timeout
```

运行超时：

```text
running 超过 2 小时仍未成功或失败
  -> ai_run.status = timeout
```

## 14. Report MCP 回写与幂等

Report MCP `write_report_result` 必须校验 run 状态、用户身份、报告类型、报告对象和报告周期。

### 14.1 允许回写

正常允许：

```text
ai_run.status = running
```

### 14.2 succeeded 幂等确认

如果同一 run 已经 `succeeded`，允许同一请求幂等确认，但不允许覆盖内容。

首次成功写入时，需要记录：

```text
report_result_hash
```

存储位置：

- 优先放 `ai_runs.output_ref_json`；或
- 放报告结果表 metadata。

后续同一 `run_id` 在 succeeded 状态下再次调用 `write_report_result`：

- 如果 `run_id + report_type + target + period_start + period_end + report_result_hash` 一致，返回幂等成功。
- 返回 `already_written=true` 或等价信息。
- 不重复覆盖报告内容。
- 如果 hash 或关键参数不一致，返回冲突或拒绝。

### 14.3 必须拒绝回写

以下情况必须拒绝：

- ai_run.status = failed
- ai_run.status = timeout
- run_id 不存在
- user_id 不一致
- report_type 不一致
- target 不一致
- period_start / period_end 不一致
- succeeded 状态下 payload hash 不一致

### 14.4 Credential TTL

Run 状态校验是本期硬要求。

credential TTL 是第二道防线。

如果当前 Report MCP credential 支持 `expires_at`：

- 本期应增加过期时间。
- TTL 建议 2 小时或 4 小时。
- MCP 鉴权时校验 TTL。

如果当前 credential 机制暂不支持 TTL：

- 标记为能力缺口 / 后续增强。
- 不能影响本期 run 状态校验落地。

## 15. 后端 preview 规则

前端预览可以展示，但后端计算结果才是权威。

推荐新增或复用 preview 接口：

```text
POST /api/v1/ai-assets/agent-schedules/preview
```

入参：

```text
schedule_type
weekdays
time_of_day
agent_id
run_kind
report_type
```

前端不传 timezone；后端使用默认 timezone。

返回：

```text
next_run_at
report_type
report_target_display
period_start
period_end
period_display
```

保存时后端必须再次计算，不信任前端 preview。

## 16. 实施步骤

### 阶段 1：收敛前端页面

- Report Agent 的 `report_config_json` 至少保存 `report_type`。
- 不保存固定 target / period。

### 阶段 3：实现 next_run_at 计算

- 创建时计算首个 `next_run_at`。
- 编辑触发规则后重算。
- 停用后 scanner 忽略。
- 重新启用后重算，不补跑。

### 阶段 5：补齐状态与回写保护

- 使用现有 `ai_runs.status`。
- pending / running / succeeded / failed / timeout 映射清楚。
- Report MCP 回写校验 run 状态。
- succeeded 支持幂等确认。
- timeout / failed 禁止回写。

## 17. 测试清单

### 17.1 前端页面测试

- 新建普通 Agent 定时任务，只展示普通 Agent 运行配置。
- 新建 Report Agent 定时任务，展示 6 种报告类型。
- Report Agent 页面不展示报告对象选择器。
- Report Agent 页面不展示报告周期规则。

- weekly schedule 可以选择任意 report_type。
- 预览能展示下次触发时间、报告对象摘要、预计报告周期。

### 17.2 调度测试

- 创建 daily schedule 后正确计算 next_run_at。
- 创建 weekly schedule 后正确计算 next_run_at。
- 编辑触发规则后 next_run_at 重算。
- 停用后 scanner 忽略。
- 重新启用后 next_run_at 重算，不补跑停用期间任务。
- 到点触发后 scheduled_trigger_at 使用旧 next_run_at。
- 到点触发后立即推进下一次 next_run_at。

自动化 / 手工验证时，可将触发时间设置为当前时间后 3 分钟，避免等待固定 19:00 或周五。

### 17.3 Report Agent 测试

- 个人日报：target 推导为本人，period 推导为触发日期全天。
- 个人周报：target 推导为本人，period 推导为触发时间所在自然周。
- 小组日报：target 推导为用户所在小组，period 推导为触发日期全天。
- 小组周报：target 推导为用户所在小组，period 推导为触发时间所在自然周。
- 部门日报：target 推导为用户所在部门，period 推导为触发日期全天。
- 部门周报：target 推导为用户所在部门，period 推导为触发时间所在自然周。
- 用户无小组 / 部门时，创建或运行对应报告类型返回明确错误。

### 17.4 身份与权限测试

- 定时任务运行时，Report MCP get_report_context 使用 schedule.user_id 权限。
- 定时任务运行时，Report MCP write_report_result 绑定 ai_run.user_id。
- 不允许 MANAGED_AGENT_TOKEN 代替用户身份访问报告上下文。

### 17.5 失败与超时测试

- Agent 平台 CreateSession / SubmitTask 明确失败时：
  - ai_run.status = failed
  - 写 error_message
  - 写 schedule.last_error
  - 不回滚 next_run_at
  - 不自动重试
- pending 超过阈值仍未拿到 external id，标记 timeout。
- running 超过阈值仍未完成，标记 timeout。
- timeout run 不允许 Report MCP 回写。
- succeeded run 重复 write_report_result 且 hash 一致，返回幂等成功。
- succeeded run 重复 write_report_result 但 hash 不一致，拒绝或冲突。

## 18. 本期不做

本期明确不做：

- 复杂执行队列
- 自动重试
- 失败自动补跑
- cron 表达式
- 多时间段触发
- 节假日跳过
- 队列监控页
- 并发配置页
- 目标对象选择器
- 报告周期规则选择器
- 时区选择器
- 让用户填写 period_json / target_json / run_id / mcp_url / credential
