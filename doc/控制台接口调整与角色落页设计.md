# 控制台接口调整与角色落页设计

版本：v0.1  
日期：2026-06-24  
范围：基于当前前端 `/dashboard` 原型、旧服务接口和现有后台代码，梳理真实落地方案。本文只做设计结论，不包含代码改动。

---

## 1. 结论

当前旧接口可以支撑一部分控制台数据，但不能直接支撑完整 `/dashboard` 原型。

建议采用：

```text
现有业务接口继续保留
  ↓
新增控制台聚合接口
  ↓
前端 dashboard 只消费聚合后的轻量数据
```

核心新增接口建议为：

```http
GET /api/v1/dashboard
```

控制台不要让前端同时拼接 `requirements`、`tasks`、`reports`、`team reports`、`tokens`、`sessions`、`teams/activity` 等多个接口，否则角色差异、风险规则、报告状态会散落在前端，后续维护成本较高。

关于“每个账号根据角色不同，登录页面也不一样”：当前代码已经有路由权限和默认可访问路由跳转，但还没有明确的“按角色指定登录后首页”设计。由于 `/dashboard` 是第一个可见菜单且没有角色限制，目前登录后基本都会进入控制台。后续应明确为：

```text
所有角色登录后默认进入 /dashboard
/dashboard 内部根据角色返回不同数据和操作入口
```

不建议 P0 做成四套完全不同首页。

## 2. 当前已有接口能力

### 2.1 用户与角色

现有接口：

```http
POST /api/v1/auth/login
GET  /api/v1/auth/me
GET  /api/v1/users
GET  /api/v1/teams
PUT  /api/v1/admin/users/{id}
POST /api/v1/admin/users/{id}/reset-password
```

现有用户角色：

```text

director
pm
team_leader

```

当前角色已经写入 JWT，也会通过 `/auth/me` 返回给前端。前端路由层已经支持 `roles` 控制访问。

需要注意：

```text
GET /api/v1/users 当前在认证中间件之外，实际是公开接口。
```

如果进入正式实现，建议至少放入登录态保护。

## 4. 报告数据模型调整建议

旧模型分为 `daily_reports` 和 `team_reports`。如果继续扩展，会很快遇到周报、部门报告、发送状态难以统一的问题。

建议新增统一报告表，旧表可迁移或保留兼容：

```sql
reports
```

建议字段：

```text
id
kind                  -- personal_daily / personal_weekly / team_daily / team_weekly / department_daily / department_weekly
scope_type            -- personal / team / department
scope_id              -- user_id / team_id / department_id，可先用 user_id 或 team_id
owner_id              -- 报告操作人或负责人
report_date
period_start
period_end
status                -- pending / generating / draft_pending / sent / generate_failed / send_failed
content
edited
feishu_doc_url
send_target
send_error
source_session_ids
source_report_ids
generated_at
sent_at
created_at
updated_at
```

P0 如果不想大迁移，也可以先在现有两张表上补字段：

```text
status
generated_at
sent_at
send_error
period_start
period_end
report_type
```

但长期看统一 `reports` 表更适合。

## 5. 角色登录落页现状

### 5.1 当前前端已有能力

当前前端已有：

```text
AuthProvider 负责登录、恢复登录态、加载当前用户
PermissionGuard 负责校验当前路由 roles
FirstAccessibleRedirect 负责从 / 跳转到第一个可访问菜单
routes.tsx 中每个路由可以配置 roles
menu.ts 会根据用户角色过滤菜单
```

当前 `appRoutes` 里 `/dashboard`：

```text
path: /dashboard
title: 控制台
hideInMenu: false
roles: 未设置
menuOrder: 10
```

因此它对所有登录用户可见，并且是第一个可见菜单。

当前登录页逻辑：

```text
如果用户访问受保护页面后跳到 /login，登录成功后回到原目标页面。
如果没有目标页面，登录成功后跳到 /。
/ 再通过 FirstAccessibleRedirect 跳到第一个可访问菜单。
```

因为 `/dashboard` 是所有角色第一个可访问菜单，所以当前实际效果是：

```text
admin       -> /dashboard
director    -> /dashboard
pm          -> /dashboard
team_leader -> /dashboard
employee    -> /dashboard
```

### 5.2 当前没有做的设计

当前没有显式配置：

```text
admin 登录默认去组织管理
director 登录默认去总监首页
pm 登录默认去 PM 首页
team_leader 登录默认去 TL 首页
employee 登录默认去个人首页
```

也没有后端返回：

```text
default_home_path
dashboard_variant
landing_page
```

当前更接近：

```text
统一进入 /dashboard，再由 dashboard 内部根据 role 展示不同内容。
```

这符合当前需求基线“不做四套独立首页”的方向。

## 6. 角色落页建议

### 6.1 P0 推荐方案

P0 推荐：

```text
所有角色登录后默认进入 /dashboard
/dashboard 是统一控制台路由
/dashboard 根据当前用户 role 请求对应聚合数据
```

角色差异体现在数据和操作入口：

| 角色 | 默认落页 | 控制台数据重点 |
| --- | --- | --- |
| employee | `/dashboard` | 我的报告、我的任务关注、我的 deadline / 依赖风险 |
| team_leader | `/dashboard` | 我的报告、组报告、组内关注、组内任务风险、成员日报覆盖 |
| pm | `/dashboard` | 我的报告、我关注 / 负责的需求、需求推进风险、缺 AC / deadline 风险 |
| director | `/dashboard` | 我的报告、部门报告、关注需求风险、部门级报告覆盖 |
| admin | `/dashboard` 或 `/organization` | P0 建议仍进 dashboard，可在 dashboard 提供组织管理入口 |

理由：

```text
控制台是登录后的首屏聚合入口。
需求基线已经明确不做四套独立首页。
统一路由方便权限、菜单、跳转、埋点和后续维护。
```

### 6.2 是否需要每个角色登录到不同页面

如果产品坚持“不同角色登录页面不一样”，建议不要做成不同 URL，而是做成：

```text
/dashboard?view=employee
/dashboard?view=team_leader
/dashboard?view=pm
/dashboard?view=director
```

或者前端内部根据角色切换视图组件。

不建议：

```text
/employee-dashboard
/tl-dashboard
/pm-dashboard
/director-dashboard
```

原因：

```text
容易形成四套页面。
重复实现报告、风险、关注模块。
后续角色变化和权限变化更难维护。
```

### 6.3 可选的默认落页配置

如果后续确实需要控制不同角色默认落页，可以新增前端映射：

```ts
const DEFAULT_HOME_BY_ROLE = {
  admin: "/dashboard",
  director: "/dashboard",
  pm: "/dashboard",
  team_leader: "/dashboard",
  employee: "/dashboard"
}
```

暂时所有角色都指向 `/dashboard`。

如果未来 admin 想默认进组织管理，可以调整为：

```ts
admin: "/organization"
```

但这应作为明确产品决策，而不是由路由顺序隐式决定。

也可以让后端 `/auth/me` 返回：

```json
{
  "default_home_path": "/dashboard"
}
```

P0 不建议增加这个字段，前端静态映射足够。

## 8. 与现有接口的落地关系

`GET /api/v1/dashboard` 内部可以复用：

```text
/auth/me
/reports/today
/reports
/reports/team/today
/reports/team/members
/reports/team
/requirements
/tasks
/tokens
/tokens/sessions
/sessions
/teams/activity
```

但这些复用应发生在后端查询层，不建议由前端页面拼接。

后端聚合时负责：

```text
角色范围过滤
报告状态映射
风险实时计算
关注对象聚合
token/session 摘要聚合
控制台列表排序
跳转 URL 生成
```

前端负责：

```text
展示
交互
打开报告弹窗
跳转需求看板
提交报告生成 / 编辑 / 发送动作
```

## 9. P0 实施顺序

建议顺序：

```text
1. 明确登录后所有角色默认进入 /dashboard。
2. 新增 /api/v1/dashboard 聚合接口。
3. 新增报告状态字段，先支持个人日报状态闭环。
4. 新增 follows 表和关注接口。
5. 在 dashboard 聚合接口中计算 deadline / 依赖阻塞风险。
6. 补充组日报入口和成员提交覆盖。
7. 再扩展个人周报、组周报、部门日报、部门周报。
```

P0 可以先不做：

```text
独立风险表
复杂风险生命周期
自定义报告模板
飞书深度发送
四套独立首页
复杂 Token 大盘
```

## 10. 最终建议

最终方案应定为：

```text
登录落页：
所有角色默认进入 /dashboard。

页面形态：
统一控制台，不做多套首页。

角色差异：
由 /api/v1/dashboard 按当前用户 role 返回不同数据范围和可见操作。

接口策略：
旧接口继续作为业务详情和基础能力接口。
```
