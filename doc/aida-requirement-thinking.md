## 1. 背景判断

现有 `doc/aida-platform-summary.md` 把初版需求、会议纪要、原型和 demo 状态汇总成了一份较完整的产品文档，其中包含角色、页面、字段、流程、阶段规划和验收标准。

但当前 Aida 仍处在需求思考和方向收敛阶段，不应直接把这些内容升级为正式 PRD，也不应据此拆开发任务。现在更重要的是先确认：Aida 第一阶段到底要验证什么。

当前形成的基本判断是：

1. Aida 不应该只是 Token 看板。Token、Session、模型、活跃人数只是辅助指标和工作证据。
2. 第一阶段不应继续堆功能，而应收敛到“角色化首页工作台”。
3. 首页的核心问题不是“展示多少数据”，而是“让不同角色知道现在应该处理什么”。
4. 重点关注是一种管理信号，不是收藏，也不是正式优先级。
5. 只有“高关注 + 异常状态”才值得优先曝光或触发提醒。
6. P0 应先验证工作台、关注、风险、待办和动态是否能组成有效的角色入口。

---

## 2. 产品定位

Aida 在 P0 阶段的定位建议是：

> 面向 AI 研发组织的角色化工作台，帮助员工、TL、PM 和总监基于任务、风险、关注和工作证据，快速判断当前最需要处理的事项。

这个定位刻意弱化“数据大屏”和“Token 统计平台”的表达。

Aida 的核心价值不是统计谁用了多少 Token，而是把 AI 编码过程中的 Session、日报、任务状态和风险信号组织起来，服务管理和协作判断：

```text
项目 / 需求 / 任务
  -> 工作进展
  -> Session / 日报 / 动态
  -> 风险与阻塞
  -> 重点关注
  -> 角色化首页曝光
  -> 相关角色处理
```

P0 不追求完整项目管理闭环，也不替代现有项目管理工具。它只验证一个更小的问题：

> 不同角色进入首页后，是否能一眼知道自己现在应该看什么、点什么、处理什么。

---

## 3. P0 验证目标

P0 原型重点验证以下问题：

1. 四个角色首页差异是否明显；
2. 首页是否能回答“我现在应该处理什么”；
3. 重点关注事项是否足够显眼；
4. 高关注且有风险的事项是否能进入首屏；
5. Token / Session 是否被正确弱化为辅助信息和工作证据；
6. 今日待办、风险阻塞、最近动态、AI 摘要是否能形成清晰行动路径；
7. 固定角色模板 + 动态内容排序是否足够支撑第一阶段，而不需要自由配置 Dashboard。

P0 的结果应服务原型评审和产品判断，不直接作为开发排期。

## 4. 核心产品决策

### 4.1 P0 首页定位为角色化工作台

首页不做 Token 数据大屏，也不做传统 BI Dashboard。首页的主叙事是待办、关注、风险、动态和摘要。

Token 和 Session 只作为辅助信息，帮助解释工作量、工作证据和异常判断。

### 4.2 P0 采用固定角色模板 + 动态内容排序

员工、TL、PM、总监使用不同首页模板。模板结构相对固定，但模块内内容根据角色、权限、关注关系、风险状态、deadline 和最近动态排序。

P0 不做自由拖拽配置、自定义 Widget、自定义指标公式。

### 4.3 P0 支持重点关注机制

重点关注用于表达管理关注和协作关注。它会影响首页曝光、列表排序和风险提示，但不自动改变正式业务优先级。

关注度高只说明“这件事需要被看见”，不说明“这件事自动插队”。

### 4.4 P0 重点验证“高关注 + 风险事项”

P0 的关注机制不应停留在“谁点了关注”。更重要的是验证：当高关注事项出现异常状态时，平台能否把它推到相关角色首页。

异常状态包括长时间无进展、临近 deadline、阻塞、缺负责人、缺 AC、缺任务拆解等。

缺少 Session / 工作记录不天然等于风险。它只是工作证据不足的信号，只有与高关注、临近 deadline、长时间无进展、阻塞等条件叠加时，才作为异常信号进入首页优先曝光。

### 4.5 P0 不做完整项目管理系统

P0 可以展示项目、需求、任务相关信息，但不应扩展为完整需求管理、任务管理、排期、甘特图或资源计划系统。

原型只需要支撑首页场景跳转和必要的状态理解。

### 4.6 P0 允许最小点击承接

P0 首页点击可以进入最小详情页或右侧抽屉，用来承接首页事项的上下文查看和轻量处理。

这些承接页只服务首页原型验证，例如查看关注原因、风险说明、最近动态、关联任务、Session 摘要、日报状态等。不做完整需求管理、任务管理、日报管理闭环。

### 4.7 P0 不做外部自动化闭环

P0 不做真实飞书推送、数字分身、自动派单、自动优先级调整、自动排期。

这些能力需要等真实试点证明工作台价值后再讨论。

### 4.8 P0 不把 Session、日报和文档作为重点关注对象

P0 的重点关注对象限定为项目 / 产品、需求、任务。

Session、日报、文档仍作为工作证据和动态来源展示，但不作为重点关注主对象。

## 5. 四类角色使用场景

### 5.1 员工

场景一：任务被上级关注但缺少工作证据

1. 进入首页前：员工昨天完成了一部分任务，但没有绑定 Session，也没有确认日报；该任务被 TL 重点关注。
2. 进入首页后第一眼看到：今日待办中出现“1 个被 TL 关注的任务缺少 Session 支撑”，并在重点关注模块中标记“上级关注”。
3. 用户点击：点击该待办进入任务工作记录区域，选择当天相关 Session 并绑定。
4. 处理后状态变化：任务从“缺少工作证据”变为“已有 Session 支撑”；该异常提示从员工首页待办中消失，TL 首页对应事项的证据不足提示降低。

场景二：下游任务等待自己的输出

1. 进入首页前：员工负责的任务是另一个跨团队任务的前置依赖，deadline 临近。
2. 进入首页后第一眼看到：风险与阻塞模块提示“你的任务影响 2 个下游任务”，并显示其中 1 个为高关注需求。
3. 用户点击：点击风险项进入任务详情，更新任务状态或补充阻塞说明。
4. 处理后状态变化：平台记录状态更新；下游相关角色在最近动态中看到该任务已有新进展或阻塞说明。

场景三：日报待确认且 AI 摘要提示异常

1. 进入首页前：系统根据 Session 和任务状态生成了个人日报草稿，但其中有一项任务进展为空。
2. 进入首页后第一眼看到：今日待办提示“确认今日日报”，AI 工作摘要提示“1 个进行中任务今日无记录”。
3. 用户点击：点击日报待办，确认已有内容，并对无记录任务补充说明。
4. 处理后状态变化：日报状态变为已确认；无记录任务不再作为当日异常提示。

### 5.2 TL

场景一：团队高关注任务阻塞超过阈值

1. 进入首页前：团队成员标记某任务阻塞已超过 1 天，该任务属于 PM 关注的需求。
2. 进入首页后第一眼看到：首屏风险与阻塞模块显示“高关注需求下有 1 个阻塞任务”，并标注负责人和阻塞时长。
3. 用户点击：点击阻塞任务进入任务详情，查看阻塞说明和上下游依赖。
4. 处理后状态变化：TL 更新处理意见或调整负责人；该事项进入最近动态，PM 首页的风险状态同步更新。

场景二：团队日报需要 Review

1. 进入首页前：多名员工提交了日报，其中两人日报缺少任务关联。
2. 进入首页后第一眼看到：今日待办显示“2 份日报待 Review，其中 2 份缺任务关联”。
3. 用户点击：点击待 Review 项进入日报 Review 列表。
4. 处理后状态变化：通过的日报变为已 Review；缺任务关联的日报被打回或要求补充，员工首页出现对应待办。

场景三：任务进行中但没有 Session 支撑

1. 进入首页前：某成员负责的任务连续两天处于进行中，但没有关联 Session 或工作记录。
2. 进入首页后第一眼看到：风险 / 异常模块提示“3 个进行中任务缺少工作证据”，其中 1 个被总监关注。
3. 用户点击：点击该任务查看负责人、deadline、关联需求和最近动态。
4. 处理后状态变化：TL 可提醒负责人补充记录，或更新任务说明；若该任务同时存在高关注、临近 deadline 或长时间无进展，才继续作为风险事项保留在首页。

### 5.3 PM

场景一：高关注需求缺少 AC

1. 进入首页前：总监关注了一个新需求，但该需求还没有补充 AC / 验收标准。
2. 进入首页后第一眼看到：重点关注模块顶部展示该需求，并标记“总监关注 + 缺 AC”。
3. 用户点击：点击需求进入需求详情或 AC 编辑入口。
4. 处理后状态变化：补齐 AC 后，该需求从“高关注且缺 AC”风险中移除，但仍保留重点关注状态。

场景二：跨团队依赖长时间未处理

1. 进入首页前：一个需求涉及两个团队，下游团队等待上游任务输出超过 1 天。
2. 进入首页后第一眼看到：风险与阻塞模块显示“跨团队依赖未处理”，并出现在高关注风险列表中。
3. 用户点击：点击该阻塞进入依赖详情，查看双方 TL、任务状态和最近说明。
4. 处理后状态变化：PM 添加协调结论或指定下一步负责人；总监首页可看到该风险已有 PM 介入。

场景三：需求有工作记录但进度无变化

1. 进入首页前：某需求本周 Session 数较多，但 AC 和任务状态没有推进。
2. 进入首页后第一眼看到：AI 工作摘要提示“高工作量但低进展需求”，AI 使用小组件只作为辅助展示。
3. 用户点击：点击摘要中的需求，查看任务状态、Session 支撑和最近动态。
4. 处理后状态变化：PM 判断是否需要调整范围、补充 AC 或要求 TL 更新任务拆解；平台记录处理动态。

### 5.4 总监

场景一：重点项目出现重大风险

1. 进入首页前：总监关注的重点项目中，一个核心需求临近 deadline 且存在跨团队阻塞。
2. 进入首页后第一眼看到：首屏重点关注模块显示“高关注 + deadline 风险 + 跨团队阻塞”，并标注 PM 是否已介入。
3. 用户点击：点击该事项进入项目或需求风险视图。
4. 处理后状态变化：总监可标记“需要 PM 跟进”或记录决策意见；PM 首页出现对应待办或高优先级风险项。

场景二：部门多个高关注事项无进展

1. 进入首页前：多个被 PM / TL 关注的需求超过 24 小时无状态变化。
2. 进入首页后第一眼看到：AI 工作摘要概括“3 个高关注事项无进展”，风险模块列出影响范围。
3. 用户点击：点击摘要进入高关注无进展列表。
4. 处理后状态变化：总监可以查看负责人、PM、TL 和最近动态，并决定是否要求 PM 介入；状态变化进入最近动态。

场景三：查看部门 AI 工作情况但不以 Token 为中心

1. 进入首页前：总监想了解部门本周 AI 工作活跃度，但不希望只看 Token 消耗。
2. 进入首页后第一眼看到：首页主区域仍是重点项目、风险和待决策事项；AI 使用小组件显示 Session 数、活跃人数、Token 趋势。
3. 用户点击：点击 AI 使用小组件查看辅助维度。
4. 处理后状态变化：不会改变业务状态；该行为只帮助总监理解工作证据背景。

## 6. 首页模块设计依据

首页模块不应从功能清单出发，而应从上面的角色场景反推。

| 模块 | 解决的问题 | 主要服务角色 | 点击后进入 |
| --- | --- | --- | --- |
| 今日待办 | 告诉用户现在要处理的动作，例如确认日报、绑定 Session、Review 日报、补 AC、处理阻塞 | 员工、TL、PM、总监 | 对应处理页或详情页 |
| 重点关注 | 暴露被自己、上级、PM、总监关注的项目 / 需求 / 任务 | 全角色 | 项目 / 需求 / 任务详情 |
| 风险 / 阻塞 | 把异常事项集中展示，尤其是高关注且有风险的事项 | TL、PM、总监，部分员工 | 风险详情、任务详情、依赖详情 |
| AI 工作摘要 | 用自然语言概括当前角色权限范围内的关键变化和异常 | 全角色 | 摘要对应的事项列表或详情 |
| 最近动态 | 帮助用户理解事项为什么变化、谁处理过、最近发生了什么 | 全角色 | 动态关联对象详情 |
| AI 使用小组件 | 提供 Session、Token、活跃人数等背景数据，但不作为首页主视觉 | 全角色 | AI 使用辅助视图 |
| 筛选与折叠 | 支持时间范围、关注范围、风险范围和模块折叠，避免首页过载 | 全角色 | 留在首页刷新内容 |

原型评审时应重点观察：

1. 四个角色首页的第一屏是否明显不同；
2. 高关注 + 风险事项是否能进入首屏；
3. 用户是否能从每个模块看出下一步动作；
4. Token 是否只是辅助，不抢占主叙事；
5. 点击路径是否能从首页自然进入处理动作。

## 7. P0 原型范围

P0 原型只验证以下内容：

1. 角色化首页；
2. 重点关注模块；
3. 风险 / 阻塞模块；
4. 今日待办；
5. 最近动态；
6. AI 工作摘要；
7. AI 使用小组件；
8. 关注筛选：全部 / 我关注 / 高关注 / 高关注且有风险；
9. 时间范围筛选：今日 / 本周 / 本月；
10. 模块折叠 / 展开。

P0 原型允许做最小详情页或右侧抽屉承接首页点击，但只用于展示上下文和轻量状态处理，不扩展成完整需求管理、任务管理或日报管理闭环。

P0 原型可以使用 Mock 数据或静态规则，不要求真实 AI 生成、真实飞书推送或完整后端闭环。

P0 的评审输出应是产品判断：

- 角色化首页方向是否成立；
- 重点关注机制是否有管理价值；
- 高关注 + 风险的排序逻辑是否清晰；
- 首页是否足够轻，不像复杂项目管理系统；
- 下一步是否值得进入正式 PRD 讨论。

## 8. P0 原型评审标准

P0 首页原型是否成立，建议用以下标准判断：

1. 角色差异清晰：员工、TL、PM、总监的首页首屏内容明显不同，不是同一套指标换权限。
2. 下一步动作明确：每个角色进入首页后，都能在 5 秒内判断最需要处理的 1 到 3 件事。
3. 高关注 + 异常优先：高关注且存在风险或异常信号的事项能稳定进入首屏。
4. 关注不混同优先级：页面表达能让用户理解“关注是曝光和提醒信号，不是业务优先级调整”。
5. Token 被弱化：Token / Session 出现在辅助信息、证据说明或小组件中，不抢占首页主叙事。
6. 点击承接轻量：点击首页事项后能看到必要上下文和处理入口，但不会进入完整管理系统。
7. 风险口径克制：缺少工作证据不会被单独渲染成风险，必须结合高关注、deadline、长时间无进展或阻塞等条件。
8. 原型边界清楚：评审者不会把 P0 理解为要同时建设完整需求、任务、日报、飞书、移动端或自动派单能力。
9. Mock 数据可信：示例数据能覆盖真实角色、真实项目、真实风险和真实关注关系。
10. 是否值得进入 PRD：评审后能明确判断是否基于该方向进入正式 PRD 讨论。

## 9. 明确不做

P0 明确不做：

1. 不做 Token 数据大屏；
2. 不做自由拖拽 Dashboard；
3. 不做自定义首页 Widget；
4. 不做自定义指标公式；
5. 不做甘特图；
6. 不做完整项目管理系统；
7. 不做完整需求管理；
8. 不做完整任务系统；
9. 不做完整日报系统；
10. 不做真实飞书集成；
11. 不做真实消息推送；
12. 不做移动端；
13. 不做数字分身；
14. 不做自动派单；
15. 不做自动优先级调整；
16. 不做自动排期；
17. 不把 Session、日报、文档作为重点关注对象；
18. 不做复杂关注算法；
19. 不做自定义关注权重；
20. 不把关注度等同于正式业务优先级。

2. 总监首页是否需要“待决策事项”作为独立表达？
3. 员工首页是否应强调“我的待办”，弱化团队和管理视角？

### 10.3 重点关注

1. P0 是否允许员工关注需求，还是只允许关注自己的任务？
2. 关注是否需要填写原因，还是先一键关注？
3. 取消关注后是否保留历史记录用于审计？
4. 关注人的角色是否应在详情页显式展示？

### 10.4 风险与提醒

1. 多久无进展才算异常：24 小时、1 个工作日，还是按 deadline 动态判断？
2. “缺少 Session / 工作记录”与哪些条件叠加时才进入风险或异常曝光？
3. 高关注且异常时，是只做首页曝光，还是需要站内提醒？
4. 风险关闭由系统自动判断，还是由负责人 / TL / PM 手动确认？

### 10.5 工作证据与隐私

1. Session 上传是否坚持员工主动勾选？
2. TL、PM、总监分别能看到 Session 的哪些信息？
3. Token 是否允许按个人展示？如果允许，展示到什么粒度？
4. 如何在产品文案和权限设计上避免绩效监控感？

### 10.6 原型评审

1. P0 原型是否只做首页，还是需要最小化详情页承接点击？
2. Mock 数据是否需要覆盖真实团队、真实项目、真实风险样例？
3. 原型评审通过后，是否进入正式 PRD 讨论？
