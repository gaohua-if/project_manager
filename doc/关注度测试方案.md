# 关注度 P0 第一阶段接口测试方案

> 本方案用于后续执行「关注 / 自动关注 / 关注度 P0 第一阶段」完整接口测试。当前步骤只梳理测试方案，不执行测试，不修改业务代码。

## 1. 测试范围与原则

### 1.1 测试目标

验证以下产品口径是否成立：

1. 创建需求后，创建人自动关注需求。
2. 创建任务后，任务创建人自动关注任务。
3. 创建任务时有 `assignee_id`，被指派人自动关注任务。
4. 任务改派时，新 assignee 自动关注任务。
5. 任务普通编辑不恢复用户已取消的关注。
6. 改派不自动删除老 assignee 的关注。
7. 被指派也算关注，并计入关注度。
8. `admin` 权重为 0，不影响关注度。
9. 关注度按当前用户 role 实时计算。
10. `dashboard/follows` 返回 `attentionScore`、`attentionLevel`、`riskPriority`，并按规则排序。
11. `dashboard/risks` 返回 `attentionScore`、`attentionLevel`，不返回 `followCount`。
12. `dashboard/risks` 的任务风险关注度使用 `max(task_attention_score, requirement_attention_score)`，不是两者相加。
13. 前端关注列表排序不再依赖风险文案字符串推断；如仅测接口，也要检查前端代码是否残留旧排序逻辑。

### 1.2 测试约束

- 只做测试，不改业务代码。
- 使用当前服务已有账号测试；不要新建测试用户，除非当前环境完全没有可用账号。
- 测试数据不要删除，保留给人工 review。
- 所有测试数据名称必须带唯一前缀。
- 脚本只能调用接口，不要直接写数据库。
- 允许只读查询数据库辅助验证 `user_follows` 与关注度，不允许数据库写入。
- 不要清理需求、任务、`user_follows` 或数据库。

### 1.3 建议测试前缀

执行测试时生成唯一前缀：

```text
ATTN_E2E_YYYYMMDD_HHMM
```

示例：

```text
ATTN_E2E_20260626_1630
ATTN_E2E_20260626_1630_需求关注度测试
```

后续所有需求 / 任务标题统一以该前缀开头。

### 2.5 请求字段与枚举

#### 需求创建字段

```json
{
  "title": "ATTN_E2E_xxxx_REQ_PM_CREATE",
  "description": "关注度 P0 接口测试需求",
  "priority": "medium",
  "team_ids": ["<AI工程 team id>"],
  "deadline": "2026-07-10",
  "acceptance_criteria": ["验证关注度规则"]
}
```

需求优先级枚举：

```text
low / medium / high / urgent
```

需求状态枚举：

```text
todo / review / active / completed / cancelled
```

#### 任务创建字段

```json
{
  "requirement_id": "<requirementId>",
  "title": "ATTN_E2E_xxxx_TASK_TL_ASSIGN_A",
  "assignee_id": "<employeeA.id>",
  "priority": "medium",
  "due_date": "2026-07-01",
  "acceptance_criteria": ["验证任务自动关注"]
}
```

任务优先级枚举：

```text
low / medium / high
```

任务存储状态枚举：

```text
todo / in_progress / done
```

注意：`blocked` 不是可写入的任务状态，`PUT /tasks/{id}` 和 `PUT /tasks/{id}/status` 都不接受 `blocked`。`blocked` 是由未完成依赖派生出的 `display_status` / `risk_types`。

### 2.6 风险制造方式

当前风险由后端实时派生：

1. **超期风险**：任务 `status != done`，且 `due_date` 早于当前 UTC 日期，派生 `risk_types` 包含 `overdue`。
2. **阻塞风险**：任务依赖的上游任务状态不是 `done`，派生 `risk_types` 包含 `blocked`，展示状态为 `display_status=blocked`。

推荐测试优先使用超期风险，造数最稳定：

```json
{
  "due_date": "<昨天 yyyy-mm-dd>",
  "status": "todo"
}
```

如需制造阻塞风险：先创建一个未完成上游任务，再创建或更新目标任务依赖该上游任务。

### 2.7 关注度权重与等级

权重：

| role | 权重 |
| --- | ---: |
| `director` | 100 |
| `team_leader` | 50 |
| `pm` | 40 |
| `employee` | 10 |
| `admin` | 0 |

等级：

| 分数区间 | attentionLevel |
| --- | --- |
| 0-39 | `normal` |

    "requirementId": "...",

排序规则应为：

1. `riskPriority DESC`
2. `attentionScore DESC`
3. `dueDate ASC NULLS LAST`，接口中展示字段为 `deadline`，后端内部用真实 `SortDueDate`
4. `updatedAt DESC`，接口中不直接暴露，后端内部用真实 `SortUpdatedAt`

说明：`dashboard/follows` 本身就是“我的关注列表”，当前没有 `isFollowedByMe` 字段；如需等价验证，可用该 item 是否出现在当前用户 `dashboard/follows` / `GET /follows` 中判断。

`DashboardRiskItem` 不返回 `followCount`，也不返回 `riskLevelPriority`。排序验证时可按代码规则从 `riskType` 推导：

| riskType | 内部 riskLevelPriority |
| --- | ---: |

排序规则应为：

1. `riskLevelPriority DESC`
2. `attentionScore DESC`
3. `dueDate ASC NULLS LAST`

### 2.9 前端排序检查点

当前代码未发现名为 `sortFollowItems` 的函数；Dashboard 中关注列表使用接口返回的 `followsQuery.data` 顺序。需要在报告中记录：

```sql

FROM user_follows f
JOIN users u ON u.id = f.user_id
WHERE f.target_id IN ('<requirementId>', '<taskId>')
ORDER BY f.target_type, f.target_id, u.role, u.name;
```

关注度只读核对 SQL：

```sql
SELECT COALESCE(SUM(CASE u.role
  WHEN 'director' THEN 100
  WHEN 'team_leader' THEN 50
  WHEN 'pm' THEN 40
  WHEN 'employee' THEN 10
  ELSE 0
END), 0) AS attention_score
FROM user_follows f
JOIN users u ON u.id = f.user_id
WHERE f.target_type = '<requirement|task>' AND f.target_id = '<id>';
```

## 4. 测试用例

- dashboard/follows 中该需求 item JSON

### Case 3：admin 关注需求，但不影响关注度

操作：

1. 如果 admin 存在，admin 关注 Case 1 需求。
2. PM 或 director 请求 `GET /api/v1/dashboard/follows`，找到该需求。

预期：

- admin 可以成功关注。
- `attentionScore` 仍为 `140`。
- `attentionLevel` 仍为 `important`。
- admin 不贡献业务关注权重。

如果 admin 不存在：

- 该 Case 标记为 `SKIPPED`，不算失败。

记录：

- admin follow 响应
- dashboard/follows 中该需求 item JSON

### Case 6：employeeA 取消关注后，普通编辑不能恢复关注

操作：

1. employeeA 取消关注 Case 5 任务：

```http
DELETE /api/v1/follows/task/<taskId>
```

2. employeeA 请求 `GET /api/v1/dashboard/follows`，确认该任务消失。
3. TL 对该任务做普通编辑，不改 `assignee_id`：

```http
PUT /api/v1/tasks/<taskId>

{
  "title": "ATTN_E2E_xxxx_TASK_TL_ASSIGN_A_EDITED",
  "progress": 20
}
```

4. employeeA 再次请求 `GET /api/v1/dashboard/follows`。
5. TL 请求 `GET /api/v1/dashboard/follows`，查看该任务关注度。

预期：

- employeeA 取消关注后，该任务不再出现在 employeeA `dashboard/follows`。
- TL 普通编辑后，该任务仍然不出现在 employeeA `dashboard/follows`。
- 任务 `attentionScore` 从 60 降为 50。
- `attentionLevel` 仍为 `notable`。
- 普通 `PUT /tasks/{id}` 不恢复用户已取消的关注。

记录：

- unfollow 响应
- 普通编辑响应
- employeeA 两次 follows 查询结果
- TL item JSON

### Case 7：任务改派给 employeeB，新 assignee 自动关注

操作：

1. TL 将 Case 5 任务从 employeeA 改派给 employeeB：

```http
PUT /api/v1/tasks/<taskId>

{
  "assignee_id": "<employeeB.id>"
}
```

2. employeeB 请求 `GET /api/v1/dashboard/follows`。
3. employeeA 请求 `GET /api/v1/dashboard/follows`。
4. TL 请求 `GET /api/v1/dashboard/follows`。

预期：

- employeeB 自动关注该任务。
- employeeB `dashboard/follows` 出现该任务。
- employeeA 因之前手动取消过关注，仍不出现该任务。
- TL 仍然关注该任务。
- `attentionScore = 50 + 10 = 60`。
- `attentionLevel = notable`。

记录：

- 改派响应
- employeeB / employeeA / TL 的 dashboard/follows 结果

### Case 8：再次改派回 employeeA，employeeA 应重新自动关注

操作：

1. TL 将任务从 employeeB 改派回 employeeA：

```http
PUT /api/v1/tasks/<taskId>

{
  "assignee_id": "<employeeA.id>"
}
```

2. employeeA 请求 `GET /api/v1/dashboard/follows`。
3. employeeB 请求 `GET /api/v1/dashboard/follows`。
4. TL 请求 `GET /api/v1/dashboard/follows`。

预期：

- employeeA 因再次被明确指派，重新自动关注。
- employeeB 的关注不被系统自动删除。
- TL 仍然关注。
- `attentionScore = 50 + 10 + 10 = 70`。
- `attentionLevel = notable`。

该用例验证：

- 改派不删除老 assignee 关注。
- 再次指派可以重新自动关注。

记录：

- 改派响应
- employeeA / employeeB / TL 的 follows item JSON
- 可选只读 `user_follows` 查询结果

### Case 9：制造任务风险，验证 dashboard/risks 使用 max 关注度

操作：

1. 在 Case 1 高关注需求下，TL 创建风险任务：

```http
POST /api/v1/tasks
```

请求体摘要：

```json
{
  "requirement_id": "<requirementId>",
  "title": "ATTN_E2E_xxxx_TASK_RISK_HIGH_REQ",
  "assignee_id": "<employeeA.id>",
  "priority": "medium",
  "due_date": "<昨天 yyyy-mm-dd>",
  "acceptance_criteria": ["验证风险关注度 max 规则"]
}
```

2. 如创建时 due_date 未生效，可 TL 更新任务：

```http
PUT /api/v1/tasks/<riskTaskId>

{
  "due_date": "<昨天 yyyy-mm-dd>",
  "status": "todo"
}
```

3. 请求 `GET /api/v1/tasks/<riskTaskId>`，确认：

- `risk_types` 包含 `overdue`
- `display_status` 不是 `done`

4. 请求 `GET /api/v1/dashboard/risks`。
5. 找到该风险任务对应的风险项。
6. 分别通过 `dashboard/follows` 或只读 SQL 记录父需求关注度和任务自身关注度。

预期：

- 风险项出现在 `dashboard/risks`。
- 风险项返回 `attentionScore` 和 `attentionLevel`。
- 风险项不返回 `followCount`。
- 如果父需求 `attentionScore = 190`，任务自身 `attentionScore = 60`，则风险项：
  - `attentionScore = 190`
  - `attentionLevel = high`
- 不允许返回 `250`，即不能把任务关注度和需求关注度简单相加。

记录：

- `riskTaskId`
- `requirementAttentionScore`
- `taskAttentionScore`
- `riskAttentionScore`
- 是否符合 `max(task, requirement)` 规则
- dashboard/risks item JSON

### Case 10：创建一个低关注风险任务，验证风险排序

操作：

1. 使用 PM 创建一个低关注需求：

```http
POST /api/v1/requirements
```

标题示例：

```text
ATTN_E2E_xxxx_REQ_LOW_RISK_SORT
```

2. 使用 PM 或 TL 在该低关注需求下创建风险任务。
3. 让该任务 `due_date` 为昨天，且 `status != done`。
4. 确保该低关注风险项 `attentionScore` 低于 Case 9 风险项。
5. 请求 `GET /api/v1/dashboard/risks`。

预期排序：

1. `riskLevelPriority` 更高的风险优先。接口不返回该字段，按 `riskType` 推导：`deadline=100`、`dependency_blocker=90`。
2. `riskLevelPriority` 相同时，`attentionScore` 更高的风险优先。
3. 排序方向必须是 `riskLevelPriority DESC`，不是 ASC。

推荐造数方式：

- Case 9 和 Case 10 都制造超期风险，使两者 `riskType=deadline`，内部 `riskLevelPriority=100` 相同。
- 这样排序主要验证同风险级别下 `attentionScore DESC`。

记录：

- 低关注 `requirementId`
- 低关注 `taskId`
- dashboard/risks 返回顺序中本轮 `ATTN_E2E` 项的位置
- 每项 `riskType`、推导的 `riskLevelPriority`、`attentionScore`、`attentionLevel`

### Case 11：dashboard/follows 排序验证

操作：

1. 使用 PM、TL、employeeA 分别请求：

```http
GET /api/v1/dashboard/follows
```

2. 找到本轮创建的多个 `ATTN_E2E` 项。
3. 检查返回顺序。
4. 检查前端代码是否还存在旧的文本排序逻辑。

预期：

`dashboard/follows` 排序符合：

1. `riskPriority DESC`
2. `attentionScore DESC`
3. `dueDate ASC NULLS LAST`
4. `updatedAt DESC`

接口返回字段中：

- `riskPriority` 直接返回。
- `attentionScore` 直接返回。
- due date 展示为 `deadline`，后端内部使用真实日期排序。
- `updatedAt` 不直接返回在 dashboard item 中，若要严格验证兜底排序，可通过创建 / 编辑时间和返回顺序间接核对，或只读 DB 辅助核对。

前端检查预期：

- 不存在 `sortFollowItems` 或类似基于 `risk` 文案字符串的排序函数。
- 如存在前端排序，只能使用结构化字段。
- 若仅有 `getFollowTone()` 基于风险文案决定卡片颜色，不判定为接口排序失败，但应在报告中作为前端展示观察项记录。

记录：

- PM / TL / employeeA 各自 follows 返回顺序
- 每项 `key`、`type`、`title`、`riskPriority`、`attentionScore`、`attentionLevel`、`deadline`
- 前端排序检查结果
