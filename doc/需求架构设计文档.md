# 需求架构设计文档

> 状态：P0 已落地实施口径
> 基线日期：2026-06-24
> 适用范围：`requirements` 主数据域 + `dashboard` 投影域
> 依据：`doc/需求看板_P0_需求文档.md`、`doc/需求看板_P0_架构文档.md`，以及当前仓库 `api/`、`web/` 实际代码

## 1. 总则

1. `requirements` 是需求、任务、依赖、进度、关注、风险派生的唯一主数据域。
2. `dashboard` 是当前用户视角的 projection，**不是独立业务域**：不复制主数据，不维护独立关注或风险表，不在前端重新计算最终状态。
3. Requirements 与 Dashboard 共享同一份事实数据 (`requirements` / `tasks` / `task_dependencies` / `user_follows`) 与统一的派生服务 (`api/service/requirements.go`)。
4. 风险（`blocked` / `overdue` / `due_soon`）只派生不落库；blocked 不是任务的存储状态。
5. Dashboard 只提供跳转入口；任务/需求的写操作必须经过 Requirements/Task 命令接口。

## 2. 分层结构

| 层 | 代码位置 | 职责 |
|---|---|---|
| DB Model | `api/db/migrations/*.sql` | 持久化 `requirements`、`tasks`、`task_dependencies`、`user_follows`、`requirement_teams`、`sessions`、`token_usage` |
| Domain Service | `api/service/requirements.go` | `DeriveTaskRisks`、`DisplayTaskStatus`、`AggregateRequirementProgress`、`SummarizeRequirementTasks` |
| HTTP Handler | `api/handler/requirement.go`、`api/handler/task.go`、`api/handler/follow.go`、`api/handler/dashboard.go` | 鉴权、参数校验、SQL 编排、调用 Domain 派生 |
| API DTO | `api/model/models.go` | 对前端的稳定字段契约 |
| Web 类型 | `web/src/features/aidashboard/api/types.ts` | 与后端 DTO 同名同字段 |
| Web 客户端 | `web/src/features/aidashboard/api/client.ts` | 调用 `/api/v1/...`，所有写操作不绕过后端规则 |
| Frontend ViewModel | `web/src/features/aidashboard/requirements/*`、`.../dashboard/*` | 文案、排序、URL 跳转；不计算最终风险或聚合 |

## 3. 数据库模型（P0 落地版）

迁移文件：`api/db/migrations/007_requirements_p0.sql`。

### 3.1 `requirements` 表

- `status` CHECK：`todo | review | active | completed | cancelled`，默认 `todo`。
- 新增字段 `completed_at TIMESTAMPTZ`。`status = 'completed'` 的历史行回填 `updated_at`。
- `progress` 由所有任务 `progress` 的 `FLOOR(AVG(progress))` 聚合产生（参见 `TaskHandler.updateRequirementProgress`），前端、Dashboard 都不允许直接写入。

### 3.2 `tasks` 表

- 新增 `progress INTEGER NOT NULL DEFAULT 0 CHECK (0..100)`。
- 新增 `completed_at TIMESTAMPTZ`，`status = done` 时回填。
- 历史 `blocked` 状态在迁移中改写为 `in_progress`。
- `status` CHECK 收紧为 `todo | in_progress | done`；**blocked 不再是存储状态**，由 Domain 派生。
- 新增部分索引：`idx_tasks_due_date ON tasks (due_date) WHERE status <> 'done'`，加速风险投影。

### 3.3 `user_follows` 表（新增）

```sql
CREATE TABLE user_follows (
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    target_type TEXT NOT NULL CHECK (target_type IN ('requirement', 'task')),
    target_id   UUID NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, target_type, target_id)
);
CREATE INDEX idx_user_follows_target ON user_follows (target_type, target_id);
```

- 主键即幂等约束，重复关注不报错。
- 取消关注是物理 `DELETE`；`ON DELETE CASCADE` 跟随用户删除自动清理。
- 不存在 Dashboard 自有表（无 `dashboard_*`），不存在风险表。

### 3.4 复用且未变更的相关表

`requirement_teams`（多对多）、`task_dependencies`（同需求内上游）、`sessions`（含 `requirement_id` / `task_id`）、`token_usage`、`daily_reports`、`team_reports`、`users` / `teams`。

## 4. Domain：风险与聚合（`api/service/requirements.go`）

### 4.1 `DeriveTaskRisks(task, now) []string`

- `task.Status == "done"` → 返回空数组。
- 任一 dependency `status != "done"` → 追加 `blocked`（仅追加一次）。
- 若 `due_date` 存在且能解析（`YYYY-MM-DD` 或 RFC3339）：
  - `due < today(UTC)` → `overdue`
  - `due <= today + 48h` → `due_soon`
- overdue 与 due_soon 互斥，blocked 可与日期类风险共存。

### 4.2 `DisplayTaskStatus(task) string`

- `done` → `done`
- 含 `blocked` 风险 → `blocked`
- 否则 → 原始 `status`（`todo` / `in_progress`）

### 4.3 `AggregateRequirementProgress(tasks) int`

- 没有任务 → 0
- 否则取 `sum(progress) / len(tasks)` 的整数部分。
- 数据库侧 `TaskHandler.updateRequirementProgress` 用 `FLOOR(AVG(progress))::int` 等价实现，避免任务变更时反复跨表回查。

### 4.4 `SummarizeRequirementTasks(tasks)`

- `RequirementTaskSummary{ Total, Done, Blocked }`
- `RequirementRiskSummary{ Blocked, Overdue, DueSoon }`

> 单元测试覆盖：`api/service/requirements_test.go`。任何调整必须保持上述四个函数语义一致。

## 5. API 契约（已落地）

所有路由注册在 `api/main.go`，挂在 `/api/v1` 下，使用 `AuthMiddleware`。

### 5.1 Requirements

| Method | Path | Handler | 说明 |
|---|---|---|---|
| GET | `/requirements` | `RequirementHandler.List` | 支持 `status`、`team_id` 过滤；`team_leader` 自动按 `team_id` 收敛；按 `created_at DESC` 返回 |
| POST | `/requirements` | `RequirementHandler.Create` | 表单提交 `acceptance_criteria`；为空且 AI 可用时尝试 `claude -p` 生成；最终回退占位 |
| GET | `/requirements/{id}` | `RequirementHandler.Get` | 单条详情 |
| PUT | `/requirements/{id}` | `RequirementHandler.Update` | 支持 title/description/status/priority/deadline/feishu/AC；`status=completed` 同时落 `completed_at`，其它阶段清空 |
| GET | `/requirements/{id}/ac` | `RequirementHandler.GetAC` | AC + 关联任务完成度 |
| POST | `/requirements/{id}/regenerate-ac` | `RequirementHandler.RegenerateAC` | `director/pm/team_leader` 权限，调用 AI 重新生成 |

每条 Requirement 响应都经过 `loadTeams` + `loadProjection`：
- 关联团队从 `requirement_teams` 读取并写入 `team_ids` / `team_names`。
- 对所属任务遍历并 `enrichTask`（依赖、风险、display_status、token 来源、关注态），再调用 Domain：
  - `progress` ← `AggregateRequirementProgress`
  - `task_summary` / `risk_summary` ← `SummarizeRequirementTasks`
- `is_followed`：来自 `user_follows`。
- `token_source_ids`：聚合 `s.requirement_id = $1 AND s.task_id IS NULL` 且有 `token_usage` 的 session。

### 5.2 Tasks

| Method | Path | Handler | 说明 |
|---|---|---|---|
| GET | `/tasks` | `TaskHandler.List` | 支持 `requirement_id`、`assignee_id`、`status` 过滤；按角色收敛：`team_leader` 看本团队成员或自己创建的，`employee` 仅看分配给自己的（`scope=requirements` 时放宽） |
| POST | `/tasks` | `TaskHandler.Create` | 权限：`admin/director/pm/team_leader`；`team_leader` 仅能给本团队员工创建且需求需归属本团队；依赖必须同需求 |
| GET | `/tasks/{id}` | `TaskHandler.Get` | 单条详情 |
| PUT | `/tasks/{id}` | `TaskHandler.Update` | 拒绝 `status=blocked`；`progress` 0–100；`status=done` 自动 `progress=100, completed_at=now()`；其它状态清空 `completed_at` |
| PUT | `/tasks/{id}/status` | `TaskHandler.UpdateStatus` | 同上单字段语义 |
| PUT | `/tasks/{id}/progress` | `TaskHandler.UpdateProgress` | 单字段进度更新；不自动改 status |
| POST | `/tasks/{id}/dependencies` | `TaskHandler.AddDependency` | 校验同需求、非自依赖、无环（递归 CTE） |
| DELETE | `/tasks/{id}/dependencies/{dep_id}` | `TaskHandler.RemoveDependency` | 物理删除依赖 |

每次写操作后调用 `updateRequirementProgress` 同事务刷新需求 `progress`。

`enrichTask(task, userID)` 负责：
- 调用 `loadDeps` 填充 `dependencies` + `blocking`。
- `risk_types` = `DeriveTaskRisks(*task, time.Now())`；nil 归一化为 `[]`。
- `display_status` = `DisplayTaskStatus(*task)`。
- `token_source_ids`：本任务关联的有 token_usage 的 session 列表。
- `is_followed`：来自 `user_follows`。

### 5.3 Follows（新增）

| Method | Path | Handler | 说明 |
|---|---|---|---|
| GET | `/follows` | `FollowHandler.List` | 列出当前用户全部关注 |
| POST | `/follows` | `FollowHandler.Follow` | body `{target_type, target_id}`；`ON CONFLICT DO NOTHING` 实现幂等；写入前校验目标可见 |
| DELETE | `/follows/{target_type}/{target_id}` | `FollowHandler.Unfollow` | 物理删除 |

可见性 (`targetVisible`)：
- `requirement` + `team_leader` → 必须挂在本团队 `requirement_teams` 上。
- `task` + `team_leader` → 必须是本团队成员负责或自己作为 TL 创建。
- 其他角色（含 `admin` / `director` / `pm` / `employee`）默认放行存在性检查（`employee` 的可见性由 List/Get 上层 SQL 约束）。

### 5.4 Dashboard 投影（新增）

| Method | Path | Handler | 说明 |
|---|---|---|---|
| GET | `/dashboard/follows` | `DashboardHandler.Follows` | 把当前用户的 `user_follows` 投影为 `DashboardFollowItem[]` |
| GET | `/dashboard/risks` | `DashboardHandler.Risks` | 计算当前用户作用域内未完成任务的风险条目（Cartesian 展开为一条风险一行） |

#### 5.4.1 `/dashboard/follows` 投影规则

- 遍历当前用户的关注记录。
- `requirement`：跳过 `completed` / `cancelled`；调用 `loadProjection` 取得 `RiskSummary` 后通过 `requirementRiskLabel` 输出中文标签。
- `task`：跳过 `done`；通过 `enrichTask` 得到 `display_status` 与 `risk_types`，并把未完成依赖名拼接为 `dependency` 字段。
- 输出对象统一带 `navigation`：
  - 需求：`/requirements?requirementId={id}`
  - 任务：`/requirements?requirementId={rid}&taskId={tid}`
- 状态文案：`requirementStatusLabel` / `taskStatusLabel`（含 `blocked → 阻塞`）。

#### 5.4.2 `/dashboard/risks` 投影规则

SQL 范围：`tasks.status <> 'done' AND requirements.status NOT IN ('completed','cancelled')`。

按角色叠加：
- `employee`：负责自己 + 关注（task 或 requirement 任一）。
- `team_leader`（含 `team_id`）：assignee 在本团队、或自己作为 TL 创建、或关注本任务/所属需求。
- 其它角色（`admin` / `director` / `pm`）：全量。

排序：`due_date NULLS LAST, updated_at DESC`。

每条 task `enrichTask` 后，按 `risk_types` 展开（一种风险一行）：
- `blocked` → `dependency_blocker` / 高 / red，原因带未完成上游名称。
- `overdue` → `deadline` / 高 / red。
- `due_soon` → `deadline` / 中 / orange。

风险条目附带 `navigation`、`targetUrl`，前端只做跳转。

## 6. Web 端契约（与后端字段一一对应）

`web/src/features/aidashboard/api/types.ts`：

- `RequirementStatus`、`TaskStatus`（含 `blocked` 仅用于 `display_status`）。
- `StoredTaskStatus` 排除 `blocked`，用于写操作类型约束。
- `Requirement`：含 `task_summary`、`risk_summary`、`is_followed`、`token_source_ids`、`completed_at`。
- `Task`：含 `display_status`、`risk_types`、`progress`、`dependencies` / `blocking`、`is_followed`、`token_source_ids`、`completed_at`。
- 类型别名 `RequirementListItemDTO`、`RequirementDetailDTO`、`RequirementTaskDTO`、`TaskDependencyDTO` 与后端一致。
- `RequirementFollowStateDTO`、`DashboardNavigationTargetDTO`、`DashboardFollowItemDTO`、`DashboardRiskItemDTO`。

`web/src/features/aidashboard/api/client.ts` 暴露的关键调用：
- `updateTaskProgress(id, progress)`
- `fetchFollows()` / `followTarget(type, id)` / `unfollowTarget(type, id)`
- `fetchDashboardFollows()` / `fetchDashboardRisks()`

前端职责仅限于：
- 文案与排序（如 FollowCard、RiskCard 的排序权重）。
- 跳转：消费 `navigation` 中的 URL 或参数。
- TanStack Query 在写操作后失效 `requirements` / `tasks` / `follows` / `dashboard/*` query key。

前端**禁止**：直接写 `requirement.progress`、推导 `display_status`、根据 Session/Token 自动改任务进度、维护独立关注列表。

## 7. 关键业务规则

### 7.1 需求阶段

| 页面 | 存储 |
|---|---|
| 待开始 | `todo` |
| 评审 | `review` |
| 进行中 | `active` |
| 完成 | `completed`（落 `completed_at`） |
| 已取消（仅筛选） | `cancelled` |

主看板列默认展示 `todo/review/active/completed`；完成列只展示最近 7 天、最多 10 条由前端筛选实现；已取消不进入主看板列，仅筛选可见。

### 7.2 任务存储状态

- 存储仅 `todo | in_progress | done`。
- `blocked` 是 `display_status`，由 Domain 派生，不接受写入（API 返回 400：`invalid status; blocked is derived from dependencies`）。
- `status = done` 时强制 `progress = 100` 且写 `completed_at`；非 `done` 时清空 `completed_at`。
- `progress = 100` 不自动 `done`，仍可保留 in_progress。

### 7.3 依赖

- 仅同 `requirement_id` 内的上游。
- 拒绝自依赖；通过递归 CTE 拒绝环。
- 增删依赖立即影响下一次查询时派生的 `blocked` 与 `display_status`，无额外持久化。

### 7.4 风险

- 三种类型由 Domain 统一派生：`blocked`、`overdue`、`due_soon`（48 小时窗口）。
- 风险出现在以下三处，使用同一来源：
  - `Task.risk_types`
  - `Requirement.risk_summary`（按需求聚合）
  - `/dashboard/risks` 列表
- 没有人工 `blocked`，没有风险写接口，没有风险落库。

### 7.5 关注

- Requirements 列表/详情页是关注维护入口；Task 同理。
- Dashboard “我关注的事项”严格只用 `/dashboard/follows` 投影，不读取本地缓存。
- 关注完成或已取消的 Requirement 仍允许保留关注关系，但默认不进入 Dashboard 关注投影（`requirementFollowItem` 过滤 `completed`/`cancelled`；`taskFollowItem` 过滤 `done`）。

### 7.6 跳转契约

```
/requirements?requirementId={requirement_id}
/requirements?requirementId={requirement_id}&taskId={task_id}
```

Requirements 页负责解析参数并打开对应抽屉。Dashboard 卡片不携带完整业务对象，只携带 ID + URL。

## 8. 写路径与一致性

- Task 写操作（status/progress/dependency 变化）均在同一 handler 调用中触发 `updateRequirementProgress`，保证读到的 Requirement.progress 与任务一致。
- 风险/Display status：不在写路径落库，**读时派生**；删除依赖、上游完成、修改截止日期后下一次查询自然消失。
- Follow 写入幂等；前端在成功回调中并行失效 `requirements`、`tasks`、`follows`、`dashboard/follows`、`dashboard/risks` 查询。

## 9. 权限与角色作用域

- `admin` 与 `director` / `pm` 行为一致：全量。
- `team_leader`：按 `team_id` 收敛 Requirement 与 Task，可关注本团队范围内的任务/需求。
- `employee`：仅看分配给自己的任务；Requirement 列表通过团队收敛（无 team 时为空）。
- 角色判定由 handler 内 `u.Role` 分支显式编写在 SQL 中（参见 `RequirementHandler.List` / `TaskHandler.List` / `DashboardHandler.Risks`）。

## 10. 与日报 / Token / Session 的边界

- Dashboard 的“今日报告”、“Token 统计”、“Session 上传概览”继续复用现有 `/reports/*`、`/tokens`、`/sessions` 接口。
- LLM、Session、Token **不能**自动改写任务 `progress` 或 `status`；任何同步必须经 `PUT /tasks/{id}/progress` 或 `/status`，由人工确认触发。
- TokenSource 在本期实现为只读关联：Requirement 关联 `s.requirement_id=$1 AND s.task_id IS NULL` 的 session，Task 关联 `s.task_id=$1` 的 session，结果为 `token_source_ids`。

## 11. P0 明确不做（与代码现状一致）

- 手动 archive / unarchive，无 `archived_at` 列。
- 自动归档已完成需求。
- 独立关注中心、风险中心、通知中心。
- 评论、注释、审批。
- 跨需求依赖、多级任务树、人工阻塞字段。
- 周报 / 部门报告新接口、完整日报系统重构。
- 飞书发送、Skill 上传管理。
- Token 明细页与权限管理页重构。
- Dashboard 私有业务表。

## 12. 验收清单

1. Requirement / Task 列表与详情字段含 `task_summary`、`risk_summary`、`token_source_ids`、`is_followed`、`completed_at`。
2. 任意 Task 写操作后，所属 Requirement 的 `progress` 立即一致。
3. `PUT /tasks/{id}*` 拒绝 `status = blocked`。
4. 上游未完成 → 该任务 `risk_types` 含 `blocked`、`display_status = blocked`；上游完成或依赖删除后，下一次查询消失。
5. 未完成任务 `due_date < today` → `overdue`；`due_date ∈ (today, today+48h]` → `due_soon`；二者互斥。
6. `POST /follows` 幂等；`DELETE /follows/...` 物理删除；可见性按角色生效。
7. `/dashboard/follows` 排除 `completed`/`cancelled` 需求与 `done` 任务，并附带 `navigation`。
8. `/dashboard/risks` 仅返回未完成任务、且所属需求未关闭；按角色作用域过滤；按 `due_date NULLS LAST, updated_at DESC` 排序。
9. Dashboard 跳转 URL 与 Requirements 页面参数协议一致。
10. 前端不在浏览器中重算 `blocked` / `overdue` / `due_soon`，不直接写 `requirement.progress`。
