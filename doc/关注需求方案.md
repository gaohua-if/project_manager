# 关注 / 自动关注 / 关注度方案

## 1. 当前关注逻辑现状

### 1.1 `user_follows` 表结构

当前关注关系由 `user_follows` 统一承载，定义在 `api/db/migrations/007_requirements_p0.sql`：

```sql
CREATE TABLE IF NOT EXISTS user_follows (
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    target_type TEXT NOT NULL CHECK (target_type IN ('requirement', 'task')),
    target_id   UUID NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, target_type, target_id)
);

CREATE INDEX IF NOT EXISTS idx_user_follows_target
    ON user_follows (target_type, target_id);
```

字段能力：

- `user_id`：关注人，外键到 `users(id)`，用户删除时级联删除关注。
- `target_type`：关注对象类型，目前只允许 `requirement` / `task`。
- `target_id`：关注对象 ID，但没有数据库外键，因为同一字段复用两类对象。
- `created_at`：关注关系创建时间。
- 主键 `(user_id, target_type, target_id)`：天然保证同一用户对同一需求/任务只有一条关注记录。
- `idx_user_follows_target`：支持按对象反查关注人，后续计算关注人数 / 关注度可复用。

当前没有以下字段：

- 没有 `source` / `manual` / `auto_creator` / `auto_assignee`。
- 没有取消关注 tombstone。
- 没有关注人数、关注度、关注等级的冗余字段。

### 1.2 手动关注 / 取消关注接口

后端接口注册在 `api/main.go`：

- `GET /api/v1/follows` → `FollowHandler.List`
- `POST /api/v1/follows` → `FollowHandler.Follow`
- `DELETE /api/v1/follows/{target_type}/{target_id}` → `FollowHandler.Unfollow`

实现位置：`api/handler/follow.go`。

手动关注：

- 校验 `target_type` 只能是 `requirement` / `task`。
- 调用 `targetVisible` 判断当前用户是否能看到目标。
- 插入：

```sql
INSERT INTO user_follows (user_id, target_type, target_id)
VALUES ($1, $2, $3)
ON CONFLICT DO NOTHING
```

取消关注：

```sql
DELETE FROM user_follows
WHERE user_id = $1 AND target_type = $2 AND target_id = $3
```

前端调用在 `web/src/features/aidashboard/api/client.ts`：

- `fetchFollows()` → `GET /follows`
- `followTarget()` → `POST /follows`
- `unfollowTarget()` → `DELETE /follows/{targetType}/{targetId}`

需求看板里 `requirementsBoardApi.toggleFavorite()` 会先 `fetchFollows()`，判断当前是否已关注，再决定 follow / unfollow。

### 1.3 自动关注现状

当前代码已经有自动关注的雏形，函数在 `api/handler/follow.go`：

```go
func autoFollow(db *sql.DB, userID, targetType, targetID string) {
    if userID == "" || targetID == "" {
        return
    }
    _, _ = db.Exec(`
        INSERT INTO user_follows (user_id, target_type, target_id)
        VALUES ($1, $2, $3)
        ON CONFLICT DO NOTHING`, userID, targetType, targetID)
}
```

已接入的位置：

- 创建需求后，创建人自动关注需求：`RequirementHandler.Create`。
- 创建任务后，任务创建人自动关注任务：`TaskHandler.Create`。
- 创建任务时，assignee 自动关注任务：`TaskHandler.Create`。
- 任务更新时，如果 `assignee_id` 发生变化，新 assignee 自动关注任务：`TaskHandler.Update`。

当前行为基本符合这次产品口径，但还没有关注度计算和展示。

### 1.4 删除需求 / 删除任务时是否清理关注

会清理。

需求删除：`RequirementHandler.Delete` 在真实删除需求前执行：

```sql
DELETE FROM user_follows WHERE target_type = 'requirement' AND target_id = $1
```

任务删除：`TaskHandler.Delete` 在真实删除任务前执行：

```sql
DELETE FROM user_follows WHERE target_type = 'task' AND target_id = $1
```

注意：

- 需求只有在没有任务 / session / token_usage / document 关联时才允许物理删除；否则提示 cancel。
- 任务删除前还会解除 sessions / token_usage / documents 对任务的引用，再删除任务关注。

### 1.5 `dashboard/follows` 当前查询、排序、过滤

后端：`DashboardHandler.Follows`。

流程：

1. 查询当前用户自己的关注关系：

```sql
SELECT target_type, target_id
FROM user_follows
WHERE user_id = $1
ORDER BY created_at DESC
```

2. 逐条按对象类型加载展示 DTO：
   - `requirementFollowItem`
   - `taskFollowItem`
3. 返回 `DashboardFollowItem[]`。

当前后端过滤：

- 已完成 / 已取消需求不返回。
- 已完成任务不返回。
- 父需求已完成 / 已取消的任务不返回。

当前后端排序：

- 初始顺序是用户关注时间倒序。
- 但最终没有基于风险或关注度排序。

当前前端还有一层排序：`DashboardPage.tsx` 里 `sortFollowItems()` 根据风险优先级排序：

- 超期风险优先。
- 依赖 / 阻塞其次。
- 进行中再次。
- 已完成靠后。

也就是说，当前关注首页不是单纯按接口返回顺序展示，而是前端会按风险文本再排一次。

### 1.6 `dashboard/risks` 当前如何使用关注关系

后端：`DashboardHandler.Risks`。

当前风险列表只针对任务风险生成条目，风险来源为：

- 任务超期。
- 任务被依赖阻塞。

查询范围：

```sql
FROM tasks t
JOIN requirements r ON r.id = t.requirement_id
LEFT JOIN users a ON a.id = t.assignee_id
WHERE t.status <> 'done'
  AND r.status NOT IN ('completed', 'cancelled')
```

关注关系当前用于扩展部分角色的可见范围：

- `employee`：可见自己负责的任务，或自己关注的任务，或自己关注需求下的任务。
- `team_leader`：可见本团队成员任务、自己创建的任务，或自己关注的任务，或自己关注需求下的任务。
- `pm` / `director` / `admin`：当前没有额外 WHERE 限制，看到全局风险任务。

当前排序：

```sql
ORDER BY t.due_date NULLS LAST, t.updated_at DESC
```

当前没有关注度提升风险优先级。

### 1.7 需求列表 / 任务列表是否展示关注状态

后端模型已经返回个人关注状态：

- `model.Requirement.IsFollowed` → JSON `is_followed`
- `model.Task.IsFollowed` → JSON `is_followed`

计算位置：

- 需求：`RequirementHandler.loadProjection` 查询当前用户是否存在 `user_follows`。
- 任务：`TaskHandler.enrichTask` 查询当前用户是否存在 `user_follows`。

前端 API 类型也已有：

- `Requirement.is_followed`
- `Task.is_followed`

但需求看板当前实际 UI 主要依赖 `GET /follows` 生成 `favoriteRequirementIds` / `favoriteTaskIds`，再展示星标与“关注”筛选；没有直接基于 `is_followed` 做全部展示逻辑。

### 1.8 当前是否已有关注人数、关注度、高关注标识

没有。

代码搜索结果显示：

- 后端没有 `follow_count`、`attention_score`、`attention_level` 等字段。
- 前端 DTO 没有关注人数 / 关注度字段。
- `RoleHomepagePrototype.tsx` 里有“高关注”等静态 prototype 文案，但不是实际数据逻辑。

### 1.9 当前角色模型

后端 `users.role` 是单个 `TEXT` 字段。

初始迁移 `001_init.sql`：

```sql
role TEXT NOT NULL CHECK (role IN ('director', 'team_leader', 'pm', 'employee'))
```

认证迁移 `005_user_auth.sql` 扩展为：

```sql
CHECK (role IN ('admin','director','team_leader','pm','employee'))
```

前端类型：`web/src/shared/auth/types.ts`

```ts
export type UserRole = "admin" | "director" | "pm" | "team_leader" | "employee";
```

本次关注度权重映射建议：

- `director`：总监，100
- `team_leader`：TL，50
- `pm`：PM，40
- `employee`：员工，10
- `admin`：管理员，0；不参与业务关注度计算，避免管理员测试关注污染首页和风险排序。

### 1.10 一个用户是否可能有多个角色

当前不会。

依据：

- 数据库 `users.role` 是单字段。
- JWT / 当前用户模型也是单个 `role`。
- 前端 `User.role` 是单个 `UserRole`。

因此关注度按用户当前的 `users.role` 实时计算，不记录关注发生时的角色快照；用户角色变化后，其历史关注关系在下一次查询时按新角色权重生效。不需要处理多角色合并或最高权重。

## 2. 当前代码风险点

1. **自动关注已存在，但是静默失败。**  
   `autoFollow` 忽略 `Exec` 错误。作为附带行为可以接受，但如果后续关注度成为核心排序依据，建议至少记录日志或让调用方可感知失败。

2. **没有 tombstone，但当前规则暂时不需要。**  
   用户取消关注后记录被删除。普通编辑不会恢复关注，因为当前自动关注只在创建和 assignee 明确变化时触发；但如果未来新增“编辑时同步参与人自动关注”等逻辑，就可能破坏“普通编辑不能恢复取消关注”的口径。

3. **关注状态有两套前端来源。**  
   后端列表返回 `is_followed`，但需求看板实际用 `/follows` 单独生成关注集合。后续加 `follow_count` / `attention_score` 时，建议统一以列表 DTO 字段为主，`/follows` 只用于“我的关注关系列表”。

4. **`dashboard/follows` 当前 N+1 查询明显。**  
   先查关注关系，再逐条查需求 / 任务。关注度接入后，如果继续逐条查，会进一步放大查询次数。建议这期至少在 SQL 层批量聚合关注度，避免再增加 N+1。

5. **`target_id` 无外键。**  
   这是统一关注表的自然结果。当前通过需求 / 任务删除时手动清理规避脏数据，但如果有新的删除路径，也要记得清理。

6. **`dashboard/risks` 当前只输出任务风险。**  
   需求关注度很高但风险来自需求整体时，目前没有 requirement-level 风险项，只有任务风险项。

7. **前端 prototype 中有“高关注”静态文案。**  
   容易和真实逻辑混淆。正式接入时应只在真实接口字段存在时展示。

## 3. 产品口径确认

本期建议确认并沿用以下口径：

1. 关注不是单纯收藏，而是用户与需求/任务存在关注关系。
2. 关注关系继续统一存储在 `user_follows`。
3. 不拆 `manual` / `auto_creator` / `auto_assignee` 等来源字段。
4. 只要 `user_follows` 中存在一条 `(user_id, target_type, target_id)`，就认为该用户关注该对象。
5. 关注可由用户手动产生，也可由系统在明确业务事件中自动产生。
6. 被指派任务算关注，并计入关注度。
7. 用户可以手动取消关注。
8. 用户取消关注后，普通编辑不能恢复关注。
9. 只有再次发生明确的创建 / 指派事件，才可以重新自动关注。
10. 暂不 backfill 历史数据，避免旧关注关系或历史指派突然改变首页排序。

## 4. 自动关注规则

### 4.1 落点

继续放在 handler 层即可，不建议本期新增复杂 service。

原因：

- 当前需求 / 任务创建和任务改派逻辑都集中在 handler。
- 自动关注是这些写操作的附带结果，触发点清晰。
- 当前项目没有统一领域 service 层，强行拆 service 会扩大改动面。

具体落点：

| 事件 | 当前/建议落点 | 目标 |
| --- | --- | --- |
| 创建需求 | `RequirementHandler.Create` | 创建人自动关注需求 |
| 创建任务 | `TaskHandler.Create` | 创建人自动关注任务 |
| 创建任务且有 assignee | `TaskHandler.Create` | assignee 自动关注任务 |
| 任务改派 | `TaskHandler.Update` | 新 assignee 自动关注任务 |

### 4.2 插入方式

统一使用：

```sql
INSERT INTO user_follows (user_id, target_type, target_id)
VALUES ($1, $2, $3)
ON CONFLICT DO NOTHING
```

这满足：

- 自己创建并指派给自己时只保留一条记录。
- 手动关注后再触发自动关注不会重复。
- 自动关注后再手动关注不会重复。

### 4.3 创建需求

当前已有：创建需求成功后执行：

```go
autoFollow(h.db, u.ID, "requirement", reqID)
```

建议保留。

### 4.4 创建任务

当前已有：创建任务成功后执行：

```go
autoFollow(h.db, u.ID, "task", taskID)
autoFollow(h.db, *req.AssigneeID, "task", taskID)
```

建议保留。

注意：当前创建任务要求 `assignee_id` 必填；员工创建任务时如果为空，会默认指派给自己。因此“创建任务时如果有 assignee”在当前实现里几乎总是会发生。

### 4.5 任务改派

当前已有逻辑：

```go
if req.AssigneeID != nil && *req.AssigneeID != "" &&
    (!task.AssigneeID.Valid || task.AssigneeID.String != *req.AssigneeID) {
    autoFollow(h.db, *req.AssigneeID, "task", id)
}
```

它满足：

- `null -> X`：自动关注新 assignee。
- `A -> B`：自动关注新 assignee。
- `A -> A`：不自动关注。
- 普通任务更新不触发自动关注。
- 改派时不删除老 assignee 的关注。

### 4.6 如何保证普通编辑不恢复取消关注

不新增来源字段的前提下，核心约束是：**只能在明确事件中调用 `autoFollow`**。

建议把规则固化为代码约定：

- `RequirementHandler.Update` 不调用 `autoFollow`。
- `TaskHandler.Update` 只有 `assignee_id` 非空且值变化时调用 `autoFollow`。
- `UpdateStatus`、`UpdateProgress`、依赖增删、AC 更新等普通编辑不调用 `autoFollow`。

这样用户取消关注后：

- 普通编辑不会插回 `user_follows`。
- 只有后续再次创建新对象，或任务明确改派给该用户，才会重新自动关注。

## 5. 关注度计算规则

### 5.1 权重

建议先硬编码在后端统一函数 / SQL CASE 中：

| role | 中文角色 | 权重 |
| --- | --- | ---: |
| `director` | 总监 | 100 |
| `team_leader` | TL | 50 |
| `pm` | PM | 40 |
| `employee` | 员工 | 10 |
| `admin` | 管理员 | 0 |

关注度：

```text
attention_score = 关注该需求/任务的所有用户当前角色权重之和
```

关注人数：

```text
follow_count = COUNT(DISTINCT user_id)
```

因此，admin 权重本期明确按 0 处理，管理员关注关系仍可存在、仍影响 `is_followed` 和个人关注列表，但不贡献 `attention_score`。由于 `user_follows` 主键已保证同一用户同一对象唯一，`COUNT(*)` 在按对象聚合时也可以。

### 5.2 关注等级

建议后端统一返回英文枚举，前端负责中文展示：

| 分数区间 | 建议枚举 | 中文展示 |
| --- | --- | --- |
| 0-39 | `normal` | 普通 |
| 40-79 | `notable` | 关注 |
| 80-149 | `important` | 重点关注 |
| 150+ | `high` | 高关注 |

注意：

- 不使用 `followed` 作为等级枚举，避免和 `is_followed` 混淆。
- 对象没有任何关注时：`follow_count=0`，`attention_score=0`，`attention_level=normal`。

### 5.3 总监关注是否直接视为高关注

不建议本期默认启用，但可以作为产品确认项。

优点：

- 总监关注通常代表管理关注，强提醒更符合业务直觉。
- 单个总监关注无需等其它角色叠加，就能在首页明显浮出。

缺点：

- 会破坏简单分数模型：总监权重 100 按当前区间只是“重点关注”，不是“高关注”。
- “高关注”的含义从“多人/多角色加权关注”变成“特定角色一票否决”，解释成本上升。
- 未来如果 admin 按总监处理，容易误把管理员测试关注也推成高关注。

建议：

- 本期先只使用分数区间，不加“总监一票高关注”。
- 如果产品强需求，可增加一个展示层规则：`has_director_follow=true` 时显示“总监关注”标签，但不直接改 `attention_level`。

## 6. `dashboard/follows` 排序方案

### 6.1 排序目标

关注列表不再只是“我的收藏时间线”，而是“我关注且需要优先处理/感知的对象”。排序上需要兼顾风险和关注度，明确高风险 / 阻塞 / 超期事项不能被低风险高关注事项完全压下去。

建议排序优先级：

1. 风险优先。
2. 关注度其次。
3. 截止时间再次。
4. 最近更新或关注时间兜底。

### 6.2 最小排序规则

后端 `dashboard/follows` 返回时建议按：

```text
risk_priority DESC,
attention_score DESC,
due_date ASC NULLS LAST,
updated_at DESC,
follow_created_at DESC
```

其中：

- `risk_priority` 建议定义为数值，数值越大越靠前，例如：超期 = 100，阻塞/依赖 = 80，进行中 = 40，普通 = 20，已完成 = 0。
- 当前关注列表已过滤已完成/已取消项，`已完成 = 0` 主要用于前端兜底或未来复用。
- 如果产品后续坚持关注度第一，再调整为：

```text
attention_score DESC,
risk_priority DESC,
due_date ASC NULLS LAST,
updated_at DESC,
follow_created_at DESC
```

### 6.3 推荐落点

建议排序尽量放后端：

- 同一接口多页面复用时顺序一致。
- 后续分页更容易。
- 避免前端只拿到部分字段时排序不完整。

但考虑当前 `dashboard/follows` 是逐条拼 DTO，最小实现可以分两步：

1. 后端 DTO 增加 `follow_count` / `attention_score` / `attention_level` / `risk_priority`。
2. 前端 `sortFollowItems()` 改为先按 `risk_priority DESC`，再按 `attention_score DESC`，保留接口排序作为主顺序。

## 7. `dashboard/risks` 接入方案

### 7.1 是否应该按关注度提升风险优先级

应该接入，但不要让关注度覆盖真实风险严重性。

建议：

- 风险是否进入列表仍由真实风险决定：超期 / 阻塞。
- 关注度只影响风险列表排序和展示突出程度。
- 高关注风险可以在 UI 上加标签，例如“高关注 · 180”。

### 7.2 排序建议

当前排序：

```text
due_date ASC NULLS LAST,
updated_at DESC
```

建议调整为：

```text
risk_level_priority ASC,
attention_score DESC,
due_date ASC NULLS LAST,
updated_at DESC
```

由于当前所有 `dashboardRiskItem` 的 `Level` 都是“高”，本期实际效果可以是：

```text
attention_score DESC,
due_date ASC NULLS LAST,
updated_at DESC
```

但要注意：

- 超期很久但关注度低的任务不应完全沉底。
- 可以保留 due_date 作为关注度之后的关键排序。

### 7.3 需求关注如何影响任务风险

当前风险项都是任务。需求关注应该影响其下任务风险的排序和展示，但本期不要简单相加：

```text
risk_attention_score = max(task_attention_score, requirement_attention_score)
```

理由：

- 如果用户关注的是需求，不一定关注具体任务，但风险发生在需求下的任务时，也应体现需求关注度。
- 如果同一用户同时关注了需求和任务，简单 `task_attention_score + requirement_attention_score` 会重复计分，导致关注度虚高。
- 本期用 `max(...)` 能避免大部分重复计分问题，改动也比 union 去重更小。
- 后续如果确实需要更精确的综合关注度，再做按用户 union 去重：同一用户同时关注父需求和任务时只计算一次，权重取该用户当前角色权重。

接口本期建议只返回风险排序使用的综合字段：

- `attention_score`：`max(task_attention_score, requirement_attention_score)`。
- `attention_level`：基于综合 `attention_score` 计算。
- `follow_count`：可先返回与综合分来源一致的对象关注人数；如需精确综合人数，后续再做 union 去重。

如果需要调试或解释排序，可选返回：

- `task_attention_score`：任务自身关注度。
- `requirement_attention_score`：父需求关注度。

但这会扩大前端展示和解释成本，本期最小建议只返回综合 `attentionScore`。

## 8. API 返回字段建议

### 8.1 需求 / 任务通用字段

建议在 `Requirement` 和 `Task` DTO 上增加：

```json
{
  "follow_count": 3,
  "attention_score": 100,
  "attention_level": "important",
  "is_followed": true
}
```

字段说明：

- `follow_count`：关注人数。
- `attention_score`：角色权重求和。
- `attention_level`：`normal` / `notable` / `important` / `high`。
- `is_followed`：当前用户是否关注，现有字段保留。

命名建议：

- 当前模型已有 `IsFollowed`，可新增 `FollowCount` / `AttentionScore` / `AttentionLevel`。
- JSON 使用 snake_case，与现有 `is_followed` 风格一致。

### 8.2 `DashboardFollowItem`

建议增加：

```json
{
  "followCount": 3,
  "attentionScore": 100,
  "attentionLevel": "important"
}
```

注意：当前 `DashboardFollowItem` 使用的是 camelCase JSON：`requirementId`、`taskId`。因此这里建议沿用 camelCase。

### 8.3 `DashboardRiskItem`

建议增加：

```json
{
  "followCount": 5,
  "attentionScore": 180,
  "attentionLevel": "high"
}
```

如果风险综合了任务和需求关注度，可考虑增加：

```json
{
  "targetAttentionScore": 80,
  "requirementAttentionScore": 100
}
```

但这会扩大前端展示和解释成本。本期最小建议只返回综合 `attentionScore`。

### 8.4 前端展示建议

需求/任务卡片：

- 星标继续表示“我是否关注”。
- 卡片右上或状态区展示关注等级 Tag：
  - `normal` 可不展示，避免噪音。
  - `notable` 展示“关注 40”。
  - `important` 展示“重点关注 100”。
  - `high` 展示“高关注 180”。
- Tooltip 可展示“3 人关注”。

首页关注列表：

- 在标题或 meta 区展示关注等级。
- 高关注项可使用更醒目的 tag，但不改变整体卡片结构。

风险列表：

- 在风险级别旁展示“高关注”。
- 排序上让高关注风险更靠前。

## 9. 数据库 / 查询方案

### 9.1 实时 SQL 聚合

推荐本期采用实时 SQL 聚合，不新增冗余字段。

聚合 SQL 可抽象为：

```sql
SELECT
  f.target_type,
  f.target_id,
  COUNT(*) AS follow_count,
  COALESCE(SUM(CASE u.role
    WHEN 'director' THEN 100
    WHEN 'team_leader' THEN 50
    WHEN 'pm' THEN 40
    WHEN 'employee' THEN 10
    WHEN 'admin' THEN 0
    ELSE 0
  END), 0) AS attention_score
FROM user_follows f
JOIN users u ON u.id = f.user_id
GROUP BY f.target_type, f.target_id
```

按单个对象查询时：

```sql
SELECT COUNT(*), COALESCE(SUM(...), 0)
FROM user_follows f
JOIN users u ON u.id = f.user_id
WHERE f.target_type = $1 AND f.target_id = $2
```

按列表查询时建议用 CTE 或 LEFT JOIN 聚合结果，避免每条 N+1 查询。

### 9.2 实时聚合优点

- 不改表结构，最小化迁移风险。
- 不需要处理角色变更后的缓存刷新。
- 不需要处理关注插入 / 删除时同步更新冗余字段。
- 当前数据量预期不大，`idx_user_follows_target` 足够支撑按对象聚合。
- 符合“不新增复杂模型”的要求。

### 9.3 实时聚合缺点

- 列表接口会增加 JOIN / GROUP BY 成本。
- 如果关注数据量很大，dashboard 聚合可能变慢。
- 多处接口重复写 CASE 权重，容易不一致。

规避建议：

- 后端封装一个 attention SQL 片段或 helper，统一权重 CASE 和等级计算。
- 关注度按用户当前 `users.role` 实时计算，不记录关注发生时的角色快照；角色调整后，已有关注关系按新角色权重参与下一次查询。
- dashboard/follows、dashboard/risks、requirements list、tasks list 复用同一套计算。

### 9.4 冗余字段缓存方案

可选方案：在 `requirements` / `tasks` 增加：

- `follow_count`
- `attention_score`
- `attention_level`

或新增聚合表：

- `target_attention(target_type, target_id, follow_count, attention_score, updated_at)`

优点：

- 查询快，排序简单。
- dashboard 大列表更稳定。

缺点：

- 增加迁移和同步复杂度。
- 用户角色变化时要重算历史关注度。
- 关注插入/删除都要维护缓存。
- 任务/需求删除要清理缓存。
- 与本期“不复杂化”目标不符。

结论：本期不建议做缓存字段，先实时聚合。

## 10. 最小实现范围

### 10.1 第一优先级：打通首页关注和风险

后端：

1. 保留现有 `autoFollow` 规则，不重复大改自动关注链路。
2. 重点检查现有自动关注触发点：
   - 普通编辑不会恢复已取消关注。
   - `assignee_id` 变化判断准确覆盖 `null -> X` 和 `A -> B`。
   - `A -> A`、只改标题/状态/进度/截止日期不会触发自动关注。
3. 增加关注度计算 helper：
   - role → weight；`admin=0`。
   - score → level；`normal` / `notable` / `important` / `high`。
4. `DashboardFollowItem` 增加关注度字段，并在 `dashboard/follows` 中返回。
5. `DashboardRiskItem` 增加关注度字段，并在 `dashboard/risks` 中返回。
6. `dashboard/follows` 排序接入：`risk_priority DESC`、`attention_score DESC`、`due_date ASC`、`updated_at DESC`。
7. `dashboard/risks` 排序接入关注度；任务风险综合关注度先用 `max(task_attention_score, requirement_attention_score)`。

前端：

1. API 类型为 `DashboardFollowItemDTO` / `DashboardRiskItemDTO` 增加关注度字段。
2. 首页关注列表展示关注等级，并按风险优先、关注度其次排序。
3. 风险列表展示关注等级，并使用接口排序或前端兜底排序。
4. 现有星标关注 / 取消关注流程不改语义。

### 10.2 第二优先级：需求/任务列表卡片展示

为避免本期改动面过大，需求/任务列表和卡片展示作为第二优先级：

1. `Requirement` / `Task` 模型增加：
   - `follow_count`
   - `attention_score`
   - `attention_level`
2. `RequirementHandler.List/Get/Create` 的投影补齐关注度。
3. `TaskHandler.List/Get` 的投影补齐关注度。
4. 需求卡片 / 任务卡片展示关注等级 Tag。

测试 / 验证：

1. 创建需求后，创建人出现在 `user_follows`。
2. 创建任务后，创建人和 assignee 出现在 `user_follows`，同人只一条。
3. 任务 `null -> X`、`A -> B` 时新 assignee 自动关注。
4. `A -> A`、只改标题/状态/进度/截止日期时不恢复已取消关注。
5. 取消关注后关注度下降。
6. 角色权重计算正确。
7. dashboard/follows 和 dashboard/risks 排序符合预期。

## 11. 不做项

本期不建议做：

1. 不新增 `follow_source` / `source` / `manual` / `auto_creator` / `auto_assignee`。
2. 不新增取消关注 tombstone。
3. 不 backfill 历史创建人 / 历史 assignee 关注关系。
4. 不做复杂权重配置后台。
5. 不做按团队 / 部门 / 时间衰减的复杂关注度模型。
6. 不做“总监一票高关注”的特殊规则，除非产品明确确认。
7. 不做关注通知、站内信、飞书推送。
8. 不做关注来源解释 UI。
9. 不做历史关注度趋势图。
10. 不把关注解释回“收藏”，文案应统一为“关注”。

## 12. 需要产品确认的问题

1. **总监关注是否直接高关注？**  
   建议本期不启用，只按分数区间；如需体现，可单独展示“总监关注”标签。

2. **关注等级中文名是否确认？**  
   当前建议：普通 / 关注 / 重点关注 / 高关注。  
   接口枚举使用 `normal` / `notable` / `important` / `high`，避免与 `is_followed` 混淆。

3. **如果产品坚持关注度第一，dashboard/follows 是否调整排序？**  
   当前建议风险优先：`risk_priority DESC`、`attention_score DESC`。如果后续产品坚持关注度第一，再调整为 `attention_score DESC`、`risk_priority DESC`。

4. **风险排序中关注度和截止日期哪个优先？**  
   当前建议风险类型先决定是否进入列表，列表内优先关注度，再看截止日期；同时 dashboard/follows 保持风险优先，避免低风险高关注压过高风险事项。

5. **需求关注是否应该影响其下任务风险的关注度？**  
   建议应该影响，但本期综合分用 `max(task_attention_score, requirement_attention_score)`，不做简单相加；后续如需精确，再做按关注用户 union 去重。

6. **关注人数是否对所有角色可见？**  
   建议可见，因为只展示聚合数量和分数，不暴露具体关注人。

7. **是否需要展示具体关注人列表？**  
   本期不建议做；如果后续要做，需要权限和隐私口径。

8. **取消关注后再次被指派是否重新关注？**  
   当前产品口径是“只有再次发生明确创建/指派事件才可以重新自动关注”，因此 `A -> B -> A` 应重新关注 A。

## 结论

本期建议继续以 `user_follows` 作为唯一关注关系，不拆来源，不 backfill，不加冗余缓存。自动关注规则当前代码已基本具备，不重复大改；本期第一优先级是打通 dashboard/follows 与 dashboard/risks 的关注度字段、风险优先排序和展示，需求/任务列表卡片展示作为第二优先级推进。
