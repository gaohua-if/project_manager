# Dashboard 风险列表架构说明

## 1. 产品口径摘要

`/dashboard/risks` 是 Dashboard 首页的“我的风险提示”接口。

它不代表全局风险池、团队风险池，也不代表管理视图里的完整风险列表。所有角色，包括 `director`、`pm`、`team_leader`、`employee`，都使用同一套“我的风险事实”范围，不因角色权限自动扩大为全局或团队风险。

P0 只实现 3 类风险：

| riskType | 层级 | 含义 |
| --- | --- | --- |
| `requirement_overdue` | requirement | 需求超期 |
| `deadline` | task | 任务超期 |
| `dependency_blocker` | task | 任务存在未完成上游依赖 |

## 2. `/dashboard/risks` 返回模型

接口返回 Dashboard 风险组数组，而不是逐条 task risk item。

核心结构：

```json
{
  "key": "requirement:<requirementId>",
  "displayType": "requirement_group",
  "requirementId": "...",
  "requirementTitle": "...",
  "riskTypes": ["requirement_overdue", "deadline"],
  "requirementOverdue": true,
  "deadlineTaskCount": 2,
  "dependencyBlockerCount": 1,
  "representativeTask": {
    "taskId": "...",
    "title": "...",
    "deadline": "2000-01-01",
    "riskTypes": ["deadline", "dependency_blocker"],
    "unfinishedDependencyCount": 2
  },
  "summary": "需求超期 · 2 个任务超期 · 1 个依赖阻塞；重点任务：...",
  "deadline": "2000-01-01",
  "level": "高",
  "tone": "red",
  "attentionScore": 80,
  "attentionLevel": "important",
  "actionText": "查看需求",
  "targetUrl": "/requirements?requirementId=...",
  "navigation": {
    "requirementId": "...",
    "url": "/requirements?requirementId=..."
  }
}
```

`taskId` 不再是风险组的必填字段，只在 `single_task` 的 `representativeTask` 和 `navigation.taskId` 中出现。

## 3. 风险事实模型

服务端内部先计算风险事实，再聚合为 Dashboard 风险组。

需求风险事实：

```text
requirement.status NOT IN ('completed', 'cancelled')
AND requirements.deadline IS NOT NULL
AND requirements.deadline < 今天
AND 当前用户创建或关注该 requirement
```

任务风险事实：

```text
parent requirement.status NOT IN ('completed', 'cancelled')
AND task.status <> 'done'
AND task 属于当前用户任务风险范围
AND task 命中 deadline 或 dependency_blocker
```

其中：

- `deadline`：`tasks.due_date < 今天`；
- `dependency_blocker`：该任务存在至少一个未完成的上游依赖；
- 依赖阻塞归属到被阻塞的下游任务；
- 一个任务有多个未完成依赖，只生成一个 `dependency_blocker`，数量放到 `unfinishedDependencyCount`。

## 4. 可见范围过滤规则

需求级风险可见范围：

```text
1. requirement.creator_id = 当前用户
2. 当前用户关注了该 requirement
```

任务级风险可见范围：

```text
1. tasks.assignee_id = 当前用户
2. tasks.creator_tl_id = 当前用户
3. 当前用户关注了该 task
4. task 所属 requirement.creator_id = 当前用户
5. 当前用户关注了 task 所属 requirement
```

过滤必须发生在聚合之前。聚合不能扩大可见范围：如果用户只负责同一需求下的一个任务，只能聚合这个任务自己的风险，不能把同需求下其他不可见任务带入风险组。

## 5. Requirement 聚合规则

所有通过“我的风险事实”过滤后的风险事实，再按 `requirementId` 聚合。

一个风险组可以包含：

- 该 requirement 自身的 `requirement_overdue`；
- 当前用户可见的任务 `deadline` 风险；
- 当前用户可见的任务 `dependency_blocker` 风险；
- 当前用户可见任务中的代表任务。

同一任务同时命中 `deadline` 和 `dependency_blocker` 时，底层事实仍然区分两个风险类型，但 Dashboard 展示合并在同一个任务摘要中，不拆成两张卡片。

代表任务只能从当前用户可见任务风险中选择，不能引用不可见任务。

## 6. `displayType` 判定规则

```text
包含 requirement_overdue
=> requirement_group

无 requirement_overdue，但同一 requirement 下有多个可见风险任务
=> requirement_group

无 requirement_overdue，且只有一个可见风险任务
=> single_task
```

注意：一个任务同时命中 `deadline` 和 `dependency_blocker`，仍然算一个可见风险任务。

## 7. 跳转与操作规则

`requirement_group`：

```text
actionText = 查看需求
targetUrl = /requirements?requirementId=<requirementId>
navigation.taskId 为空
```

`single_task`：

```text
actionText = 查看任务
targetUrl = /requirements?requirementId=<requirementId>&taskId=<taskId>
navigation.taskId = <taskId>
```

如果 `single_task` 只有 `dependency_blocker`，按钮可以显示为“处理依赖”，跳转仍定位到任务。

## 8. 排序规则

P0 排序只使用当前用户可见风险事实计算：

1. 包含超期风险优先；
2. 最早超期日期更早优先；
3. `AttentionScore` 更高优先；
4. 最近更新时间更近优先。

排序不能引用当前用户不可见任务的截止日期或更新时间。

## 9. AttentionScore 边界

`AttentionScore` 只影响排序和热度展示，不参与风险来源判定。

无关 requirement 或 task 即使关注度很高，也不能出现在当前用户 Dashboard 风险列表。风险组的关注分只从该 requirement 本身和当前用户可见风险任务中取值。

## 10. 前端展示逻辑

Dashboard 风险模块标题使用“我的风险提示”。

空状态文案：

```text
暂无需要处理的风险
```

`requirement_group` 展示：

- 标题：`requirementTitle`；
- 标签：需求超期、N 个任务超期、N 个依赖阻塞；
- 摘要：`summary` 和代表任务；
- 操作：查看需求。

`single_task` 展示：

- 标题：`representativeTask.title`；
- 副信息：所属需求 `requirementTitle`；
- 标签：任务超期、依赖阻塞；
- 操作：查看任务或处理依赖。

## 11. 关键边界场景

- `requirements.deadline = 今天` 不算需求超期；
- `requirements.deadline` 为空不生成需求超期；
- `tasks.due_date = 今天` 不算任务超期；
- `tasks.due_date` 为空不生成任务超期；
- `completed` / `cancelled` requirement 不生成需求风险，也不生成其下任务风险；
- `done` task 不生成任务风险；
- 需求超期不是任务超期，任务超期也不是需求超期；
- 用户只负责或只关注某个 task 时，不自动看到同 requirement 下其他 task 风险；
- 关注 requirement 后，可以看到该 requirement 本身风险和其下任务风险；
- director / pm / team_leader 不因角色默认看到全局或团队风险。

## 12. 测试覆盖说明

后端 `api/handler/dashboard_test.go` 覆盖了以下核心行为：

- director / pm / team_leader / employee 使用同一套“我的风险事实”范围；
- requirement 创建或关注后生成 `requirement_overdue`；
- 无关风险不会进入结果；
- 同一任务同时命中 `deadline` 和 `dependency_blocker` 时合并到一个 `representativeTask`；
- 多个未完成依赖只计一个 `dependency_blocker`，并保留 `unfinishedDependencyCount`；
- 聚合只使用已过滤的可见任务事实；
- 排序只基于可见风险事实；
- `single_task` 和 `requirement_group` 的跳转形态正确。

前端验证覆盖：

- TypeScript 类型检查；
- Dashboard 风险卡兼容风险组 DTO；
- `requirement_group` 与 `single_task` 两种展示形态；
- 空状态和模块标题文案。

## 13. 后续扩展建议

以下能力不属于 P0，本次不实现：

- 需求缺 AC；
- 需求未拆任务；
- 需求停滞；
- 智能进展异常；
- 高关注需求异常；
- 团队风险池；
- 全局风险池。

如后续需要团队或全局风险，应新增独立入口和接口，不复用 Dashboard 首页“我的风险提示”的语义。
