# 需求看板独立需求基线说明

版本：v0.1
模块：需求看板
定位：作为 AIcoding 管理平台中的独立核心页面，承载需求、任务、依赖、阻塞、deadline、验收标准、关注与风险处理。

---

## 1. 模块背景

当前平台整体重新收敛为两个核心部分：

```text
AIcoding 管理平台
├─ 控制台
└─ 需求看板
```

其中：

* 控制台负责聚合提醒、我关注的、风险项、日报入口；
* 需求看板负责真实工作承载，包括需求管理、任务拆解、任务依赖、阻塞处理、deadline、验收标准、关注状态和风险定位。

本需求文档只讨论「需求看板」，不展开控制台和日报生成能力。

---

## 2. 产品定位

需求看板是一个轻量研发协作看板，类似 Jira / Linear / 飞书项目 / TAPD / ONES 中的需求与任务管理部分，但第一版不追求完整项目管理系统。

它主要解决：

1. 需求可以结构化管理；
2. 需求可以拆解为任务树；
3. 任务可以设置负责人、状态、deadline、验收标准；
4. 任务之间可以存在依赖关系；
5. 任务可以被标记为阻塞；
6. 阻塞、依赖、deadline、缺少验收标准可以形成风险；
7. 用户可以关注需求或任务；
8. 控制台中的关注项和风险项可以跳转到需求看板定位处理。

一句话定义：

> 需求看板是围绕“需求拆解、任务执行、依赖阻塞、风险处理”的轻量工作承载页面。

## 5. 需求与任务结构

需求可以拆解为任务。

任务可以继续拆解为子任务。

P0 建议支持两层任务结构：

```text
需求
├─ 任务 1
│  ├─ 子任务 1.1
│  └─ 子任务 1.2
├─ 任务 2
└─ 任务 3
```

P0 不建议支持无限层级任务树，避免交互和维护成本过高。

---

## 6. 需求字段

需求是任务树的根节点，表达一个业务目标、产品目标、技术目标或交付物。

### 6.1 需求核心字段

| 字段       | 是否必填 | 说明                    |
| -------- | ---- | --------------------- |
| 需求名称     | 是    | 需求标题                  |
| 需求描述     | 否    | 补充说明                  |
| 负责人      | 是    | 需求 owner              |
| 状态       | 是    | 需求当前推进状态              |
| 验收标准 AC  | 否    | 需求级验收标准               |
| deadline | 否    | 需求整体截止时间              |
| 关注状态     | 否    | 当前用户是否关注              |
| 风险数量     | 系统生成 | 由任务、阻塞、deadline、AC 聚合 |
| 阻塞任务数量   | 系统生成 | 当前需求下阻塞任务数量           |
| 创建人      | 是    | 系统记录                  |
| 创建时间     | 是    | 系统记录                  |
| 更新时间     | 是    | 系统记录                  |

### 6.2 需求状态

P0 状态建议：

```text
未开始
进行中
已完成
已暂停
已取消
```

需求是否有风险，不建议放进状态枚举，而应作为独立风险标识。

---

## 7. 任务字段

任务是需求拆解后的执行项，可以有子任务。

### 7.1 任务核心字段

| 字段       | 是否必填 | 说明        |
| -------- | ---- | --------- |
| 任务名称     | 是    | 任务标题      |
| 任务描述     | 否    | 补充说明      |
| 所属需求     | 是    | 任务归属的需求   |
| 父任务      | 否    | 子任务才有     |
| 负责人      | 是    | 任务 owner  |
| 状态       | 是    | 当前任务状态    |
| 验收标准 AC  | 否    | 任务级验收标准   |
| deadline | 否    | 任务截止时间    |
| 是否阻塞     | 否    | 是否被标记阻塞   |
| 阻塞原因     | 否    | 阻塞时填写     |
| 上游依赖任务   | 否    | 当前任务依赖的任务 |
| 关注状态     | 否    | 当前用户是否关注  |
| 创建人      | 是    | 系统记录      |
| 创建时间     | 是    | 系统记录      |
| 更新时间     | 是    | 系统记录      |

### 7.2 任务状态

P0 状态建议：

```text
未开始
进行中
已完成
已暂停
已取消
```

任务是否阻塞不建议作为任务状态，而应作为独立字段。

因为实际可能出现：

```text
进行中 + 阻塞
未开始 + 依赖未完成
进行中 + 已超期
进行中 + 缺少验收标准
```

状态、阻塞、deadline、风险应该解耦。

---

## 8. 验收标准 AC

需求和任务都可以有验收标准。

### 8.1 字段口径

验收标准 AC 是选填字段。

但在某些情况下，缺少 AC 可以形成风险。

### 8.2 缺 AC 形成风险的场景

```text
缺 AC 风险
├─ 高关注需求缺少 AC
├─ 已进入执行阶段但缺少 AC
├─ 临近 deadline 但缺少 AC
└─ 任务已经开始但缺少必要验收标准
```

普通草稿需求缺少 AC，不一定算风险。

---

## 9. deadline

需求和任务都可以设置 deadline。

### 9.1 需求 deadline

需求 deadline 表达整体交付时间。

### 9.2 任务 deadline

任务 deadline 表达具体执行时间约束。

风险判断中，任务 deadline 优先级高于需求 deadline。

### 9.3 deadline 风险

```text
deadline 风险
├─ 任务即将到期
├─ 任务已超期
├─ 需求即将到期但关键任务未完成
└─ 上游任务超期影响下游任务
```

P0 不做复杂排期和自动计划，只做简单到期和超期提醒。

---

## 10. 任务依赖

任务之间可以存在依赖关系。

依赖关系用于表达：

```text
任务 A 依赖任务 B
任务 B 未完成 / 阻塞 / 超期
        ↓
任务 A 存在依赖风险或依赖阻塞
```

### 10.1 P0 依赖范围

P0 建议：

```text
P0 依赖能力
├─ 支持同一需求内任务依赖
├─ 支持一个任务依赖多个上游任务
├─ 上游任务未完成时，下游展示“依赖未完成”
├─ 上游任务阻塞时，下游展示“依赖阻塞”
└─ 上游任务超期时，下游展示“依赖超期风险”
```

### 10.2 P0 不做

```text
P0 不做
├─ 跨需求依赖
├─ 复杂依赖网络图
├─ 关键路径分析
├─ 自动排期
└─ 依赖冲突自动修复
```

跨需求依赖可作为 P1。

---

## 11. 阻塞

阻塞表示任务当前无法继续推进。

### 11.1 阻塞来源

```text
阻塞来源
├─ 手动阻塞
│  ├─ 负责人主动标记阻塞
│  └─ 填写阻塞原因
│
└─ 依赖阻塞
   ├─ 上游任务未完成
   ├─ 上游任务已阻塞
   └─ 上游任务已超期
```

P0 以手动阻塞为主，系统依赖判断为辅。

### 11.2 手动阻塞

用户可以在任务上标记阻塞。

标记阻塞时应填写阻塞原因。

可选字段：

```text
阻塞信息
├─ 是否阻塞
├─ 阻塞原因
├─ 阻塞时间
├─ 阻塞标记人
└─ 是否已解除
```

P0 可先只做：

```text
是否阻塞
阻塞原因
```

### 11.3 阻塞解除

当任务状态更新、阻塞原因解决后，用户可以取消阻塞。

取消阻塞后：

* 当前任务不再计入阻塞任务数量；
* 下游依赖任务的依赖阻塞风险重新计算；
* 控制台风险项相应减少。

---

## 12. 风险

风险不是用户手动维护的主字段，而是系统根据结构化数据聚合出来。

### 12.1 风险类型

```text
风险类型
├─ AC 缺失风险
├─ 手动阻塞风险
├─ 依赖未完成风险
├─ 依赖阻塞风险
├─ 依赖超期风险
├─ deadline 即将到期风险
└─ deadline 已超期风险
```

### 12.2 风险展示位置

风险需要在两个层级展示：

```text
风险展示
├─ 需求列表
│  ├─ 风险数量
│  ├─ 阻塞任务数量
│  └─ 最高风险等级
│
└─ 需求详情 / 任务树
   ├─ 具体风险项
   ├─ 具体任务
   ├─ 风险原因
   └─ 处理入口
```

### 12.3 风险处理

风险本身不做独立工作流。

风险点击后定位到对应需求或任务，由用户修改任务状态、deadline、阻塞状态、AC 或依赖关系来解决风险。

---

## 13. 关注

用户可以关注需求或任务。

### 13.1 关注对象

```text
关注对象
├─ 需求
└─ 任务
```

### 13.2 关注规则

* 用户只能关注自己有权限访问的需求或任务；
* 关注不等于权限；
* 取消关注不影响需求或任务本身；
* 关注对象会出现在控制台“我关注的”模块；
* 任务出现在关注列表时，必须展示所属需求。

### 13.3 默认关注视角

不同角色默认关注对象不同，但所有角色都可以关注需求或任务。

| 角色 | 默认关注对象  |
| -- | ------- |
| 总监 | 需求      |
| TL | 需求 + 任务 |
| PM | 需求      |
| 个人 | 任务      |

需求看板本身不需要做四套页面，只需要支持不同默认筛选或默认视角。

## 14. 页面结构建议

### 14.1 推荐页面结构

需求看板建议采用：

```text
需求看板
├─ 顶部区域
│  ├─ 页面标题
│  ├─ 新建需求按钮
│  ├─ 视图切换：列表 / 看板 / 我关注的
│  └─ 基础筛选
│
├─ 左侧或主区域：需求列表
│  ├─ 需求名称
│  ├─ 状态
│  ├─ 负责人
│  ├─ deadline
│  ├─ 风险数量
│  ├─ 阻塞任务数量
│  └─ 关注状态
│
└─ 右侧或详情区：需求详情
   ├─ 基础信息
   ├─ 验收标准
   ├─ 任务树
   ├─ 风险 / 阻塞汇总
   └─ 操作区
```

### 14.2 两种可选布局

#### 方案 A：列表 + 详情抽屉

```text
需求列表
└─ 点击需求 → 右侧抽屉展示需求详情和任务树
```

优点：

* 轻量；
* 不打断列表浏览；
* 适合后台管理系统；
* 适合 P0 快速落地。

缺点：

* 任务树很复杂时，抽屉可能不够承载；
* 深层编辑体验可能一般。

#### 方案 B：列表 + 独立详情页

```text
需求列表
└─ 点击需求 → 进入需求详情页
```

优点：

* 承载能力强；
* 适合复杂任务树；
* 适合后续扩展依赖、风险、历史记录。

缺点：

* 跳转层级更深；
* 返回和定位需要做好。

### 14.3 P0 建议

P0 推荐：

```text
需求列表 + 需求详情页
```

原因：

* 任务树、依赖、阻塞、AC、deadline 都需要较大空间；
* 抽屉容易显得拥挤；
* 后续从控制台跳转定位任务时，详情页更容易承载。

## 16. 任务树展示要求

任务树要重点解决：

```text

├─ 这个需求拆了哪些任务
├─ 每个任务谁负责
```

## 17. 控制台跳转到需求看板

虽然本需求只讨论需求看板，但需求看板需要支持从控制台跳转定位。

### 17.1 跳转类型

```text
控制台跳转
├─ 点击关注的需求
│  └─ 进入需求详情页
│
├─ 点击关注的任务
│  └─ 进入需求详情页，并定位到该任务
│
├─ 点击风险项：任务阻塞
│  └─ 进入需求详情页，并展开对应任务
│
├─ 点击风险项：依赖阻塞
│  └─ 进入需求详情页，并展示当前任务和上游任务
│
└─ 点击风险项：deadline 风险
   └─ 进入需求详情页，并定位到对应任务或需求
```

### 17.2 URL 参数建议

原型阶段可以考虑类似：

```text
/requirements
/requirements/:requirementId
/requirements/:requirementId?taskId=xxx
/requirements/:requirementId?riskId=xxx
```

## 21. 同类产品调研方向

后续做同类产品调研时，不要泛泛比较完整项目管理平台，而是重点看以下能力：

### 21.1 Jira

重点看：

```text
Jira 调研重点
├─ issue 层级
├─ issue 状态
├─ blocker 标记
├─ linked issue / dependency
├─ backlog 和 board 的关系
├─ 详情页信息密度
└─ 风险和状态标签呈现方式
```

不要直接照搬 Jira 的复杂工作流和配置能力。

### 21.2 Linear

重点看：

```text
Linear 调研重点
├─ 列表体验
├─ 快速创建任务
├─ 任务详情侧栏
├─ 状态标签
├─ 负责人展示
├─ deadline / due date
└─ 低噪音的信息组织
```

Linear 的价值在于轻量、高效、低干扰，适合作为交互复杂度控制参考。

### 21.3 飞书项目

重点看：

```text
飞书项目调研重点
├─ 国内用户熟悉的信息结构
├─ 需求 / 任务 / 子任务关系
├─ 视图切换
├─ 字段展示
├─ 负责人和状态筛选
└─ 与飞书生态的协作入口
```

但不要让 P0 被完整协同套件能力拖重。

### 21.4 TAPD / ONES

重点看：

```text
TAPD / ONES 调研重点
├─ 需求管理
├─ 任务拆解
├─ 缺陷 / 风险标签
├─ 研发团队使用习惯
├─ 国内后台系统信息密度
└─ 需求详情页结构
```

适合作为国内研发团队习惯参考。

## 22. 原型设计重点

需求看板原型设计不要一开始做视觉发散，优先确定信息架构。

### 22.1 第一优先级

```text
第一优先级
├─ 需求列表怎么展示
├─ 需求详情页怎么承载任务树
├─ 任务树怎么展示状态、负责人、deadline、阻塞
├─ 风险项怎么在需求和任务层级显示
├─ 依赖关系怎么轻量表达
└─ 从控制台跳转后怎么定位任务
```

### 22.2 第二优先级

```text
第二优先级
├─ 筛选器
├─ 视图切换
├─ 批量操作
├─ 高级字段
├─ 统计卡片
└─ 复杂图表
```

### 22.3 原型不建议一开始做

```text
不要一开始做
├─ 大量筛选项
├─ 类 Jira 全功能配置
├─ 甘特图
├─ 复杂 dashboard
├─ 多角色多套页面
└─ 花哨但不解决信息结构的图表
```


## 24. 当前阶段决策

当前阶段先固定以下决策：

```text
已确认
├─ 需求看板是独立核心页面
├─ 需求可以拆解任务
├─ 任务最多先支持两层
├─ 任务可以有依赖

├─ 需求和任务都有 AC，但 AC 选填
├─ 缺 AC 在特定场景下形成风险
├─ 需求和任务都有 deadline
├─ 风险来自 AC、阻塞、依赖、deadline
├─ 用户可以关注需求或任务

```

## 26. 一句话基线

需求看板不是普通需求列表，也不是完整 Jira。

它是一个轻量研发协作页面，核心主线是：

```text
需求
  ↓ 拆解
任务树
  ↓ 存在依赖
阻塞 / deadline / AC 风险
  ↓ 定位
在需求详情中处理
```

P0 的目标不是功能多，而是让用户能清楚看到：

```text
有哪些需求
拆了哪些任务
谁负责
什么时候到期
哪里阻塞
依赖谁
缺什么
下一步该处理什么
```
