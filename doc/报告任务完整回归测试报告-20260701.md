## 6. 新发现 Bug

### BUG-1 P0: Report Agent session message 覆盖 Start Prompt Template，导致 run_id 不进入 Agent 初始消息

证据：

- Aida 本地 `ai_runs.input_ref_json.start_prompt_values.run_id` 存在，值等于 `ai_runs.id`。
- 平台 session 结果显示 `run_id does not exist` 或 `缺少 run_id 参数`。
- Agent 平台源码 `ResolveInitialMessage(template, values, editedMessage)` 逻辑为：`message` 非空时直接返回 `message`，不会渲染 Start Prompt Template。
- Aida schedule path 创建 session 时总是传 `Message=sessionMessage`；当 schedule message 为空时，Aida 会填 fallback message。因此默认 Report Agent Start Prompt Template 中的 `run_id={{ run_id }}` 被绕过。

影响：

- schedule run-now、schedule auto、50 真实模型同刻触发大量出现 completed 但无 MCP 写回。
- syncer 修复后能正确把这类问题标为 failed，但端到端报告生成仍失败。

建议修复：

- Report Agent 创建 session 时，不要让普通 message 覆盖系统参数模板。
- 可选方案 1：Aida 自己渲染包含 `run_id/report_type/period/target/mcp_url` 的最终 message，再追加用户自定义 message。
- 可选方案 2：传空 `Message`，让平台渲染 Start Prompt Template；用户自定义 message 改为 start prompt value 的一个字段，由模板显式引用。
- 修复后必须重跑 `TC-SCH-CONC-074` 50 真实模型并发。

### BUG-2 P0/P1: 50 并发下 Agent 平台沙箱容量不足或缺少重试/限流

证据：

- 50 真实模型并发中 8 个 session 返回 `opensandbox returned 504`，错误包含 `KUBERNETES::POD_READY_TIMEOUT`。

影响：

- 即使 Aida 参数注入修复，50 同刻真实模型并发仍可能因平台沙箱容量失败。

建议：

- Aida 侧增加 schedule 并发触发的队列/限速/重试策略，或平台侧提升 sandbox 并发容量。
- 测试期可以记录为平台容量限制；生产期需要明确 SLA。

### BUG-3 P1: Agent 找到已有报告后未调用 write_report_result

证据：

- 手动 `team_daily` session 平台结果显示找到现有团队日报，但 Aida run 最终为 `completed without report writeback`。

建议：

- Report Agent 指令应明确：即使复用已有报告内容，也必须调用 `write_report_result` 完成 run 与报告表绑定。
