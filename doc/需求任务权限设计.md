# 需求 / 任务权限设计

## 1. 背景

当前需求 / 任务模块的权限实现不统一，主要问题包括：

* 前端 route roles、页面按钮显隐、后端 handler 判断不一致。
* 后端多数接口仅做登录鉴权，缺少资源级权限判断。
* employee 当前可以通过直接调用 API 更新 / 取消任意需求、更新任意任务状态 / 进度 / 依赖。
* PM 后端具备任务管理能力，但前端 `/tasks` 相关路由不允许 PM 进入。
* TL、PM、director 在真实业务中都可能参与需求和任务的增删改查。
* employee 在真实业务中可以看到自己团队的需求和任务，也可以管理自己的任务，但不能编辑或删除别人的任务。
* 需求删除、取消、归档没有清晰区分。

本文作为需求 / 任务模块权限改造的设计基准。

---

## 2. 设计目标

本次改造目标：

1. 统一需求 / 任务的后端权限判断。
2. 修复 employee 可越权修改任意需求 / 任意任务的问题。
3. 支持 TL / PM / director 在真实业务场景下具备需求和任务管理能力。
4. 支持 employee 查看自己团队需求 / 任务，并管理自己的任务。
5. 通过资源范围约束，避免 TL 跨团队误改、误删不相关资源。
6. 统一前端 route、按钮显隐和后端权限。
7. 删除、取消、归档语义分离，避免把业务取消误做成硬删除。

---

## 3. 核心权限原则

需求 / 任务权限采用三层判断：

```text
角色权限 + 资源范围 + 状态 / 关联数据约束
```

### 3.1 角色权限

角色决定用户是否具备某类操作能力。

```text
admin / director / pm / team_leader = 管理角色
employee = 团队执行角色
```

管理角色可以具备需求 / 任务的创建、查看、编辑、状态流转、取消、恢复、删除等能力。

employee 不是需求管理角色，但不是只读用户。employee 可以：

```text
1. 查看自己团队的需求；
2. 查看自己团队的任务；
3. 创建自己的任务；
4. 查看自己的任务；
5. 编辑自己的任务；
6. 删除自己的任务；
7. 更新自己的任务状态 / 进度；
8. 不能编辑或删除别人的任务；
9. 不能创建、编辑、取消、恢复、删除需求。
```

### 3.2 资源范围

角色有操作能力，不代表能操作所有资源。

推荐范围：

| 角色          | 资源范围                                  |
| ----------- | ------------------------------------- |
| admin       | 全部需求 / 全部任务                           |
| director    | 全部需求 / 全部任务                           |
| pm          | P0 阶段暂定全部需求 / 全部任务                    |
| team_leader | 自己团队相关需求 / 自己创建需求 / 自己团队相关任务 / 自己创建任务 |
| employee    | 自己团队相关需求 / 自己团队相关任务 / 自己负责或自己创建的任务    |

说明：

* 当前如果系统没有 `requirement_owner_id` / `pm_id` 字段，PM 暂时按全量管理处理。
* 后续如补充需求负责人字段，可将 PM 范围收敛为“自己负责的需求及其任务”。
* TL 的权限重点是“团队范围内管理”，不是全平台管理。
* employee 可以看团队范围内需求和任务，但只能改自己的任务。

### 3.3 “自己的任务”定义

P0 推荐统一定义：

```text
自己的任务 = assignee_id = 当前用户 ID
```

如果现有表里有 creator 字段，也可以补充：

```text
自己的任务 = assignee_id = 当前用户 ID OR creator_id = 当前用户 ID
```

但 P0 为了避免歧义，建议优先使用 `assignee_id = 当前用户 ID` 判断 employee 的编辑 / 删除权限。

### 3.4 状态 / 关联数据约束

即使角色和范围满足，也不代表所有操作都允许。

例如：

* 已存在关联任务、session、token_usage、documents 的需求，不允许硬删除。
* 任务存在 session、token_usage、documents 时，删除需要按当前系统策略处理，例如置空关联或拒绝删除。
* `blocked` 是依赖推导状态，不允许直接手动设置。
* 删除是危险操作，应与取消、归档区分。

## 4. 角色定位

### 4.1 admin

系统兜底管理员。

权限：

* 全量查看需求 / 任务。
* 全量管理需求 / 任务。
* 可恢复、删除、修复异常数据。
* 不受团队范围限制。

### 4.2 director

管理层 / 方向负责人。

权限：

* 全量查看需求 / 任务。
* 全量管理需求 / 任务。
* 可以创建、编辑、取消、恢复、删除需求。
* 可以创建、编辑、删除任务。

说明：

director 在真实使用中可能不频繁操作，但权限上保留全量管理能力。

### 4.3 pm

需求推进负责人。

权限：

* P0 阶段全量查看和管理需求 / 任务。
* 可以创建、编辑、取消、恢复、删除需求。
* 可以创建、编辑、删除任务。
* 可以管理需求下的任务拆分和状态推进。

说明：

当前没有明确 PM 负责范围字段，因此 P0 不做 PM 资源范围收窄。

### 4.4 team_leader

团队负责人。

权限：

* 可以创建需求。
* 可以管理自己团队相关或自己创建的需求。
* 可以创建、编辑、删除自己团队相关任务。
* 可以给自己团队成员分配任务。
* 可以更新自己团队任务状态 / 进度。
* 不允许跨团队修改不相关需求 / 任务。

说明：

真实场景中，很多需求可能是讨论后由 TL 创建，因此 TL 不应被降级为只能管理任务的执行角色。

### 4.5 employee

团队执行人。

权限：

* 可以查看自己团队的需求。
* 可以查看自己团队的任务。
* 可以创建自己的任务。
* 可以编辑自己的任务。
* 可以删除自己的任务。
* 可以更新自己的任务状态 / 进度。
* 不允许编辑或删除别人的任务。
* 不允许创建、编辑、取消、恢复、删除需求。
* 不允许修改别人的任务负责人、依赖、状态、进度等信息。

说明：

employee 不是只读角色。员工在团队内部可以自我维护任务，但不能影响别人的任务和需求主数据。

## 5. 需求权限设计

### 5.1 需求可见范围

| 角色          | 可见范围              |
| ----------- | ----------------- |
| admin       | 全部需求              |
| director    | 全部需求              |
| pm          | P0 阶段全部需求         |
| team_leader | 自己团队参与的需求；自己创建的需求 |
| employee    | 自己团队参与的需求         |

说明：

employee 可以看到自己团队的需求，但只是查看，不具备需求管理权限。

### 5.2 需求操作权限矩阵

| 操作           | admin | director | pm |        team_leader | employee |
| ------------ | ----: | -------: | -: | -----------------: | -------: |
| 查看列表         |    全部 |       全部 | 全部 |         本团队 / 自己创建 |      本团队 |
| 查看详情         |    全部 |       全部 | 全部 |         本团队 / 自己创建 |      本团队 |
| 创建需求         |     ✅ |        ✅ |  ✅ |                  ✅ |        ❌ |
| 编辑需求基础信息     |     ✅ |        ✅ |  ✅ |         本团队 / 自己创建 |        ❌ |
| 修改需求状态 / 阶段  |     ✅ |        ✅ |  ✅ |         本团队 / 自己创建 |        ❌ |
| 取消需求         |     ✅ |        ✅ |  ✅ |         本团队 / 自己创建 |        ❌ |
| 恢复需求         |     ✅ |        ✅ |  ✅ |         本团队 / 自己创建 |        ❌ |
| 删除需求         |     ✅ |        ✅ |  ✅ | 本团队 / 自己创建，且满足删除条件 |        ❌ |
| 查看 AC        |     ✅ |        ✅ |  ✅ |         本团队 / 自己创建 |      本团队 |
| 编辑 / 重新生成 AC |     ✅ |        ✅ |  ✅ |         本团队 / 自己创建 |        ❌ |

### 5.3 TL 创建需求规则

TL 可以创建需求，但需要约束资源范围。

P0 推荐规则：

```text
TL 创建需求时，默认必须包含自己的 team_id。
TL 不允许创建完全不包含自己团队的需求。
```

如果当前创建表单允许选择多个 team_ids：

```text
TL 可选择自己团队；
是否允许选择其他团队，P0 建议先不开放。
跨团队需求由 PM / director 补充或调整。
```

如果当前业务必须允许 TL 创建跨团队需求，则后端至少要保证：

```text
TL 创建的需求必须包含自己的 team_id；
其他 team_ids 可以存在，但后续跨团队管理边界需要单独评审。
```

### 5.4 需求删除规则

删除不等同于取消。

#### 5.4.1 硬删除

硬删除只用于误创建、测试数据、无业务痕迹的需求。

允许硬删除的条件：

```text
1. 用户具备删除权限；
2. 资源在用户可管理范围内；
3. 需求没有关联任务；
4. 需求没有关联 session；
5. 需求没有关联 token_usage；
6. 需求没有关联 documents；
7. 需求没有其他强业务关联数据。
```

不满足条件时，返回 conflict，不允许硬删除。

#### 5.4.2 取消需求

取消表示业务上不再推进。

规则：

```text
有管理权限的用户可以将可管理范围内的需求置为 cancelled。
取消不是删除，历史数据保留。
```

#### 5.4.3 归档需求

归档表示从主看板收起。

规则：

```text
已完成或已取消的需求可以归档。
归档不删除数据。
P0 如果没有归档字段，可以先不实现归档，只保留设计口径。
```

## 6. 任务权限设计

### 6.1 任务可见范围

| 角色          | 可见范围                             |
| ----------- | -------------------------------- |
| admin       | 全部任务                             |
| director    | 全部任务                             |
| pm          | P0 阶段全部任务                        |
| team_leader | 自己团队成员负责的任务；自己创建的任务；自己团队相关需求下的任务 |
| employee    | 自己团队相关任务                         |

说明：

employee 可以看到自己团队的任务，包括别人负责的任务。
但 employee 只能编辑 / 删除自己的任务，别人的任务只读。

### 6.2 任务操作权限矩阵

| 操作        | admin | director | pm |       team_leader | employee |
| --------- | ----: | -------: | -: | ----------------: | -------: |
| 查看列表      |    全部 |       全部 | 全部 |        本团队 / 自己创建 |      本团队 |
| 查看详情      |    全部 |       全部 | 全部 |        本团队 / 自己创建 |      本团队 |
| 创建任务      |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅给自己创建 |
| 编辑任务基础信息  |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅自己的任务 |
| 修改负责人     |     ✅ |        ✅ |  ✅ | 仅可分配给本团队 employee |        ❌ |
| 修改任务 AC   |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅自己的任务 |
| 修改任务状态    |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅自己的任务 |
| 修改任务进度    |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅自己的任务 |
| 添加 / 删除依赖 |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅自己的任务 |
| 删除任务      |     ✅ |        ✅ |  ✅ |             本团队相关 |   仅自己的任务 |

### 6.3 employee 任务操作边界

employee 可以查看自己团队内所有任务。

employee 对别人的任务：

```text
1. 可以查看；
2. 不能编辑；
3. 不能删除；
4. 不能修改状态；
5. 不能修改进度；
6. 不能添加 / 删除依赖；
7. 不能修改负责人。
```

employee 对自己的任务：

```text
1. 可以创建；
2. 可以查看；
3. 可以编辑；
4. 可以删除；
5. 可以修改状态；
6. 可以修改进度；
7. 可以维护任务 AC；
8. 可以维护任务依赖。
```

P0 推荐限制：

```text
employee 创建任务时：
1. requirement_id 必须属于自己团队可见需求；
2. assignee_id 必须是当前用户自己；
3. 不允许 employee 给别人创建任务；
4. 不允许 employee 创建到其他团队需求下。
```

employee 编辑自己的任务时：

```text
允许修改：
1. title
2. priority
3. due_date
4. acceptance_criteria
5. status
6. progress
7. dependencies

不允许修改：
1. assignee_id
2. requirement_id
3. creator_id
4. 别人的任务字段
```

说明：

如果当前系统暂时没有 `creator_id`，则 employee 创建任务后以 `assignee_id = 当前用户 ID` 作为“自己的任务”判断依据。

### 6.4 blocked 状态规则

`blocked` 是依赖关系推导状态，不允许用户直接设置。

允许直接写入的任务存储状态：

```text
todo
in_progress
done
```

不允许直接写入：

```text
blocked
```

如果任务因依赖阻塞展示为 blocked，应由后端根据依赖关系计算或投影。

## 7. 后端权限函数设计

不要继续在每个 handler 内散写角色判断。建议新增统一权限判断方法。

### 7.1 需求权限函数

建议抽象：

```go
CanViewRequirement(ctx, user, requirementID) bool
CanCreateRequirement(ctx, user, teamIDs) bool
CanUpdateRequirement(ctx, user, requirementID) bool
CanChangeRequirementStatus(ctx, user, requirementID) bool
CanCancelRequirement(ctx, user, requirementID) bool
CanRestoreRequirement(ctx, user, requirementID) bool
CanDeleteRequirement(ctx, user, requirementID) bool
CanViewRequirementAC(ctx, user, requirementID) bool
CanManageRequirementAC(ctx, user, requirementID) bool
```

### 7.2 任务权限函数

建议抽象：

```go
CanViewTask(ctx, user, taskID) bool
CanCreateTask(ctx, user, requirementID, assigneeID) bool
CanUpdateTaskMeta(ctx, user, taskID) bool
CanReassignTask(ctx, user, taskID, assigneeID) bool
CanUpdateTaskStatus(ctx, user, taskID) bool
CanUpdateTaskProgress(ctx, user, taskID) bool
CanManageTaskDependencies(ctx, user, taskID, dependencyTaskID) bool
CanDeleteTask(ctx, user, taskID) bool
```

### 7.3 employee 任务权限判断

employee 的任务权限必须明确区分“可见”和“可编辑”。

```text
CanViewTask(employee, taskID):
  任务属于 employee 所在团队范围即可

CanUpdateTaskMeta(employee, taskID):
  task.assignee_id = employee.id

CanUpdateTaskStatus(employee, taskID):
  task.assignee_id = employee.id

CanUpdateTaskProgress(employee, taskID):
  task.assignee_id = employee.id

CanManageTaskDependencies(employee, taskID):
  task.assignee_id = employee.id

CanDeleteTask(employee, taskID):
  task.assignee_id = employee.id
```

### 7.4 权限判断失败返回

统一返回：

```text
401：未登录
403：已登录，但无权限
404：资源不存在，或出于安全考虑不暴露资源存在性
409：有关联数据，不允许删除
400：请求参数非法
```

推荐：

* 列表接口按权限过滤，不返回 403。
* 详情接口如无查看权限，可返回 404，避免暴露 ID 是否存在。
* 修改 / 删除接口如无操作权限，返回 403。
* 删除因关联数据失败，返回 409。

## 8. 后端接口改造要求

### 8.1 需求接口

#### GET /requirements

必须按角色过滤：

```text
admin/director/pm：返回全部
team_leader：返回自己团队参与或自己创建的需求
employee：返回自己团队参与的需求
```

不允许普通用户通过 `team_id` 参数扩大可见范围。

#### GET /requirements/{id}

必须调用：

```go
CanViewRequirement(...)
```

employee 可以查看自己团队范围内的需求。

无权限返回 404 或 403，推荐 404。

#### POST /requirements

必须调用：

```go
CanCreateRequirement(...)
```

规则：

```text
admin/director/pm：允许创建
team_leader：允许创建，但必须包含自己团队范围
employee：禁止创建
```

同时补充：

```text
1. team_ids 必须非空；
2. team_ids 必须存在；
3. TL 创建时必须包含自己的 team_id；
4. 插入 requirement_teams 必须检查 Exec 错误；
5. creator_id 写当前用户 ID；
6. creator_role 写当前用户 role。
```

#### PUT /requirements/{id}

必须调用：

```go
CanUpdateRequirement(...)
```

规则：

```text
admin/director/pm：允许更新全部需求
team_leader：只能更新自己团队相关或自己创建的需求
employee：禁止更新需求
```

#### PUT /requirements/{id}/restore

必须调用：

```go
CanRestoreRequirement(...)
```

允许：

```text
admin/director/pm：全量
team_leader：本团队 / 自己创建
employee：禁止
```

仅允许恢复 `cancelled` 状态的需求。

#### DELETE /requirements/{id}

必须调用：

```go
CanDeleteRequirement(...)
```

允许：

```text
admin/director/pm：全量
team_leader：本团队 / 自己创建
employee：禁止
```

同时保留强关联检查：

```text
有关联任务、session、token_usage、documents 时不允许硬删除，返回 409。
```

#### GET /requirements/{id}/ac

必须调用：

```go
CanViewRequirementAC(...)
```

employee 可以查看自己团队需求的 AC。

#### POST /requirements/{id}/regenerate-ac

必须调用：

```go
CanManageRequirementAC(...)
```

允许：

```text
admin/director/pm：全量
team_leader：本团队 / 自己创建
employee：禁止
```

注意：

```text
admin 必须允许 regenerate AC。
```

### 8.2 任务接口

#### GET /tasks

必须按角色过滤：

```text
admin/director/pm：返回全部
team_leader：返回自己团队成员负责的任务、自己创建的任务、自己团队相关需求下任务
employee：返回自己团队相关任务
```

重要：

```text
employee 可以查看自己团队任务；
但任何 scope 参数都不能让 employee 查看其他团队任务。
```

如果 `scope=requirements` 是为了需求看板拉任务，也必须叠加当前用户团队范围。

#### GET /tasks/{id}

必须调用：

```go
CanViewTask(...)
```

employee 可以查看自己团队范围内任务，包括别人负责的任务。

无权限返回 404 或 403，推荐 404。

#### POST /tasks

必须调用：

```go
CanCreateTask(...)
```

规则：

```text
admin/director/pm：允许创建
```

## 10. 前端路由权限

建议统一 route roles。

| 页面                     | 允许角色                                           |
| ---------------------- | ---------------------------------------------- |
| `/requirements`        | admin / director / pm / team_leader / employee |
| `/requirements/create` | admin / director / pm / team_leader            |
| `/requirements/:id`    | admin / director / pm / team_leader / employee |
| `/tasks`               | admin / director / pm / team_leader / employee |
| `/tasks/create`        | admin / director / pm / team_leader / employee |
| `/tasks/:id`           | admin / director / pm / team_leader / employee |

说明：

* PM 必须允许进入 `/tasks`、`/tasks/create`、`/tasks/:id`。
* employee 可以进入 `/tasks`，看到自己团队任务。
* employee 可以进入 `/tasks/create`，但只能给自己创建任务。
* employee 可以进入 `/requirements`，看到自己团队需求。
* employee 不允许进入 `/requirements/create`。
* route 只控制页面入口，不代表具体按钮权限。
* 具体按钮必须基于后端返回的权限字段或前端同构权限函数控制。

## 14. 验收标准

### 14.1 employee 权限验收

employee 应满足：

* 可以查看自己团队的需求。
* 可以查看自己团队的任务。
* 可以创建自己的任务。
* 可以编辑自己的任务。
* 可以删除自己的任务。
* 可以修改自己的任务状态。
* 可以修改自己的任务进度。
* 可以维护自己的任务依赖。
* 不能创建需求。
* 不能编辑需求。
* 不能取消需求。
* 不能恢复需求。
* 不能删除需求。
* 不能给别人创建任务。
* 不能编辑别人的任务。
* 不能删除别人的任务。
* 不能修改别人的任务状态 / 进度。
* 不能添加 / 删除别人任务的依赖。
* 不能通过 `scope=requirements` 查看其他团队任务。

### 14.2 TL 权限验收

TL 应满足：

* 可以创建需求。
* 创建需求时必须包含自己团队范围。
* 可以查看自己团队相关或自己创建的需求。
* 可以编辑自己团队相关或自己创建的需求。
* 可以取消 / 恢复 / 删除自己团队相关或自己创建的需求。
* 删除需求时仍需满足无强关联数据。
* 可以创建自己团队相关任务。
* 只能给自己团队 employee 分配任务。
* 可以编辑、删除自己团队相关任务。
* 不能管理其他团队无关需求 / 任务。

### 14.3 PM 权限验收

PM 应满足：

* 可以进入 `/tasks`、`/tasks/create`、`/tasks/:id`。
* 可以创建、编辑、删除需求。
* 可以创建、编辑、删除任务。
* P0 阶段可以查看和管理全部需求 / 任务。

### 14.4 director 权限验收

director 应满足：

* 可以查看和管理全部需求 / 任务。
* 可以创建、编辑、取消、恢复、删除需求。
* 可以创建、编辑、删除任务。

### 14.5 admin 权限验收

admin 应满足：

* 可以查看和管理全部需求 / 任务。
* 可以 regenerate AC。
* 可以执行所有兜底管理动作。

### 14.6 删除验收

需求硬删除：

* 无关联任务 / session / token_usage / documents 时允许。
* 有关联数据时返回 409。
* 不允许 employee 删除。
* TL 只能删除自己范围内需求。

任务删除：

* employee 只能删除自己的任务。
* employee 不能删除别人的任务。
* TL 只能删除自己团队相关或自己创建任务。
* 删除后 session / token_usage / documents 的 task_id 置空。
* 删除后重算 requirement progress。
