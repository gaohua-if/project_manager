# 报告任务测试用例

## 1. 目标

本用例用于指导 Codex 基于 Aida 当前环境，对 **AI Assets / Agent 定时任务** 做一轮高覆盖、可复现的端到端测试。

本轮测试目标不是只验证“接口 200”，而是尽量完整覆盖下面四层：

- Report Agent 直接手动运行，生成 6 类 `report_type` 报告
- Report Agent 定时任务创建、编辑、启停、删除、立即运行、自动触发
- 6 类 `report_type` 在手动运行和定时任务两类入口下的调度能力
- 不同角色与 `report_type` 的允许 / 拒绝矩阵
- 执行后的 `ai_runs / schedule / report` 联动结果
- Agent 生成内容的轻度合理性检查

测试必须以 **Aida 当前真实接口** 为准，不直接调用远端 Agent 平台私有接口，不手工伪造 Report MCP 回写。

重要口径：本测试用例以本文件和 [定时任务.md](/home/intellif/dev/project_manager/doc/定时任务.md) 中定义的产品 / 技术契约作为验收基线。当前代码形态只能作为定位接口、字段和实现路径的证据，不能被视为 100% 正确。如果实际代码行为与契约冲突，应在执行记录中标注为 **实现缺陷 / 产品口径缺口 / 可观测性缺口**，不得因为“代码当前就是这样”而降低测试预期。

## 2. 适用范围

本用例覆盖的报告类型：

- `personal_daily`
- `personal_weekly`
- `team_daily`
- `team_weekly`
- `department_daily`
- `department_weekly`

本用例默认当前 6 类报告主流程已经实现并可正式支持。

本用例覆盖两类最常见用户入口：

- **直接手动运行 Report Agent**：用户在 AI Assets 中选择 Report Agent，选择 `report_type`、报告周期和补充要求后立即运行。
- **设置定时器运行 Report Agent**：用户创建 Agent 定时任务，由 schedule 按 daily / weekly 频率触发对应 `report_type`。

这两类入口的 period 来源不同：直接手动运行使用用户提交的 `period`；定时器运行使用 `report_type + scheduled_trigger_at` 推导。测试用例必须分别验证，不能混为一类。

## 4. 测试原则

### 4.1 执行方式

优先顺序：

1. API 为主
2. 页面为辅

建议：

- 创建默认 Report Agent、创建定时任务、手动触发、查询结果，优先用 API 批量执行
- 页面抽样必须覆盖本文件 8.12 的字段隐藏、表单分支、预览和列表验收，不能只做“看一眼页面能打开”

### 4.2 时间原则与缺陷判定

不要直接假设“今天”一定有可用 session，也不要把历史 `seed_date` 误当成所有运行方式的报告归属日期。

权威规则以 [定时任务.md](/home/intellif/dev/project_manager/doc/定时任务.md) 为准：

- 直接手动运行 Report Agent，即 `POST /ai-assets/report-agents/{agentId}/runs`，报告周期由用户提交的 `period` 决定。
- 定时任务运行 Report Agent，即 `POST /ai-assets/agent-schedules/{id}/runs` 或 scanner 自动触发，报告周期由 `report_type + scheduled_trigger_at` 推导。
- 定时任务的 `manual` / `save_and_run` 运行，`scheduled_trigger_at` 应是用户触发时的当前时间，不应使用 `next_run_at`。
- 定时任务的 `scheduled` 自动触发，`scheduled_trigger_at` 应是 scanner 领取前的旧 `next_run_at`。

Codex 在执行前必须先做数据预检查并记录：

- `execution_date`：本轮实际触发时间在后端默认时区下的日期。
- `execution_week_start` / `execution_week_end`：`execution_date` 所在自然周，周一到周日。
- `seed_date`：最近 14 天内，`307 / 308 / 311 / 312` 都有 session 的共同日期。
- `seed_week_start` / `seed_week_end`：`seed_date` 所在自然周，周一到周日。

要求：

- 如果使用直接手动 Report Agent Run，报告期望基于请求 payload 中显式提交的 `period`，可以使用 `seed_date` / `seed_week_start` 来验证历史数据。
- 如果使用定时任务的 `manual` 或 `save_and_run` 验证报告内容，报告期望必须基于 `execution_date` / `execution_week_start`，不能硬套历史 `seed_date`。
- 如果 `execution_date` 上 307 / 308 / 311 / 312 没有共同 session，团队 / 部门“内容来源完整性”应判定为 **测试环境前置数据不足**，不要用历史 `seed_date` 冒充本次运行周期。
- 历史 `seed_date` 可用于直接手动 Report Agent Run 的显式 `period`。但对定时任务运行而言，除非测试手段能让 `scheduled_trigger_at` 合法落到该日期，否则不能作为本次定时任务写回的期望周期。
- 如果产品需要按历史日期生成报告，但当前公开接口无法表达该能力，应记录为 **产品 / 接口能力缺口**。
- 如果报告查询接口宣称支持 `date` / `week_start` 但实际忽略参数，应记录为 **实现缺陷或可观测性缺口**，不得静默改成“按当前代码行为通过”。

### 4.3 清理原则

测试前应尽量清理本地 Aida 数据，避免旧 schedule / ai_run 干扰：

- 仅清理 Aida 本地库中本轮测试账号的：
  - `managed_agent_schedules`
  - `managed_agent_profiles`
  - `ai_runs`
- 不要求清远端 Agent 平台历史资产

若远端已有未归档默认 Report Agent，可直接复用；若没有，则通过 Aida 接口创建。

### 8.2 定时任务通用创建步骤

除生命周期、直接手动 Report Agent Run 和负向用例外，所有定时任务正向用例统一按下面步骤执行：

1. 用对应账号获取或确认默认 Report Agent
2. 调 `/ai-assets/agent-schedules/preview`
3. 创建定时任务 `POST /ai-assets/agent-schedules`
4. 立即运行 `POST /ai-assets/agent-schedules/{id}/runs`
5. 轮询：
   - `/ai-assets/agent-runs`
   - `/ai-assets/agent-runs/{runId}`
   - `/ai-assets/agent-schedules`
6. 再调对应报告查询接口核对写回结果

### 8.3 定时任务通用创建 payload 约束

Report Agent 定时任务 payload 必须满足：

- `run_kind = report_agent`
- `trigger_config.schedule_type` 与 `report_type` 不绑定
- `run_config.report_config.report_type` 必填
- 不提交：
  - `target`
  - `target_json`
  - `period_json`
  - `period_start`
  - `period_end`
  - `scheduled_trigger_at`
  - `run_id`
  - `mcp_url`
  - `credential`
  - `credential_slot`
  - `AIDA_REPORT_MCP_AUTH`

### 8.11 schedule_type / report_type 解耦用例

目标：

- 明确验证 `schedule_type=daily / weekly` 只表示触发频率。
- 明确验证 `report_type` 才决定日报 / 周报，以及个人 / 小组 / 部门目标。
- 如果当前实现把 `daily schedule` 强制等同日报，或把 `weekly schedule` 强制等同周报，应记录为 **实现缺陷**。

建议至少覆盖：

| Case ID | 执行账号 | schedule_type | report_type | 核心验证 |
| --- | --- | --- | --- | --- |
| TC-DC-PD-WEEKLY | 309 | weekly | personal_daily | 每周触发一次个人日报，报告周期仍为触发日全天 |
| TC-DC-PW-DAILY | 309 | daily | personal_weekly | 每天刷新当前自然周个人周报，报告周期仍为自然周 |

- `POST /ai-assets/agent-schedules/preview` 返回的 `next_run_at` 只由 `schedule_type / weekdays / time_of_day` 决定。

- 创建 schedule 成功后，保存的 `schedule_type` 与 `report_config.report_type` 分别保持用户输入值。
- 立即运行后，`ai_runs.input_ref_json.report_type` 与创建时一致。
- 立即运行后，日报类 period 是 `execution_date` 全天，周报类 period 是 `execution_week_start ~ execution_week_end`。

## 9. 负向与边界用例

以下负向用例建议至少覆盖一次。

### 9.1 角色权限不匹配

用例：

- employee 手动运行或创建 schedule 使用 `team_daily`
- employee 手动运行或创建 schedule 使用 `department_daily`
- TL 手动运行或创建 schedule 使用 `department_weekly`
- director 手动运行或创建 schedule 使用 `team_weekly`，除非产品明确允许 director 代小组生成

预期：

- 手动运行应直接失败，不生成 `ai_run`，或生成失败态 `ai_run` 且错误明确。
- 创建 schedule 应直接失败，或保存后运行阶段失败且错误明确；推荐创建阶段失败。
- 返回 400 / 403 或明确业务错误信息。
- 不生成脏 schedule，不写入错误报告。

### 9.2 系统变量覆盖

对直接手动 Report Agent Run 和 Report Agent schedule 创建，故意在 `start_prompt_values` 中提交：

- `report_type`
- `period_json`
- `target_json`
- `period_start`
- `period_end`
- `scheduled_trigger_at`
- `run_id`
- `mcp_url`
- `credential`
- `credential_slot`
- `AIDA_REPORT_MCP_AUTH`

预期：

- 直接手动运行应失败，或创建 schedule 应失败。
- 返回保留字段错误。

### 9.3 Agent 不存在

使用一个假的 `agent_id` 直接手动运行 Report Agent，并使用同一个假的 `agent_id` 创建 schedule。

预期：

- 手动运行和创建 schedule 均失败。
- 返回 `agent not found` 或明确等价错误。

### 9.4 调度超时与失败展示

如果环境允许，可额外制造 1 条失败 schedule，例如：

- 先创建 schedule
- 再让其绑定的 Agent 不可用

预期：

- `last_run_status = failed` 或 `timeout`
- `last_error` 有明确原因
- 不影响其他 schedule 正常执行
- 如果失败发生在 Agent 平台提交阶段，`ai_runs.error_message` 与 `schedule.last_error` 都应可定位原因

### 9.5 调度参数与 Agent 类型边界

至少覆盖：

- `run_kind=report_agent` 绑定普通 Agent。
- `run_kind=generic_agent` 绑定 Report Agent。
- `schedule_type=weekly` 但 `weekdays` 为空。
- `weekdays` 包含 0、8 或重复值。
- `time_of_day` 非 `HH:mm`，例如 `24:00`、`9:00`、`19:99`。
- `agent_id` 为已归档 Agent。

预期：

- 创建或编辑失败。
- 返回明确错误。
- 不生成脏 schedule。
- 如果实际代码接受非法配置，应记录为实现缺陷，而不是调整测试期望。

### 9.6 报告对象无法推导

针对 Report Agent schedule，构造或查找无有效组织关系的账号，至少覆盖：

- 无 `team_id` 用户创建 `team_daily` / `team_weekly`。
- 非 director 用户创建 `department_daily` / `department_weekly`。
- director 账号无法推导所属部门或部门成员范围。

预期：

- 创建或运行阶段返回明确错误。
- 前端不应出现“报告对象选择器”让用户手工绕过。
- 如果无法通过现有测试账号构造，应记录为 **测试数据缺口**。

### 9.7 Report MCP 回写保护与幂等

本组用例优先通过真实 Report Agent run 产生的 MCP credential 或后端集成测试验证；不要在共享环境手工伪造 Report MCP 回写。

至少覆盖：

- `run_id` 不存在，`write_report_result` 被拒绝。
- `run_id` 属于其他用户，`write_report_result` 被拒绝。
- `report_type` 与 run 中记录不一致，回写被拒绝。
- `target` 与 run 中记录不一致，回写被拒绝。
- `period` 与 run 中记录不一致，回写被拒绝。
- `ai_run.status=failed` 或 `timeout` 后，回写被拒绝。
- 同一个 succeeded run 使用相同内容重复回写，返回幂等成功，不覆盖内容。
- 同一个 succeeded run 使用不同内容重复回写，返回冲突或拒绝。

如果当前公开测试手段无法触达这些验证点，应在执行记录中标记为 **可观测性 / 测试能力缺口**，不能视为通过。

## 10. 结果验证口径

### 10.1 调度层

每个定时任务正向用例至少验证：

- schedule 创建成功。
- `next_run_at` 合法。
- `manual` / `save_and_run` 不应错误使用 `next_run_at` 作为报告归属时间。
- `scheduled` 自动触发时，`scheduled_trigger_at` 应等于触发前旧 `next_run_at`，且 `next_run_at` 应立即推进。
- 手动运行或自动触发后：
  - `last_ai_run_id` 已更新。
  - `last_run_status` 进入 `running` 或 `succeeded`。
  - 失败时 `last_error` 有明确文案。
  - disabled schedule 不应被 scanner 触发。

### 10.2 运行层

每个 run 至少验证：

- `user_id` 与当前执行用户一致。
- `agent_id` 与本次选择的 Report Agent 或 schedule 绑定 Agent 一致。
- `trigger_source` 正确：直接手动 Report Agent Run 为 `manual`；定时任务立即运行为 `manual` / `save_and_run`；自动触发为 `scheduled`。
- 直接手动 Report Agent Run 不要求 `schedule_id` 或 `scheduled_trigger_at`，但 `period` 必须等于请求 payload。
- 定时任务 run 必须有 `schedule_id` 和 `scheduled_trigger_at`，且 `period` 由 `report_type + scheduled_trigger_at` 推导。
- `report_type` 正确。
- `target` 正确：直接手动 Run 按当前用户和 payload target 推导；定时任务 run 按 `schedule.user_id + report_type` 推导。
- `external_session_id` 或 `external_task_id` 存在。
- Report Agent run 不应在可见字段中泄露原始 credential。

### 10.3 报告写回层

每个正向 case 至少验证：

- 对应报告查询接口能读到内容。
- 报告时间 / 周期与预期一致。
- `managed_agent_run_id` 或等价字段指向本次 run。
- 写回目标没有串人、串组、串部门。
- Report MCP `write_report_result` 成功后，`ai_run.status` 最终为 `succeeded`。

### 10.4 轻度质量检查

Codex 需要对生成内容做轻度合理性检查，不要求人工文学评审，但至少检查：

1. 内容非空：
   - 不应为空字符串
   - 不应只有标题
2. 无系统参数泄露：
   - 不出现 `run_id`
   - 不出现 `mcp_url`
   - 不出现 `target_json`
   - 不出现 `period_json`
   - 不出现 `AIDA_REPORT_MCP_AUTH`
3. 无明显模板残留：
   - 不出现 `{{`
   - 不出现 `}}`
   - 不出现 `TODO`
4. 范围合理：
   - 个人报告不应写成小组 / 部门口径
   - 小组报告不应写成部门口径
   - 部门报告不应只覆盖单个小组
5. 日期 / 周期合理：
   - 直接手动 Report Agent Run 应对应请求 payload 中的 `period`，可为 `seed_date` 或 `seed_week_start ~ seed_week_end`。
   - 定时任务日报应对应 `execution_date`。
   - 定时任务周报应对应 `execution_week_start ~ execution_week_end`。
6. 如果前置 session 明确存在：
   - 报告不应整篇只有“暂无记录”

建议额外抽查：

- 小组日报 / 周报内容中，至少出现一个该组成员昵称或工作项痕迹
- 部门日报 / 周报内容中，至少体现两个小组的汇总痕迹，例如 `小组A / 小组B`

如果内容结构基本正确，但表达较泛，可记为：

- `warning`

如果出现范围错误、日期错误、系统变量泄露、整篇空洞模板，则记为：

- `fail`

### 10.5 缺陷判定口径

执行记录中必须区分下面几类结果：

- `pass`：行为符合本测试用例和 [定时任务.md](/home/intellif/dev/project_manager/doc/定时任务.md) 的契约。
- `warning`：主链路成功，但内容质量泛、环境数据稀疏或可观测性不足。
- `implementation bug`：当前代码行为与契约不一致。
- `product / contract gap`：开发文档和实际产品需求不一致，或契约没有定义清楚。
- `test data gap`：测试账号、session、组织关系等前置数据不足。
- `observability gap`：行为可能正确，但公开 API / 日志 / 字段不足以证明。

不得把“当前代码就是这样”直接记为通过。若代码不合理，应明确记录 bug 或缺口，并保留接口返回、字段值、run id、schedule id 作为证据。
