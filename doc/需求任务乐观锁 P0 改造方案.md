# 需求任务乐观锁 P0 改造方案

## 背景与目标

当前需求 / 任务保存链路已暴露 `updated_at`，但没有 `version`、`base_version`、`If-Match` 或类似机制。多人同时编辑时，后保存的请求可能静默覆盖先保存的结果，尤其是编辑弹窗会提交整组表单字段。

本期目标：只做行级 `version` 乐观锁，解决多人同时编辑需求 / 任务时的静默覆盖问题。

P0 明确不做：实时协同、WebSocket、CRDT、字段级自动合并、冲突差异对比、草稿恢复、自动重试、自动覆盖。

---

## 0. P0 版本边界口径

### 0.1 version 保护范围

P0 使用两个独立行级版本：

- `requirements.version`：只保护需求本体字段。
- `tasks.version`：保护任务本体字段和任务依赖关系。

需求本体字段包括：

- `title`
- `description`
- `feishu_doc_url`
- `acceptance_criteria`
- `status`
- `priority`
- `deadline`
- `completed_at`

任务本体字段包括：

- `title`
- `acceptance_criteria`
- `assignee_id`
- `status`
- `priority`
- `progress`
- `due_date`
- `completed_at`
- 任务依赖集合：`task_dependencies` 中 `task_id = 当前任务 id` 的上游依赖集合

### 0.2 任务变化不 bump 父需求 version

任务新增 / 编辑 / 删除 / 状态 / 进度 / 依赖变化，不默认 bump 父需求 `requirements.version`。

原因：

- `requirements.version` 只保护需求本体字段。
- 任务变化不应导致正在编辑需求标题 / 描述 / 优先级的人保存时触发 409。
- 任务列表和任务详情由 `tasks.version` 自己保护。

如果任务变化需要更新父需求投影，可以更新：

- `requirements.progress`
- `requirements.updated_at`

但 P0 不递增 `requirements.version`。

说明：当前 `task_summary` / `risk_summary` 主要是读取时投影汇总，不是独立持久字段；如果未来新增持久投影字段，P0 仍按“更新投影但不 bump requirements.version”的口径处理。

### 0.3 空更新口径

P0 不允许空更新静默成功。

空更新定义：请求体中除了 `base_version` 外，没有任何业务字段可更新。

处理规则：

- `PUT /requirements/{id}`：如果只有 `base_version`，没有 `title` / `description` / `status` / `priority` / `deadline` / `feishu_doc_url` / `acceptance_criteria`，返回 `400`。
- `PUT /tasks/{id}`：如果只有 `base_version`，没有 `title` / `acceptance_criteria` / `assignee_id` / `status` / `priority` / `due_date` / `progress`，返回 `400`。
- 错误码建议：`code: no_fields_to_update`。
- 不递增 `version`。
- 不返回成功实体。

目的：避免前端误以为保存成功，也避免“只有 base_version 的请求”掩盖前端表单 / payload 组装问题。

## 1. 数据库改造

### 1.1 新增 migration

预计新增：

- `api/db/migrations/016_requirement_task_versions.sql`

内容方向：

```sql
ALTER TABLE requirements
    ADD COLUMN IF NOT EXISTS version BIGINT NOT NULL DEFAULT 1;

ALTER TABLE tasks
    ADD COLUMN IF NOT EXISTS version BIGINT NOT NULL DEFAULT 1;
```

### 1.2 是否需要 migration 回填

需要让历史数据具备初始版本，但不需要额外 `UPDATE` 回填语句。

原因：

- `ADD COLUMN ... NOT NULL DEFAULT 1` 会让既有行具备默认值 `1`。
- 所有历史需求 / 任务从版本 `1` 开始即可。
- 后续每次受乐观锁保护的修改成功后递增到 `2、3、...`。

如果担心旧 PostgreSQL 大表加非空默认值的表重写成本，可拆成三步：

1. `ADD COLUMN IF NOT EXISTS version BIGINT DEFAULT 1`
2. `UPDATE requirements SET version = 1 WHERE version IS NULL` / `UPDATE tasks SET version = 1 WHERE version IS NULL`
3. `ALTER COLUMN version SET NOT NULL`

本项目 P0 可直接使用 `NOT NULL DEFAULT 1`。

### 1.3 task_dependencies 是否需要 version

P0 不给 `task_dependencies` 单独加 `version`。

依赖关系作为任务详情的一部分，由 owner task 的 `tasks.version` 保护：

- `POST /tasks/{id}/dependencies`：校验并在依赖确实新增时递增 `{id}` 对应任务的 `tasks.version`。
- `DELETE /tasks/{id}/dependencies/{dep_id}`：校验并在依赖确实删除时递增 `{id}` 对应任务的 `tasks.version`。
- 不校验 `dep_id` 对应任务的 `version`，避免一次依赖操作需要两个 base version。
- `Blocking` 是反向投影，刷新任务列表 / 详情即可更新；P0 不因为反向 `blocking` 变化阻塞 `dep_id` 任务的元信息编辑。

### 2.4 DELETE 接口统一用 query 传 base_version

约定：

- `DELETE /requirements/{id}?base_version=123`
- `DELETE /tasks/{id}?base_version=123`
- `DELETE /tasks/{id}/dependencies/{dep_id}?base_version=123`

后端统一解析：

- 缺失、非数字或小于等于 0：`400 Bad Request`
- 行不存在：`404 Not Found`
- 版本不一致：`409 Conflict`，body 返回 `code: EDIT_CONFLICT`

### 2.5 409 返回格式

统一格式：

```json
{
  "code": "EDIT_CONFLICT",
  "error": "content has been updated by another user",
  "current_version": 3
}
```

要求：

- `code` 固定为 `EDIT_CONFLICT`。
- `error` 固定为英文稳定文案。
- `current_version` 返回数据库当前版本号。
- 如果行已不存在，不返回 409，返回 404。

前端统一展示：

> 内容已被其他人更新，请刷新后再操作

## 3. 后端 handler 改造

通用原则：

1. 所有会修改既有需求 / 任务 / 任务依赖的接口必须携带 `base_version`。
2. `base_version <= 0` 返回 `400`。
3. 普通 UPDATE 从 `WHERE id = ?` 改为 `WHERE id = ? AND version = ?`。
4. 更新成功时同时执行：
   - `version = version + 1`
   - `updated_at = now()`
5. `RowsAffected == 0` 时：
   - 先查 `id` 是否存在。
   - 不存在返回 `404`。
   - 存在返回 `409 EDIT_CONFLICT`，并带 `current_version`。
6. 所有成功写接口必须让前端拿到最新版本：
   - 非 DELETE：优先返回完整 `Requirement` / `Task`。
   - 依赖 DELETE：返回最新 `Task`。
   - 物理删除 `Requirement` / `Task` 后目标行已不存在，返回删除状态并由前端移除缓存 / 刷新 query；这类接口没有“最新实体 version”。
7. 空更新返回 `400 no_fields_to_update`，不递增 version，不静默成功。
8. 任务相关写操作不 bump 父需求 `requirements.version`。
9. 如任务变化需要重算父需求 `progress` 或更新父需求 `updated_at`，可以更新这些投影字段，但不递增父需求 `version`。

建议在 handler 层抽内部 helper：

- `writeEditConflict(w, currentVersion)`
- `parseBaseVersionFromQuery(r)`
- `requireBaseVersion(w, baseVersion)`
- `currentVersionForRequirement(id)`
- `currentVersionForTask(id)`
- `writeNotFoundOrEditConflict(w, table, id)`

P0 可放在 `api/handler` 内部，不引入复杂 service 层。

### 3.1 `PUT /requirements/{id}`

预计修改：

- `api/handler/requirement.go`：`RequirementHandler.Update`
- `api/model/models.go`：`UpdateRequirementRequest`

改造点：

- 请求体增加 `base_version`。
- `base_version <= 0` 返回 400。
- 更新 SQL 条件：`WHERE id = $N AND version = $N+1`。
- `SET` 追加：`version = version + 1, updated_at = now()`。
- 当前字段级 SET 逻辑保留。
- 如果请求体除了 `base_version` 外没有任何业务字段，返回 `400`，`code: no_fields_to_update`。
- 空更新不递增版本，不返回成功实体。
- `RowsAffected == 0` 时区分 404 / 409，并在 409 中返回 `current_version`。
- 成功后返回完整最新 Requirement。

注意：

- `status` 通过此接口更新时也受 `base_version` 保护。
- `cancelRequirement` 当前也是提交 `{ status: "cancelled" }`，前端需同时带 `base_version`。

### 3.2 `POST /requirements/{id}/regenerate-ac`

预计修改：

- `api/handler/requirement.go`：`RequirementHandler.RegenerateAC`
- `api/model/models.go`：新增 `RegenerateACRequest`

改造点：

- 请求体读取 `base_version`。
- AI 生成前先做一次版本检查：
  1. 读取当前需求 `version`、`title`、`description`。
  2. 不存在返回 404。
  3. 当前 `version != base_version` 直接返回 `409 EDIT_CONFLICT`，不调用 AI。
- 通过首次版本检查后再调用 AI 生成 AC。
- AI 生成完成后，写入时仍做最终保护：
  - `UPDATE requirements SET acceptance_criteria = $1, version = version + 1, updated_at = now() WHERE id = $2 AND version = $3`
- `RowsAffected == 0` 时再次区分 404 / 409，并返回 `current_version`。
- 成功后返回完整最新 Requirement。

原因：

- 首次检查避免在明显过期的编辑上下文中浪费 AI 调用。
- 最终 `WHERE id AND version` 防止 AI 生成期间其他用户又保存了需求。

### 3.3 `PUT /requirements/{id}/restore`

预计修改：

- `api/handler/requirement.go`：`RequirementHandler.Restore`
- `api/model/models.go`：新增 `RequirementVersionRequest`

改造点：

- body 传 `base_version`。
- SQL 条件从：
  - `WHERE id = $1 AND status = 'cancelled'`
- 改成：
  - `WHERE id = $1 AND status = 'cancelled' AND version = $2`
- SET：
  - `status = 'todo'`
  - `completed_at = NULL`
  - `version = version + 1`
  - `updated_at = now()`
- `RowsAffected == 0` 时判断：
  1. `id` 不存在：404。
  2. `version` 不匹配：409 `EDIT_CONFLICT` + `current_version`。
  3. `status <> 'cancelled'`：保留当前业务 409，如 `code: not_cancelled`。
- 成功后返回完整最新 Requirement。

### 3.4 `DELETE /requirements/{id}`

预计修改：

- `api/handler/requirement.go`：`RequirementHandler.Delete`

改造点：

- query 传 `base_version`。
- 事务内先锁行：`SELECT version FROM requirements WHERE id = $1 FOR UPDATE`。
- 不存在：404。
- version 不等于 `base_version`：409 `EDIT_CONFLICT` + `current_version`。
- version 通过后再做现有关联检查：tasks / sessions / token_usage / documents。
- 若有关联，保留当前业务 409：`has_associations`。
- 无关联时删除 user_follows / requirement_teams / requirements。
- 删除成功返回 `{ "status": "deleted", "id": "..." }` 即可，前端随后刷新 query。

删除成功无需 `version + 1`，因为目标需求行已删除。

### 3.5 `PUT /tasks/{id}`

预计修改：

- `api/handler/task.go`：`TaskHandler.Update`
- `api/model/models.go`：`UpdateTaskRequest`

改造点：

注意：

### 3.8 `DELETE /tasks/{id}`

预计修改：

### 3.9 `POST /tasks/{id}/dependencies`

预计修改：

- `api/handler/task.go`：`TaskHandler.AddDependency`
- `api/model/models.go`：`AddDependencyRequest`

必须使用事务流程：

1. 读取请求体 `depends_on_id + base_version`。
2. 校验 `base_version > 0`。
3. 开启事务。
4. `SELECT version FROM tasks WHERE id = $taskID FOR UPDATE`。
5. 不存在：404。
6. 当前 version 不等于 `base_version`：409 `EDIT_CONFLICT` + `current_version`。
7. 校验依赖合法性。
8. `INSERT INTO task_dependencies (...) VALUES (...) ON CONFLICT DO NOTHING`。
9. 只有 insert rows affected > 0 时，执行：
   - `UPDATE tasks SET version = version + 1, updated_at = now() WHERE id = $taskID`
10. commit。
11. 返回完整最新 Task。

要求：

- 依赖已存在时，不 bump `tasks.version`。
- 依赖确实新增时，bump owner task 的 `tasks.version`。
- 不 bump 父需求 `requirements.version`。

### 3.10 `DELETE /tasks/{id}/dependencies/{dep_id}`

预计修改：

- `api/handler/task.go`：`TaskHandler.RemoveDependency`

DELETE query：

- `DELETE /tasks/{id}/dependencies/{dep_id}?base_version=123`

必须使用事务流程：

1. 从 query 解析 `base_version`。
2. 校验 `base_version > 0`。
3. 开启事务。
4. `SELECT version FROM tasks WHERE id = $taskID FOR UPDATE`。
5. 不存在：404。
6. 当前 version 不等于 `base_version`：409 `EDIT_CONFLICT` + `current_version`。
7. `DELETE FROM task_dependencies WHERE task_id = $1 AND depends_on_id = $2`。
8. 只有 delete rows affected > 0 时，执行：
   - `UPDATE tasks SET version = version + 1, updated_at = now() WHERE id = $taskID`
9. commit。
10. 返回完整最新 Task。

要求：

- 依赖本来不存在时，不 bump `tasks.version`。
- 依赖确实删除时，bump owner task 的 `tasks.version`。
- 不 bump 父需求 `requirements.version`。

- 任务新增不 bump 父需求 `requirements.version`。

### 4.6 成功响应必须更新本地 version

所有成功写接口如果目标实体仍存在，应返回完整实体并更新本地缓存：

- `PUT /requirements/{id}`：完整 Requirement。
- `POST /requirements/{id}/regenerate-ac`：完整 Requirement。
- `PUT /requirements/{id}/restore`：完整 Requirement。
- `PUT /tasks/{id}`：完整 Task。
- `PUT /tasks/{id}/status`：完整 Task。
- `PUT /tasks/{id}/progress`：完整 Task。
- `POST /tasks/{id}/dependencies`：完整 Task。
- `DELETE /tasks/{id}/dependencies/{dep_id}`：完整 Task。

目的：避免前端刚操作成功但本地仍持有旧 `version`，导致下一次操作自己触发 409。

乐观 UI 约束：

- 需求状态流转、任务状态流转、任务进度等如果使用乐观 UI，成功后必须用后端返回的完整 `Requirement` / `Task` 修正 query cache。
- 不能只在本地改 `status` / `progress` / `title` 等业务字段而忽略 `version`。
- 如果暂时不做精确 cache patch，也必须 invalidate/refetch 对应 query，确保下一次操作拿到最新 `version`。
- 失败回滚只恢复 UI，不改变“409 必须刷新 query”的规则。

物理删除接口例外：

- `DELETE /requirements/{id}`：删除后无最新 Requirement；前端移除 / 刷新。
- `DELETE /tasks/{id}`：删除后无最新 Task；前端移除 / 刷新。

### 4.7 统一处理 409

当前 `HttpError` 已有 `status`：

- `web/src/shared/request/types.ts`

建议在 `web/src/shared/request/apiError.ts` 增加判断函数：

- `isEditConflict(error)`：必须同时满足 `HttpError.status === 409` 且 payload `code === "EDIT_CONFLICT"`。
- 可选读取 `current_version`，但 P0 不展示详细版本号。

需求 / 任务页面 mutation 的 `onError`：

- 如果是并发 409，即 `status === 409 && code === "EDIT_CONFLICT"`：
  - 提示：`内容已被其他人更新，请刷新后再操作`
  - 刷新相关 query。
  - 不自动重试。
  - 不自动覆盖。
  - 不做 diff。
- 如果是业务 409，例如 `has_associations`、`not_cancelled`：
  - 不使用并发冲突提示。
  - 继续按原业务错误提示处理。
- 其他错误保持当前错误提示。

### 4.8 409 后 query 刷新策略

409 后最小刷新：

- 列表页：刷新 `requirements-board` 相关 query。
- 详情页：刷新当前 requirement 和 tasks query。
- 任务卡片操作：刷新 requirement list / tasks，确保拿到最新 `version`。

当前很多 mutation 成功后已经 invalidate，P0 需要把 409 的失败路径也做对应 invalidate。

## 5. 风险和测试用例

### 5.1 后端必须覆盖的并发测试

#### A/B 同时改同一需求字段

步骤：

1. 读取需求，得到 `version = 1`。
2. A 用 `base_version = 1` 修改 `title`，成功，返回完整 Requirement，`version = 2`。
3. B 仍用 `base_version = 1` 修改 `title`。

期望：

- B 返回 `409`。
- body 包含 `code: EDIT_CONFLICT`。
- body 包含 `current_version: 2`。
- 数据库保留 A 的修改。

#### A 改需求优先级，B 改需求标题

步骤：

1. A/B 都读取同一需求 `version = 1`。
2. A 修改 `priority`，成功到 `version = 2`。
3. B 修改 `title`，但提交整组表单，带旧 `priority` 和 `base_version = 1`。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 返回 `current_version = 2`。
- A 的 `priority` 不被覆盖。

#### A/B 同时改同一任务状态

步骤：

1. A/B 都读取任务 `version = 1`。
2. A 改为 `in_progress`，成功到 `version = 2`。
3. B 改为 `done`，仍用 `base_version = 1`。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 最终状态为 A 的结果。

#### A 改任务负责人，B 改任务优先级

步骤：

1. A/B 都读取任务 `version = 1`。
2. A 更新 `assignee_id`，成功到 `version = 2`。
3. B 更新 `priority`，仍用 `base_version = 1`。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- A 的负责人修改不被 B 的旧表单覆盖。
- 父需求 `requirements.version` 不变化。

#### A 删除任务，B 修改任务

两个时序都要测。

时序 1：A 先删。

1. A 用 `base_version = 1` 删除任务，成功。
2. B 用 `base_version = 1` 修改任务。

期望：

- B 返回 `404`，因为任务已不存在。
- 父需求 `requirements.version` 不因任务删除递增。

时序 2：B 先改。

1. B 用 `base_version = 1` 修改任务，成功到 `version = 2`。
2. A 用旧 `base_version = 1` 删除任务。

期望：

- A 返回 `409 EDIT_CONFLICT`。
- 返回 `current_version = 2`。
- 任务不被删除。

#### A 重新生成 AC，B 保存旧 AC

步骤：

1. A/B 都读取需求 `version = 1`。
2. A 调用重新生成 AC，AI 生成前检查通过。
3. A 写入 AC 时再次 `WHERE id AND version` 检查通过，成功到 `version = 2`。
4. B 保存旧编辑弹窗，payload 中带旧 `acceptance_criteria` 和 `base_version = 1`。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 返回 `current_version = 2`。
- A 生成的新 AC 不被覆盖。

补充测试：

1. A 开始 regenerate-ac，base version = 1。
2. 在 AI 生成期间 B 保存需求成功到 version = 2。
3. A 最终写入时命中最终 `WHERE id AND version` 保护。

期望：

- A 返回 `409 EDIT_CONFLICT`。
- 不覆盖 B 的修改。

#### regenerate-ac 生成前已过期

步骤：

1. A 读取需求 `version = 1`。
2. B 保存需求成功到 `version = 2`。
3. A 调用 regenerate-ac，仍传 `base_version = 1`。

期望：

- 后端在调用 AI 前直接返回 `409 EDIT_CONFLICT`。
- 不触发 AI 调用。

#### A 新增依赖，B 删除依赖

场景 1：依赖原本不存在。

1. A/B 都读取 owner task `version = 1`。
2. A 新增依赖，事务内 `SELECT version FOR UPDATE` 校验通过，insert rows affected > 0，任务 version 到 2。
3. B 用旧 `base_version = 1` 删除同一依赖。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 返回 `current_version = 2`。
- 依赖保持存在。

场景 2：依赖原本存在。

1. A/B 都读取 owner task `version = 1`。
2. A 删除依赖，delete rows affected > 0，任务 version 到 2。
3. B 用旧 `base_version = 1` 新增同一依赖。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 返回 `current_version = 2`。
- 依赖保持删除状态。

#### 依赖无实际变化不 bump version

新增已存在依赖：

1. 任务当前 `version = 1`，依赖 A 已存在。
2. 请求新增依赖 A，`base_version = 1`。

期望：

- 返回最新 Task。
- `version` 仍为 1。

删除不存在依赖：

1. 任务当前 `version = 1`，依赖 A 不存在。
2. 请求删除依赖 A，`base_version = 1`。

期望：

- 返回最新 Task。
- `version` 仍为 1。

#### 空更新返回 400

需求空更新：

1. 请求 `PUT /requirements/{id}`，body 只有 `{ "base_version": 1 }`。

期望：

- 返回 `400`。
- body 包含 `code: no_fields_to_update`。
- `requirements.version` 不递增。

任务空更新：

1. 请求 `PUT /tasks/{id}`，body 只有 `{ "base_version": 1 }`。

期望：

- 返回 `400`。
- body 包含 `code: no_fields_to_update`。
- `tasks.version` 不递增。

#### A/B 同时新增任务

步骤：

1. A/B 同时在同一需求下新增任务。
2. `POST /tasks` 不要求 `base_version`。

期望：

- 两个创建都成功。
- 不发生旧 task list 覆盖新 task list。
- 新建任务返回最新 Task 或前端后续 GET 拿到 `version = 1`。
- 父需求 `requirements.version` 不因任务新增递增。
- 父需求进度 / summary / risk 投影刷新后正确。

说明：

- P0 不阻止重复任务名。
- P0 不做“任务列表已变化”的创建前提醒。
