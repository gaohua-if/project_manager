# AIcoding 管理平台需求基线说明

版本：v0.1
状态：需求基线稿
目标：统一产品方向，避免后续设计和实现跑偏

---

## 1. 背景

当前 AIcoding 管理平台的需求方向重新收敛。

原先按不同角色分别设计复杂首页，容易导致页面职责不清、角色逻辑重复、控制台过重、产品方向发散。新的产品形态统一调整为：

> 每个账号登录后，核心只围绕两个页面展开：**控制台** 与 **需求看板**。

控制台负责聚合提醒、关注对象、风险项、日报入口。
需求看板负责承载真实工作，包括需求拆解、任务树、依赖关系、阻塞、deadline、验收标准和处理动作。

除核心页面外，系统可保留少量辅助页面，例如用户管理、个人角色信息、个人设置等。

---

## 2. 产品定位

本平台是一个围绕研发协作场景的轻量管理平台，核心解决：

1. 用户登录后能快速知道自己当前应关注什么；
2. 系统能聚合展示当前风险、阻塞、超期、缺失信息；
3. 用户能基于 session 生成、编辑、发送日报；
4. 需求可以拆解为任务树；
5. 任务之间可以存在依赖关系；
6. 任务阻塞、deadline、验收标准可以沉淀为结构化风险；
7. 不同角色看到的默认关注对象和数据范围不同，但不做四套独立产品。

一句话定义：

> 这是一个围绕“需求拆解、任务依赖、阻塞风险、日报生成”的轻量研发协作平台。

---

## 3. 设计原则

### 3.1 控制台只做聚合入口

控制台不承担复杂处理，不做重型筛选，不做完整工作流。

控制台主要负责：

* 提醒
* 聚合
* 摘要
* 跳转
* 日报入口

真正处理动作应跳转到需求看板或日报编辑视图完成。

### 3.2 需求看板是唯一事实源

需求、任务、关注状态、风险状态、阻塞状态、deadline、验收标准等核心工作数据，均以需求看板为准。

控制台展示的数据，来自需求看板和日报状态的聚合结果。

### 3.3 角色影响默认视图，不产生四套首页

不同角色登录后看到的默认关注对象、数据范围、日报周报范围不同，但控制台结构保持统一。

### 3.4 关注不是权限

关注只是用户个人订阅行为。

用户是否能查看某个需求或任务，由权限决定；是否出现在“我关注的”，由关注关系决定。

### 3.5 P0 优先保证闭环，不做复杂 Jira

P0 不追求完整 Jira 能力，不做复杂工作流配置、甘特图、关键路径算法、复杂依赖网络图。

P0 先完成：

* 控制台聚合
* 需求任务树
* 关注
* 风险
* 阻塞
* deadline
* 日报生成编辑发送

## 4. 核心页面

系统核心页面只有两个：

```text
核心页面
├─ 控制台
└─ 需求看板
```

辅助页面可包括：

```text
辅助页面
├─ 用户管理
├─ 个人角色信息
├─ 个人设置
└─ 日报 / 周报详情或编辑视图
```

辅助页面不应改变“控制台 + 需求看板”的主结构。

## 5. 页面一：控制台

### 5.1 页面定位

控制台是登录后的第一屏，回答三个问题：

1. 我现在应该关注什么？
2. 当前有没有风险？
3. 今天或本周的日报是否需要生成、编辑、发送？

### 5.2 控制台核心模块

```text
控制台
├─ 日报生成区
├─ 我关注的
├─ 风险项
└─ 日报 / 周报查看入口
```

### 5.3 日报生成区

所有角色都需要日报生成能力。

基础流程：

```text
选择 / 上传 session
        ↓
选择日报生成 Skill
        ↓
生成日报草稿
        ↓
人工编辑
        ↓
确认完成
        ↓
发送日报
```

P0 口径：

* 系统内置默认日报 Skill；
* 用户可以基于 session 生成日报草稿；
* 日报生成后可以编辑；
* 编辑完成后可以发送；
* 发送到飞书文档、飞书群或站内保存的最终方式待定；
* 飞书文档能力不作为 P0 强依赖，站内保存应作为兜底。

P1 口径：

* 支持用户上传自定义日报 Skill；
* 支持选择不同日报模板；
* 支持团队或部门维度日报周报汇总；
* 支持飞书文档深度集成。

### 5.4 我关注的

“我关注的”展示用户主动关注的需求或任务。

不同角色有默认关注对象，但用户可以关注任意有权限访问的需求或任务。

默认展示规则：

| 角色 | 默认关注对象  |
| -- | ------- |
| 总监 | 需求      |
| TL | 需求 + 任务 |
| PM | 需求      |
| 个人 | 任务      |

展示内容应支持需求和任务混合展示。

任务类型关注项必须展示所属需求，避免上下文丢失。

示例：

```text
我关注的
├─ 需求：AI 日报生成
│  ├─ 状态：进行中
│  ├─ deadline：2026-06-30
│  └─ 风险：2 个阻塞任务
│
├─ 任务：补充日报验收标准
│  ├─ 所属需求：AI 日报生成
│  ├─ 负责人：张三
│  └─ 状态：阻塞
│
└─ 任务：飞书发送联调
   ├─ 所属需求：日报发送
   ├─ 依赖：接口完成
   └─ deadline：明天
```

### 5.5 风险项

风险项是系统聚合出来的异常、阻塞、缺失、超时提醒。

风险项不是独立工作流，不建议在控制台做大量处理按钮。

风险项点击后应跳转到需求看板或日报详情定位处理。

风险来源包括：

```text
风险项来源
├─ AC 风险
│  ├─ 高关注需求缺少验收标准
│  ├─ 已进入执行阶段但缺少验收标准
│  └─ 临近 deadline 但缺少验收标准
│
├─ 阻塞风险
│  ├─ 任务被手动标记阻塞
│  ├─ 阻塞超过阈值
│  ├─ 高关注需求下存在阻塞任务
│  └─ 依赖的上游任务阻塞
│
├─ deadline 风险
│  ├─ 任务即将到期
│  ├─ 任务已超期
│  ├─ 需求即将到期但关键任务未完成
│  └─ 上游任务超期影响下游任务
│
└─ 日报风险
   ├─ 今日日报未生成
   ├─ 日报生成失败
   ├─ 日报草稿未发送
   └─ 组内成员日报缺失
```

### 5.6 日报 / 周报查看入口

总监和 TL 需要查看组或部门的日报周报。

查看范围：

| 角色 | 日报 / 周报查看范围              |
| -- | ------------------------ |
| 总监 | 部门 / 多个组                 |
| TL | 自己组                      |
| PM | 默认查看自己相关需求或个人日报，是否查看团队待定 |
| 个人 | 自己的日报                    |

P0 可先支持：

* 我的日报；
* 我负责范围内的日报；
* 日报状态概览。

P1 再支持：

* 周报汇总；
* 部门级汇总；
* 飞书文档同步；
* 多维度筛选统计。

## 6. 页面二：需求看板

### 6.1 页面定位

需求看板是系统的真实工作承载页面，类似轻量 Jira。

它负责管理：

* 需求；
* 任务；
* 子任务；
* 任务依赖；
* 阻塞；
* deadline；
* 验收标准；
* 关注状态；
* 风险定位与处理。

### 6.2 页面结构

```text
需求看板
├─ 需求列表
│  ├─ 需求名称
│  ├─ 状态
│  ├─ 负责人
│  ├─ deadline
│  ├─ 风险数量
│  ├─ 阻塞任务数量
│  └─ 关注状态
│
└─ 需求详情 / 抽屉 / 详情页
   ├─ 需求基础信息
   ├─ 需求验收标准
   ├─ 需求 deadline
   ├─ 任务树
   │  ├─ 任务
   │  ├─ 子任务
   │  ├─ 状态
   │  ├─ 负责人
   │  ├─ deadline
   │  ├─ 阻塞状态
   │  └─ 上游依赖
   └─ 风险 / 阻塞汇总
```

### 6.3 需求列表

需求列表应让用户一眼看清：

* 当前有哪些需求；
* 每个需求处于什么状态；
* 谁负责；
* deadline 是什么；
* 是否被关注；
* 是否存在风险；
* 是否存在阻塞任务；
* 点击后能展开或进入详情查看任务树。

### 6.4 任务树

需求可以拆解为任务。

任务可以继续拆解为子任务。

P0 建议任务树最多支持两层：

```text
需求
├─ 任务
│  └─ 子任务
└─ 任务
```

不建议 P0 支持无限层级，避免交互复杂度和数据维护复杂度过高。

### 6.5 任务依赖

任务之间可以存在依赖关系。

依赖关系用于表达：

```text
任务 A 依赖任务 B
任务 B 未完成 / 阻塞 / 超期
        ↓
任务 A 存在依赖风险或依赖阻塞
```

P0 建议：

* 支持同一需求内任务依赖；
* 支持一个任务依赖多个上游任务；
* 上游任务未完成时，下游任务展示“依赖未完成”；
* 上游任务阻塞时，下游任务展示“依赖阻塞”；
* 上游任务超期时，下游任务展示“依赖超期风险”。

P1 再考虑：

* 跨需求依赖；
* 依赖网络图；
* 关键路径分析；
* 自动排期。

## 7. 核心对象模型

### 7.1 需求 Requirement

需求是目标或交付物，是任务树的根节点。

核心字段：

| 字段       | 必填   | 说明                     |
| -------- | ---- | ---------------------- |
| 需求名称     | 是    | 需求标题                   |
| 需求描述     | 否    | 补充说明                   |
| 负责人      | 是    | 需求 owner               |
| 状态       | 是    | 当前推进状态                 |
| 验收标准 AC  | 否    | 需求级验收标准                |
| deadline | 否    | 需求截止时间                 |
| 关注状态     | 否    | 当前用户是否关注               |
| 风险数量     | 系统生成 | 由任务、阻塞、deadline、AC 等聚合 |
| 阻塞任务数量   | 系统生成 | 当前需求下阻塞任务数量            |
| 创建人      | 是    | 创建需求的人                 |
| 创建时间     | 是    | 系统记录                   |
| 更新时间     | 是    | 系统记录                   |

### 7.2 任务 Task

任务是需求拆解后的执行项，可以有子任务。

核心字段：

| 字段       | 必填 | 说明        |
| -------- | -- | --------- |
| 任务名称     | 是  | 任务标题      |
| 任务描述     | 否  | 补充说明      |
| 所属需求     | 是  | 任务归属的需求   |
| 父任务      | 否  | 子任务才有     |
| 负责人      | 是  | 任务 owner  |
| 状态       | 是  | 当前任务状态    |
| 验收标准 AC  | 否  | 任务级验收标准   |
| deadline | 否  | 任务截止时间    |
| 是否阻塞     | 否  | 是否被标记阻塞   |
| 阻塞原因     | 否  | 阻塞时需要填写   |
| 上游依赖任务   | 否  | 当前任务依赖的任务 |
| 关注状态     | 否  | 当前用户是否关注  |
| 创建人      | 是  | 创建任务的人    |
| 创建时间     | 是  | 系统记录      |
| 更新时间     | 是  | 系统记录      |

### 7.3 关注 Follow

关注用于记录用户主动关注的需求或任务。

核心字段：

| 字段           | 说明 |
| ------------ | -- |
| 用户           |    |
| 对象类型：需求 / 任务 |    |
| 对象 ID        |    |
| 创建时间         |    |

规则：

* 用户可关注有权限访问的需求或任务；
* 关注不等于权限；
* 取消关注不影响对象本身；
* 控制台“我关注的”来自关注关系聚合。

### 7.4 阻塞 Blocker

阻塞用于表达任务无法继续推进。

阻塞来源分为两类：

```text
阻塞来源
├─ 手动阻塞
│  ├─ 负责人主动标记阻塞
│  └─ 填写阻塞原因
│
└─ 依赖阻塞
   ├─ 上游任务未完成
   ├─ 上游任务阻塞
   └─ 上游任务超期
```

P0 以手动阻塞为主，依赖阻塞作为系统提示。

### 7.5 日报 Daily Report

日报由 session 和日报 Skill 生成。

核心字段：

| 字段         | 说明 |
| ---------- | -- |
| 日报标题       |    |
| 归属用户       |    |
| 归属日期       |    |
| 来源 session |    |
| 使用的 Skill  |    |
| 日报内容       |    |
| 状态         |    |
| 发送目标       |    |
| 创建时间       |    |
| 更新时间       |    |
| 发送时间       |    |

日报状态：

```text
未生成
生成中
草稿
已发送
发送失败
```

P0 至少支持：

```text
草稿
已发送
发送失败
```

## 8. 状态定义

### 8.1 需求状态

P0 可先采用简单状态：

```text
未开始
进行中
已完成
已暂停
已取消
```

### 8.2 任务状态

P0 可先采用简单状态：

```text
未开始
进行中
已完成
已暂停
已取消
```

任务是否阻塞不建议放进状态枚举，而应作为独立字段。

原因：

```text
进行中 + 阻塞
未开始 + 依赖未完成
进行中 + 已超期
```

这些组合状态比单一枚举更清楚。

### 8.3 风险状态

风险不建议作为用户手动维护字段，而应由系统基于数据聚合生成。

风险类型：

```text
AC 缺失风险
手动阻塞风险
依赖阻塞风险
deadline 即将到期风险
deadline 已超期风险
日报缺失风险
日报发送失败风险
```

### 9.1 角色列表

系统当前涉及角色：

```text
总监

PM
个人
```

### 9.2 角色默认关注对象

| 角色 | 默认关注对象  | 说明                   |
| -- | ------- | -------------------- |
| 总监 | 需求      | 关注需求整体进展、风险、部门/组日报周报 |
| TL | 需求 + 任务 | 关注组内需求推进、具体任务阻塞、负责人  |

### 9.3 角色差异

角色差异主要体现在：

```text
角色差异
├─ 默认关注对象不同
├─ 控制台默认数据范围不同
├─ 风险项排序权重不同
└─ 日报 / 周报查看范围不同
```

不应体现为四套独立首页。

## 10. 核心用户流程

### 10.1 登录后查看控制台

```text
用户登录
  ↓
进入控制台
  ↓
查看日报状态
  ↓
查看我关注的
  ↓
查看风险项
  ↓
点击关注项或风险项
  ↓
跳转到需求看板定位处理
```

### 10.2 生成日报

```text
用户进入控制台
  ↓
点击生成日报
  ↓
选择 / 上传 session
  ↓
选择默认日报 Skill
  ↓
系统生成日报草稿
  ↓
用户编辑日报
  ↓
用户确认完成
  ↓
发送日报
  ↓
日报状态变为已发送
```

### 10.3 查看并处理风险

```text
控制台展示风险项
  ↓
用户点击风险项
  ↓
跳转到需求看板
  ↓
定位到对应需求 / 任务
  ↓
展开任务树
  ↓
查看阻塞原因 / deadline / 依赖关系
  ↓
用户更新任务状态或阻塞信息
  ↓
风险项重新计算
```

### 10.4 需求拆解任务

```text
用户进入需求看板
  ↓
创建 / 选择需求
  ↓
填写需求基础信息
  ↓
拆解任务
  ↓
设置负责人、deadline、验收标准
  ↓
必要时设置任务依赖
  ↓
保存任务树
```

### 10.5 任务发生阻塞

```text
任务负责人进入任务详情
  ↓
标记任务阻塞
  ↓
填写阻塞原因
  ↓
系统将该任务计入风险项
  ↓
如果该任务被其他任务依赖
  ↓
下游任务展示依赖阻塞风险
```

## 11. 跳转规则

控制台不负责复杂处理，所有关注项和风险项都应可跳转定位。

```text
控制台
├─ 我关注的需求
│  └─ 点击 → 需求看板定位到该需求
│
├─ 我关注的任务
│  └─ 点击 → 需求看板定位到所属需求，并展开任务树定位该任务
│
├─ 风险项：任务阻塞
│  └─ 点击 → 展开对应任务，展示阻塞原因
│
├─ 风险项：依赖阻塞
│  └─ 点击 → 展示当前任务和上游依赖任务
│
├─ 风险项：deadline 风险
│  └─ 点击 → 定位到对应需求或任务
│
```

## 15. 关键产品口径

### 15.1 AC 是选填，但缺 AC 可以形成风险

验收标准 AC 字段本身是选填。

但在特定场景下，缺少 AC 可以被系统识别为风险。

建议规则：

```text
缺 AC 算风险的场景
├─ 高关注需求缺 AC
├─ 已进入执行阶段但缺 AC
├─ 临近 deadline 但缺 AC
└─ 任务已开始但缺少必要验收标准
```

普通草稿需求缺 AC 不一定算风险。

### 15.2 阻塞以手动为主，系统判断为辅

P0 阻塞来源：

```text
阻塞来源
├─ 用户手动标记阻塞
└─ 系统基于依赖关系给出依赖阻塞提示
```

不建议 P0 做复杂自动判定。

### 15.3 deadline 优先作用于任务

需求和任务都可以有 deadline。

风险判断中，任务 deadline 优先级更高。

需求 deadline 更适合表达整体交付时间，任务 deadline 更适合表达真实执行风险。

### 15.4 控制台不做工作流

控制台只显示摘要和入口。

处理动作回到需求看板或日报编辑视图完成。

### 15.5 任务必须有所属需求上下文

任何任务在控制台、风险项、关注列表中出现时，都必须展示所属需求。

## 17. 验收标准

### 17.1 控制台验收标准

* 用户登录后可以看到控制台；
* 控制台展示日报生成入口；
* 控制台展示当前用户关注的需求或任务；
* 控制台展示当前用户可见范围内的风险项；
* 关注项点击后可跳转到需求看板并定位；
* 风险项点击后可跳转到需求看板或日报编辑视图；
* 总监和 TL 可看到日报 / 周报查看入口；
* 控制台不出现复杂工作流和重型筛选。

### 17.2 需求看板验收标准

* 用户可以查看需求列表；
* 用户可以创建或编辑需求；
* 用户可以在需求下创建任务；
* 用户可以在任务下创建子任务；
* 任务可以设置负责人、状态、AC、deadline；
* 任务可以标记阻塞；
* 任务阻塞时可以填写阻塞原因；
* 任务可以设置同需求内上游依赖；
* 上游任务阻塞时，下游任务可以展示依赖阻塞提示；
* 用户可以关注或取消关注需求 / 任务；
* 需求列表可以展示风险数量和阻塞任务数量。

### 17.3 日报验收标准

* 用户可以从控制台进入日报生成流程；
* 用户可以选择或上传 session；
* 系统可以使用默认日报 Skill 生成日报草稿；
* 用户可以编辑日报草稿；
* 用户可以发送日报；
* 日报发送后状态变为已发送；
* 发送失败时状态变为发送失败；
* 日报记录可被后续查看。

## 18. 当前阶段结论

当前产品主线固定为：

```text
需求
  ↓ 拆解
任务树
  ↓ 存在依赖
阻塞 / deadline / AC 风险
  ↓ 聚合
控制台提醒
  ↓ 点击
回到需求看板处理
```

控制台负责：

```text
提醒
聚合
日报入口
跳转
```

需求看板负责：

```text
需求管理
任务拆解
依赖关系
阻塞处理
deadline 管理
验收标准管理
关注管理
```

日报负责：

```text
session 输入
Skill 生成
人工编辑
发送
状态记录
```

角色负责：

```text
默认视图
数据范围
日报周报查看范围
```

不再按角色设计四套独立首页。
