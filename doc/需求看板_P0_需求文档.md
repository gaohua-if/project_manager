# AIcoding 管理平台 - 需求看板 P0 需求文档

> 适用范围：需求看板 / 任务树 / 需求详情 / 任务详情 / 新建需求 / 添加任务 / Token 来源关联。  
> 不包含：控制台日报重构、日报任务进展卡、Token 明细页、权限后台、复杂项目管理能力。  
> 目标读者：Claude / Codex / 前端实现人员。  
> 当前版本：P0 可上线闭环。

---

## 1. 背景

AIcoding 管理平台需要提供一个轻量的需求推进视图，用于帮助 PM / TL / 总监 / 执行人员查看需求整体推进情况、任务拆解情况、依赖阻塞情况，以及和 AI 编码工作记录相关的 Token 消耗来源。

此前页面容易被设计成完整项目管理系统，包括复杂看板、依赖视图、Session 视图、人工阻塞、评论、日报联动等能力。P0 阶段不做大而全项目管理工具，只做一条最小可用闭环。

P0 核心闭环：

```text
创建需求
  ↓
补充验收标准
  ↓
需求进入看板阶段
  ↓
在需求详情中拆分任务
  ↓
任务设置负责人、截止日期、上游依赖
  ↓
任务推进进度
  ↓
上游未完成时下游任务依赖阻塞
  ↓
需求聚合任务完成情况、阻塞情况、Token 摘要
  ↓
需求 / 任务都可以关联 Token 来源
```

---

## 2. 产品目标

P0 目标不是完整替代 Jira / TAPD / ONES，而是解决 AIcoding 管理平台内部的几个实际问题：

1. **需求推进可见**  
   PM / TL 能快速看到需求处于哪个阶段、是否需要继续推进。

2. **需求可拆成任务**  
   一个需求可以拆成多个可执行任务，任务有负责人、进度、截止日期和上游依赖。

3. **依赖阻塞可见**  
   如果任务依赖的上游任务未完成，当前任务展示为依赖阻塞。

4. **Token 来源可归集**  
   Session 不是管理对象，只是 Token 消耗和工作证据来源。需求和任务都可以关联 Token 来源，用于查看 AI 编码投入。

5. **已完成不会污染主看板**  
   完成列保留，但只展示最近完成需求。历史完成需求通过筛选 / 归档查询。

## 3. P0 范围

### 3.1 做什么

P0 包含以下功能：

1. 需求看板
2. 任务树
3. 新建需求
4. 需求详情抽屉
5. 添加任务弹窗
6. 任务详情抽屉
7. 任务进度维护
8. 上游依赖与自动阻塞展示
9. 需求 / 任务关联 Token 来源
10. Token 来源明细展示
11. 已完成需求最近展示与历史查询

### 3.2 不做什么

P0 明确不做：

1. 控制台日报交互重构
2. 日报里的任务进展卡
3. LLM 自动更新任务进度
4. 人工标记阻塞
5. 阻塞原因填写
6. 评论 / 注释功能
7. 独立依赖视图
8. 独立 Session 视图
9. Session 数统计
10. 复杂风险记录页面
11. 甘特图 / 燃尽图
12. 字段级权限
13. 审批流
14. Token 去重分摊规则
15. 无限层级任务
16. 任务级验收标准强绑定
17. 已完成自动归档配置
18. 自定义完成保留天数

## 4. 核心对象

### 4.1 需求 Requirement

需求是业务目标或交付对象，是看板上的主对象。

需求包含：

| 字段 | 说明 |
|---|---|
| 标题 | 必填，需求名称 |
| 描述 | 必填，说明背景、目标和范围 |
| 阶段 | 待开始 / 评审 / 进行中 / 完成 |
| 优先级 | 高 / 中 / 低 |
| 截止日期 | 可选 |
| 参与团队 | 必填，支持一个或多个团队 |
| 创建人 | 系统记录 |
| 更新时间 | 系统记录 |
| 完成时间 | 进入完成阶段时记录 |
| 验收标准 | 需求级验收条件 |
| 任务列表 | 需求下拆分出的执行任务 |
| Token 来源 | 直接关联到需求的 Token 来源 |

### 4.2 任务 Task

任务是需求下的执行拆解。P0 只支持“需求 + 任务”两层结构，不做子任务。

任务包含：

| 字段 | 说明 |
|---|---|
| 任务标题 | 必填 |
| 所属需求 | 必填 |
| 负责人 | 必填 |
| 状态 | 待办 / 进行中 / 阻塞 / 完成 |
| 进度 | 0-100 数字 |
| 优先级 | 高 / 中 / 低 |
| 截止日期 | 可选 |
| 上游依赖 | 可选，仅支持同一需求内任务依赖 |
| Token 来源 | 关联到该任务的 Token 来源 |
| 更新时间 | 系统记录 |
| 完成时间 | 进入完成状态时记录 |

### 4.3 验收标准 Acceptance Criteria

统一文案为 **验收标准**。页面中不得出现 AC、任务级 AC、需求级 AC 等文案。

P0 验收标准属于需求级信息，不强制绑定任务。

验收标准用于说明需求完成需要满足的条件，例如：

```text
1. 用户可以创建需求并补充验收标准
2. 需求可以拆分为任务并设置负责人
3. 上游任务未完成时，下游任务展示依赖阻塞
4. Token 来源可以关联到需求或任务
```

### 4.4 Token 来源 Token Source

Token 来源来自已上报的 AI 编码工作记录 / Session。  
注意：**Session 不是页面核心管理对象，只是 Token 来源。**

Token 来源字段：

| 字段 | 说明 |
|---|---|
| 时间 | Session / 工作记录时间 |
| 工具 | Claude Code / Codex 等 |
| 上传人 | 来源上传人 |
| Token | 消耗 Token 数 |
| 摘要 | AI 或系统生成摘要 |
| 关联对象 | 需求或任务 |
| 关联类型 | 需求关联 / 来自任务 |

## 5. 状态模型

## 5.1 需求阶段

需求看板阶段固定为四列：

```text
待开始 / 评审 / 进行中 / 完成
```

| 阶段 | 含义 |
|---|---|
| 待开始 | 需求已创建，但尚未进入评审或执行 |
| 评审 | 正在确认范围、验收标准和拆分任务 |
| 进行中 | 已有任务进入执行，需求正在推进 |
| 完成 | 已完成交付或验收 |

### 已取消处理

已取消不是主看板阶段，不新增“已取消”看板列。  
已取消需求仅通过筛选或归档查询。  
顶部“有效需求”统计应排除已取消需求。

---

## 5.2 任务状态

任务状态建议：

```text
待办 / 进行中 / 阻塞 / 完成
```

| 状态 | 含义 |
|---|---|
| 待办 | 任务尚未开始 |
| 进行中 | 任务正在执行 |
| 阻塞 | 任务存在未完成上游依赖 |
| 完成 | 任务已完成 |

P0 不做人工阻塞。  
阻塞只由上游依赖自动产生。

---

## 5.3 依赖阻塞规则

P0 只支持同一需求内任务依赖。

规则：

```text
如果任务 A 依赖任务 B，且任务 B 未完成，则任务 A 展示为依赖阻塞。
```

展示规则：

1. 任务行展示阻塞标签。
2. 需求卡片展示“1 个依赖阻塞”等摘要。
3. 需求顶部统计中的“依赖阻塞”统计来自任务依赖关系。
4. 不提供“手动解除阻塞”。
5. 当上游任务完成后，下游任务的依赖阻塞自动解除。

---

## 5.4 进度规则

任务进度由用户手动维护。

规则：

1. 进度范围为 0-100。
2. UI 使用 Slider + InputNumber 联动。
3. 进度不依赖 Token 来源。
4. 没有 Token 来源的任务也可以更新进度。
5. 需求进度不单独手动维护，由任务聚合展示。
6. 需求卡片可以展示任务完成情况，但不提供需求级进度编辑。

任务完成建议：

```text
当任务状态改为完成时，进度应为 100%。
当进度改为 100% 时，可以提示用户是否同步状态为完成，但 P0 不强制。
```

## 6. 功能说明

## 6.1 需求看板

### 6.1.1 定位

需求看板用于查看需求整体阶段推进。

看板只展示摘要，不承载复杂编辑。

用户在看板中可以：

1. 查看需求分布。
2. 查看需求当前阶段。
3. 查看是否有依赖阻塞。
4. 查看任务完成情况。
5. 查看 Token 摘要小字。
6. 点击需求卡片打开需求详情抽屉。
7. 拖动需求卡片调整阶段。

### 6.1.2 看板列

固定四列：

```text
待开始 / 评审 / 进行中 / 完成
```

列副标题：

| 列 | 副标题 |
|---|---|
| 待开始 | 尚未进入执行 |
| 评审 | 确认范围与验收标准 |
| 进行中 | 任务正在推进 |
| 完成 | 最近完成 |

### 6.1.3 完成列展示规则

完成列必须保留，但不能展示全部历史完成需求。

规则：

1. 完成列只展示最近完成需求。
2. 默认展示最近 7 天内完成的需求。
3. 如果最近 7 天超过 10 条，只展示最近 10 条。
4. 如果没有 completedAt，可兼容使用 updatedAt。
5. 完成列空状态为：`暂无最近完成需求`。
6. 如果存在更多历史完成需求，完成列底部展示：`查看全部已完成`。
7. 点击 `查看全部已完成` 后，进入任务树 / 列表视图，并自动设置阶段筛选为“完成”。
8. P0 不做自定义保留天数，不做自动归档配置。

### 6.1.4 需求卡片展示

需求卡片主信息：

1. 标题
2. 描述一行
3. 优先级 / 风险标签
4. 任务完成情况，例如 `任务 1/2`
5. 依赖阻塞提示，例如 `1 个依赖阻塞`
6. 截止日期
7. Token 小字，仅有值时展示，例如 `Token 160K`

禁止展示：

1. Session 数，例如 `5 Session`
2. `0 Session`
3. `0 Token`
4. 大段说明文
5. 卡片内复杂编辑表单

### 6.1.5 未拆任务展示

如果需求尚未拆分任务：

1. 不显示 `0/0`。
2. 不显示 `0%`。
3. 显示 `未拆分任务`。
4. 提供轻量入口 `添加任务`。

---

## 6.2 任务树

### 6.2.1 定位

任务树用于展开需求，查看任务拆解与执行进度。

P0 只支持需求 + 任务两层。

### 6.2.2 表格列

建议列：

| 列 | 说明 |
|---|---|
| 需求 / 任务 | 展示需求行与任务行 |
| 负责人 / 团队 | 需求展示团队，任务展示负责人 |
| 状态 | 需求阶段或任务状态 |
| 进度 | 任务进度；需求行可展示任务完成情况 |
| 上游依赖 | 任务上游依赖数或依赖任务 |
| 最近更新 | 最近更新时间与 Token 小字 |
| 操作 | 查看详情 |

不建议将“进度 / 验收标准”混在同一列。  
验收标准主要在需求详情中查看。

---

## 6.3 新建需求

新建需求使用独立页面。

字段：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 标题 | 是 | 需求标题 |
| 描述 | 是 | 背景、目标、范围 |
| 验收标准 | 是 | 多行文本 |
| 优先级 | 是 | 高 / 中 / 低 |
| 截止日期 | 否 | 需求期望完成日期 |
| 飞书文档 | 否 | 外部文档链接 |
| 参与团队 | 是 | 一个或多个团队 |

创建成功后：

1. 不要只返回看板。
2. 提供明确下一步：
   - `继续拆分任务`
   - `返回看板`
3. 更推荐创建成功后打开需求详情抽屉，并聚焦执行拆解区域。

---

## 6.4 需求详情抽屉

### 6.4.1 定位

需求详情抽屉是需求日常查看与维护的主入口。

建议宽度：720px。

### 6.4.2 内容结构

需求详情抽屉包含：

1. 顶部标题区
2. 基础信息
3. 验收标准
4. 执行拆解
5. Token 摘要

### 6.4.3 基础信息

展示：

1. 创建者
2. 参与团队
3. 截止日期
4. 更新时间
5. 飞书文档
6. 阶段标签
7. 优先级标签

### 6.4.4 验收标准

展示需求验收标准列表。

文案统一：

```text
验收标准
```

不得出现 AC。

### 6.4.5 执行拆解

展示：

1. 任务总数
2. 已完成任务数
3. 依赖阻塞任务数
4. 任务摘要列表
5. 添加任务按钮

无任务时展示：

```text
该需求尚未拆分任务
拆分任务后可聚合进度、依赖阻塞和 Token。
[添加第一个任务]
```

添加任务按钮文案：

```text
添加任务
```

不建议写成“继续添加任务”作为唯一文案，除非已有任务时作为补充。

### 6.4.6 Token 摘要

展示：

1. 合计 Token
2. 关联 Token 来源按钮
3. Token 来源明细

文案：

```text
Token 摘要
合计 323K Token
关联 Token 来源
Token 来源明细
```

无 Token 来源时：

```text
暂无关联 Token 来源
```

禁止显示：

```text
0 Session
```

### 6.4.7 Token 汇总规则

需求 Token 汇总包括：

```text
需求直接关联 Token + 下属任务关联 Token
```

P0 不做复杂去重和成本分摊。

## 6.5 添加任务

### 6.5.1 交互形态

添加任务不使用独立页面作为主路径。

推荐：从需求详情抽屉点击 `添加任务`，打开居中的添加任务弹窗。

原因：任务依附于需求，不应让用户跳出需求上下文。

### 6.5.2 字段

添加任务弹窗字段：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 任务标题 | 是 | 任务名称 |
| 负责人 | 是 | 任务负责人 |
| 优先级 | 是 | 高 / 中 / 低 |
| 截止日期 | 否 | 任务截止日期 |
| 上游依赖 | 否 | 同一需求内可选任务 |

P0 不把关联验收标准作为主流程必填。  
如果保留，只能作为可选折叠项。

### 6.5.3 创建成功后

1. 弹窗关闭。
2. 停留在当前需求详情抽屉。
3. 执行拆解列表新增任务。
4. 需求卡片聚合信息更新。

---

## 6.6 任务详情抽屉

### 6.6.1 定位

任务详情抽屉用于维护任务执行信息。

建议宽度：640px。

### 6.6.2 内容结构

任务详情包含：

1. 基础信息
2. 状态与进度
3. 上游依赖
4. Token 来源

### 6.6.3 状态与进度

任务进度编辑：

1. 使用 Slider + InputNumber。
2. 支持 0-100。
3. 保存按钮：`保存进度`。
4. 不要求关联 Token 来源。
5. 没有 Token 来源也可以更新进度。

### 6.6.4 上游依赖

展示上游依赖任务。

如果上游未完成：

```text
依赖阻塞
```

P0 不提供人工阻塞、阻塞说明、评论。

### 6.6.5 Token 来源

任务详情展示：

1. 当前任务已关联 Token
2. 关联 Token 来源按钮
3. Token 来源明细
4. 移除关联操作

任务级 Token 自动汇总到所属需求。

---

## 6.7 关联 Token 来源

### 6.7.1 定位

Token 来源关联用于将已上传的 AI 编码工作记录归集到需求或任务。

注意：

```text
Session 只是 Token 来源，不是页面主对象。
```

### 6.7.2 入口

入口有两个：

1. 需求详情抽屉 → Token 摘要 → `关联 Token 来源`
2. 任务详情抽屉 → Token 来源 → `关联 Token 来源`

### 6.7.3 弹窗

弹窗标题：

```text
关联 Token 来源
```

说明文案：

```text
选择已上报的工作记录作为当前需求或任务的 Token 来源。
```

表格列：

| 列 | 说明 |
|---|---|
| 选择 | 多选 |
| 时间 | 工作记录时间 |
| 工具 | Claude Code / Codex |
| 上传人 | 来源上传人 |
| Token | Token 消耗 |
| 摘要 | 工作摘要 |

操作：

1. 选择一条或多条来源。
2. 点击 `关联`。
3. 关联成功后更新 Token 摘要。
4. 支持移除关联。

### 6.7.4 关联语义

1. 任务级关联优先，用于把 Token 归因到具体执行任务。
2. 需求级关联作为兜底，用于未拆任务、预研、会议、需求澄清等无法准确归到任务的工作。
3. 同一个 Token 来源可以被关联到需求或任务。
4. P0 不做复杂去重和成本分摊。
5. 没有 Token 来源不代表没有进展。

## 6.8 关注能力

关注是 P0 核心能力，不能降级为普通筛选。

### 6.8.1 关注需求

1. 需求看板卡片右上角展示星标按钮。
2. 需求详情抽屉标题旁展示星标按钮。
3. 点击星标可以关注 / 取消关注当前需求。
4. 星标只展示当前用户的关注状态。

### 6.8.2 关注任务

1. 任务树任务行展示星标按钮。
2. 任务详情抽屉标题旁展示星标按钮。
3. 点击星标可以关注 / 取消关注当前任务。
4. 星标只展示当前用户的关注状态。

### 6.8.3 「我关注的」筛选

1. 在需求看板和任务树当前页面提供「我关注的」筛选入口。
2. 启用后只展示当前用户关注的需求，或关注任务所属的需求。
3. 筛选状态通过 URL 参数 `favorite=1` 同步。
4. 不跳出当前页面，不另开关注管理页。

### 6.8.4 关注规则

1. 关注只对当前用户生效，不共享给他人。
2. 关注不改变需求 / 任务的阶段、状态、优先级、风险、权限。
3. 只有当前用户可见的需求 / 任务才能被关注。
4. 关注操作是幂等的：再次点击星标即取消关注。
5. 取消关注不删除任何业务数据。

### 6.8.5 首页关注列表关系

1. 关注关系作为首页「我的关注」列表的数据源。
2. 本期不修改首页，不实现首页的关注列表 UI，不调整首页代码。
3. 首页接入由后续迭代完成，基于同一份关注关系数据。

### 6.8.6 不做项

1. 不做关注通知 / 订阅消息。
2. 不展示关注人列表 / 关注人数。
3. 不做代为关注 / 给他人加关注。
4. 不做关注动态流。
5. 不做关注独立管理页面。

## 7. 权限口径

P0 只做基础权限口径，不做字段级权限和审批流。

建议：

| 操作 | 管理员 | 需求负责人 / PM | 任务负责人 | 普通成员 |
|---|---:|---:|---:|---:|
| 查看需求 | 是 | 是 | 是 | 是 |
| 创建需求 | 是 | 是 | 否 | 可配置 |
| 编辑需求阶段 | 是 | 是 | 否 | 否 |
| 补充验收标准 | 是 | 是 | 否 | 否 |
| 添加任务 | 是 | 是 | 可配置 | 否 |
| 设置任务依赖 | 是 | 是 | 否 | 否 |
| 更新任务进度 | 是 | 是 | 仅自己负责 | 否 |
| 关联 Token 来源 | 是 | 是 | 仅自己负责或参与 | 否 |
| 移除 Token 来源 | 是 | 是 | 仅自己负责或参与 | 否 |

P0 可先只在 UI 上做基础显隐和提示，后端权限按现有体系校验。

## 8. 关键边界规则

1. 需求看板只展示需求，不展示任务卡片。
2. 任务树用于查看需求下任务拆解。
3. 需求进度不手动维护，由任务聚合。
4. 任务进度由用户手动维护。
5. Token 来源不等于任务进度。
6. 没有 Token 来源不代表没有进展。
7. Session 数不作为指标展示。
8. 完成列只展示最近完成。
9. 已取消不进入主看板列。
10. 日报模块另行设计，不纳入本需求看板 P0。

## 9. 验收标准

### 9.1 主链路验收

1. 用户可以创建需求，并填写标题、描述、验收标准、优先级、截止日期、参与团队。
2. 创建成功后可以继续拆分任务。
3. 用户可以在需求详情中添加任务。
4. 任务可以设置负责人、优先级、截止日期和上游依赖。
5. 用户可以在任务详情中更新进度。
6. 上游任务未完成时，下游任务展示依赖阻塞。
7. 上游任务完成后，下游任务依赖阻塞解除。
8. 需求看板能聚合展示任务完成情况和依赖阻塞。
9. 需求和任务都可以关联 Token 来源。
10. 任务 Token 能汇总到需求 Token 摘要中。

### 9.2 看板验收

1. 看板列为待开始、评审、进行中、完成。
2. 已取消需求不进入主看板列。
3. 完成列只展示最近完成需求。
4. 历史完成需求可以通过“查看全部已完成”查询。
5. 未拆任务需求不显示 0/0 和 0%，而是显示“未拆分任务”。
6. 看板卡片不展示 Session 数。
7. 没有 Token 时不显示 0 Token。
8. 需求看板卡片右上角展示关注星标，点击可关注 / 取消关注。
9. 任务树任务行展示关注星标，点击可关注 / 取消关注。
10. 「我关注的」筛选启用后只展示当前用户关注的需求或关注任务所属的需求。

### 9.3 文案验收

1. 页面不出现 AC 字样。
2. 所有 AC 统一为验收标准。
3. 不展示 0 Session / N Session 作为主指标。
4. Token 相关文案统一为 Token 摘要、Token 来源、来源明细、关联 Token 来源。
5. 不出现“人工标记阻塞”“阻塞原因”“评论”“注释”等 P0 外功能。
