# 需求 / 任务字段校验规则设计

## 1. 背景

当前需求 / 任务模块已经具备基础的新增、编辑能力，但后端入参校验偏弱。

当前主要请求结构包括：

* 需求新增：`CreateRequirementRequest`
* 需求编辑：`UpdateRequirementRequest`
* 任务新增：`CreateTaskRequest`
* 任务编辑：`UpdateTaskRequest`

现有校验主要集中在少量必填字段：

* 需求：标题、描述、团队必填；
* 任务：需求、标题、负责人必填。

为了避免脏数据入库、无效 UUID 导致数据库转换异常、无效枚举污染状态，同时避免过度限制正常业务文本，本方案对需求 / 任务字段校验进行统一收敛。

---

## 2. 设计目标

### 2.1 目标

1. 统一需求 / 任务新增、编辑接口的字段校验规则。
2. 避免无效 UUID、无效日期、无效枚举打到数据库层后变成 500。
3. 保持业务文本输入足够宽松，不误伤中文、英文、标点、emoji、括号、斜杠等正常内容。
4. 区分基础入参校验和业务规则校验，避免 validator 过度膨胀。
5. 为前端表单错误定位提供明确字段名。

### 2.2 非目标

1. 不做复杂文本安全策略，不限制只能输入中文、英文、数字。
2. 不做敏感词、XSS、HTML 标签语义校验。
3. 不强制截止日期必须大于今天。
4. 不在基础校验层处理复杂业务权限。
5. 不引入过度复杂的校验框架。

---

## 3. 校验分层

字段校验分为两层。

### 3.1 基础入参校验

基础入参校验只处理字段格式和结构合法性，包括：

* trim；
* 必填非空；
* 字符长度；
* UUID 格式；
* 日期格式；
* 枚举值；
* 数组去空、去重、数量限制；
* 标题禁止换行和 tab；
* 控制字符过滤。

这部分应该在 handler 或统一 request validator 中完成。

### 3.2 业务规则校验

业务规则校验依赖数据库、权限、当前用户上下文，包括：

* team 是否存在；
* 当前用户是否有权限给指定 team 创建需求；
* requirement 是否存在；
* assignee 是否存在；
* assignee 是否属于合法团队范围；
* depends_on_ids 引用的任务是否存在；
* depends_on_ids 是否属于同一需求；
* depends_on_ids 是否产生循环依赖；
* 当前用户是否有权限引用或修改对应任务。

这部分应该放在业务逻辑中处理，不建议混入通用字段 validator。

---

## 4. 编辑接口字段语义

编辑接口必须能区分：

1. 字段未传：不修改；
2. 字段传空：按字段类型决定是报错还是清空；
3. 字段传非空：执行对应校验后更新。

因此，`UpdateRequirementRequest` 和 `UpdateTaskRequest` 中的可编辑字段建议使用 pointer 类型。

示例：

```go
type UpdateRequirementRequest struct {
    Title              *string   `json:"title"`
    Description        *string   `json:"description"`
    FeishuDocURL       *string   `json:"feishu_doc_url"`
    Priority           *string   `json:"priority"`
    Status             *string   `json:"status"`
    Deadline           *string   `json:"deadline"`
    AcceptanceCriteria *[]string `json:"acceptance_criteria"`
}
```

数组字段建议使用 `*[]string`：

* `nil`：字段未传，不修改；
* `[]`：显式清空；
* `["", " "]`：过滤空项后为空，等同清空；
* `["标准1", ""]`：过滤后保存 `["标准1"]`。

如果当前结构体无法区分“未传”和“传空”，需要先调整请求结构，否则编辑语义会不准确。

---

## 5. 通用校验规则

### 5.1 字符串 trim

所有字符串入库前统一执行 `strings.TrimSpace`。

规则：

* 新增必填字段：trim 后为空则报错；
* 编辑必填字段：字段传入后 trim 为空则报错；
* 编辑可空字段：字段传入后 trim 为空则清空，存 NULL；
* 字段未传：不修改。

校验顺序：

1. 先执行 trim；
2. 再检查长度、内部换行、内部 tab、控制字符等规则。

因此，首尾空白会被清理；只有 trim 后内容中仍包含非法字符时才报错。例如 `"\t需求标题\t"` 会保存为 `"需求标题"`，但 `"需求\t标题"` 应报错。

---

### 5.2 文本字符规则

标题、描述、验收标准不做“只能中文、英文、数字”的正则限制。

允许输入：

* 中文；
* 英文；
* 数字；
* 空格；
* 常用标点；
* 括号；
* 斜杠；
* emoji；
* `<`、`>`、`&` 等普通文本字符。

只拦截不可见控制字符：

```text
\x00-\x08
\x0B
\x0C
\x0E-\x1F
\x7F
```

字段差异：

| 字段类型                | 规则                 |
| ------------------- | ------------------ |
| title               | 不允许 `\r`、`\n`、`\t` |
| description         | 允许换行和 tab          |
| acceptance_criteria | 允许换行和 tab          |

建议将 `\r\n` 统一规范化为 `\n`。

---

### 5.3 字符长度统计

文本长度按 Unicode 字符数统计，不按 byte 统计。

Go 实现建议使用：

```go
utf8.RuneCountInString(s)
```

原因：

* 中文是多字节字符；
* emoji 是多字节字符；
* 如果使用 `len(s)`，会误伤正常中文输入。

URL 长度可以按字符串长度或 byte 长度控制，主要用于 2048 上限保护。

---

### 5.4 UUID 校验

以下字段必须校验 UUID 格式：

* `requirement_id`
* `assignee_id`
* `team_ids[]`
* `depends_on_ids[]`

建议使用宽松 UUID 正则：

```regex
^[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}$
```

说明：

* 不强制 UUID version；
* 只保证格式合法；
* 避免数据库 UUID 类型转换异常导致 500。

---

### 5.5 日期字段校验

日期字段：

* 需求：`deadline`
* 任务：`due_date`

规则：

1. 可空；
2. 空字符串表示清空；
3. 非空时必须是 `YYYY-MM-DD`；
4. 必须是真实日期；
5. 不强制大于今天；
6. 不强制晚于创建时间。

格式正则：

```regex
^\d{4}-\d{2}-\d{2}$
```

正则通过后，还需要用日期解析确认真实日期。

非法示例：

```text
2026-02-31
2026-13-01
2026-00-10
2026/01/01
```

---

### 5.6 URL 字段校验

字段：

* `feishu_doc_url`

规则：

1. 可空；
2. trim 后为空则存 NULL；
3. 最大长度 2048；
4. 非空时必须是 `http://` 或 `https://` URL；
5. URL 必须包含 host；
6. 不限制必须是飞书域名；
7. 不请求外部校验 URL 是否可访问。

Go 实现建议：

* 使用 `url.Parse`；
* 校验 `scheme`；
* 校验 `host` 非空。

合法示例：

```text
https://example.com/doc?id=1#section
http://example.com/doc/123
```

非法示例：

```text
javascript:alert(1)
www.example.com/doc
/doc/123
```

---

### 5.7 枚举字段校验

枚举字段必须严格校验。

#### 需求 status

允许值：

```text
todo
review
active
completed
cancelled
```

#### 需求 priority

允许值：

```text
low
medium
high
urgent
```

#### 任务 status

允许值：

```text
todo
in_progress
done
```

说明：

* `blocked` 是派生展示状态，表示任务被未完成的前置依赖阻塞；
* 用户不能手动把任务状态改成 `blocked`；
* 后端不应把 `blocked` 写入 `tasks.status`；
* 接口响应中可以通过 `display_status` 返回 `blocked` 供前端展示。

#### 任务 priority

允许值：

```text
low
medium
high
```

#### 空值规则

新增场景：

* `priority` 为空时默认 `medium`。

编辑场景：

* 字段未传：不修改；
* `priority` / `status` 一旦显式传入，trim 后不能为空；
* 传空字符串应报错；
* 不能把空字符串解释为默认值或不修改。

---

### 5.8 数组字段校验

适用字段：

* `team_ids`
* `acceptance_criteria`
* `depends_on_ids`

通用规则：

1. 每项 trim；
2. ID 数组中的空项报错；
3. 文本数组中的空项过滤；
4. ID 数组去重；
5. 数量不得超过上限；
6. 去重或过滤后，如果字段业务要求必填，则必须仍然至少保留 1 项。

#### acceptance_criteria 语义

| 输入            | 语义           |
| ------------- | ------------ |
| 字段未传          | 不修改          |
| `[]`          | 清空验收标准       |
| `["", "  "]`  | 过滤后为空，等同清空   |
| `["标准1", ""]` | 保存 `["标准1"]` |
| 超过 50 条       | 报错           |
| 单条超过 1000 字符  | 报错           |

---

## 6. 需求新增校验规则

请求字段：

```go
type CreateRequirementRequest struct {
    Title              string   `json:"title"`
    Description        string   `json:"description"`
    FeishuDocURL       *string  `json:"feishu_doc_url"`
    Priority           string   `json:"priority"`
    Deadline           *string  `json:"deadline"`
    TeamIDs            []string `json:"team_ids"`
    AcceptanceCriteria []string `json:"acceptance_criteria"`
}
```

字段规则：

| 字段                  | 规则                                                |
| ------------------- | ------------------------------------------------- |
| title               | 必填；trim 后 1-120 字符；不允许换行；不允许 tab；不允许控制字符          |
| description         | 必填；trim 后 1-10000 字符；允许换行和 tab；不允许控制字符            |
| feishu_doc_url      | 可空；trim 后为空则 NULL；最大 2048 字符；必须是 http 或 https URL |
| priority            | 可空；空则默认 medium；传值必须是 low / medium / high / urgent |
| deadline            | 可空；格式 YYYY-MM-DD；必须是真实日期                          |
| team_ids            | 必填；至少 1 个；每个必须是 UUID；去重后仍至少 1 个                   |
| acceptance_criteria | 可空；最多 50 条；空项过滤；每条 1-1000 字符；允许换行和 tab            |

业务校验：

* team 是否存在；
* 当前用户是否有权限给对应 team 创建需求；
* 多 team 场景下是否符合当前需求权限设计。

---

## 7. 需求编辑校验规则

请求字段建议：

```go
type UpdateRequirementRequest struct {
    Title              *string   `json:"title"`
    Description        *string   `json:"description"`
    FeishuDocURL       *string   `json:"feishu_doc_url"`
    Priority           *string   `json:"priority"`
    Status             *string   `json:"status"`
    Deadline           *string   `json:"deadline"`
    AcceptanceCriteria *[]string `json:"acceptance_criteria"`
}
```

字段规则：

| 字段                  | 规则                                                              |
| ------------------- | --------------------------------------------------------------- |
| title               | 未传不改；传入后 trim 必须 1-120 字符；不允许换行；不允许 tab                         |
| description         | 未传不改；传入后 trim 必须 1-10000 字符                                     |
| feishu_doc_url      | 未传不改；传空则清空；非空必须是 http 或 https URL，最大 2048                       |
| priority            | 未传不改；传入后不能为空；必须是 low / medium / high / urgent                   |
| status              | 未传不改；传入后不能为空；必须是 todo / review / active / completed / cancelled |
| deadline            | 未传不改；传空则清空；非空必须是合法 YYYY-MM-DD                                   |
| acceptance_criteria | 未传不改；传入后最多 50 条；空项过滤；每条 1-1000 字符                               |

说明：

* 当前需求编辑不处理 `team_ids`；
* `deadline` 和 `feishu_doc_url` 如果传空字符串，应在规范化后写入 NULL，而不是把空字符串直接入库；
* 如果后续支持需求团队变更，需要单独设计权限和数据迁移规则。

## 8. 任务新增校验规则

请求字段：

```go
type CreateTaskRequest struct {
    RequirementID      string   `json:"requirement_id"`
    Title              string   `json:"title"`
    AcceptanceCriteria []string `json:"acceptance_criteria"`
    AssigneeID         *string  `json:"assignee_id"`
    Priority           string   `json:"priority"`
    DueDate            *string  `json:"due_date"`
    DependsOnIDs       []string `json:"depends_on_ids"`
}
```

字段规则：

| 字段                  | 规则                                       |
| ------------------- | ---------------------------------------- |
| requirement_id      | 必填；必须是 UUID                              |
| title               | 必填；trim 后 1-120 字符；不允许换行；不允许 tab；不允许控制字符 |
| acceptance_criteria | 可空；最多 50 条；空项过滤；每条 1-1000 字符             |
| assignee_id         | 必填；必须是 UUID                              |
| priority            | 可空；空则默认 medium；传值必须是 low / medium / high |
| due_date            | 可空；格式 YYYY-MM-DD；必须是真实日期                 |
| depends_on_ids      | 可空；最多 50 个；每个必须是 UUID；不允许空项；去重           |

业务校验：

* requirement 是否存在；
* 当前用户是否有权限在该 requirement 下创建任务；
* assignee 是否存在；
* assignee 是否属于合法团队范围；
* depends_on_ids 对应任务是否存在；
* depends_on_ids 是否属于同一 requirement；
* depends_on_ids 是否形成循环依赖。

## 9. 任务编辑校验规则

请求字段建议：

```go
type UpdateTaskRequest struct {
    Title              *string   `json:"title"`
    AcceptanceCriteria *[]string `json:"acceptance_criteria"`
    AssigneeID         *string   `json:"assignee_id"`
    Status             *string   `json:"status"`
    Priority           *string   `json:"priority"`
    DueDate            *string   `json:"due_date"`
    Progress           *int      `json:"progress"`
}
```

字段规则：

| 字段                  | 规则                                                     |
| ------------------- | ------------------------------------------------------ |
| title               | 未传不改；传入后 trim 必须 1-120 字符；不允许换行；不允许 tab                |
| acceptance_criteria | 未传不改；传入后最多 50 条；空项过滤；每条 1-1000 字符                      |
| assignee_id         | 未传不改；传空报错；非空必须是 UUID                                   |
| status              | 未传不改；传入后不能为空；必须是 todo / in_progress / done；不允许 blocked |
| priority            | 未传不改；传入后不能为空；必须是 low / medium / high                   |
| due_date            | 未传不改；传空则清空；非空必须是合法 YYYY-MM-DD                          |
| progress            | 未传不改；传入后必须是 0-100                                      |

说明：

* 当前任务编辑接口不直接处理 `depends_on_ids`；
* 任务依赖通过单独的新增 / 删除依赖接口维护；
* 如果后续需要在任务编辑接口中支持“批量替换依赖”，应先扩展 `UpdateTaskRequest` 和后端 replace-all 逻辑，再复用第 10 节的依赖数组校验规则。

业务校验：

* 当前用户是否有权限编辑该任务；
* 非本人任务是否允许编辑；
* assignee 是否存在。

---

## 10. depends_on_ids 校验细则

`depends_on_ids` 需要分为基础校验和业务校验。

当前 P0 接口语义：

* 任务新增接口支持通过 `depends_on_ids` 初始化依赖；
* 任务编辑接口暂不支持直接提交 `depends_on_ids` 批量替换依赖；
* 已存在任务的依赖关系通过单独的新增依赖、删除依赖接口维护；
* 如果后续新增“编辑任务时批量替换依赖”能力，应复用本节规则，并额外处理 replace-all 的事务一致性。

### 10.1 基础入参校验

规则：

1. 字段未传：新增任务时表示不设置依赖；未来任务编辑批量替换依赖时表示不修改依赖；
2. 传空数组：新增任务时表示不设置依赖；未来批量替换依赖时表示清空依赖；
3. 每个 ID trim；
4. 空 ID 报错；
5. 每个 ID 必须是 UUID；
6. 去重；
7. 最多 50 个；
8. 编辑任务时，不允许包含当前任务自身 ID。

### 10.2 业务规则校验

规则：

1. 每个依赖任务必须存在；
2. 每个依赖任务必须与当前任务属于同一个 requirement；
3. 不允许形成循环依赖；
4. 当前用户必须有权限引用这些依赖任务。

---

## 11. team_ids 校验细则

### 11.1 基础入参校验

规则：

1. 需求新增时必填；
2. 每个 ID trim；
3. 空 ID 报错；
4. 每个 ID 必须是 UUID；
5. 去重；
6. 去重后仍至少 1 个。

### 11.2 业务规则校验

规则：

1. team 必须存在；
2. 当前用户必须有权限在指定 team 下创建需求；
3. 多团队需求必须符合当前权限设计。

---

## 12. 错误返回规范

### 12.1 P0 采用格式

P0 建议先采用单字段轻量格式，减少对当前前端和 handler 的改造范围：

```json
{
  "error": "validation failed",
  "field": "title",
  "message": "title must be 1-120 characters"
}
```

要求：

* 每次返回一个优先级最高的字段错误；
* `field` 必须对应请求字段名；
* `message` 面向开发和前端定位，后续可由前端映射为更友好的中文提示。

### 12.2 推荐格式

如果后续需要一次性返回多个字段错误，可以升级为结构化字段错误：

```json
{
  "error": "validation failed",
  "fields": {
    "title": "标题不能为空，且不能超过 120 个字符",
    "deadline": "截止日期格式必须为 YYYY-MM-DD",
    "priority": "优先级必须是 low、medium、high 或 urgent"
  }
}
```

优点：

* 前端可以定位到具体表单字段；
* 多字段错误可以一次性展示；
* 后续扩展更稳定。

### 12.3 兼容格式

如果当前项目暂时不适合统一错误结构，可以先使用轻量格式：

```json
{
  "error": "validation failed",
  "field": "title",
  "message": "title must be 1-120 characters"
}
```

### 12.4 最小格式

如果只保持当前简单错误风格，至少保证错误信息包含字段名：

```json
{
  "error": "title must be 1-120 characters"
}
```

### 12.5 HTTP 状态码

字段校验失败统一返回：

```text
400 Bad Request
```

不要把数据库转换错误、枚举错误、日期解析错误暴露成 500。

---

## 13. 推荐实现方式

### 13.1 建议新增 validator helper

建议抽出轻量 helper，不要在 handler 中堆大量重复逻辑。

示例 helper：

```go
func trimPtr(s *string) *string
func validateRequiredText(field, value string, min, max int, allowMultiline bool) error
func validateOptionalText(field string, value *string, max int, allowMultiline bool) (*string, error)
func validateUUID(field, value string) error
func validateOptionalUUID(field string, value *string) error
func validateDateYYYYMMDD(field string, value *string) (*string, error)
func validateEnum(field, value string, allowed map[string]bool) error
func normalizeStringList(values []string, maxItems int, maxItemLen int) ([]string, error)
func normalizeUUIDList(field string, values []string, maxItems int) ([]string, error)
```

### 13.2 不建议引入重型校验框架

P0 不建议引入复杂校验框架，原因：

1. 当前字段数量有限；
2. 业务语义较多，框架标签表达能力有限；
3. 编辑接口需要精确处理“未传”和“传空”；
4. 手写 helper 更容易控制错误信息和边界行为。

---

## 14. 落地优先级

第一期建议按以下顺序落地：

1. 调整编辑接口字段语义，确保能区分“未传”和“显式传空”。
2. 所有字符串统一 trim。
3. 必填字段 trim 后非空校验。
4. UUID 格式校验，避免无效 ID 打到数据库后变 500。
5. priority / status 枚举校验。
6. deadline / due_date 日期格式和真实日期校验。
7. 文本长度限制，按 Unicode 字符数统计。
8. acceptance_criteria 数量和单项长度限制。
9. team_ids / depends_on_ids 数组去重、空项处理、数量限制。
10. 补充 team、requirement、assignee、depends_on_ids 的业务存在性和权限校验。

## 16. 验收标准

完成后应满足：

1. 需求新增接口对 title、description、team_ids、priority、deadline、feishu_doc_url、acceptance_criteria 有明确校验。
2. 需求编辑接口能区分字段未传和显式传空。
3. 任务新增接口对 requirement_id、title、assignee_id、priority、due_date、depends_on_ids、acceptance_criteria 有明确校验。
4. 任务编辑接口能区分字段未传和显式传空。
5. 非法 UUID 不再进入数据库转换流程。
6. 非法日期不再进入数据库日期字段。
7. 非法 status / priority 不允许入库。
8. title 不允许换行和 tab。
9. description / acceptance_criteria 允许正常多行文本。
10. 文本长度按 Unicode 字符数统计，不误伤中文和 emoji。
11. blocked 不允许作为任务 status 直接写入。
12. 所有校验失败返回 400，不返回 500。
13. 错误信息至少包含字段名，方便前端定位。
14. 不引入过重的校验框架，不扩大 P0 产品范围。

---

## 17. 结论

本方案作为 P0 字段校验基准是可行的。

它保留了业务输入的灵活性，同时补齐了以下关键风险：

1. 无效 UUID 导致数据库层 500；
2. 无效日期入库；
3. 非法枚举污染状态；
4. 编辑接口无法区分未传和传空；
5. title 换行、tab 破坏列表展示；
6. acceptance_criteria、depends_on_ids 等数组字段缺少数量和空项控制。

第一期按“基础入参校验优先、业务校验逐步补齐”的方式落地即可，不需要引入过度复杂的规则。
