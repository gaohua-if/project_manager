## 1. 背景与结论

日报 / 周报模块的新产品方向是从“用户选择来源和 Skill 后生成”调整为“对应报告 Agent 自动生成，用户查看、修订、确认，已生成内容可被上级汇总使用”。

当前分支已经具备以下基础：

1. 日报三层链路：个人日报、小组日报、部门日报。
2. 周报三层链路：个人周报、小组周报、部门周报。
3. 个人周报表 `personal_weekly_reports`、小组周报表 `team_weekly_reports`、部门周报表 `department_weekly_reports`。
4. 外部 Managed Agent client、AI 资产页、`ai_runs` 运行记录、定时任务 `managed_agent_schedules`、日报 MCP。
5. 个人日报 Managed Agent run 预留接口：`POST /reports/today/managed-agent-runs`、`GET /reports/managed-agent-runs/{runId}`。

但当前代码仍未完整实现新产品设计要求：

1. 报告主流程仍以旧的 report generator / daemon 生成接口为主。
2. 个人日报弹窗仍保留 session 选择、Skill 上传等旧心智。
3. 周报生成仍要求用户选择来源，尚未切换为 Agent 自动取数。
4. 报告状态只覆盖 `saved` / `submitted` / 部分归档语义，尚未覆盖“AI 已生成、本人已确认、本人已修改、手写报告、生成失败、Agent 未配置”等新状态。
5. 汇总来源仍主要依赖“提交 / 发送”语义，没有完全切换为“已生成即可汇总，确认只影响可信度”。

因此本设计采用渐进式架构：保留现有接口和生成链路可用性，同时新增统一的 Report Agent Action 层作为日报周报 Agent 化的长期主入口。Agent 相关接口可以先预留和部分实现，不要求一次性完成外部平台、定时生成、状态回写的全部闭环。

需要特别明确：日报周报 Agent 化后，Aida 当前服务不只是“调用 Agent 的客户端”，还需要作为 **Report MCP Server** 对 Agent 暴露业务工具。Agent 通过 MCP 读取日报 / 周报上下文、来源完整度、可信度分布，并通过 MCP 将生成结果回写到 Aida。前端和业务 API 仍以 Aida 数据库为准，Managed Agent 平台不保存日报周报事实源。

## 2. 设计目标

### 2.1 产品目标

1. 六类报告统一成“查看 / 编辑 / 生成 / 确认 / 手写”心智。
2. 移除报告弹窗中的 Skill 选择、Skill 上传、手动选择生成器等旧心智。
3. Agent 成功生成即形成可用报告，可参与上级汇总。
4. 用户确认是可信度标记，不作为上级汇总的强阻塞条件。
5. Agent 不可用时，保留手写兜底。
6. 汇总报告展示来源完整度和可信度分布。

### 2.2 技术目标

1. 前端报告页只触发业务动作，不直接拼 Managed Agent 平台接口。
2. 后端通过 Action Registry 解析报告生成动作、权限、上下文、Prompt、运行时和输出处理器。
3. Aida 服务提供 Report MCP Server，供 Agent 在运行时读取上下文和回写报告。
4. Managed Agent 平台只作为外部运行时，Aida 保留报告事实源和权限事实源。
5. `ai_runs` 记录所有 Agent 运行，报告表记录业务结果和当前可信度状态。
6. 当前 report generator / daemon 链路作为过渡 fallback，避免未合并 Agent 代码阻塞业务。
7. 接口设计允许先返回“运行已创建 / 生成中”，再通过轮询、状态同步或 MCP 回写刷新报告。

## 3. 领域模型

### 3.1 报告类型

| 报告类型 | 周期 | 归属 | 主要使用角色 | 上级汇总 |
| --- | --- | --- | --- | --- |
| 我的日报 | 天 | 用户 | Employee / TL / PM / Director / Admin | 小组日报或部门日报 |
| 小组日报 | 天 | Team | TL | 部门日报 |
| 部门日报 | 天 | 部门 | Director / Admin | 最终报告 |
| 我的周报 | 周 | 用户 | Employee / TL / PM / Director / Admin | 小组周报或部门周报 |
| 小组周报 | 周 | Team | TL | 部门周报 |
| 部门周报 | 周 | 部门 | Director / Admin | 最终报告 |

PM 作为独立个人来源，不进入 TL 小组范围；PM 的个人日报 / 周报如需进入部门汇总，应直接作为部门独立个人来源参与，而不是挂到某个小组下。

### 3.2 业务实体

现有表与目标职责：

| 表 | 当前职责 | 目标职责 |
| --- | --- | --- |
| `daily_reports` | 个人日报正文、保存 / 提交状态、session 来源、Managed Agent 字段 | 我的日报事实源，承载 AI 生成状态、确认状态、手写状态、运行引用 |
| `team_reports` | 小组日报正文、成员日报来源、保存 / 提交状态 | 小组日报事实源，承载来源完整度、可信度摘要、运行引用 |
| `department_reports` | 部门日报正文、小组日报来源、保存 / 归档状态 | 部门日报事实源，承载部门汇总结果、来源完整度、可信度摘要 |
| `personal_weekly_reports` | 个人周报正文、日报来源、保存 / 提交状态 | 我的周报事实源 |
| `team_weekly_reports` | 小组周报正文、个人周报来源、提交时间 | 小组周报事实源，需要补齐状态和运行引用 |
| `department_weekly_reports` | 部门周报正文、小组周报来源、归档时间 | 部门周报事实源，需要补齐状态和运行引用 |
| `ai_runs` | Managed Agent 运行记录 | 所有报告 Agent Action 的运行审计和状态索引 |
| `managed_agent_schedules` | 用户级 Agent 定时任务 | 过渡能力；长期应收敛到 Action subscription / schedule policy |

## 4. 状态模型

### 4.1 产品状态

六类报告统一使用以下产品状态：

| 产品状态 | 含义 | 是否可汇总 | 建议存储语义 |
| --- | --- | --- | --- |
| 未生成 | 当前周期没有报告内容 | 否 | 无报告记录，或记录 `status = null / missing` |
| 生成中 | Agent run 已创建，未完成 | 否 | `ai_runs.status in pending/running`，报告记录可不存在 |
| AI 已生成 | Agent 成功生成报告，用户未确认 | 是 | `status = ai_generated`，`generation_mode = managed_agent` |
| 本人已确认 | 用户确认 AI 内容可用 | 是 | `status = confirmed` |
| 本人已修改 | 用户编辑保存过内容 | 是 | `status = edited` 或 `edited = true` + `status = saved` |
| 手写报告 | 用户跳过 Agent 直接手写 | 是 | `status = manual`，`generation_mode = manual` |
| 生成失败 | Agent run 失败或超时 | 否 | `ai_runs.status in failed/timeout`，报告记录可不存在 |
| Agent 未配置 | 找不到对应 Action 或 Agent | 否 | 接口计算态，不一定落库 |

当前代码只支持部分 `saved` / `submitted` / `archived` 状态。为了兼容已有页面和数据，不建议直接删除旧状态，而是增加一个面向新产品的状态计算层：

```text
ReportProductStatus = f(report_record, ai_run, generation_mode, edited, saved/submitted fields)
```

对前端暴露时建议返回 `product_status`、`trust_level`、`usable_for_rollup`，而不是要求前端理解所有历史字段。

### 4.2 可信度等级

汇总报告需要能展示来源可信度。建议后端统一计算：

| 可信度 | 来源状态 | 说明 |
| --- | --- | --- |
| `edited` | 本人已修改 | 人工修订过，可信度最高 |
| `confirmed` | 本人已确认 | 用户认可 AI 内容 |
| `manual` | 手写报告 | 人工直接填写 |
| `ai_generated` | AI 已生成 | 可用但未确认 |
| `missing` | 未生成 / 失败 / 未配置 | 不参与正文汇总，只参与缺失提示 |

汇总完整度口径：

```text
可用来源 = AI 已生成 + 本人已确认 + 本人已修改 + 手写报告
缺失来源 = 未生成 + 生成失败 + Agent 未配置且未手写
```

## 5. 总体架构

### 5.1 目标链路

```text
前端报告弹窗
  -> POST /api/v1/report-actions/{action_key}/runs
  -> Report Action Registry
  -> 权限校验
  -> Prompt / Params Renderer
  -> Managed Agent Client
  -> ai_runs
  -> Managed Agent Runtime
  -> Agent 调用 Aida Report MCP
     -> report_get_context / report_get_sources / report_save_generated
  -> 轮询 / 状态同步 / MCP 回写确认
  -> Report Output Handler
  -> daily_reports / team_reports / department_reports / weekly tables
  -> 前端刷新报告详情、来源完整度、Dashboard
```

说明：Context Provider 不再只是 Action Runner 内部函数。目标形态下，它主要沉淀为 MCP tools 背后的服务能力，供 Agent 按需调用。Action Runner 只传递最小启动参数，例如报告类型、周期、用户身份、运行 id 和输出约束。

### 5.2 模块职责

| 模块 | 职责 | 当前代码基础 |
| --- | --- | --- |
| Report 页面 | 展示报告、触发生成、手写、编辑、确认、查看来源完整度 | `ReportsPage.tsx`、`WeeklyReportsPage.tsx`、`DailyReportGenerateModal.tsx` |
| Report API | 查询 / 保存 / 确认报告事实源 | `api/handler/report.go` |
| Report Action Runner | 统一触发报告 Agent Action | 待新增，可参考 `AgentAction触发架构与实现方案.md` |
| Report MCP Server | 向 Agent 暴露上下文读取、来源摘要、报告回写工具 | 当前已有 `DailyReportMCPHandler`，需扩展到六类报告 |
| Context Provider | MCP tool 背后的上下文装配能力，按报告类型和权限组装上下文 | 当前散落在 `report.go`、`daily_report_mcp.go`、`daemon/server_reports.go` |
| Managed Agent Client | 调外部平台创建任务、查询结果 | `api/service/managed_agent.go` |
| Run Store | 保存本地运行记录 | `ai_runs`、`managed_agent.go` |
| Output Handler | 校验 Agent 输出并写入报告表 | 日报已有 MCP `saveDailyReportDraft`，周报待补 |
| Scheduler | 定时触发报告生成 | 当前有 `ManagedAgentScheduleRunner`，长期应改为 Action schedule |

### 5.3 为什么需要 Report Action 层

报告页面不应该直接选择 Agent、Skill、MCP 或外部平台任务接口。原因：

1. 六类报告共享生成、运行、状态、回写、失败处理逻辑。
2. 不同角色的权限和上下文不同，必须由 Aida 后端统一判断。
3. 外部 Managed Agent 平台是运行时，不应成为业务事实源。
4. Agent 运行时通过 Aida MCP 读取和写入报告业务数据，页面不能直接绕过 MCP 约束。
5. 定时生成、手动生成、失败重试应复用同一套业务 Action。
6. 当前 Agent 代码未完整合入，Action 层可以先预留，用 report generator fallback 承接。

## 6. Report Action 设计

### 6.1 Action Key

建议定义六个业务 Action：

| action_key | 报告类型 | 触发方式 | 权限 |
| --- | --- | --- | --- |
| `report.daily.personal.generate` | 我的日报 | manual / schedule | self |
| `report.daily.team.generate` | 小组日报 | manual / schedule | team_leader |
| `report.daily.department.generate` | 部门日报 | manual / schedule | director / admin |
| `report.weekly.personal.generate` | 我的周报 | manual / schedule | self |
| `report.weekly.team.generate` | 小组周报 | manual / schedule | team_leader |
| `report.weekly.department.generate` | 部门周报 | manual / schedule | director / admin |

后续如果需要“确认”“重试”“重新生成”，不建议都建成 Agent Action；确认和手写是报告业务写操作，生成 / 重试才是 Agent Action。

### 6.2 Action Definition

P0 可先用 Go 代码或 YAML 注册，不需要后台配置页。

```yaml
action_key: report.weekly.team.generate
surface: reports
period_type: weekly
report_scope: team
permission_scope: team_leader
trigger_types: [manual, schedule]
context_provider: team_weekly_report_context
managed_agent_ref:
  required: false
  fallback: report_generator
output_handler: save_team_weekly_ai_report
```

`managed_agent_ref.required = false` 表示当前阶段允许 Agent 未配置时 fallback 到现有 report generator，或直接返回 Agent 未配置并允许用户手写。不同环境可以通过配置控制。

### 6.3 Action Run 状态

接口返回应以本地 `ai_runs` 为准：

| 状态 | 说明 |
| --- | --- |
| `pending` | 本地运行记录已创建，等待外部任务 |
| `running` | 外部任务执行中 |
| `succeeded` | 外部任务完成且输出已处理 |
| `failed` | 外部任务失败或输出不合法 |
| `timeout` | 超时 |

`ai_runs.business_type` 建议使用六类稳定值：

```text
daily_personal_report
daily_team_report
daily_department_report
weekly_personal_report
weekly_team_report
weekly_department_report
```

`ai_runs.business_id` 在输出落库后指向对应报告记录 id。

## 7. 上下文设计

### 7.1 上下文原则

1. Context Provider 必须在 Aida 后端执行权限过滤。
2. Context Provider 优先通过 MCP tool 对 Agent 暴露，而不是把完整上下文塞进启动 prompt。
3. Agent 只能看到当前用户有权使用的报告事实。
4. 汇总类报告不要求所有来源确认，但需要携带来源状态和可信度。
5. Prompt 中明确要求不得编造未提供的任务、阻塞、计划或成员信息。
6. 输出必须是结构化 JSON，正文放入 `report_markdown`。

### 7.2 六类上下文

| 报告 | 上下文来源 | 当前代码基础 | 目标调整 |
| --- | --- | --- | --- |
| 我的日报 | 本人当日 session、关联任务、需求、token 信息 | `loadDraftSessions`、`DailyReportMCPHandler.getDailyReportContext` | Agent 自动取数，前端不再手选 session |
| 小组日报 | 小组成员可用个人日报、TL 本人日报、缺失成员 | `GetTeamReportSources` | 来源扩大到 AI 已生成 / 确认 / 修改 / 手写，而不只看 submitted |
| 部门日报 | 可用小组日报、独立个人日报、缺失来源 | `GetDepartmentReportSources` | 增加 PM / 独立个人来源口径 |
| 我的周报 | 本周个人可用日报 | `GetPersonalWeeklyReportSources` | Agent 自动取本周可用日报，前端不再强制选择 |
| 小组周报 | TL 本人和成员可用个人周报、缺失成员 | `GetTeamWeeklyReportSources` | 从 submitted-only 改为 usable-for-rollup |
| 部门周报 | 可用小组周报、独立个人周报、缺失来源 | `GetDepartmentWeeklyReportSources` | 增加独立个人周报来源 |

### 7.3 汇总来源摘要

汇总接口应返回以下摘要，供前端展示：

```json
{
  "source_total": 10,
  "usable_count": 8,
  "missing_count": 2,
  "trust_summary": {
    "edited": 2,
    "confirmed": 3,
    "manual": 1,
    "ai_generated": 2,
    "missing": 2
  }
}
```

## 8. Report MCP 设计

### 8.1 MCP 在架构中的位置

Report MCP 是 Agent 访问 Aida 报告业务数据的唯一工具入口。它解决三个问题：

1. Agent 不直接访问 Aida 内部 HTTP 业务接口，避免绕过权限、状态和来源口径。
2. Agent 不需要前端传入完整上下文，只需要通过 MCP 按报告类型读取事实。
3. Agent 生成结果通过 MCP 回写，Aida 可以统一校验输出、关联 `ai_runs`、更新报告表和来源快照。

目标链路：

```text
Action Runner 创建 ai_runs
  -> Managed Agent Task 启动，携带 report_type / scope / period / run_id
  -> Agent 调用 Aida Report MCP
     -> aida_report_get_context
     -> aida_report_get_sources
     -> aida_report_save_generated
  -> Aida 写报告事实源
  -> 前端轮询 run 或刷新报告详情
```

### 8.2 MCP Endpoint

当前已有：

```http
POST /api/v1/mcp/daily-report
```

目标建议新增或演进为：

```http
POST /api/v1/mcp/reports
```

兼容策略：

1. P0 保留 `/mcp/daily-report`，只承载个人日报。
2. P1 新增 `/mcp/reports`，覆盖日报 / 周报六类报告。
3. `/mcp/daily-report` 后续可以作为 `/mcp/reports` 的兼容包装。

### 8.3 MCP 鉴权

MCP 请求必须带 Aida 用户身份，不能只信任外部 Agent 平台的 task id。

推荐鉴权方式：

1. Managed Agent 平台在调用 MCP 时带 Aida 用户 token。
2. Aida MCP 复用现有 `AuthMiddleware` 的用户解析能力。
3. MCP tool 内部再次按 `report_type`、`report_scope`、`period` 做业务权限校验。
4. `run_id` / `external_task_id` 只用于运行关联和审计，不作为授权依据。

这对产品的影响是：Agent “自动取数”仍严格等价于当前用户在 Aida 里能看到的数据，不会因为 Agent 拥有平台能力而越权读取小组或部门报告。

### 8.4 Tool 列表

建议 Report MCP 暴露以下工具：

| Tool | 职责 | 适用范围 |
| --- | --- | --- |
| `aida_report_get_context` | 获取生成某类报告所需的完整上下文 | 六类报告 |
| `aida_report_get_sources` | 获取来源列表、完整度、可信度分布 | 小组 / 部门汇总，个人周报 |
| `aida_report_save_generated` | 保存 Agent 生成结果，状态为 `ai_generated` | 六类报告 |
| `aida_report_get_current` | 获取当前周期已有报告内容和状态 | 六类报告 |
| `aida_report_get_output_contract` | 获取结构化输出协议和写作约束 | 六类报告 |

P0 可以只实现最小三件套：

```text
aida_report_get_context
aida_report_save_generated
aida_report_get_output_contract
```

### 8.5 Tool 参数

通用参数：

```json
{
  "report_type": "daily|weekly",
  "report_scope": "personal|team|department",
  "period_date": "2026-06-29",
  "period_start": "2026-06-29",
  "run_id": "uuid",
  "external_task_id": "task_xxx"
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `report_type` | 日报或周报 |
| `report_scope` | 我的 / 小组 / 部门 |
| `period_date` | 日报日期 |
| `period_start` | 周报周起始日期 |
| `run_id` | Aida 本地 `ai_runs.id`，用于关联输出 |
| `external_task_id` | Managed Agent 平台 task id，用于审计 |

### 8.6 Context 返回结构

`aida_report_get_context` 返回结构建议统一：

```json
{
  "report": {
    "type": "weekly",
    "scope": "team",
    "period_start": "2026-06-29",
    "period_end": "2026-07-05",
    "current_status": "missing"
  },
  "actor": {
    "id": "123",
    "name": "张三",

    "team_id": "team_uuid"
  },
  "sources": {
    "source_total": 8,
    "usable_count": 6,
    "missing_count": 2,
    "trust_summary": {
      "edited": 1,
```

```json
{
```

### 8.7 结果回写

`aida_report_save_generated` 请求：

```json
{
  "report_type": "weekly",
  "report_scope": "team",
  "period_start": "2026-06-29",
  "run_id": "uuid",
  "external_task_id": "task_xxx",
  "agent_id": "agent_xxx",
  "model_id": "Kimi-K2.6",
  "report_markdown": "...",
  "source_ids": ["uuid1", "uuid2"],
  "source_snapshot": {}
}
```

服务端回写规则：

1. 校验调用者是否有权写对应报告。
2. 校验 `report_markdown` 非空，且只引用可用来源。
3. upsert 当前周期报告。
4. 设置 `generation_mode = managed_agent`。
5. 设置产品状态为 `ai_generated`。
6. 关联 `managed_agent_run_id` / `agent_id` / `model_id`。
7. 写入来源 id 和来源快照。
8. 更新 `ai_runs.business_id`、`output_ref_json` 和终态。

### 8.8 对产品交互的影响

服务提供 MCP 后，产品交互需要同步收敛：

1. 前端不再要求用户选择 session、日报、周报作为生成输入；来源选择最多作为高级调整能力。
2. “Agent 自动生成”不是前端把材料传给 Agent，而是 Agent 通过 MCP 自己拉取有权限的上下文。
3. 报告弹窗应展示 MCP 计算出的来源完整度和可信度，而不是展示“已发送 / 未发送”作为唯一依据。
4. 生成失败可能来自 Agent 运行失败，也可能来自 MCP 鉴权、上下文为空、输出校验失败；错误信息需要区分。
5. 手写、编辑、确认仍走普通报告业务接口，不需要通过 MCP。

## 9. 接口设计

### 9.1 当前可保留接口

当前接口继续保留，用于兼容已实现页面和 fallback：

| 能力 | 接口 |
| --- | --- |
| 个人日报草稿 | `POST /api/v1/reports/today/draft` |
| 个人日报 Managed Agent run | `POST /api/v1/reports/today/managed-agent-runs` |
| 查询个人日报 Managed Agent run | `GET /api/v1/reports/managed-agent-runs/{runId}` |
| 个人周报生成预览 | `POST /api/v1/reports/weekly/mine/current/generate` |
| 小组周报生成预览 | `POST /api/v1/reports/team/weekly/current/generate` |
| 部门周报生成 | `POST /api/v1/reports/department/weekly/current/generate` |

这些接口不必立即删除，但前端新弹窗应逐步切到统一 Action 接口。

### 9.2 新增统一 Action 接口

#### 查询可用 Action

```http
GET /api/v1/report-actions?surface=reports&period=daily|weekly
```

返回：

```json
{

```

```http
{
  "period_date": "2026-06-29",
  "period_start": "2026-06-29",
  "trigger_source": "manual",

```

返回：

```json
{
  "run_id": "uuid",
  "action_key": "report.weekly.personal.generate",
  "status": "pending",
  "report": null
}
```

如果 P0 采用同步 fallback，也允许直接返回报告：

```json
{
```

### 9.3 报告业务接口补充

#### 保存手写或编辑

现有 `PUT` 保存接口可继续使用，但需要能表达来源：

```json
{
  "content": "...",
  "save_mode": "manual|edit"
}
```

服务端规则：

1. 无 Agent 生成记录时保存，状态为 `manual`。
2. 已有 AI 生成记录后修改保存，状态为 `edited`。
3. 兼容旧字段时可继续写 `status = saved`、`edited = true`。

#### 确认报告

建议新增确认接口，而不是复用提交：

```http
POST /api/v1/reports/{report_scope}/{id}/confirm
```

或按现有路由拆分：

```http
POST /api/v1/reports/{id}/confirm
POST /api/v1/reports/team/{id}/confirm
POST /api/v1/reports/department/{id}/confirm
POST /api/v1/reports/weekly/mine/current/confirm
POST /api/v1/reports/team/weekly/{id}/confirm
POST /api/v1/reports/department/weekly/{id}/confirm
```

确认只改变可信度，不改变“是否可汇总”。是否保留“发送给上级”按钮由产品决定；如果保留，发送应代表通知 / 快照，不再代表汇总资格。

## 10. 数据模型调整建议

### 10.1 最小兼容方案

在不大改表结构的前提下，先增加计算字段和少量列：

1. 对所有报告详情 / 列表接口增加响应字段：
   - `product_status`
   - `trust_level`
   - `usable_for_rollup`
   - `generation_mode`
   - `managed_agent_run_id`
2. 已有 `daily_reports` 已具备 `generation_mode`、`managed_agent_run_id`、`agent_id`、`agent_version_id`、`model_id`。
3. 周报和汇总表建议补齐同类字段。

### 10.2 推荐新增列

对 `team_reports`、`department_reports`、`personal_weekly_reports`、`team_weekly_reports`、`department_weekly_reports` 补齐：

```sql
generation_mode TEXT NOT NULL DEFAULT 'manual',
managed_agent_run_id UUID REFERENCES ai_runs(id),
agent_id TEXT,
agent_version_id INTEGER,
model_id TEXT,
confirmed_at TIMESTAMPTZ,
confirmed_by BIGINT REFERENCES users(id)
```

对周报汇总表补齐状态：

```sql
status TEXT CHECK (
  status IS NULL OR status IN (
    'ai_generated',
    'confirmed',
    'edited',
    'manual',
    'saved',
    'submitted',
    'archived'
  )
)
```

是否保留 `submitted_at` / `submitted_content`：

1. 可保留，作为“发送给上级时的快照”。
2. 不再作为上级汇总唯一条件。
3. 汇总查询应改用 `usable_for_rollup` 状态判断。

## 11. 前端架构

### 11.1 页面结构

日报和周报可以继续保留两个路由：

```text
/reports
/reports/weekly
```

但弹窗结构应统一为：

```text
标题区：报告类型、日期 / 周期、产品状态
状态区：Agent 状态、来源完整度、可信度分布
正文区：Markdown 编辑器 / 预览
来源区：可折叠来源列表，只展示不强制选择
操作区：Agent 生成 / 重新生成 / 保存 / 确认 / 手写
```

### 11.2 前端状态流

```text
打开弹窗
  -> 查询当前报告
  -> 查询来源摘要
  -> 查询可用 Action
  -> 如果已有内容：展示正文和状态
  -> 如果无内容：展示“生成”与“手写”

点击生成
  -> POST report-actions/{action_key}/runs
  -> status=pending/running 时轮询 run
  -> succeeded 后刷新当前报告、来源摘要、Dashboard
  -> failed 后展示失败原因，保留手写入口

编辑保存
  -> PUT 当前报告接口
  -> 刷新当前报告、上级受影响来源查询、Dashboard

确认
  -> POST confirm
  -> 刷新当前报告、上级受影响来源查询、Dashboard
```

### 11.3 查询刷新范围

任何日报 / 周报变更都要刷新受影响查询：

| 变更 | 必须刷新 |
| --- | --- |
| 我的日报 | 我的日报详情 / 列表、Dashboard、小组日报来源、部门独立个人来源 |
| 小组日报 | 小组日报详情 / 列表、Dashboard、部门日报来源 |
| 部门日报 | 部门日报详情 / 列表、Dashboard |
| 我的周报 | 我的周报详情 / 列表、Dashboard、小组周报来源、部门独立个人来源 |
| 小组周报 | 小组周报详情 / 列表、Dashboard、部门周报来源 |
| 部门周报 | 部门周报详情 / 列表、Dashboard |

## 12. 定时生成

### 12.1 目标

定时生成不应绕过业务 Action。

```text
schedule tick
  -> 找到订阅用户和 action_key
  -> RunReportAction(action_key, trigger_source=schedule)
```

### 12.2 当前代码过渡

当前 `ManagedAgentScheduleRunner` 直接读取 `managed_agent_schedules` 并提交外部 Managed Agent task，适合作为过渡能力，但长期存在两个问题：

1. 它不知道报告业务 Action 和输出处理器。
2. 它只保存 `ai_runs`，不天然保证报告表状态回写。

建议 P0 先保留 `managed_agent_schedules`，但新增约定：

```json
{
  "action_key": "report.daily.personal.generate",
  "period_strategy": "today"
}
```

Runner 发现 `action_key` 后走 Report Action Runner；没有 `action_key` 的旧 schedule 继续按现有逻辑执行。

## 13. MCP 与输出回写

本章保留对当前实现和演进路径的简要说明；详细工具协议以第 8 章 Report MCP 设计为准。

### 13.1 当前日报 MCP

当前 `DailyReportMCPHandler` 已提供：

1. `aida_daily_report_get_context`
2. `aida_daily_report_save_draft`

`save_draft` 可以将 Managed Agent 输出写入 `daily_reports`，并关联 `ai_runs`。

### 13.2 目标扩展

长期建议统一为 Report MCP：

```text
aida_report_get_context
aida_report_save_generated
```

参数中携带：

```json
{
  "report_type": "daily|weekly",
  "report_scope": "personal|team|department",
  "period_date": "2026-06-29",
  "period_start": "2026-06-29"
```

P0 不强制一次性实现完整通用 MCP。可先按优先级扩展：

3. 先实现个人日报 / 个人周报的 get context 和 save generated。
4. 再实现小组 / 部门报告的 sources、trust summary 和 source snapshot。

## 14. 分期落地

### 14.1 P0：接口预留与状态计算

目标：不破坏现有日报周报功能，先让新产品状态和 MCP contract 可被 Agent / 前端消费。

1. 新增状态计算函数，输出 `product_status`、`trust_level`、`usable_for_rollup`。
2. 报告详情 / 列表 / 来源接口返回上述字段。
3. 新增或预留 `/api/v1/mcp/reports`，定义 tools/list、鉴权、通用参数和输出协议。
4. 新增 `report-actions` 路由骨架，可先只支持个人日报 Agent run。
5. 保留旧生成接口作为 fallback。
6. 前端弹窗逐步隐藏 Skill 上传和生成器选择。

### 14.2 P1：个人日报 / 个人周报 Agent 化

1. 我的日报切到 `report.daily.personal.generate`。
2. 我的周报切到 `report.weekly.personal.generate`。
3. Agent 通过 Report MCP 读取个人上下文并回写报告。
4. Agent 成功输出后写入报告表，状态为 `ai_generated`。
5. 保存 / 手写 / 确认状态闭环。
6. Dashboard 使用真实报告状态。

### 14.3 P2：小组 / 部门汇总 Agent 化

1. 小组日报 / 小组周报按可用来源汇总，不要求全部确认。
2. 部门日报 / 部门周报增加独立个人来源，覆盖 PM 口径。
3. Report MCP 为汇总类报告返回来源完整度、可信度分布和缺失来源。
4. 汇总报告保存来源快照和可信度摘要。
5. 汇总生成失败不影响手写。

### 14.4 P3：定时生成与治理

1. 定时任务统一走 Report Action Runner。
2. 提供用户级开关：是否自动生成日报 / 周报。
3. 管理员可配置默认 Action、默认 Agent、模型和 schedule policy。
4. `managed_agent_schedules` 逐步收敛到 Action subscription。

## 15. 兼容与迁移策略

1. 历史 `saved` 报告默认映射为 `manual` 或 `edited`，具体取决于 `generation_mode` 和 `edited`。
2. 历史 `submitted` 报告映射为可汇总，可信度至少为 `confirmed`。
3. 没有 `generation_mode` 的周报默认按 `manual` 处理。
4. 旧的 report generator 接口保留到六类 Report Action 全部可用后再下线。
5. 前端先隐藏旧 Skill 心智，不必立即删除后端兼容字段。

## 16. 风险与待确认

1. `submitted` 是否继续作为“发送给上级”的产品动作需要确认；本设计建议它不再决定汇总资格。
2. PM 独立个人来源进入部门日报 / 周报的具体展示位置需要产品确认。
3. 个人日报是否完全取消 session 手选，还是保留“来源明细可调整”的高级入口，需要产品确认。
4. Agent 输出 JSON contract 需要统一，否则 Output Handler 难以稳定解析。
5. 周报和汇总表缺少 Managed Agent 运行引用字段，落地前需要迁移补齐。
6. 当前 `managed_agent_schedules` 是通用 Agent 定时任务，不是报告 Action 定时任务；迁移期间需要避免重复生成。
7. Report MCP 的鉴权必须复用 Aida 当前账号体系，不能引入远程分支中的 AIHub SSO 口径污染。
8. 如果 Agent 通过 MCP 回写报告，需要明确重复回写、重新生成、用户已编辑后再回写的覆盖规则。
