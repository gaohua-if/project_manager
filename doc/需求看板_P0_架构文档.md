# AIcoding 管理平台 - 需求看板 P0 架构文档

> 适用范围：前端页面结构、组件拆分、状态模型、数据流、交互边界。  
> 目标读者：Claude / Codex / 前端实现人员。  
> 目标：在现有代码基础上实现 P0 需求看板闭环，避免重写和过度设计。

---

## 1. 架构目标

本模块不是完整项目管理系统，而是 AIcoding 管理平台中的一个需求推进模块。

架构目标：

1. 保持现有平台布局和路由体系。
2. 在现有需求看板页面内完成 P0 功能闭环。
3. 保持模块职责清晰，避免控制台日报、Token 明细、组织管理等模块污染。
4. 前端交互以“主页面 + 右侧抽屉 + 弹窗”为主。
5. 支持后续接入真实 API，但 P0 可以先基于现有 mock / service 数据模型实现。

## 5. 数据模型

以下为前端建议模型，字段名可按现有代码适配。

## 5.1 Requirement

```ts
export type RequirementStage = 'todo' | 'review' | 'in_progress' | 'done' | 'cancelled';
export type Priority = 'high' | 'medium' | 'low';

export interface Requirement {
  id: string;
  title: string;
  description: string;
  stage: RequirementStage;
  priority: Priority;
  ownerId?: string;
  ownerName?: string;
  teams: string[];
  dueDate?: string;
  feishuUrl?: string;
  acceptanceCriteria: string[];
  createdBy: string;
  createdAt: string;
  updatedAt: string;
  completedAt?: string;
  cancelledAt?: string;
  taskIds: string[];
  directTokenSourceIds: string[];
}
```

说明：

1. `cancelled` 不是主看板列。
2. `completedAt` 用于完成列最近展示。
3. 无 `completedAt` 时可用 `updatedAt` 兼容。
4. `directTokenSourceIds` 是直接挂到需求的 Token 来源。

---

## 5.2 Task

```ts
export type TaskStatus = 'todo' | 'in_progress' | 'blocked' | 'done';

export interface Task {
  id: string;
  requirementId: string;
  title: string;
  ownerId: string;
  ownerName: string;
  status: TaskStatus;
  priority: Priority;
  progress: number; // 0-100
  dueDate?: string;
  upstreamTaskIds: string[];
  tokenSourceIds: string[];
  createdAt: string;
  updatedAt: string;
  completedAt?: string;
}
```

说明：

1. P0 只支持同一需求内上游任务依赖。
2. `blocked` 可以是计算展示状态，也可以是保存状态，但展示必须以依赖规则为准。
3. `progress` 由用户维护，不由 Token 来源自动计算。

export interface TokenSource {

3. P0 不做复杂去重和成本分摊。

## 5.4 FavoriteTarget

关注关系表，用于记录用户对需求 / 任务的关注。

```ts
export type FavoriteTargetType = 'requirement' | 'task';

export interface FavoriteTarget {
  id: string;
  userId: string;
  targetType: FavoriteTargetType;
  targetId: string;
  createdAt: string;
}
```

后端表结构建议 `favorite_targets`：

```sql
CREATE TABLE favorite_targets (
  id           BIGSERIAL PRIMARY KEY,
  user_id      BIGINT       NOT NULL,
  target_type  VARCHAR(16)  NOT NULL CHECK (target_type IN ('requirement','task')),
  target_id    VARCHAR(64)  NOT NULL,
  created_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  UNIQUE (user_id, target_type, target_id)
);
CREATE INDEX idx_favorite_targets_user ON favorite_targets (user_id, target_type);
```

说明：

1. 如已有同义关注表（如 `user_favorites`），优先复用；只新增本表所列字段或索引，不重复建表。
2. `target_type` 仅 `requirement` 与 `task` 两种。
3. `(user_id, target_type, target_id)` 唯一约束保证幂等。
4. 取消关注通过物理删除该行实现，不留软删除标记。
5. 仅当前用户可见的对象才能成为关注目标，鉴权在写入前由 handler 校验。
6. 关注关系不影响阶段、状态、优先级、风险、权限的计算。
7. 同一份 `favorite_targets` 关系将作为首页「我的关注」列表的数据源，本期不修改首页。

## 6. 派生数据与计算规则

## 6.1 有效需求

```ts
const activeRequirements = requirements.filter(r => r.stage !== 'cancelled');
```

顶部“有效需求”排除已取消。

## 6.2 看板分组

```ts
const boardStages = ['todo', 'review', 'in_progress', 'done'];
```

已取消不进主看板列。

## 6.3 完成列最近展示

```ts
function getRecentDoneRequirements(requirements: Requirement[]) {
  const done = requirements.filter(r => r.stage === 'done');

  const sorted = done.sort((a, b) => {
    const at = a.completedAt || a.updatedAt;
    const bt = b.completedAt || b.updatedAt;
    return new Date(bt).getTime() - new Date(at).getTime();
  });

  const sevenDaysAgo = Date.now() - 7 * 24 * 60 * 60 * 1000;
  const inSevenDays = sorted.filter(r => {
    const t = new Date(r.completedAt || r.updatedAt).getTime();
    return t >= sevenDaysAgo;
  });

  return inSevenDays.slice(0, 10);
}
```

判断是否存在历史完成：

```ts
const allDoneCount = requirements.filter(r => r.stage === 'done').length;
const recentDoneCount = getRecentDoneRequirements(requirements).length;
const hasMoreDone = allDoneCount > recentDoneCount;
```

## 6.4 任务阻塞计算

```ts
function isTaskDependencyBlocked(task: Task, allTasks: Task[]) {
  return task.upstreamTaskIds.some(id => {
    const upstream = allTasks.find(t => t.id === id);
    return upstream && upstream.status !== 'done';
  });
}
```

展示状态：

```ts
function getTaskDisplayStatus(task: Task, allTasks: Task[]): TaskStatus {
  if (task.status === 'done') return 'done';
  if (isTaskDependencyBlocked(task, allTasks)) return 'blocked';
  return task.status;
}
```

## 6.5 需求聚合

```ts
function getRequirementTaskSummary(requirementId: string, tasks: Task[]) {
  const list = tasks.filter(t => t.requirementId === requirementId);
  const total = list.length;
  const done = list.filter(t => t.status === 'done').length;
  const blocked = list.filter(t => getTaskDisplayStatus(t, list) === 'blocked').length;

  return { total, done, blocked };
}
```

未拆任务：

```ts
if (total === 0) show "未拆分任务";
```

禁止展示 `0/0` 和 `0%`。

## 6.6 Token 汇总

需求 Token 汇总：

```ts
function getRequirementTokenTotal(requirement: Requirement, tasks: Task[], tokenSources: TokenSource[]) {
  const directIds = requirement.directTokenSourceIds;
  const taskIds = tasks
    .filter(t => t.requirementId === requirement.id)
    .flatMap(t => t.tokenSourceIds);

  const ids = [...directIds, ...taskIds];

  // P0 可先简单去重，避免同一来源重复显示过大；如当前业务希望重复归因，也可不去重。
  const uniqueIds = Array.from(new Set(ids));

  return uniqueIds.reduce((sum, id) => {
    const source = tokenSources.find(s => s.id === id);
    return sum + (source?.tokenCount || 0);
  }, 0);
}
```

如果产品坚持 P0 不做去重，可改为直接累加，但文档中应说明 Token 仅为辅助归集，不作为严格成本核算。

## 7. 交互流程

## 7.1 创建需求流程

```text
点击「新建需求」
  ↓
进入新建需求页
  ↓
填写标题、描述、验收标准、优先级、截止日期、参与团队
  ↓
创建需求
  ↓
创建成功
  ↓
打开需求详情抽屉，并聚焦执行拆解
  ↓
用户可选择添加任务或返回看板
```

## 7.2 添加任务流程

```text
打开需求详情抽屉
  ↓
点击「添加任务」
  ↓
打开添加任务弹窗
  ↓
填写任务标题、负责人、优先级、截止日期、上游依赖
  ↓
点击「创建任务」
  ↓
弹窗关闭
  ↓
需求详情抽屉保持打开
  ↓
执行拆解列表新增任务
  ↓
看板卡片聚合信息更新
```

## 7.3 任务进度更新流程

```text
点击任务行或任务摘要卡
  ↓
打开任务详情抽屉
  ↓
使用 Slider + InputNumber 更新进度
  ↓
点击「保存进度」
  ↓
任务进度更新
  ↓
所属需求聚合信息更新
```

## 7.4 上游依赖阻塞流程

```text
任务 B 依赖任务 A
  ↓
任务 A 未完成
  ↓
任务 B 展示依赖阻塞
  ↓
需求卡片展示 1 个依赖阻塞
  ↓
任务 A 完成
  ↓
任务 B 依赖阻塞解除
```

## 7.5 关联 Token 来源流程

```text
打开需求详情或任务详情
  ↓
点击「关联 Token 来源」
  ↓
打开 Token 来源弹窗
  ↓
选择一条或多条来源
  ↓
点击「关联」
  ↓
弹窗关闭
  ↓
详情中刷新 Token 摘要与来源明细
```

## 7.6 查看全部已完成流程

```text
需求看板完成列
  ↓
完成列仅展示最近完成
  ↓
存在历史完成需求时，底部显示「查看全部已完成」
  ↓
点击后切换到任务树 / 列表视图
  ↓
自动设置阶段筛选为「完成」
  ↓
展示全部已完成需求
```

## 7.7 关注 / 取消关注流程

```text
用户在看板卡片 / 任务行 / 详情抽屉看到星标
  ↓
点击星标
  ↓
前端 toggleFavorite(target_type, target_id)
  ↓
后端 upsert favorite_targets，若已存在则删除（幂等）
  ↓
返回 { favorited: boolean }
  ↓
前端刷新 favorites 查询
  ↓
星标视觉态切换 + 「我关注的」筛选结果更新
```

## 7.8 「我关注的」筛选流程

```text
点击工具栏「我关注的」
  ↓
URL 写入 favorite=1
  ↓
filteredRequirements 计算时合入 favoriteMatched
  ↓
看板 / 任务树只展示当前用户关注的需求,或关注任务所属的需求
  ↓
再次点击「我关注的」清除 favorite 参数,恢复全量视图
```

## 10. Mock 数据建议

Mock 数据应覆盖以下场景：

1. 待开始需求：未拆任务。
2. 评审需求：有验收标准，有 Token 来源。
3. 进行中需求：有多个任务，其中一个依赖阻塞。
4. 完成需求：有 completedAt。
5. 历史完成需求：completedAt 超过 7 天，用于验证“查看全部已完成”。
6. 已取消需求：不出现在主看板列。
7. 有 Token 来源的任务。
8. 无 Token 来源但有进度的任务。
9. 需求直接关联 Token 来源。
10. 任务关联 Token 来源并汇总到需求。
