# 日报 / 周报 Agent 化产品设计文档

版本：v1.1  
状态：产品层修订稿  
范围：我的日报、我的周报、小组日报、小组周报、部门日报、部门周报 6 类报告弹窗与报告生成心智  
不包含：Agent 配置页、Agent 取数逻辑、Agent 执行链路、接口设计、数据库设计、代码实现方案

---

## 1. 背景

当前日报 / 周报模块已经具备报告生成、手写、编辑、保存、提交、查看等能力。历史实现中，报告生成主要依赖 Skill：用户在报告弹窗中选择来源、选择 Skill 预设或上传 `skill.md`，再触发生成。

后续日报 / 周报生成能力将迁移到用户自己的 Agent。Agent 根据当前用户身份、角色和权限，自动读取相关 session、日报、周报等上下文，并生成对应报告内容。

因此，日报 / 周报模块的产品心智需要从：

> 用户选择来源并把材料交给 Skill 生成报告

调整为：

> Agent 自动生成报告内容，用户可查看、修订、确认，报告内容可继续被上级汇总使用

本次产品设计只定义日报 / 周报业务模块的交互与状态，不定义 Agent 平台内部能力。

---

## 2. 产品目标

### 2.1 核心目标

1. 移除日报 / 周报弹窗中的 Skill 生成心智。
2. 移除用户手动选择来源、上传 Skill、选择生成预设等旧交互。
3. 将 6 类报告弹窗统一为“报告查看 / 编辑 / 生成 / 确认”入口。
4. 支持 Agent 自动生成的报告内容直接参与上级汇总，避免因个人未确认导致小组 / 部门报告阻塞。
5. 保留用户修订和确认能力，用于提升报告可信度，而不是作为汇总链路的强前置条件。
6. 在 Agent 未配置、生成失败、内容缺失等场景下，保留手写兜底能力。

### 2.2 非目标

本次不设计以下内容：

1. Agent 如何配置。
2. Agent 如何选择模型、凭据或工具。
3. Agent 如何拉取 session、日报、周报。
4. Agent 定时任务如何运行。
5. Agent 失败如何重试、排队、调度。
6. Agent 与日报 / 周报后端接口如何对接。
7. 数据表、接口字段、权限校验、运行记录等技术实现。

---

## 3. 产品原则

### 3.1 报告模块只负责报告业务闭环

日报 / 周报模块只关心：

- 当前报告有没有内容；
- 内容是否由 AI 生成；
- 用户是否确认或修改过；
- 当前报告是否可以用于上级汇总；
- 用户是否可以继续编辑、保存、确认。

日报 / 周报模块不关心：

- Agent 用了哪些工具；
- Agent 从哪些接口取数；
- Agent 如何组织上下文；
- Agent 的内部运行细节。

### 3.2 Agent 生成即形成可用报告

Agent 生成的日报 / 周报不应被视为“不可用草稿”。

只要 Agent 成功生成报告内容，该内容就应该成为一份可被上级汇总使用的报告。用户后续可以确认、修改、重新生成，但未确认不应该阻断上级小组 / 部门报告生成。

### 3.3 用户确认是可信度标记，不是流程阻断

用户确认或修改报告的意义是提升内容可信度，而不是决定报告是否存在。

例如：

- AI 已生成：可用于汇总，但可信度较低；
- 本人已确认：可用于汇总，可信度较高；
- 本人已修改：可用于汇总，可信度最高。

### 3.4 手写能力必须保留

Agent 未配置、生成失败、内容质量不佳、用户临时不想使用 Agent 时，都必须允许用户直接手写报告。

日报 / 周报不能因为 Agent 不可用而无法完成。

### 3.5 汇总报告不等待所有人确认

小组日报、小组周报、部门日报、部门周报生成时，应允许使用下级已生成但未确认的报告内容。

否则会出现：

- 某个成员忘记确认个人日报；
- TL 的小组日报无法生成；
- 部门日报继续被阻塞；
- 整个汇报链路被一个人的确认动作卡死。

因此，汇总报告应关注“来源完整度”和“来源可信度”，而不是只关注“是否全部人工确认”。

## 4. 核心产品心智

### 4.1 旧心智

历史 Skill 生成模式下，用户需要：

1. 选择 session / 日报 / 周报来源；
2. 选择 Skill 或上传 Skill；
3. 点击生成；
4. 查看生成结果；
5. 保存或提交。

该心智下，报告页承担了“选择材料”和“配置生成器”的职责。

### 4.2 新心智

Agent 模式下，用户不再手动选择来源，也不再上传 Skill。

新的心智是：

1. Agent 根据用户身份和权限自动生成报告内容；
2. 报告页展示生成结果和状态；
3. 用户可以查看、修改、确认、重新生成或手写；
4. 上级汇总可直接使用已生成报告；
5. 用户确认只影响可信度，不阻断汇总。

### 4.3 核心表达

产品表达建议统一为：

> 系统会使用你已配置的日报 / 周报 Agent，根据你的权限自动生成报告内容。你可以查看、修改或确认；即使未确认，已生成内容也可用于上级汇总。

## 5. 报告类型范围

本次统一覆盖 6 类报告：

| 报告类型 | 使用角色 | 内容来源心智 | 上级汇总关系 |
|---|---|---|---|
| 我的日报 | 员工 / TL / PM / 总监 / Admin | 根据本人当天工作记录生成 | 员工 / TL 可进入小组或部门汇总；PM 作为独立个人来源，不进入小组汇总 |
| 我的周报 | 员工 / TL / PM / 总监 / Admin | 根据本人本周工作记录和日报生成 | 员工 / TL 可进入小组或部门汇总；PM 作为独立个人来源，不进入小组汇总 |
| 小组日报 | TL | 根据 TL 所管理小组成员日报生成 | 可被部门日报汇总使用 |
| 小组周报 | TL | 根据 TL 所管理小组成员周报 / 日报生成 | 可被部门周报汇总使用 |
| 部门日报 | 总监 / Admin | 根据部门范围内小组报告和独立个人报告生成 | 部门层最终报告 |
| 部门周报 | 总监 / Admin | 根据部门范围内小组报告和独立个人报告生成 | 部门层最终报告 |

### 5.1 PM 角色特殊口径

PM 不归属于任何小组，不承担小组日报 / 小组周报的生成职责。

产品上，PM 应被视为一个独立个人用户：

- PM 可以生成、查看、修改、确认自己的日报 / 周报；
- PM 不进入 TL 的小组成员范围；
- PM 不拥有小组日报 / 小组周报入口；
- PM 的个人日报 / 周报如需进入部门汇总，应作为“部门独立个人来源”直接参与部门日报 / 部门周报，而不是先进入某个小组报告。

这避免把 PM 错误归入某个小组，也避免部门报告只汇总小组报告时遗漏 PM 的个人工作内容。

## 6. 报告状态模型

### 6.1 状态列表

日报 / 周报弹窗统一使用以下产品状态：

| 状态 | 含义 | 是否可用于上级汇总 | 用户动作 |
|---|---|---:|---|
| 未生成 | 当前周期没有报告内容 | 否 | Agent生成、直接手写 |
| 生成中 | Agent 正在生成报告内容 | 否 | 等待、关闭弹窗 |
| AI 已生成 | Agent 已生成内容，用户尚未确认 | 是 | 查看、确认、修改、重新生成 |
| 本人已确认 | 用户确认 AI 生成内容无误 | 是 | 查看、修改、重新生成 |
| 本人已修改 | 用户编辑并保存过内容 | 是 | 查看、继续修改、重新生成 |
| 手写报告 | 用户跳过 Agent 直接手写并保存 | 是 | 查看、继续修改、重新生成 |
| 生成失败 | Agent 未能生成内容 | 否 | 重试、直接手写 |
| Agent 未配置 | 当前用户未配置对应报告 Agent | 否 | 去配置、直接手写 |

### 6.2 状态判断原则

1. “AI 已生成”不是草稿阻断态，而是有效报告态。
2. “本人已确认”和“本人已修改”是更高可信度状态。
3. “生成失败”不影响用户手写报告。
4. “Agent 未配置”不影响用户手写报告。
5. 上级汇总时应可识别来源报告的状态，而不是简单判断是否提交。

## 7. 汇总来源完整度与可信度

### 7.1 来源完整度

汇总类报告应展示来源完整度，用于帮助 TL / 总监判断当前汇总质量。

示例：

> 来源完整度：8 / 10 人已有报告

这里的“已有报告”包括：

- AI 已生成；
- 本人已确认；
- 本人已修改；
- 手写报告。

不包括：

- 未生成；
- 生成失败；
- Agent 未配置且未手写。

### 7.2 来源可信度

汇总类报告应展示来源可信度分布。

示例：

> 5 人已确认，2 人已修改，1 人 AI 已生成未确认，2 人缺失

可信度从高到低建议为：

1. 本人已修改；
2. 本人已确认；
3. 手写报告；
4. AI 已生成未确认；
5. 未生成 / 生成失败。

说明：手写报告一般由用户主动完成，可信度应高于 AI 未确认内容。但如果手写内容为空或极短，应在质量规则中另行判断，本次产品层不展开。

### 7.3 汇总不阻塞原则

汇总报告不要求所有来源都已确认。

只要存在可用来源，TL / 总监就可以生成汇总报告。缺失来源只作为风险提示或完整度提示，不阻断生成。

## 8. 弹窗统一结构

6 类报告弹窗统一改为“报告弹窗”，不再使用“选择来源 → 确认日报”的步骤条。

### 8.1 弹窗基础结构

```text
标题区
- 报告名称：我的日报 / 我的周报 / 小组日报 / 小组周报 / 部门日报 / 部门周报
- 当前日期或周期
- 当前状态

说明区
- 当前报告由对应 Agent 自动生成
- 汇总类报告展示来源完整度和可信度

内容区
- 报告 Markdown 编辑区
- 无内容时展示空状态
- 生成中时展示生成中状态
- 失败时展示失败状态

操作区
- Agent生成
- 重新生成
- 直接手写
- 确认无误
- 保存修改
- 关闭
```

实际按钮根据当前状态动态展示，不是所有按钮同时出现。

## 9. 各状态下的弹窗交互

### 9.1 未生成

适用场景：当前周期没有任何报告内容。

页面表现：

```text
当前还没有报告内容。
系统可以使用你的 Agent 自动生成，也可以直接手写。

[Agent生成] [直接手写]
```

交互规则：

1. 点击“Agent生成”后进入生成中。
2. 点击“直接手写”后进入空编辑区。
3. 不展示来源选择。
4. 不展示 Skill 预设。
5. 不展示上传 Skill。

### 9.2 生成中

适用场景：Agent 正在生成报告。

页面表现：

```text
正在生成报告内容，请稍候。
你可以关闭弹窗，稍后回来查看结果。
```

交互规则：

1. 用户可以关闭弹窗。
2. 不提供保存或确认动作。
3. 不展示来源选择。
4. 生成完成后进入“AI 已生成”。
5. 生成失败后进入“生成失败”。

### 9.3 AI 已生成

适用场景：Agent 已生成报告，用户尚未确认或修改。

页面表现：

```text
AI 已生成报告内容，可用于上级汇总。
你可以确认无误，也可以修改后保存。
```

按钮建议：

```text
[确认无误] [保存修改] [重新生成]
```

交互规则：

1. 用户不操作时，该报告仍可用于上级汇总。
2. 点击“确认无误”后状态变为“本人已确认”。
3. 修改内容并保存后状态变为“本人已修改”。
4. 点击“重新生成”时需要提示可能覆盖当前内容。

### 9.4 本人已确认

适用场景：用户确认 AI 生成内容无误。

页面表现：

```text
你已确认本报告内容。
```

按钮建议：

```text
[保存修改] [重新生成]
```

交互规则：

1. 用户继续编辑并保存后，状态变为“本人已修改”。
2. 重新生成需要二次确认。

### 9.5 本人已修改

适用场景：用户编辑并保存过报告内容。

页面表现：

```text
你已修改并保存本报告内容。
```

按钮建议：

```text
[保存修改] [重新生成]
```

交互规则：

1. 继续修改后仍保持“本人已修改”。
2. 重新生成需要明确提示会覆盖人工修改内容。
3. 人工修改内容优先级高于 AI 原始生成内容。

### 9.6 手写报告

适用场景：用户跳过 Agent，直接手写报告。

页面表现：

```text
当前报告由你手写完成。
```

按钮建议：

```text
[保存修改] [Agent生成]
```

交互规则：

1. 手写内容可用于上级汇总。
2. 如果用户在已有手写内容时点击“Agent生成”，必须提示可能覆盖当前内容。

### 9.7 生成失败

适用场景：Agent 生成失败。

页面表现：

```text
Agent生成失败，你可以重试，也可以直接手写。
```

按钮建议：

```text
[重新生成] [直接手写]
```

交互规则：

1. 生成失败不阻断用户写报告。
2. 失败状态不作为可汇总来源。
3. 如果之前已有旧报告内容，失败不能清空旧内容。

### 9.8 Agent 未配置

适用场景：用户未配置当前报告类型所需 Agent。

页面表现：

```text
你还没有配置对应的报告 Agent，暂时无法Agent生成。
你仍然可以直接手写报告。
```

按钮建议：

```text
[去配置 Agent] [直接手写]
```

交互规则：

1. Agent 未配置不阻断手写。
2. 报告弹窗不承载 Agent 配置表单。
3. 点击“去配置 Agent”跳转到 Agent 配置页面。

## 10. 6 类报告弹窗差异

### 10.1 我的日报

定位：个人当天工作报告。

说明文案：

> 系统将根据你今天的工作记录生成日报内容。AI 已生成内容可用于上级汇总，你可以确认或修改。

重点展示：

- 当前日期；
- 当前状态；
- 报告内容；
- 是否本人确认 / 修改。

不再展示：

- 选择 session；
- 上传 Skill；
- 选择 Skill；
- 发送给 TL / 总监。

### 10.2 我的周报

定位：个人本周工作总结。

说明文案：

> 系统将根据你本周的工作记录和日报生成周报内容。AI 已生成内容可用于上级汇总，你可以确认或修改。

重点展示：

- 当前周周期；
- 当前状态；
- 报告内容；
- 是否本人确认 / 修改。

不再展示：

- 手动选择来源日报；
- 选择 Skill；
- 上传 Skill；
- 发送给上级。

### 10.3 小组日报

定位：TL 查看并生成自己所管理小组的当天汇总。

说明文案：

> 系统将根据 TL 所管理小组成员今日已有报告生成小组日报。成员 AI 已生成但未确认的日报也会参与汇总。PM 不归属小组，不参与小组日报。

重点展示：

- 小组名称；
- 来源完整度；
- 来源可信度分布；
- 小组日报内容；
- 当前状态。

示例来源提示：

> 来源完整度：8 / 10 人已有日报；其中 5 人已确认，2 人已修改，1 人 AI 已生成未确认，2 人缺失。

不再展示：

- 手动勾选成员日报；
- 等待所有人提交；
- 发送给总监按钮作为主流程。

### 10.4 小组周报

定位：TL 查看并生成自己所管理小组的本周汇总。

说明文案：

> 系统将根据 TL 所管理小组成员本周已有报告生成小组周报。成员 AI 已生成但未确认的周报也会参与汇总。PM 不归属小组，不参与小组周报。

重点展示：

- 小组名称；
- 周周期；
- 来源完整度；
- 来源可信度分布；
- 小组周报内容。

不再展示：

- 手动选择成员周报；
- 等待所有成员提交；
- Skill 生成入口。

### 10.5 部门日报

定位：总监 / Admin 查看并生成部门当天汇总。

说明文案：

> 系统将根据部门范围内已有报告生成部门日报。未确认但已生成的下级报告也会参与汇总。

重点展示：

- 部门范围；
- 来源完整度；
- 来源可信度分布；
- 部门日报内容；
- 当前状态。

不再展示：

- 手动选择小组日报；
- 等待所有小组提交；
- 发送给谁。

### 10.6 部门周报

定位：总监 / Admin 查看并生成部门本周汇总。

说明文案：

> 系统将根据部门范围内本周已有报告生成部门周报。未确认但已生成的下级报告也会参与汇总。

重点展示：

- 部门范围；
- 周周期；
- 来源完整度；
- 来源可信度分布；
- 部门周报内容。

不再展示：

- 手动选择小组周报；
- 等待所有小组提交；
- Skill 生成入口。

## 11. 操作按钮设计

### 11.1 按钮集合

报告弹窗允许出现以下按钮：

| 按钮 | 含义 | 出现场景 |
|---|---|---|
| Agent生成 | 使用对应 Agent 生成报告内容 | 未生成、手写报告、有内容但用户主动重生成 |
| 重新生成 | 使用 Agent 重新生成报告内容 | 已有 AI 内容、已确认、已修改 |
| 直接手写 | 跳过 Agent，进入编辑区 | 未生成、生成失败、Agent 未配置 |
| 确认无误 | 用户确认当前内容无需修改 | AI 已生成 |
| 保存修改 | 保存用户编辑后的内容 | 编辑区有内容变更 |
| 去配置 Agent | 跳转到 Agent 配置页 | Agent 未配置 |
| 关闭 | 关闭弹窗 | 全状态 |

### 11.2 按钮优先级

不同状态下的主按钮建议：

| 状态 | 主按钮 | 次按钮 |
|---|---|---|
| 未生成 | Agent生成 | 直接手写 |
| 生成中 | 无 | 关闭 |
| AI 已生成 | 确认无误 | 保存修改、重新生成 |
| 本人已确认 | 保存修改 | 重新生成 |
| 本人已修改 | 保存修改 | 重新生成 |
| 手写报告 | 保存修改 | Agent生成 |
| 生成失败 | 重新生成 | 直接手写 |
| Agent 未配置 | 去配置 Agent | 直接手写 |

### 11.3 重新生成保护

当当前报告已有人工编辑内容时，点击“重新生成”必须提醒：

> 重新生成可能覆盖当前已编辑内容，是否继续？

对于“本人已修改”和“手写报告”状态，重新生成必须使用更强的二次确认。

## 12. 旧能力移除范围

日报 / 周报弹窗中移除以下旧交互：

1. 步骤条：选择来源 → 确认报告。
2. session 手动选择区。
3. 来源日报 / 周报手动勾选区。
4. Skill 预设选择。
5. 上传 `skill.md`。
6. 通过 Skill 生成。
7. 发送给 TL / 发送给总监 / 发送给谁。
8. 把“提交给某人”作为报告主流程。

说明：

- “来源范围”和“来源完整度”可以展示，但不作为用户选择项。
- “上级汇总关系”可以弱提示，但不作为“发送”动作。
- 报告内容保存后即可作为上级汇总来源，不再依赖用户主动发送。

## 13. 提交与确认关系

### 13.1 不再以“发送给谁”为核心

历史日报流程中，“保存并发送给 TL / 总监”承担了报告上报语义。

Agent 化后，产品语义调整为：

- 报告存在后即可被上级汇总；
- 用户确认只是标记可信度；
- 不再强调手动发送给某个角色。

### 13.2 确认的产品含义

“确认无误”表示：

> 用户已查看当前报告内容，并认可该内容可以代表自己的工作情况。

它不表示：

- 才能被上级看到；
- 才能被上级汇总；
- 才算报告存在。

### 13.3 修改的产品含义

“保存修改”表示：

> 用户对报告内容进行了人工修订，后续汇总应优先使用修订后的内容。

人工修订后的内容优先级高于 AI 原始生成内容。

## 14. 典型用户流程

### 14.1 个人日报自动生成后用户未确认

1. Agent 定时生成员工 A 的个人日报。
2. 员工 A 当天没有进入系统确认。
3. 该日报状态为“AI 已生成”。
4. TL 生成小组日报时，该日报参与汇总。
5. 小组日报来源提示中标记：员工 A 的日报为“AI 已生成未确认”。

结论：小组日报不被员工 A 的未确认动作阻塞。

### 14.2 个人发现 AI 日报有误并修改

1. 员工进入“我的日报”。
2. 看到 AI 已生成内容。
3. 修改日报正文。
4. 点击“保存修改”。
5. 状态变为“本人已修改”。
6. 后续小组汇总使用修改后的日报。

结论：人工修改内容优先。

### 14.3 Agent 生成失败

1. 用户进入“我的日报”。
2. 页面显示“Agent生成失败”。
3. 用户可以点击“重新生成”。
4. 用户也可以点击“直接手写”。
5. 手写保存后，该报告可用于上级汇总。

结论：Agent 失败不阻断日报完成。

### 14.4 小组日报来源不完整

1. 小组共 10 人。
2. 5 人已确认日报。
3. 2 人已修改日报。
4. 1 人 AI 已生成未确认。
5. 2 人未生成。
6. TL 进入小组日报弹窗。
7. 页面展示来源完整度：8 / 10。
8. TL 仍可生成小组日报。

结论：来源不完整只提示，不阻断。

### 14.5 用户未配置 Agent

1. 用户进入“我的周报”。
2. 页面显示未配置周报 Agent。
3. 用户可以点击“去配置 Agent”。
4. 用户也可以点击“直接手写”。

结论：Agent 配置缺失不阻断报告编写。

## 15. 页面文案建议

### 15.1 通用说明

> 系统会使用你已配置的报告 Agent 自动生成内容。你可以查看、修改或确认；已生成内容可用于上级汇总。

### 15.2 未生成

> 当前还没有报告内容。你可以Agent生成，也可以直接手写。

### 15.3 AI 已生成

> AI 已生成报告内容，可用于上级汇总。你可以确认无误，也可以修改后保存。

### 15.4 已确认

> 你已确认本报告内容。

### 15.5 已修改

> 你已修改并保存本报告内容。

### 15.6 生成失败

> Agent生成失败。你可以重新生成，也可以直接手写。

### 15.7 Agent 未配置

> 你还没有配置对应的报告 Agent，暂时无法Agent生成。你仍然可以直接手写报告。

### 15.8 汇总来源提示

> 来源完整度：{已有数量} / {应有数量}。其中 {已确认数量} 人已确认，{已修改数量} 人已修改，{AI生成数量} 人 AI 已生成未确认，{缺失数量} 人缺失。

## 16. 权限与角色产品口径

### 16.1 员工

员工可操作：

- 我的日报；
- 我的周报。

员工不可操作：

- 小组日报；
- 小组周报；
- 部门日报；
- 部门周报。

### 16.2 TL

TL 可操作：

- 我的日报；
- 我的周报；
- 小组日报；
- 小组周报。

TL 在小组报告中可以看到自己所管理小组成员的来源完整度和可信度。

### 16.3 PM

PM 是独立于小组的个人用户，不归属于任何 TL 小组。

PM 可操作：

- 我的日报；
- 我的周报。

PM 不可操作：

- 小组日报；
- 小组周报；
- 部门日报；
- 部门周报。

产品口径：

- PM 的日报 / 周报是个人报告，不是小组报告；
- PM 不应出现在 TL 的小组成员来源中；
- PM 不应生成或管理小组日报 / 小组周报；
- 如部门日报 / 部门周报需要覆盖 PM 工作内容，应将 PM 个人报告作为部门独立个人来源参与部门汇总。

### 16.4 总监 / Admin

总监 / Admin 可操作：

- 我的日报；
- 我的周报；
- 部门日报；
- 部门周报。

总监 / Admin 在部门报告中可以看到部门范围内来源完整度和可信度。

部门报告的来源可以包括：

- TL 小组日报 / 小组周报；
- 小组成员个人报告；
- 独立个人报告，例如 PM 的个人日报 / 周报。

是否展示到“成员个人级”还是“先按小组聚合后展示”，属于后续开放设计阶段的展示粒度问题，本产品口径仅确认 PM 不归入小组。

## 17. 边界规则

### 17.1 已有内容时重新生成

如果当前已有报告内容，重新生成不得静默覆盖。

需要区分：

- AI 已生成但未编辑：轻确认；
- 本人已确认：中等确认；
- 本人已修改 / 手写报告：强确认。

### 17.2 生成失败不得清空旧内容

如果此前已经存在报告内容，新的生成失败不能清空旧内容。

页面应继续保留旧内容，并提示最近一次生成失败。

### 17.3 来源缺失不得阻断汇总

汇总类报告在来源不完整时仍可生成，但必须展示完整度提示。

### 17.4 AI 未确认内容必须可识别

上级查看汇总来源时，应能区分：

- 本人已确认；
- 本人已修改；
- AI 已生成未确认；
- 缺失。

这用于帮助上级判断汇总内容可靠性。

### 17.5 手写优先于 AI

用户手写或人工修改后的内容，应优先作为有效报告内容。

### 17.6 Agent 配置不进入报告弹窗

报告弹窗只提供“去配置 Agent”入口，不在弹窗内承载配置表单。

## 18. 产品验收标准

### 18.1 交互验收

1. 6 类报告弹窗不再展示 Skill 预设、上传 Skill、来源手动选择。
2. 6 类报告弹窗不再使用“选择来源 → 确认报告”的步骤条。
3. 未生成状态下，用户可以Agent生成或直接手写。
4. Agent 未配置状态下，用户可以去配置 Agent 或直接手写。
5. 生成失败状态下，用户可以重新生成或直接手写。
6. AI 已生成内容默认可用于上级汇总。
7. 用户可以对 AI 已生成内容确认无误。
8. 用户可以修改 AI 已生成内容并保存。
9. 已修改或手写内容重新生成前必须有覆盖提醒。
10. 汇总类报告展示来源完整度和可信度分布。

### 18.2 产品语义验收

1. “草稿未提交不可汇总”的旧语义被移除。
2. “发送给 TL / 总监”的旧语义被移除。
3. “用户选择材料生成”的旧语义被移除。
4. “Agent 生成即形成可用报告”的新语义成立。
5. “人工确认 / 修改只是可信度标记”的新语义成立。
6. “来源不完整不阻断汇总”的新语义成立。

## 20. 最终结论

日报 / 周报模块的产品边界应收敛为：

> 报告页只负责展示和维护报告内容；Agent 负责生成内容。Agent 生成的报告默认有效，可用于上级汇总；用户确认或修改只提升可信度，不作为汇总链路的阻断条件。

因此，6 类报告弹窗统一从“Skill 生成流程”调整为“报告内容管理弹窗”：

- 不再选择来源；
- 不再上传 Skill；
- 不再发送给某人；
- 不再把未确认内容视为不可用草稿；
- 保留Agent生成、重新生成、直接手写、确认无误、保存修改；
- 汇总类报告展示来源完整度和可信度；
- Agent 配置和取数逻辑放在 Agent 模块内解决。
