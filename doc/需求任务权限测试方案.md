# 需求 / 任务权限测试方案

## 1. 测试目标

本方案用于让 Claude 或自动化脚本使用当前服务中的真实账号，对需求 / 任务权限改造做端到端验证。

测试目标：

1. 验证后端接口权限兜底，不能只依赖前端按钮显隐。
2. 验证前端 route、按钮、可操作入口与后端权限一致。
3. 验证不同角色的资源可见范围符合《需求任务权限设计.md》。
4. 验证 employee 不是只读用户：可以管理自己的任务，但不能影响别人的任务或需求主数据。
5. 验证 TL 只能管理自己团队范围或自己创建范围内的需求 / 任务。
6. 验证 PM / director / admin 在 P0 阶段具备全量管理能力。
7. 覆盖越权、跨团队、状态、依赖、删除关联数据等边界。

本方案优先以 **API 自动化测试** 作为权限验收基准，再用 **浏览器真实 UI 测试** 验证入口和按钮一致性。

## 2. 测试范围

### 2.1 覆盖模块

- 登录鉴权
- 用户 / 团队辅助数据
- 需求列表 / 详情 / 创建 / 编辑 / 状态流转 / 取消 / 恢复 / 删除 / AC
- 任务列表 / 详情 / 创建 / 编辑 / 负责人变更 / 状态 / 进度 / 依赖 / 删除
- 前端路由权限
- 前端按钮显隐和只读态

### 2.2 不覆盖模块

本次不以以下模块为主验收对象：

- 日报 / 周报
- Token 明细统计
- session 上传链路
- 风险通知完整链路
- 组织用户管理后台
- 归档字段实现；归档只保留产品语义，不纳入本轮 P0 权限测试

但如果需求 / 任务删除会影响 session / token / documents 关联字段，应覆盖删除副作用。

### 2.3 统一状态码规则

本方案所有 API 用例统一使用以下状态码口径，后续用例必须写成明确、唯一的预期状态码：

| 场景 | 明确预期 |
|---|---|
| 未登录、缺 token、token 过期、token 被篡改 | 401 |
| 列表接口越权范围 | 200，但只返回当前用户可见范围内的数据；越权数据不得出现 |
| 详情 GET 无查看权限 | 404，避免暴露资源存在性 |
| 修改 / 删除 / 状态变更 / 依赖维护无操作权限 | 403 |
| 删除因有关联任务 / session / token_usage / document 失败 | 409 |
| 参数非法、字段非法、状态非法、依赖非法 | 400 |
| 资源真实不存在 | 404 |

重点收敛：

- `GET /requirements/{id}` 跨团队不可见：404。
- `GET /tasks/{id}` 跨团队不可见：404。
- `PUT /requirements/{id}` 越权：403。
- `PUT /tasks/{id}` 越权：403。
- `DELETE` 越权：403。
- 删除有关联数据：409。
- 权限判断必须优先于业务副作用；无权限请求不能因为 AI 未配置、有关联数据、依赖非法等原因泄露资源状态。

## 3. 测试环境要求

### 3.1 服务要求

建议使用本地 docker compose 环境：

```bash
docker compose up -d db api web
```

默认地址：

```text
API: http://localhost:8080/api/v1
Web: http://localhost:5173 或 docker 暴露的 Web 地址
```

本文档中的接口路径默认都是相对 `AIDA_API_BASE` 的路径。`AIDA_API_BASE` 默认是 `http://localhost:8080/api/v1`。自动化脚本拼接 URL 时只能使用 `AIDA_API_BASE + 本文档路径`，例如 `AIDA_API_BASE + /users`，不要再额外拼接 `/api/v1`。

如果端口不同，执行测试前设置：

```bash
export AIDA_API_BASE="http://localhost:8080/api/v1"
export AIDA_WEB_BASE="http://localhost:5173"
```

建议使用干净数据库执行完整权限验收：

## 4. 自动化执行原则

### 4.1 Claude 执行要求

让 Claude 执行本方案时，应明确要求：

```text
请使用真实服务和真实账号执行，不要 mock API，不要只读代码推断。
优先用 API 自动化脚本完成权限矩阵验证，再用浏览器检查关键 UI 入口和按钮。
所有测试数据必须带唯一前缀，执行结束后清理。
遇到失败不要跳过，记录请求、角色、资源、期望状态码、实际状态码和响应体。
```

### 4.2 自动化脚本建议

推荐写一个临时测试脚本，例如：

```text
/tmp/aida-permission-e2e.<timestamp>.js
```

脚本职责：

1. 登录全部测试账号并保存 token。
2. 创建跨角色、跨团队测试夹具。
3. 按矩阵执行 API 请求。
4. 断言 HTTP 状态码和关键响应字段。
5. 输出 Markdown 或 JSON 报告。
6. 按依赖顺序清理测试数据。

禁止：

- 禁止使用 mock 数据。
- 禁止修改非测试前缀数据。
- 禁止为了通过测试而放宽断言。
- 禁止在测试失败后继续错误清理导致证据丢失。

## 6. API 测试基础流程

### 6.1 登录并保存 token

对所有测试账号执行：

```http
POST /auth/login
Content-Type: application/json

{
  "employee_id": "zhangsan",
  "password": "123"
}
```

断言：

- HTTP 200
- 返回 token
- 返回 user.role 正确
- 返回 team_id 与种子数据一致

负向登录：

| 场景 | 请求 | 期望 |
|---|---|---|
| 错误密码 | `password=wrong` | 401 |
| 不存在 employee_id | `employee_id=no_such_user` | 401 |
| 缺少 token 访问 `/requirements` | 无 Authorization | 401 |
| 非 Bearer 格式 | `Authorization: token xxx` | 401 |

### 6.2 辅助数据接口

| 接口 | 角色 | 期望 |
|---|---|---|
| `GET /users` 未登录 | anonymous | 401 |
| `GET /users` 已登录 | admin / li_director / chen_pm / liu_tl / zhangsan | 200，返回用户基础列表；该接口只做登录鉴权，不作为任务负责人候选范围断言 |
| `GET /task-assignees` admin/director/pm | admin / li_director / chen_pm | 200，仅返回全部 employee 用户 |
| `GET /task-assignees` TL | liu_tl | 200，仅返回自己团队 employee |
| `GET /task-assignees` employee | zhangsan | 200，仅返回当前登录用户自己 |
| `GET /teams` 未登录 | anonymous | 401 |
| `GET /teams` 登录用户 | 任意 | 200，返回全部团队基础信息 |

`/users` 只验证登录鉴权，不用于负责人选择器；`/task-assignees` 的范围是强断言，前端负责人选择器只能使用 `/task-assignees`。

## 10. 自动化断言规范

### 10.1 状态码断言

| 场景 | 明确状态码 |
|---|---|
| 未登录 / token 过期 / token 被篡改 | 401 |
| 列表接口越权范围 | 200，结果按当前用户权限过滤，越权数据不返回 |
| GET 详情无查看权限 | 404 |
| 修改 / 删除 / 状态变更 / 依赖维护无操作权限 | 403 |
| 资源真实不存在 | 404 |
| 参数非法 / 字段非法 / 状态非法 / 依赖非法 | 400 |
| 删除存在强关联 | 409 |
| 有权限调用 regenerate AC 且 AI 正常 | 200 |
| 有权限调用 regenerate AC 但 AI 未配置 | 503 |

权限判断顺序强断言：无权限请求必须优先返回 403 或详情 GET 的 404，不能先返回 409、400、503 等业务错误。

### 10.3 列表断言方式

不要只断言列表 HTTP 200。必须断言：

1. 应该可见的测试资源 ID 存在。
2. 不应该可见的测试资源 ID 不存在。
3. `team_id`、`scope`、`assignee_id`、`status` 等查询参数不能扩大权限范围。

### 10.4 can_xxx 权限字段断言

后端必须在需求 / 任务列表项和详情中返回 `can_xxx` 权限字段，前端按钮显隐以这些字段为验收基准，不再把权限字段视为可选增强。

最终报告汇总：

```text

失败详情：...
测试数据前缀：PERM-E2E-...
开始时间：...
```

失败时必须保留：

- 请求 method/path/body
- 使用账号
- 期望结果
- 实际状态码
- 实际响应体
- 相关资源 ID

## 12. 清理策略

清理顺序：

1. 删除测试任务依赖。
2. 删除测试任务。
3. 恢复被取消但仍需删除的测试需求到可删除状态，或直接删除无关联需求。
4. 删除无关联测试需求。
5. 对因为关联数据无法硬删除的需求，改为 `cancelled`，并在报告中列为残留测试数据。

清理要求：

- 只清理 title 包含本轮 `PERM-E2E-...` 前缀的数据。
- 如果删除返回 409，不要强制破坏业务关联；改为取消需求并记录。
- 如果测试失败，应优先保留现场，除非用户明确要求清理。

## 13. 最小冒烟用例集

如果时间有限，至少执行以下用例：

1. 登录所有角色成功，错误密码失败。
2. employee 不能创建需求。
3. employee 不能编辑 / 取消本团队需求。
4. employee 不能查看跨团队需求详情。
5. TL 不能创建不含自己团队的需求。
6. TL 不能编辑跨团队需求。
7. PM 可以进入并调用任务创建。
8. employee 可以创建 assignee 为自己的任务。
9. employee 不能给别人创建任务。
10. employee 可以编辑自己的任务。
11. employee 不能编辑同团队别人的任务。
12. employee 不能更新别人任务状态 / 进度 / 依赖。
13. `GET /tasks?scope=requirements` 不能让 employee 看到可见集合之外的任务；多团队参与需求下其他团队任务可见但只读。
14. TL 只能给本团队 employee 分配任务。
15. `blocked` 不能直接设置。
16. 依赖不能跨需求、不能自依赖、不能形成环。
17. 有关联任务的需求删除返回 409。
18. 无关联需求可被有权限角色删除。
19. admin 可以 regenerate AC 或在 AI 未配置时返回非权限错误。
20. 前端 employee 不显示需求管理按钮，自己的任务显示操作，别人的任务只读。

## 15. 验收通过标准

本轮权限验收通过需要同时满足：

1. 所有 P0 权限矩阵 API 用例通过。

## 16. 待产品最终确认但 P0 已给默认值

以下事项如产品未另行确认，自动化验收按 P0 默认值执行：

| 事项 | P0 默认值 | 是否待确认 |
|---|---|---|
| 多团队需求下 employee 是否能看到其他团队 assignee 的任务 | 能；参与团队 employee 可见该需求下全部拆解任务，但只读非自己任务 | 否 |
| 多团队需求下未分配任务是否可见 | 当前用户团队在 requirement_teams 中即可见 | 否 |
| employee 创建任务不传 assignee_id | 后端默认当前登录用户 | 否 |
| employee 创建任务时传 status / progress | 返回 400，不允许创建时覆盖初始状态 / 进度 | 否 |
| TL 创建需求是否可选择其他团队 | P0 默认不允许，只能选择自己团队 | [待确认] |
| 取消需求后是否允许继续创建任务 | 不允许，返回 409 | [待确认] |
| 依赖是否允许跨需求 | 不允许，返回 400 | 否 |
| employee 是否可添加跨团队不可见任务作为依赖 | 不允许，返回 404 | 否 |
| 归档字段 | P0 不实现，不纳入测试 | 否 |
