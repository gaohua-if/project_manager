# 需求 / 任务权限测试方案

## 1. 测试目标

本方案用于让 Claude 或自动化脚本使用当前服务中的真实账号，对需求 / 任务权限改造做端到端验证。

测试目标：

1. 验证后端接口权限兜底，不能只依赖前端按钮显隐。
2. 验证前端 route、按钮、可操作入口与后端权限一致。
3. 验证不同角色的资源可见范围符合《需求任务权限设计.md》。
4. 验证 employee 不是只读用户：可以管理自己的任务，但不能影响别人的任务或需求主数据。
5. 验证 TL 只能管理自己团队范围或自己创建范围内的需求 / 任务。
6. 验证 PM / director / admin 在 P0 阶段具备全量管理能力。
7. 覆盖越权、跨团队、状态、依赖、删除关联数据等边界。

本方案优先以 **API 自动化测试** 作为权限验收基准，再用 **浏览器真实 UI 测试** 验证入口和按钮一致性。

---

## 2. 测试范围

### 2.1 覆盖模块

- 登录鉴权
- 用户 / 团队辅助数据
- 需求列表 / 详情 / 创建 / 编辑 / 状态流转 / 取消 / 恢复 / 删除 / AC
- 任务列表 / 详情 / 创建 / 编辑 / 负责人变更 / 状态 / 进度 / 依赖 / 删除
- 前端路由权限
- 前端按钮显隐和只读态

### 2.2 不覆盖模块

本次不以以下模块为主验收对象：

- 日报 / 周报
- Token 明细统计
- session 上传链路
- 风险通知完整链路
- 组织用户管理后台

但如果需求 / 任务删除会影响 session / token / documents 关联字段，应覆盖删除副作用。

## 4. 自动化执行原则

### 4.1 Claude 执行要求

让 Claude 执行本方案时，应明确要求：

```text
请使用真实服务和真实账号执行，不要 mock API，不要只读代码推断。
优先用 API 自动化脚本完成权限矩阵验证，再用浏览器检查关键 UI 入口和按钮。
所有测试数据必须带唯一前缀，执行结束后清理。
遇到失败不要跳过，记录请求、角色、资源、期望状态码、实际状态码和响应体。
```

### 4.2 自动化脚本建议

推荐写一个临时测试脚本，例如：

```text
/tmp/aida-permission-e2e.<timestamp>.js
```

脚本职责：

1. 登录全部测试账号并保存 token。
2. 创建跨角色、跨团队测试夹具。
3. 按矩阵执行 API 请求。
4. 断言 HTTP 状态码和关键响应字段。
5. 输出 Markdown 或 JSON 报告。
6. 按依赖顺序清理测试数据。

禁止：

- 禁止使用 mock 数据。
- 禁止修改非测试前缀数据。
- 禁止为了通过测试而放宽断言。
- 禁止在测试失败后继续错误清理导致证据丢失。

## 6. API 测试基础流程

### 6.1 登录并保存 token

对所有测试账号执行：

```http
POST /auth/login
Content-Type: application/json

{
  "employee_id": "zhangsan",
  "password": "123"
}
```

断言：

- HTTP 200
- 返回 token
- 返回 user.role 正确
- 返回 team_id 与种子数据一致

负向登录：

| 场景 | 请求 | 期望 |
|---|---|---|
| 错误密码 | `password=wrong` | 401 |
| 不存在 employee_id | `employee_id=no_such_user` | 401 |
| 缺少 token 访问 `/requirements` | 无 Authorization | 401 |
| 非 Bearer 格式 | `Authorization: token xxx` | 401 |

### 6.2 辅助数据接口

| 接口 | 角色 | 期望 |
|---|---|---|
| `GET /users` 未登录 | anonymous | 401 或不允许公开 |
| `GET /users` admin/director/pm | 登录用户 | 200，返回范围符合设计 |
| `GET /users` TL | 刘TL | 200，仅自己团队 employee，或至少不能用于跨团队分配成功 |
| `GET /users` employee | 张三 | 200，仅自己或符合设计范围 |
| `GET /teams` 未登录 | anonymous | 401 |
| `GET /teams` 登录用户 | 任意 | 200，返回范围符合设计 |

如果 P0 只要求 `/users` 放入登录鉴权，则未登录必须 401 是强断言；登录后的返回范围按当前实现或设计结果记录。

## 12. 清理策略

清理顺序：

1. 删除测试任务依赖。
2. 删除测试任务。
3. 恢复被取消但仍需删除的测试需求到可删除状态，或直接删除无关联需求。
4. 删除无关联测试需求。
5. 对因为关联数据无法硬删除的需求，改为 `cancelled`，并在报告中列为残留测试数据。

清理要求：

- 只清理 title 包含本轮 `PERM-E2E-...` 前缀的数据。
- 如果删除返回 409，不要强制破坏业务关联；改为取消需求并记录。
- 如果测试失败，应优先保留现场，除非用户明确要求清理。
