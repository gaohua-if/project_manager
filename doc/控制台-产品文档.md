# 控制台产品文档

版本：v0.2
模块：AIcoding 管理平台 - 控制台
受众：产品评审、UI/UX、研发评审
说明：本文档只描述「控制台」要做成什么样、用户怎么用、数据怎么流动，不涉及具体技术实现。

---

## 1. 模块定位

控制台是 AIcoding 管理平台登录后的首屏，回答用户登录后最关心的三件事：

```text
1. 我的报告今天 / 本周需要处理吗？
2. 我关注的需求和任务现在发生了什么变化？
3. 当前有哪些风险是我必须处理的？
```

对 TL 和总监，还需要额外回答：

```text
4. 我管理范围内的组 / 部门报告状态如何？
5. 团队成员的 session 上报和 token 消耗情况如何？
```

一句话定义：

> 控制台是「报告状态 + 关注变化 + 风险提醒 + Token 概览」的轻量聚合首屏，不是工作处理中心，也不是数据大盘。

> 📸 **截图位置 1：控制台整体一屏**
> 截取个人角色登录后的控制台首屏完整截图，包含原型角色切换条、上方报告卡片、Token 统计卡片、风险列表、关注对象列表。
> 体现：信息密度适中、无大量空白、关键信息（待发送日报、超期风险）有视觉强调。

## 2. 页面边界

### 2.1 控制台负责

```text
控制台负责
├─ 我的报告（个人日报 / 个人周报）状态与入口
├─ 组报告 / 部门报告（TL / 总监）状态与入口
├─ 日报生成、编辑、发送闭环弹窗
├─ Token 统计概览（个人 / 组 / 部门维度）
├─ 待处理风险列表（deadline、依赖阻塞）
├─ 关注对象变化列表（需求 / 任务）
└─ 跳转到需求看板定位处理
```

### 2.2 控制台不负责

```text
控制台不负责
├─ 需求详情编辑
├─ 任务树编辑
├─ 任务依赖配置
├─ 阻塞原因详细维护
├─ deadline 修改
├─ 验收标准 AC 维护
├─ 复杂统计图表（趋势、对比）
└─ 独立风险处理工作流
```

控制台可以展示摘要，但具体处理动作必须跳转到需求看板或在专门的弹窗内完成。

## 3. 角色与可见模块

控制台只有**一套页面**，不为不同角色单独做四套界面。
角色差异仅体现在：默认展示的数据范围、是否有汇总报告模块。

| 角色 | 我的报告 | 汇总报告 | Token 统计 | 关注对象 | 风险项 |
|------|---------|---------|-----------|---------|--------|
| 个人 | ✓ | — | 个人维度 | 我关注 + 我负责的任务 | 我相关的风险 |
| PM   | ✓ | — | 个人维度 | 我关注的需求 | 我关注需求的风险 |
| TL   | ✓ | 组日报 / 组周报 | 组维度（含本人） | 组内需求 + 组内任务 | 组内风险 |
| 总监 | ✓ | 部门日报 / 部门周报 | 部门维度（按组） | 重点需求 | 高优先级风险 |

> 📸 **截图位置 2：四种角色对比**
> 用顶部「原型角色」切换条，依次切到「个人 / PM / TL / 总监」，截图四张控制台首屏。

> 页面顶部保留一个角色切换器，**仅在产品评审、原型展示、上线前调试阶段使用**，正式上线时根据用户实际角色自动判断、隐藏切换条。评审时通过它一键演示不同角色看到的画面。

## 4. 页面结构

控制台从上到下分四个区域：

```text
控制台页面
├─ 区域 A：原型角色切换条（评审阶段）
│
├─ 区域 B：报告卡片区（左右两列）
│  ├─ 左列：我的报告（所有角色）
│  └─ 右列：
│     ├─ 组报告 / 部门报告（仅 TL、总监；个人 / PM 不显示）
│     └─ Token 统计
│
├─ 区域 C：待处理风险列表
│
└─ 区域 D：关注对象变化列表
```

设计原则：

- 风险列表的视觉权重**高于**关注对象列表，因为风险是必须处理的事，关注对象只是变化提醒。
- 报告卡片不超过两行高度，保证下方风险和关注列表至少能露出 2-3 条。
- 没有风险时，风险区不空着，展示"暂无需要关注的风险 + 进入需求看板"的引导。

> 📸 **截图位置 3：页面分区示意图**
> 在控制台首屏上叠加红框 + 数字标号（A/B/C/D），标出四个区域的位置和占比。
> 体现：分区清晰、报告区紧凑、风险区视觉突出。

## 5. 模块一：报告卡片区

### 5.1 我的报告（所有角色）

我的报告卡片同时展示两项内容：

```text
我的报告卡片
├─ 主体：今日日报
│  ├─ 报告名称
│  ├─ 状态标签
│  ├─ 一句话状态说明
│  └─ 主操作按钮
│
├─ 副体：本周周报（一行紧凑提醒）
│  ├─ 报告名称
│  ├─ 状态标签
│  ├─ 简短来源说明
│  └─ 文字链入口
│
└─ 底部：两个快捷入口
   ├─ 日报记录（草稿与发送历史）
   └─ 周报记录（本周汇总与历史）
```

**关键产品规则**：

- 今日日报永远占主位，因为这是每日动作。
- 本周周报作为次级提醒行，只在到了周报生成时间或已有草稿/已发送时才有可点按钮。
- 没到生成时间时，周报展示"周五自动汇总本周日报与任务记录"的弱提醒文字，**不显示按钮**。

> 📸 **截图位置 4：我的报告卡片四种状态**
> 截四张卡片样例，分别展示：
> 1. 今日日报 = 待生成，周报 = 待生成（个人角色登录刚开始一天的状态）
> 2. 今日日报 = 草稿待确认，周报 = 待生成（系统已自动生成草稿，等用户确认）
> 3. 今日日报 = 已发送，周报 = 草稿待确认（周五下午常见状态）
> 4. 今日日报 = 生成失败（兜底重新生成入口）
> 体现：状态-按钮配套规则一目了然。

### 5.2 汇总报告（仅 TL / 总监）

TL 看到「组报告」卡片，总监看到「部门报告」卡片。卡片内部分别展示：

```text
汇总报告卡片
├─ 今日组日报 / 今日部门日报
│  ├─ 状态标签
│  ├─ 一句话说明
│  └─ 主操作按钮
│
└─ 本周组周报 / 本周部门周报
   ├─ 状态标签
   ├─ 一句话说明
   └─ 主操作按钮
```

汇总报告的生成与发送对应主操作按钮的状态变化，与个人报告共用同一套规则。

> 📸 **截图位置 5：汇总报告卡片**
> 截 TL 角色下「组报告」卡片，包含今日组日报（生成失败 / 重新生成）和本周组周报（待生成 / 生成周报）两种状态。

### 5.3 统一报告状态体系（日报 / 周报 / 汇总报告全部通用）

```text
状态                按钮规则
─────────────────  ────────────────────────────
待生成             生成报告
生成中             生成中（禁用）
草稿待确认         查看草稿 + 发送
已发送             查看报告（只读）
生成失败           重新生成
发送失败           重试发送 + 查看草稿
```

> 📸 **截图位置 6：状态-按钮对应表**
> 可以截一组并排的报告卡片缩略图，每张卡片标注其状态和对应按钮，作为视觉规则参考。

### 5.4 报告生成弹窗

点击「生成报告」按钮后，统一打开一个**三步弹窗**。不同报告类型走不同的步骤组合。

**个人日报弹窗（三步）：**

```text
步骤 1：选择 session
  ↓ 系统列出今天的所有 session（按工具 + 时间段 + 内容摘要）
  ↓ 系统根据规则默认勾选「应进入日报的 session」
  ↓ 用户可手动调整勾选
  ↓
步骤 2：编辑草稿（合并到一步呈现）
  ↓ 左侧：Markdown 文本编辑器（纯文本 textarea，仅允许标题、列表、换行）
  ↓ 右侧：任务进展建议列表（见 5.5）
  ↓
步骤 3：发送
  └─ 点击「发送日报」按钮
     └─ 报告状态 → 已发送
     └─ 任务进展建议中被修改的项一并回写到任务（见 5.5）
```

**个人周报 / 组报告 / 部门报告弹窗（两步）：**

```text
步骤 1：确认来源
  ↓ 卡片说明来源：本周个人日报、任务、风险与阻塞 / 组内成员报告 / 各组报告
  ↓ 汇总报告额外展示提交情况：应提交 X，已提交 Y，未提交 Z，发送失败 W
  ↓ 用户点击「生成草稿」
  ↓
步骤 2：编辑草稿
  ↓ 仅 Markdown 文本编辑器（不带任务进展建议）
  ↓
步骤 3：发送
```

> 截弹窗，展示 session 列表、默认勾选状态、"6 月 22 日可用 session" 标题、底部「下一步」按钮。

> 📸 **截图位置 8：日报生成弹窗 - 步骤 2 编辑草稿（含任务进展建议）**

> 体现：编辑区主导，右侧建议区视觉次一级。

### 5.5 任务进展建议（个人日报独有）

这是当前控制台与旧版需求最重要的一处差异，必须在产品评审时重点讲清楚。

**产品意图**：用户每天写日报时，让 AI 顺手把任务进展也填了，不用再切到需求看板逐个改。

**流程**：

```text
用户在日报弹窗第 2 步看到 Markdown 草稿
        ↓
弹窗右侧同时展示「任务进展建议」列表
  ├─ 每条建议关联：任务名 + 关联 session + 建议进度 + 建议状态
  └─ 这些建议由 AI 根据已选 session 推断
        ↓
用户可对每条建议点「修改」
  ├─ 弹出「修改任务进展」二级弹窗
  │  ├─ 进度（25 / 50 / 75 / 100）
  │  ├─ 状态（未开始 / 进行中 / 已完成）
  │  ├─ 关联 session 复选
  │  └─ 备注
  └─ 保存后该条建议显示「待同步」标签
        ↓
用户点「发送日报」
  ├─ 日报本身发出
  └─ 所有「待同步」的任务建议**同步回写**到对应任务的进度和状态
```

**关键产品规则**：

- 任务进展建议**不强制用户处理**，可以不改、直接发日报。
- 修改但未发送日报时，状态停留在「待同步」，不会污染任务真实数据。
- 发送日报后才统一回写。日报底部有"已修改 N 个任务，发送日报后同步任务进展"提示。
- 只有个人日报弹窗有这块；周报、组报告、部门报告**不展示**任务建议。

## 6. 模块二：Token 统计

Token 统计是控制台新增模块，让用户在首屏快速看到 AI 工具使用量。

### 6.1 不同角色看到的内容

```text
个人 / PM
├─ 时间范围切换：昨天 / 近 3 天 / 近 7 天
├─ 区间总 Token 数
├─ Session 数量
└─ 按日趋势 Mini 柱状图

TL
├─ 时间范围切换：昨天 / 近 3 天 / 近 7 天
├─ 区间组内总 Token 数
├─ 组内 session 数量
├─ 已上报成员数 / session 数 / Token 三组对比柱
└─ 子项：我的 Token（用于和组数据对照）

总监
├─ 时间范围切换：昨天 / 近 3 天 / 近 7 天
├─ 区间部门总 Token 数
├─ 部门 session 数量
├─ 按组分布的横向柱（每个组一行：组名 / 长条 / Token 数）
└─ 子项：我的 Token
```

### 6.2 数据动作

```text
Token 统计卡片
├─ 时间范围切换 → 切换上方所有数字与图表
└─ 「查看 Token 明细」文字链 → 跳转到 /tokens 页面
```

控制台本身**不做**：明细表、按用户排行、按工具拆分、按需求拆分等复杂分析。这些都在 Token 明细页面承载。

> 并排截三张：个人、TL、总监三种形态，时间范围都切到「近 7 天」。
> 体现：相同位置、相同卡片框架，但数据呈现方式按角色变化（趋势 vs 三柱对比 vs 分组分布）。

## 7. 模块三：待处理风险

### 7.1 风险类型（P0 只做两种）

当前阶段控制台风险只覆盖两类核心异常：

```text
风险类型
├─ deadline 风险
│  ├─ 已超期：需求 / 任务过了 deadline 还没完成
│  └─ 临期：48 小时内到期
│
└─ 依赖阻塞风险
   ├─ 任务上游依赖未完成
   └─ 任务上游依赖已超期
```

旧版需求中提到的「缺 AC」「日报未发送」「日报生成失败」**暂不列入风险列表**——日报相关问题已在报告卡片直接呈现，AC 问题在 P1 再考虑。

### 7.2 风险卡片展示

```text
风险列表（表格化）
├─ 表头：风险 | 标题 / 原因 | 影响对象 | 负责人 | 截止 | 操作
└─ 每行卡片：
   ├─ 风险等级 + 类型标签（如"高 · 已超期"、"高 · 依赖阻塞"）
   ├─ 风险标题（一句话描述发生了什么）
   ├─ 风险原因（一句话补充上下文，如"上游任务已超期"）
   ├─ 影响对象（所属需求 / 任务名）
   ├─ 负责人
   ├─ 截止日期
   └─ 操作按钮：「查看任务」/「处理依赖」（按风险类型展示对应文案）
```

### 7.3 风险排序

```text
排序优先级（高 → 低）
1. 已超期
2. 依赖阻塞
3. 其他（临期等）
```

### 7.4 点击跳转

```text
deadline 风险
  ↓ 点击「查看任务」
  ↓ 跳转 /requirements?requirementId={需求}&taskId={任务}&focus=deadline
  ↓ 需求看板自动展开对应需求 + 定位到该任务 + 高亮 deadline 信息

依赖阻塞风险
  ↓ 点击「处理依赖」
  ↓ 跳转 /requirements?requirementId={需求}&taskId={任务}&focus=dependency
  ↓ 需求看板自动展开对应需求 + 定位到任务 + 显示上游依赖
```

## 8. 模块四：关注对象变化

### 8.1 关注对象定义

「关注」是用户自己主动订阅的对象，分为两类：

```text
关注对象
├─ 需求（PM、TL、总监默认关注的对象类型）
└─ 任务（个人默认关注的对象类型，TL 也会关注）
```

用户也可以跨类型关注：个人可以关注需求，总监也可以关注具体任务。

### 8.2 列表展示

```text
关注对象列表（表格化）
├─ 表头：对象 / 状态 | 标题 / 所属 | 负责人 | 截止 | 变化 / 提醒 | 操作
└─ 每行卡片：
   ├─ 对象类型标签（"需求" / "任务"）+ 状态徽标（进行中 / 阻塞 / 未开始 / 已完成）
   ├─ 标题（需求名 / 任务名）
   ├─ 所属信息（任务展示"所属需求：xxx"，需求展示依赖摘要）
   ├─ 负责人
   ├─ 截止日期
   ├─ 变化 / 提醒：
   │  ├─ 风险标签（"已超期" / "临期" / "依赖阻塞" / "依赖未完成" 等）
   │  └─ 一句话变化说明（如"任务进度 62%"、"模板评审待完成"）
   └─ 操作按钮：「详情」→ 跳转需求看板定位
```

### 8.3 排序规则

```text
关注列表排序（高优先级 → 低）
1. 已超期 / 超期
2. 临期 / 明天到期
3. 依赖阻塞 / 状态为阻塞
4. 进行中
5. 其他
6. 已完成（最末）
```

### 8.4 跳转规则

```text
点击任务的「详情」
  ↓ 跳转 /requirements?requirementId={需求}&taskId={任务}
  ↓ 进入需求看板，展开该任务所属需求，定位到该任务

点击需求的「详情」
  ↓ 跳转 /requirements?requirementId={需求}
  ↓ 进入需求看板，定位到该需求
```

> 体现：表格密度合理、类型/状态可一眼区分、变化提醒列简明。

## 9. 数据流动

控制台不存储独立业务数据，所有数据来源于三个上游系统，并在跳转后回到对应处理位置。

### 9.1 数据来源

```text
控制台数据来源
├─ 需求看板（事实源）
│  ├─ 需求 / 任务 / 任务树
│  ├─ AC（验收标准）
│  ├─ deadline
│  ├─ 阻塞状态
│  ├─ 依赖关系
│  └─ 关注关系
│
├─ 报告系统（事实源）
│  ├─ 报告状态（待生成 / 草稿 / 已发送…）
│  ├─ Session 列表（按日聚合）
│  ├─ 报告草稿内容
│  └─ 汇总报告（组 / 部门）
│
├─ Token 与 Session 系统
│  ├─ 各用户 / 各组的 token 用量
│  ├─ Session 上报记录
│  └─ Session 与任务的关联关系（用于"任务进展建议"）
│
└─ 用户与角色
   ├─ 当前用户
   ├─ 角色
   ├─ 所属组
   └─ 管理范围
```

### 9.2 数据流向

```text
                    需求看板  报告系统  Token系统  用户角色
                       │        │         │         │
                       └────────┴────┬────┴─────────┘
                                     ↓
                                  控制台
                       （聚合 + 展示 + 跳转）
                                     │
              ┌──────────────────────┼──────────────────────┐
              ↓                      ↓                      ↓
        点击风险/关注           点击报告按钮           点击 Token 明细
              │                      │                      │
              ↓                      ↓                      ↓
        需求看板定位            报告生成弹窗           Token 明细页
                                     │
                              ┌──────┴──────┐
                              ↓             ↓
                          报告状态        任务进度回写
                          回写报告系统    （仅个人日报）
                                          回写需求看板
```

### 9.3 关键口径

> **需求、任务、风险、关注状态以需求看板为唯一事实源；报告状态以报告系统为唯一事实源；Token 与 session 以 Token 系统为唯一事实源。控制台只做聚合展示，不持有独立数据。**

## 10. 核心用户流程

### 10.1 个人 - 处理今日日报

```text
用户登录 → 进入控制台
        ↓
  看到「今日日报 = 草稿待确认」
        ↓
  点击「确认日报草稿」→ 弹窗打开（步骤 2 编辑草稿）
        ↓
  阅读 Markdown 草稿
        ↓
  右侧看到 2 个任务进展建议
  └─ 修改其中 1 个的进度从 75% 改到 100%（状态改为已完成）
  └─ 该条建议显示「待同步」
        ↓
  点击「发送日报」
        ↓
  日报状态 → 已发送
  + 那个修改过的任务同步回写为「已完成 100%」
        ↓
  控制台关注列表中那个任务自动消失（或移到末尾"已完成"区）
```

### 10.2 个人 - 处理依赖阻塞风险

```text
用户进入控制台
        ↓
  风险列表第一条：「飞书发送联调等待上游任务完成」（高 · 依赖阻塞）
        ↓
  点击「处理依赖」
        ↓
  跳转 /requirements?requirementId=xxx&taskId=yyy&focus=dependency
        ↓
  需求看板展开「日报发送」需求
  + 定位到「飞书发送联调」任务
  + 高亮显示上游依赖任务「发送目标确认」（已超期）
        ↓
  用户在需求看板处理依赖（推动上游、调整 deadline、改依赖关系）
        ↓
  风险解除后，控制台风险列表自动刷新
```

### 10.3 TL - 处理今日组日报

```text
TL 登录 → 控制台
        ↓
  右上角组报告卡片：「今日组日报 = 生成失败」
        ↓
  点击「重新生成」→ 弹窗打开（步骤 1 确认来源）
        ↓
  弹窗展示成员提交情况：应提交 8，已提交 6，未提交 2，失败 0
        ↓
  点击「生成草稿」→ 弹窗跳到步骤 2 编辑
        ↓
  Markdown 草稿展示在编辑器中
  （组报告**没有**任务进展建议）
        ↓
  TL 修改草稿后点「发送组日报」
        ↓
  组日报状态 → 已发送
```

### 10.4 总监 - 查看 Token 概览

```text
总监登录 → 控制台
        ↓
  Token 统计卡片默认范围：近 3 天
        ↓
  切换到「近 7 天」
  ↓ 卡片即时更新：部门总量 34.2M、261 个 session
  ↓ 下方分组柱：芯片组 12.4M / 后台组 7.9M / 平台组 6.4M / 模型组 7.5M
        ↓
  点击「查看 Token 明细」→ 跳转 /tokens 页面深度分析
```

## 11. 空状态

```text
无风险：
└─ 风险列表区显示「暂无需要关注的风险」+ 「查看需求看板」文字链入口

无关注对象：
└─ 关注列表区显示「暂无关注的需求或任务」+ 「去需求看板关注」入口

无报告：
└─ 报告卡片显示「报告尚未生成」+ 「生成报告」按钮

无 Token 数据：
└─ Token 卡片显示「暂无 session 上报记录」+ 状态标签「暂无记录」
```

设计原则：空状态不空白，必须给用户下一步行动指引。

## 14. 评审重点

希望产品评审重点关注以下问题（这些是与旧版需求差异最大、最需要确认的）：

```text
评审关注点
├─ 任务进展建议这个交互合理吗？
│  ├─ AI 推断的进度准确度能接受多少？
│  ├─ 用户不修改直接发日报，会不会"漏改"任务？
│  └─ 是否需要在日报发送前给一个确认弹窗（"将同步 X 个任务"）？
│
├─ Token 统计在控制台首屏是必要的吗？
│  ├─ 总监最关心，个人是否觉得多余？
│  └─ 是否要让个人/PM 可隐藏？
│
├─ 风险只做 deadline + 依赖阻塞 够用吗？
│  └─ 旧版的"缺 AC"为什么不进 P0？
│
├─ 关注对象与风险列表是否会重复？
│  └─ 例如一个"已超期"的任务，既出现在关注列表"变化提醒"，又出现在风险列表，需要重复展示吗？
│
└─ 报告生成弹窗的三步是否过于细碎？
   └─ 个人日报能否合并成两步（选 session + 编辑同屏）？
```

## 15. 一句话基线

> 控制台是用户的"今日状态首屏"——一眼看到：今天日报怎么处理、关注对象变了什么、有哪些必须处理的风险、AI 工具用了多少 Token。处理动作要么在弹窗内完成（报告类），要么跳转到需求看板（业务类），控制台本身不堆复杂功能。
