# 周报现状盘点与调整方案

## 1. 当前代码现状

### 1.1 路由与页面

1. `/reports/weekly` 路由已存在，定义在 `web/src/router/routes.tsx:169`，组件为 `WeeklyReportsPage`。
2. 当前路由权限只包含 `admin/director/team_leader`，不包含 `employee/pm`，见 `web/src/router/routes.tsx:174`。因此 Employee/PM 默认不会在左侧菜单看到周报，也无法按正常权限进入。
3. `WeeklyReportsPage` 当前按角色整页分流，见 `web/src/features/aidashboard/reports/pages/WeeklyReportsPage.tsx:74`：
   - `director/admin`：进入 `DirectorWeeklyReportsView`，只显示部门周报。
   - `team_leader`：进入 `TeamWeeklyReportsView`，可生成、编辑、保存、提交小组周报。
   - `pm`：进入 `TeamWeeklyReportsView`，但 `canEdit=false`，只读小组周报来源、草稿和历史。
   - 其他角色：显示“当前产品口径暂未开放个人周报 / 个人周报暂未开放”。
4. 当前 `/reports/weekly` 只实现了小组周报和部门周报视角，缺失“我的周报”。
5. Director/Admin 当前有“小组记录”一级 tab，用于查看小组周报历史；这不符合新口径，后续应收敛到“部门周报 Step 1 来源确认”里。

3. 当前 `summaryReports` 只保留 `team_daily/department_daily`，把 `team_weekly/department_weekly` 过滤掉。
4. 点击 `personal_weekly/team_weekly/department_weekly` 统一 `navigate("/reports/weekly")`，见 `DashboardPage.tsx:1229`。这不符合“Dashboard 是当前周处理入口”的产品要求。

## 2. 产品口径修正

产品口径已确认：周报交互逻辑和日报同构，只是周期从“天”变成“周”。

```text
日报：每天执行
周报：每周执行

个人日报 -> 小组日报 -> 部门日报
个人周报 -> 小组周报 -> 部门周报
```

因此周报必须具备三层能力：

```text
我的周报
小组周报
部门周报
```

个人周报不是“暂不开放”能力，而是周报链路的第一层和小组周报的主来源。

## 3. 目标角色结构

```text
Employee：
- 我的周报

PM：
- 我的周报

TL：
- 我的周报
- 小组周报

Director/Admin：
- 我的周报
- 部门周报
```

说明：

1. PM 当前代码里有“小组周报只读视角”，但产品本次没有确认 PM 管理周报职责，因此本轮不启用、不作为目标能力。
2. Director/Admin 不需要“小组周报记录”一级 tab；总监查看小组周报应在“部门周报 Step 1 来源确认”里完成。
3. 所有角色都应能处理自己的“我的周报”，包括 Employee/PM/TL/Director/Admin。

## 4. 周报三层数据流

### 4.1 我的周报

我的周报用于生成个人周报，来源可以是：

```text
本周个人日报列表
```

个人周报的主输入是用户在弹窗中自由选择的个人日报列表，不再遍历或选择 session。它和个人日报生成弹窗保持同构：日报生成选择 session list，周报生成选择 daily report list。

个人周报生成不引入额外任务列表或任务/风险摘要上下文。风险、阻塞、下周计划都应由 Agent 从用户选择的日报内容中归纳。

生成行为必须是预览，不是保存。生成接口只返回 Markdown 内容和来源信息，前端展示在 Step 2；用户关闭弹窗且未点击保存/发送，当作没发生。

```text
generate result = report_markdown + source metadata
不写 content
不写 status=saved
不写 status=submitted
不新增 draft 状态
不创建正式周报记录
```

只有点击“保存周报”时才写入：

```text
personal_weekly_reports.content = 当前内容
personal_weekly_reports.status = saved
personal_weekly_reports.saved_at = now()
按 user_id + week_start upsert
```

只有点击“保存并发送”时才写入提交快照：

```text
personal_weekly_reports.content = 当前内容
personal_weekly_reports.submitted_content = content
personal_weekly_reports.status = submitted
personal_weekly_reports.submitted_at = now()
按 user_id + week_start upsert
```

### 4.2 小组周报

小组周报来源必须是 TL 在弹窗中自由选择的“TL 本人 + 小组成员”已发送个人周报：

```text
personal_weekly_reports.status = submitted
personal_weekly_reports.submitted_content
```

它和日报/个人周报生成弹窗保持同构：

```text
日报生成：选择 session list -> Agent 生成日报
个人周报生成：选择 daily report list -> Agent 生成个人周报
小组周报生成：选择 TL/member personal weekly report list -> Agent 生成小组周报
```

不能直接读取以下内容作为主来源：

```text
daily_reports.content
team_daily_reports.content
成员保存但未发送的 personal_weekly_reports.content
```

小组周报生成不引入本周成员日报、小组日报、任务/风险摘要作为辅助上下文。风险、阻塞、下周计划都应由 Agent 从 TL 选择的本人和成员个人周报内容中归纳。

边界规则：

```text
如果本周没有任何可选个人周报：
  默认不生成小组周报
  前端提示“暂无已发送个人周报，无法生成小组周报”

如果本周有可选个人周报，但 TL 未选择任何一份：
  禁用生成按钮或返回业务错误
  前端提示“请至少选择一份个人周报”
```

TL 本人或成员未发送个人周报时只作为缺失状态展示，不参与生成。

### 4.3 部门周报

部门周报来源继续是各 TL 已发送的小组周报：

```text
team_weekly_reports.status = submitted
team_weekly_reports.submitted_content
```

不能读取 TL 只保存但未发送的小组周报 `content`。

## 5. 数据表调整方案

### 5.1 新增个人周报表

推荐表名：`personal_weekly_reports`。

理由：当前项目已有 `daily_reports` 表表示个人日报，但周报需要和 `team_weekly_reports/department_weekly_reports` 明确区分层级；`personal_weekly_reports` 更清晰，避免 `weekly_reports` 被误解为所有周报的总表。

推荐字段：

```text
id
user_id
week_start
week_end
content
submitted_content
status
saved_at
submitted_at
submitted_to
source_daily_report_ids
created_at
updated_at
```

唯一约束：

```text
UNIQUE(user_id, week_start)
```

状态：

```text
saved
submitted
```

不需要 `draft` 持久化状态。

字段语义：

```text
content = 当前保存的周报内容
submitted_content = 最近一次发送给上级的周报内容
saved = 已保存，未发送最新版本
submitted = 已保存并发送给上级
```

发送对象：

```text
Employee -> TL
PM -> Director
TL -> Director
Director/Admin -> 只保存，不发送
```

### 5.2 改造 `team_weekly_reports`

当前表保留，补齐和日报一致的字段：

```text
submitted_content
status
saved_at
submitted_to
```

保留已有：

```text
submitted_at
```

状态：

```text
saved
submitted
```

迁移策略：

```text
submitted_at IS NOT NULL：
  status = submitted
  submitted_content = content

submitted_at IS NULL 且 content 不为空：
  status = saved
```

保存行为：

```text
content = 当前内容
status = saved
saved_at = now()
不更新 submitted_content
```

保存并发送行为：

```text
submitted_content = content
status = submitted
submitted_at = now()
submitted_to = director
```

### 5.3 改造 `department_weekly_reports`

当前表保留，补齐：

```text
status
```

继续保留：

```text
archived_at
```

部门周报没有发送动作，也不需要“保存但未归档”的中间状态。状态统一使用：

```text
archived
```

前端文案统一展示：

```text
已归档
```

历史旧数据如 `archived_at IS NOT NULL`，可视为 `archived`。

当前 `department_weekly_reports` 只有 `week_start` 唯一，P0 单部门场景可以接受。若未来支持多部门或多总监，需要补充：

```text
department_id 或 director_id
UNIQUE(department_id, week_start)
```

并同步把部门周报 current 查询、归档 upsert、历史列表按部门隔离。

## 6. API 调整方案

以下路径按现有后端实际前缀可落到 `/api/v1` 下；文档只描述业务路由。

### 6.1 我的周报接口

新增完整接口组：

```http
GET  /api/v1/reports/weekly/mine?from_week=&to_week=&page=&page_size=
GET  /api/v1/reports/weekly/mine/current?week_start=
GET  /api/v1/reports/weekly/mine/sources?week_start=
POST /api/v1/reports/weekly/mine/current/generate
PUT  /api/v1/reports/weekly/mine/current
POST /api/v1/reports/weekly/mine/current/submit
```

用途：

| 接口 | 用途 |
| --- | --- |
| `mine` | 我的周报历史分页 |
| `current` | 当前周我的周报详情 |
| `sources` | 本周日报来源确认，返回可选个人日报列表 |
| `generate` | 基于用户选择的日报列表生成个人周报预览，返回 Markdown 和来源信息，不落最终状态 |
| `PUT current` | 保存当前周我的周报，按 `user_id/week_start` upsert，`status=saved` |
| `current/submit` | 保存并发送当前周我的周报，按 `user_id/week_start` upsert，`status=submitted`，`submitted_content=content` |

权限：

1. Employee/PM/TL/Director/Admin 均可访问自己的个人周报。
2. Employee submit 默认发送给 TL。
3. PM submit 默认发送给 Director。
4. TL submit 默认发送给 Director。
5. Director/Admin 无 submit 按钮；后端可拒绝 submit 或将其视为保存，建议前端不暴露。

generate 接口不得先创建正式周报记录。当前周保存/发送不应依赖 id 作为唯一入口，统一走 `current` upsert；历史记录可以继续使用 id 做详情、查看或后续管理。

个人周报生成请求：

```json
{
  "week_start": "2026-06-22",
  "source_daily_report_ids": ["..."]
}
```

其中 `source_daily_report_ids` 来自 Step 1 勾选的日报列表，是 P0 必填主来源。请求不再包含 `source_session_ids` 或 `source_task_ids`。

个人周报生成响应：

```json
{
  "report_markdown": "# 本周周报\n...",
  "week_start": "2026-06-22",
  "week_end": "2026-06-28",
  "source_daily_report_ids": ["..."]
}
```

## 7. 前端页面与弹窗交互方案

### 7.1 `/reports/weekly` 页面结构

不要继续按角色整页分流成单一视角，改为按角色展示 tab/section：

```text
Employee：
[我的周报]

PM：
[我的周报]

TL：
[我的周报] [小组周报]

Director/Admin：
[我的周报] [部门周报]
```

不为 Director/Admin 增加“小组周报记录”一级 tab。小组周报原文查看放入部门周报 Step 1 来源确认。

### 7.2 我的周报交互

和个人日报一致，两步：

```text
Step 1：确认来源
Step 2：确认周报
```

Step 1 的核心交互是选择本周个人日报列表：

```text
日报生成：选择 session list -> Agent 生成日报
周报生成：选择 daily report list -> Agent 生成周报
```

来源确认页展示当前用户本周可选日报，支持勾选/取消勾选。默认勾选本周已保存或已发送的个人日报；用户可以自由选择其中任意多条后生成周报。若没有可选日报，生成按钮禁用，并提示“本周暂无可用于生成周报的日报”。

## 8. Dashboard 周报接入方案

Dashboard 是当前周处理入口，`/reports/weekly` 是历史记录/列表页。当前周报按钮统一跳 `/reports/weekly` 不符合产品要求，不能继续用 `navigate("/reports/weekly")` 代替当前周处理弹窗。

目标：

```text
Dashboard 我的周报卡片 -> 打开我的周报弹窗
Dashboard 小组周报卡片 -> TL 打开小组周报弹窗
Dashboard 部门周报卡片 -> Director/Admin 打开部门周报弹窗
```

Dashboard 周报状态必须来自真实 weekly API，不能继续用 `ROLE_DATA` 假状态。

Dashboard 和 `/reports/weekly` 必须复用同一套周报弹窗组件和 API，避免出现两套状态机、两套按钮语义或两套保存/发送行为。

状态映射：

我的周报：

```text
无记录：待生成
status=saved：已保存
status=submitted：已发送
```

小组周报：

```text
无记录：待生成
status=saved：已保存
status=submitted：已发送
```

部门周报：

```text
无记录：待生成
status=archived：已归档
历史 archived_at IS NOT NULL：已归档
```

实现建议：

1. Dashboard 查询当前周个人周报接口，所有角色启用。
2. TL 额外查询当前周小组周报接口。
3. Director/Admin 额外查询当前周部门周报接口。
4. 周报弹窗复用 `/reports/weekly` 的两步组件和同一套 API。
5. `ROLE_DATA` 只保留非状态性的展示兜底，周报状态、时间、来源数量必须来自 API。

## 9. daemon / Agent 生成方案

### 9.1 新增个人周报生成逻辑

新增 daemon 内部入口：

```http
POST /reports/weekly/generate
```

输入：

```text
user_id
week_start
source_daily_report_ids
```

生成来源：

```text
用户选择的本周个人日报内容
```

个人周报生成不再遍历 session，也不接收 `source_session_ids/source_task_ids`。如果后续确实需要追溯日报来源 session，应通过日报自身的 `session_ids` 做只读追踪，不作为周报生成入口的选择对象。

生成结果只作为预览返回，不直接写入最终状态：

```text
report_markdown
source_daily_report_ids
不创建 personal_weekly_reports 正式记录
```

用户点击“保存周报”或“保存并发送”后，再由 API 写入 `personal_weekly_reports`。关闭弹窗且未保存时，视为未发生。

### 9.2 调整小组周报生成逻辑

小组周报生成来源改为 TL 选择的本人和成员个人周报：

```text
source_personal_weekly_report_ids
personal_weekly_reports.status = submitted
personal_weekly_reports.submitted_content
```

生成接口不再遍历 session、本周成员日报、小组日报或任务/风险摘要。TL 可以在 Step 1 自由勾选本周已发送的本人和成员个人周报，Agent 只能基于所选 `submitted_content` 归纳小组本周完成、关键进展、风险阻塞和下周计划。

如果本周没有任何可选个人周报：

```text
默认不生成小组周报
提示“暂无已发送个人周报，无法生成小组周报”
```

如果有可选个人周报但 TL 未选择任何一份：

```text
默认不生成小组周报
提示“请至少选择一份个人周报”
```

生成结果只作为预览返回：

```text
report_markdown
source_personal_weekly_report_ids
不创建 team_weekly_reports 正式记录
```

### 9.3 调整部门周报生成逻辑

部门周报继续读取：

```text
已发送小组周报 submitted_content
```

生成结果只作为预览返回：

```text
report_markdown
source_team_weekly_report_ids
不创建 department_weekly_reports 正式记录
```

归档后：

```text
department_weekly_reports.content = 当前内容
department_weekly_reports.status = archived
department_weekly_reports.archived_at = now()
```

### 9.4 当前代码按日报列表生成的修改方案

当前已实现的个人周报链路里，`sources/generate/save/submit` 同时携带了 `source_session_ids/source_daily_report_ids/source_task_ids`，前端也在“我的周报”来源页展示了 Session 指标和任务/风险辅助信息。这与新的产品口径不一致，需要调整为“日报列表为唯一生成来源”。

后端 API 调整：

1. `PersonalWeeklyReportSources` 只保留 `daily_reports/daily_count`，移除或废弃 `sessions/session_count/tasks/task_count`。
2. `GetPersonalWeeklyReportSources` 查询当前用户本周个人日报列表，返回可选日报内容、日期、状态、更新时间；不要查询 session 或任务列表。
3. `GeneratePersonalWeeklyReportPreview` 只接受 `source_daily_report_ids`，校验所选日报必须属于当前用户且位于 `week_start/week_end` 内。
4. `personalWeeklySourceIDs` 调整为只过滤日报 ID 的 helper。
5. `SavePersonalWeeklyReportCurrent` / `SubmitPersonalWeeklyReportCurrent` 持久化 `source_daily_report_ids`；如果数据库已存在 `source_session_ids/source_task_ids` 字段，可以先写空数组以兼容旧迁移，但新逻辑不再依赖它们。

daemon / Agent 调整：

1. `/reports/weekly/generate` 请求结构移除 `SourceSessionIDs/SourceTaskIDs`。
2. `generateServerPersonalWeeklyReportPreview` 不再调用 session 查询或任务查询逻辑，只读取用户选择的日报内容。
3. prompt 明确要求 Agent 只能基于所选日报内容归纳本周完成、风险阻塞和下周计划。
4. fallback 逻辑同样基于日报列表生成，不再列 session 或任务/风险摘要。

前端调整：

1. `PersonalWeeklyReportsView` 的 Step 1 改为日报列表选择器，默认勾选本周已保存或已发送日报。
2. 指标卡从 `Session` 改为 `已选日报 / 可选日报`，不展示任务/风险辅助指标。
3. 点击“生成周报预览”时提交用户勾选的 `source_daily_report_ids`，不再从 sources 自动取 session IDs。
4. Dashboard “今日报告”模块恢复“我的周报”行入口，点击后打开同一个“我的周报”弹窗。
5. 保存/发送时沿用当前编辑器内容和已选日报 ID；关闭弹窗且未保存时不落库。

兼容策略：

1. 如果线上表已经有 `source_session_ids/source_task_ids`，本轮可以不立即删字段，先写空数组，降低迁移风险。
2. API 响应可以短期继续返回空的 `source_session_ids: []` / `source_task_ids: []` 兼容旧前端，但新前端类型和业务逻辑不再依赖它们。
3. 后续清理阶段再删除 `source_session_ids/source_task_ids` 字段和相关类型。

### 9.5 当前代码按成员周报列表生成小组周报的修改方案

当前已实现的小组周报链路里，`sources/generate` 仍以本周成员日报、小组日报、任务/风险为主要来源，生成接口会直接创建或更新 `team_weekly_reports` 草稿。这与新的产品口径不一致，需要调整为“TL 自由选择本人和成员已发送个人周报列表 -> Agent 生成小组周报预览 -> 保存/发送”。

后端 API 调整：

1. 新增或调整 `TeamWeeklyReportSources`：返回 `submitted_personal_weekly_reports` 和 `missing_people`。
2. `submitted_personal_weekly_reports` 查询当前 TL 本人和当前 TL 小组内成员的 `personal_weekly_reports.status = submitted` 且 `week_start` 匹配的记录，字段包含 `report_id/user_id/user_name/source_role/week_start/week_end/submitted_at/submitted_content`，其中 `source_role` 用于区分 `leader/member`。
3. `missing_people` 返回本周没有已发送个人周报的 TL 本人和成员，用于 Step 1 展示缺失状态，不参与生成。
4. `GenerateTeamWeeklyReportPreview` 只接受 `source_personal_weekly_report_ids`，校验所选周报必须属于当前 TL 本人或当前 TL 的小组成员、同一 `week_start`、且状态为 `submitted`。
5. 如果本周没有可选个人周报，返回“暂无已发送个人周报，无法生成小组周报”；如果过滤后为空，返回“请至少选择一份个人周报”。
6. 生成接口只返回 `report_markdown/week_start/week_end/source_personal_weekly_report_ids`，不创建或更新 `team_weekly_reports`。
7. `SaveTeamWeeklyReportCurrent` / `SubmitTeamWeeklyReportCurrent` 按 `team_id + week_start` upsert，保存/发送时持久化 `source_personal_weekly_report_ids`；如果表仍使用旧字段，可以先映射到兼容字段或新增明确字段。

数据表调整：

1. 推荐在 `team_weekly_reports` 增加 `source_personal_weekly_report_ids UUID[] NOT NULL DEFAULT '{}'`，避免继续用 `source_daily_report_ids/source_team_report_ids/source_task_ids` 表达新语义。
2. 同步补齐 `status/submitted_content/saved_at/submitted_to`，让小组周报和个人周报状态模型一致。
3. 旧字段可以短期保留但新逻辑不再写入；后续清理阶段再删除或废弃旧来源字段。

daemon / Agent 调整：

1. `/reports/team/weekly/generate` 请求结构改为接收 `team_id/leader_id/week_start/source_personal_weekly_report_ids`。
2. `generateServerTeamWeeklyReport` 不再调用 TL session、成员日报、小组日报、任务/风险查询逻辑，只读取所选本人和成员个人周报的 `submitted_content`。
3. prompt 明确要求 Agent 只能基于所选本人和成员个人周报内容归纳小组完成、关键进展、风险阻塞和下周计划。
4. fallback 逻辑同样基于所选个人周报列表生成，不再列 session、日报、小组日报或任务/风险摘要。

前端调整：

1. `TeamWeeklyReportsView` 的 Step 1 改为个人周报列表选择器，默认勾选 TL 本人和成员全部已发送个人周报。
2. 支持勾选/取消勾选，用户可以自由选择任意多份个人周报后生成草稿。
3. 指标卡从 `个人日报/小组日报/任务风险` 改为 `已发送个人周报 / 已选个人周报 / 未发送人员`。
4. 点击“生成小组周报预览”时提交 `source_personal_weekly_report_ids`，不再从 sources 自动取日报、任务或小组日报来源。
5. Step 1 保留未发送人员列表和查看原文能力；未发送人员不参与生成。
6. 保存/发送时沿用当前编辑器内容和已选成员周报 ID；关闭弹窗且未保存时不落库。
7. Dashboard “小组周报”卡片和 `/reports/weekly` 小组周报 tab 复用同一套组件和 API。

兼容策略：

1. 如果线上 `team_weekly_reports` 已有 `source_daily_report_ids/source_team_report_ids/source_task_ids`，本轮可以先保留字段，停止把它们作为小组周报生成来源。
2. 若无法立即新增 `source_personal_weekly_report_ids`，可短期使用单独迁移补字段后再切逻辑，避免把个人周报 ID 塞进旧语义字段。
3. 旧的“生成即落库”行为需要改为“生成只预览，保存/发送才 upsert”，否则会继续出现关闭弹窗也留下草稿的问题。

## 11. 不做范围

1. 不改日报已确定逻辑。
2. 不接飞书。
3. 不引入审批流。
4. 不把 Dashboard 周报按钮继续做成跳转。
5. 不用 `ROLE_DATA` / 本地模板伪造周报状态。
6. 不保留 Employee/PM 周报屏蔽。
7. 不把 PM 小组周报只读视角作为本轮目标能力，除非产品另行确认。
8. 不让小组周报继续绕过个人周报主链路。
9. 不让部门周报读取 TL 保存但未发送的小组周报 `content`。
10. 不做 `draft` 持久化状态。

## 12. 待确认问题

1. PM 个人周报默认发送给 Director 是否最终确认？当前按 `PM -> Director` 设计；如果后续产品确认 PM 属于某 TL 管理，则改为 `PM -> TL`，并同步调整 `submitted_to` 和按钮文案。
2. Director/Admin 的“我的周报”是否只保存、不发送？本文按只保存设计。
3. 小组周报的 `source_personal_weekly_report_ids` 是否新增独立字段，还是通过兼容迁移替换旧来源字段？推荐新增独立字段，避免语义混淆。
4. 历史记录分页默认 `page_size`、排序和筛选字段是否完全沿用日报列表规范？

## 13. 最终结论

1. 当前真实能力：小组周报和部门周报已有真实接口、表和 daemon 生成逻辑，但状态模型不完整；个人周报没有表、接口和生成逻辑。
2. 当前个人/PM 周报不可见原因：`/reports/weekly` 路由 roles 排除了 Employee/PM；页面对 Employee 显示“个人周报暂未开放”；PM 即使进入页面也被分到小组周报只读视角。
3. 当前 mock / 假状态：Dashboard 的 `ROLE_DATA` 周报状态和默认模板是本地伪状态；`/reports/weekly` 的小组/部门周报数据是真实 API。
4. 目标周报三层链路是：`个人周报 -> 小组周报 -> 部门周报`。
5. 生成只是预览：生成接口返回 Markdown 和来源信息，不创建正式周报记录，不写 `status=saved`，不写 `status=submitted`，不新增 `draft` 状态。
6. 保存才是 `saved`：只有点击“保存周报”才写 `content/status=saved/saved_at`；当前周保存统一走 `current` upsert，按 `user_id/week_start` 或 `team_id/week_start` 定位，不依赖 id。
7. 保存并发送才是 `submitted`：只有点击“保存并发送”才写 `content/submitted_content/status=submitted/submitted_at/submitted_to`；当前周发送统一走 `current/submit` upsert。
8. 部门周报只有“保存归档”最终动作，归档后是 `status=archived/archived_at`，前端展示“已归档”；当前周归档统一走 `current/archive` upsert，P0 按 `week_start` 定位，未来多部门按 `department_id/week_start` 定位。
9. Dashboard 周报不再用跳转替代弹窗；Dashboard 当前周入口必须打开对应周报弹窗，并和 `/reports/weekly` 复用同一套弹窗组件和 API。
10. 个人周报生成必须基于用户选择的本周个人日报列表，不遍历 session，不复用日报生成的 session 选择器，也不引入额外任务/风险摘要上下文。
11. 必须新增：`personal_weekly_reports` 表、个人周报完整 API、前端个人周报类型和 API client、个人周报两步弹窗、daemon 个人周报生成入口。
12. 必须改造：小组周报来源改为 TL 自由选择的本人和成员已发送个人周报；部门周报来源改为已发送小组周报 `submitted_content`；小组/部门周报补齐同构状态字段；Dashboard 周报状态接真实 API。
13. 推荐路径：Phase 1 严格只打通所有角色“我的周报”；Phase 2 改造 TL 小组周报；Phase 3 改造总监部门周报。
