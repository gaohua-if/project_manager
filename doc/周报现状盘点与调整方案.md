# 周报现状盘点与调整方案

## 1. 当前代码现状

### 1.1 路由与页面

1. `/reports/weekly` 路由已存在，定义在 `web/src/router/routes.tsx:169`，组件为 `WeeklyReportsPage`。
2. 当前路由权限只包含 `admin/director/team_leader`，不包含 `employee/pm`，见 `web/src/router/routes.tsx:174`。因此 Employee/PM 默认不会在左侧菜单看到周报，也无法按正常权限进入。
3. `WeeklyReportsPage` 当前按角色整页分流，见 `web/src/features/aidashboard/reports/pages/WeeklyReportsPage.tsx:74`：
   - `director/admin`：进入 `DirectorWeeklyReportsView`，只显示部门周报。
   - `team_leader`：进入 `TeamWeeklyReportsView`，可生成、编辑、保存、提交小组周报。
   - `pm`：进入 `TeamWeeklyReportsView`，但 `canEdit=false`，只读小组周报来源、草稿和历史。
   - 其他角色：显示“当前产品口径暂未开放个人周报 / 个人周报暂未开放”。
4. 当前 `/reports/weekly` 只实现了小组周报和部门周报视角，缺失“我的周报”。
5. Director/Admin 当前有“小组记录”一级 tab，用于查看小组周报历史；这不符合新口径，后续应收敛到“部门周报 Step 1 来源确认”里。

2. Dashboard 只把 `personal_daily` 用真实日报接口状态覆盖；`personal_weekly` 仍是本地假状态。
3. 当前 `summaryReports` 只保留 `team_daily/department_daily`，把 `team_weekly/department_weekly` 过滤掉。
4. 点击 `personal_weekly/team_weekly/department_weekly` 统一 `navigate("/reports/weekly")`，见 `DashboardPage.tsx:1229`。这不符合“Dashboard 是当前周处理入口”的产品要求。
5. 当前未检索到 `reportStateById`；`updateReport` 已为空函数并保留旧分支注释。

## 2. 产品口径修正

产品口径已确认：周报交互逻辑和日报同构，只是周期从“天”变成“周”。

```text
日报：每天执行
周报：每周执行

个人日报 -> 小组日报 -> 部门日报
个人周报 -> 小组周报 -> 部门周报
```

因此周报必须具备三层能力：

```text
我的周报
小组周报
部门周报
```

个人周报不是“暂不开放”能力，而是周报链路的第一层和小组周报的主来源。

## 3. 目标角色结构

```text
Employee：
- 我的周报

PM：
- 我的周报

TL：
- 我的周报
- 小组周报

Director/Admin：
- 我的周报

```

说明：

1. PM 当前代码里有“小组周报只读视角”，但产品本次没有确认 PM 管理周报职责，因此本轮不启用、不作为目标能力。
2. Director/Admin 不需要“小组周报记录”一级 tab；总监查看小组周报应在“部门周报 Step 1 来源确认”里完成。
3. 所有角色都应能处理自己的“我的周报”，包括 Employee/PM/TL/Director/Admin。

## 4. 周报三层数据流

### 4.1 我的周报

我的周报用于生成个人周报，来源可以是：

```text
本周 session
本周个人日报
本周任务/风险摘要
```

生成行为必须是预览，不是保存。生成接口只返回 Markdown 内容和来源信息，前端展示在 Step 2；用户关闭弹窗且未点击保存/发送，当作没发生。

```text
generate result = report_markdown + source metadata
不写 content
不写 status=saved
不写 status=submitted
不新增 draft 状态
不创建正式周报记录
```

只有点击“保存周报”时才写入：

```text
personal_weekly_reports.content = 当前内容
personal_weekly_reports.status = saved
personal_weekly_reports.saved_at = now()
按 user_id + week_start upsert
```

只有点击“保存并发送”时才写入提交快照：

```text
personal_weekly_reports.content = 当前内容
personal_weekly_reports.submitted_content = content
personal_weekly_reports.status = submitted
personal_weekly_reports.submitted_at = now()
按 user_id + week_start upsert
```

### 4.2 小组周报

小组周报主来源必须是成员已发送个人周报：

```text
personal_weekly_reports.status = submitted
personal_weekly_reports.submitted_content
```

不能直接读取以下内容作为主来源：

```text
daily_reports.content
team_daily_reports.content
成员保存但未发送的 personal_weekly_reports.content
```

本期如果为了兼容已有能力，可以把本周成员日报、小组日报、任务/风险作为辅助输入，但不能替代“成员已发送个人周报”这条主链路。

边界规则：

```text
如果本周没有任何成员已发送个人周报：
  默认不生成小组周报
  前端提示“暂无已发送成员周报，无法生成小组周报”
```

本周成员日报、小组日报、任务/风险只能作为辅助输入或后续增强，不能替代“成员已发送个人周报”主链路。

### 4.3 部门周报

部门周报来源继续是各 TL 已发送的小组周报：

```text
team_weekly_reports.status = submitted
team_weekly_reports.submitted_content
```

不能读取 TL 只保存但未发送的小组周报 `content`。

## 5. 数据表调整方案

### 5.1 新增个人周报表

推荐表名：`personal_weekly_reports`。

理由：当前项目已有 `daily_reports` 表表示个人日报，但周报需要和 `team_weekly_reports/department_weekly_reports` 明确区分层级；`personal_weekly_reports` 更清晰，避免 `weekly_reports` 被误解为所有周报的总表。

推荐字段：

```text
id
user_id
week_start
week_end
content
submitted_content
status
saved_at
submitted_at
submitted_to
source_daily_report_ids
source_session_ids
source_task_ids
created_at
updated_at
```

唯一约束：

```text
UNIQUE(user_id, week_start)
```

状态：

```text
saved
submitted
```

不需要 `draft` 持久化状态。

字段语义：

```text
content = 当前保存的周报内容
submitted_content = 最近一次发送给上级的周报内容
saved = 已保存，未发送最新版本
submitted = 已保存并发送给上级
```

发送对象：

```text
Employee -> TL
PM -> Director
TL -> Director
Director/Admin -> 只保存，不发送
```

### 5.2 改造 `team_weekly_reports`

当前表保留，补齐和日报一致的字段：

```text
submitted_content
status
saved_at
submitted_to
```

保留已有：

```text
submitted_at
```

状态：

```text
saved
submitted
```

迁移策略：

```text
submitted_at IS NOT NULL：
  status = submitted
  submitted_content = content

submitted_at IS NULL 且 content 不为空：
  status = saved
```

保存行为：

```text
content = 当前内容
status = saved
saved_at = now()
不更新 submitted_content
```

保存并发送行为：

```text
submitted_content = content
status = submitted
submitted_at = now()
submitted_to = director
```

### 5.3 改造 `department_weekly_reports`

当前表保留，补齐：

```text
status
```

继续保留：

```text
archived_at
```

部门周报没有发送动作，也不需要“保存但未归档”的中间状态。状态统一使用：

```text
archived
```

前端文案统一展示：

```text
已归档
```

历史旧数据如 `archived_at IS NOT NULL`，可视为 `archived`。

当前 `department_weekly_reports` 只有 `week_start` 唯一，P0 单部门场景可以接受。若未来支持多部门或多总监，需要补充：

```text
department_id 或 director_id
UNIQUE(department_id, week_start)
```

并同步把部门周报 current 查询、归档 upsert、历史列表按部门隔离。

## 6. API 调整方案

以下路径按现有后端实际前缀可落到 `/api/v1` 下；文档只描述业务路由。

### 6.1 我的周报接口

新增完整接口组：

```http
GET  /api/v1/reports/weekly/mine?from_week=&to_week=&page=&page_size=
GET  /api/v1/reports/weekly/mine/current?week_start=
GET  /api/v1/reports/weekly/mine/sources?week_start=
POST /api/v1/reports/weekly/mine/current/generate
PUT  /api/v1/reports/weekly/mine/current
POST /api/v1/reports/weekly/mine/current/submit
```

用途：

| 接口 | 用途 |
| --- | --- |
| `mine` | 我的周报历史分页 |
| `current` | 当前周我的周报详情 |
| `sources` | 本周来源确认 |
| `generate` | 基于来源生成个人周报预览，返回 Markdown 和来源信息，不落最终状态 |
| `PUT current` | 保存当前周我的周报，按 `user_id/week_start` upsert，`status=saved` |
| `current/submit` | 保存并发送当前周我的周报，按 `user_id/week_start` upsert，`status=submitted`，`submitted_content=content` |

权限：

1. Employee/PM/TL/Director/Admin 均可访问自己的个人周报。
2. Employee submit 默认发送给 TL。
3. PM submit 默认发送给 Director。
4. TL submit 默认发送给 Director。
5. Director/Admin 无 submit 按钮；后端可拒绝 submit 或将其视为保存，建议前端不暴露。

generate 接口不得先创建正式周报记录。当前周保存/发送不应依赖 id 作为唯一入口，统一走 `current` upsert；历史记录可以继续使用 id 做详情、查看或后续管理。

## 8. Dashboard 周报接入方案

Dashboard 是当前周处理入口，`/reports/weekly` 是历史记录/列表页。当前周报按钮统一跳 `/reports/weekly` 不符合产品要求，不能继续用 `navigate("/reports/weekly")` 代替当前周处理弹窗。

目标：

```text
Dashboard 我的周报卡片 -> 打开我的周报弹窗
Dashboard 小组周报卡片 -> TL 打开小组周报弹窗
Dashboard 部门周报卡片 -> Director/Admin 打开部门周报弹窗
```

Dashboard 周报状态必须来自真实 weekly API，不能继续用 `ROLE_DATA` 假状态。

Dashboard 和 `/reports/weekly` 必须复用同一套周报弹窗组件和 API，避免出现两套状态机、两套按钮语义或两套保存/发送行为。

状态映射：

我的周报：

```text
无记录：待生成
status=saved：已保存
status=submitted：已发送
```

小组周报：

```text
无记录：待生成
status=saved：已保存
status=submitted：已发送
```

部门周报：

```text
无记录：待生成
status=archived：已归档
历史 archived_at IS NOT NULL：已归档
```

实现建议：

1. Dashboard 查询当前周个人周报接口，所有角色启用。
2. TL 额外查询当前周小组周报接口。
3. Director/Admin 额外查询当前周部门周报接口。
4. 周报弹窗复用 `/reports/weekly` 的两步组件和同一套 API。
5. `ROLE_DATA` 只保留非状态性的展示兜底，周报状态、时间、来源数量必须来自 API。

## 9. daemon / Agent 生成方案

### 9.1 新增个人周报生成逻辑

新增 daemon 内部入口：

```http
POST /reports/weekly/generate
```

输入：

```text
user_id
week_start
source_session_ids
source_daily_report_ids
source_task_ids
```

生成来源：

```text
本周 session
本周个人日报
本周任务/风险摘要
```

生成结果只作为预览返回，不直接写入最终状态：

```text
report_markdown
source_session_ids
source_daily_report_ids
source_task_ids
不创建 personal_weekly_reports 正式记录
```

用户点击“保存周报”或“保存并发送”后，再由 API 写入 `personal_weekly_reports`。关闭弹窗且未保存时，视为未发生。

### 9.2 调整小组周报生成逻辑

小组周报生成主来源改为：

```text
成员已发送个人周报 submitted_content
```

现有本周成员日报、小组日报、任务/风险可以作为辅助输入，但不能绕过个人周报主链路。

如果本周没有任何成员已发送个人周报：

```text
默认不生成小组周报
提示“暂无已发送成员周报，无法生成小组周报”
```

生成结果只作为预览返回：

```text
report_markdown
source_personal_weekly_report_ids
辅助来源 ids
不创建 team_weekly_reports 正式记录
```

### 9.3 调整部门周报生成逻辑

部门周报继续读取：

```text
已发送小组周报 submitted_content
```

生成结果只作为预览返回：

```text
report_markdown
source_team_weekly_report_ids
不创建 department_weekly_reports 正式记录
```

归档后：

```text
department_weekly_reports.content = 当前内容
department_weekly_reports.status = archived
department_weekly_reports.archived_at = now()
```

## 10. 分阶段实施方案

### Phase 1：个人周报最小闭环

目标：

```text
所有角色都能处理“我的周报”
```

包括：

```text
personal_weekly_reports 表
我的周报 mine/sources/current/generate/save/submit/history 接口
个人周报弹窗
/reports/weekly 我的周报 tab
Dashboard 我的周报真实状态
Employee/PM 不再被屏蔽
```

不做：

```text
小组周报入口调整
小组周报来源改造
小组周报状态逻辑改造
部门周报入口调整
部门周报来源改造
部门周报状态逻辑改造
PM 小组周报只读管理视角
```

Phase 1 严格只做“我的周报”闭环，不改现有小组周报和部门周报的入口、来源、状态逻辑，避免破坏当前已能工作的“小组周报/部门周报”能力。小组周报来源切换到 `personal_weekly_reports.submitted_content` 放到 Phase 2；部门周报来源切换到 `team_weekly_reports.submitted_content` 放到 Phase 3。

### Phase 2：小组周报同构改造

目标：

```text
TL 小组周报基于成员已发送个人周报生成
```

包括：

```text
team_weekly_reports 补 status/submitted_content/saved_at/submitted_to
小组周报 sources 改为查 personal_weekly_reports.submitted_content
小组周报保存 / 保存并发送给总监
Dashboard 小组周报真实状态
小组周报历史分页
```

### Phase 3：部门周报同构改造

目标：

```text
总监部门周报基于已发送小组周报生成
```

包括：

```text
department_weekly_reports 补 status/archived_at 语义
部门周报 sources 改为查 team_weekly_reports.submitted_content
部门周报保存归档
Dashboard 部门周报真实状态
部门周报历史分页
```

## 11. 不做范围

1. 不改日报已确定逻辑。
2. 不接飞书。
3. 不引入审批流。
4. 不把 Dashboard 周报按钮继续做成跳转。
5. 不用 `ROLE_DATA` / 本地模板伪造周报状态。
6. 不保留 Employee/PM 周报屏蔽。
7. 不把 PM 小组周报只读视角作为本轮目标能力，除非产品另行确认。
8. 不让小组周报继续绕过个人周报主链路。
9. 不让部门周报读取 TL 保存但未发送的小组周报 `content`。
10. 不做 `draft` 持久化状态。

## 12. 待确认问题

1. PM 个人周报默认发送给 Director 是否最终确认？当前按 `PM -> Director` 设计；如果后续产品确认 PM 属于某 TL 管理，则改为 `PM -> TL`，并同步调整 `submitted_to` 和按钮文案。
2. Director/Admin 的“我的周报”是否只保存、不发送？本文按只保存设计。
3. 个人周报来源里的任务/风险摘要是由 API 汇总后传给 daemon，还是 daemon 自行查库？
4. 小组周报辅助输入是否保留本周成员日报/小组日报/任务风险？如果保留，需要在生成 prompt 中明确“辅助输入，不替代成员已发送个人周报”。
5. 历史记录分页默认 `page_size`、排序和筛选字段是否完全沿用日报列表规范？

## 13. 最终结论

1. 当前真实能力：小组周报和部门周报已有真实接口、表和 daemon 生成逻辑，但状态模型不完整；个人周报没有表、接口和生成逻辑。
2. 当前个人/PM 周报不可见原因：`/reports/weekly` 路由 roles 排除了 Employee/PM；页面对 Employee 显示“个人周报暂未开放”；PM 即使进入页面也被分到小组周报只读视角。
3. 当前 mock / 假状态：Dashboard 的 `ROLE_DATA` 周报状态和默认模板是本地伪状态；`/reports/weekly` 的小组/部门周报数据是真实 API。
4. 目标周报三层链路是：`个人周报 -> 小组周报 -> 部门周报`。
5. 生成只是预览：生成接口返回 Markdown 和来源信息，不创建正式周报记录，不写 `status=saved`，不写 `status=submitted`，不新增 `draft` 状态。
6. 保存才是 `saved`：只有点击“保存周报”才写 `content/status=saved/saved_at`；当前周保存统一走 `current` upsert，按 `user_id/week_start` 或 `team_id/week_start` 定位，不依赖 id。
7. 保存并发送才是 `submitted`：只有点击“保存并发送”才写 `content/submitted_content/status=submitted/submitted_at/submitted_to`；当前周发送统一走 `current/submit` upsert。
8. 部门周报只有“保存归档”最终动作，归档后是 `status=archived/archived_at`，前端展示“已归档”；当前周归档统一走 `current/archive` upsert，P0 按 `week_start` 定位，未来多部门按 `department_id/week_start` 定位。
9. Dashboard 周报不再用跳转替代弹窗；Dashboard 当前周入口必须打开对应周报弹窗，并和 `/reports/weekly` 复用同一套弹窗组件和 API。
10. 必须新增：`personal_weekly_reports` 表、个人周报完整 API、前端个人周报类型和 API client、个人周报两步弹窗、daemon 个人周报生成入口。
11. 必须改造：小组周报来源改为成员已发送个人周报；部门周报来源改为已发送小组周报 `submitted_content`；小组/部门周报补齐同构状态字段；Dashboard 周报状态接真实 API。
12. 推荐路径：Phase 1 严格只打通所有角色“我的周报”；Phase 2 改造 TL 小组周报；Phase 3 改造总监部门周报。
