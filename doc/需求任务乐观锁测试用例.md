# 需求任务乐观锁测试用例

## 0. 测试目标

基于《需求任务乐观锁 P0 改造方案》，设计一套接口级真实模拟测试，用真实 HTTP API 调用验证：

- `requirements.version` 是否只保护需求本体字段。
- `tasks.version` 是否保护任务本体字段和任务依赖。
- 写接口是否必须携带 `base_version`。
- 版本冲突是否返回 `409 EDIT_CONFLICT`。
- `409` 是否包含 `current_version`。
- 是否能防止多人同时编辑时的静默覆盖。
- 任务变化是否不会误 bump 父需求 `requirements.version`。
- 业务 `409` 是否不会被误判为并发冲突。
- 成功写接口是否返回最新 `version`，避免连续操作自己触发 `409`。

本测试不是前端 E2E，不使用浏览器，只通过真实 HTTP API 模拟客户端 A/B。

7. 若无法选择合法 `team_id` 或 `assignee_id`，所有依赖创建需求 / 任务的用例标记 `SKIP`。

```json
  "priority": "medium",

```

```json

```

## 3. 测试脚本设计

### 3.1 结果状态

每个 case 输出：

- `PASS`：实际结果符合期望。
- `FAIL`：接口返回、最终状态或副作用不符合期望。
- `SKIP`：当前环境 / 接口能力不足以执行，需输出原因。

判定口径：

- 乐观锁 P0 已实现后，核心能力缺失必须记为 `FAIL`，不能记为 `SKIP`。例如响应缺少 `version`、写接口缺少 `base_version` 仍成功、旧版本写入返回 `200`、冲突响应没有 `current_version`。
- 只有环境不可控或公开 API 无法构造的场景才记为 `SKIP`。例如 AI/Claude CLI 不可用、无法构造 sessions/token_usage/documents 关联、缺少 mock 延迟能力。
- setup 阶段无法创建基础 Requirement / Task 时，依赖该资源的用例记为 `SKIP`，原因必须包含具体失败接口、状态码和响应体。
- 如果 setup 创建响应缺少 `version`，对应创建响应用例记为 `FAIL`；为了继续执行其他用例，可以紧接着 `GET` 详情补齐资源信息，但报告中不得把创建响应缺陷掩盖为通过。
- 业务 `409` 只有在 `code == "EDIT_CONFLICT"` 时才算并发冲突；其他业务 409 必须按业务错误校验。

### 3.2 前置健康检查

正式执行 case 前先执行 health/setup 检查：

1. `POST /auth/login`：确认主账号可登录。
2. `GET /auth/me`：确认 token 有效，并记录当前用户角色和团队。
3. `GET /teams`：确认至少有一个团队。
4. `GET /task-assignees`：确认至少有一个可指派员工。
5. `POST /requirements` + `GET /requirements/{id}`：确认能创建并读取测试需求。
6. `POST /tasks` + `GET /tasks/{id}`：确认能创建并读取测试任务。

health/setup 不计入业务 case 总数，但失败时必须输出阻塞详情。若只是一部分能力不可用，继续执行不依赖该能力的用例。

### 3.3 输出字段

每个用例输出一行表格：

| case_id | name | status | reason | related endpoint |
| --- | --- | --- | --- | --- |

失败用例必须输出：

- request method/path
- request body，隐藏 token
- status code
- response body
- final GET state

### 3.4 并发模拟方式

A/B 模拟规则：

1. A 和 B 都先 `GET` 同一资源，拿到相同 `version = v1`。
2. A 用 `base_version = v1` 提交成功。
3. B 仍用 `base_version = v1` 提交。
4. 校验 B 是否返回 `409 EDIT_CONFLICT`，最终状态是否没有静默覆盖。

真实并发压测型用例使用 Python `ThreadPoolExecutor`：

- 10 个请求使用同一个 `base_version` 同时提交。
- 统计成功数、冲突数、最终版本。

### 3.5 并发冲突判断函数

脚本必须区分业务 409 与并发 409：

```python
def is_edit_conflict(status_code, body):
    return status_code == 409 and isinstance(body, dict) and body.get("code") == "EDIT_CONFLICT"
```

不能只用 `status_code == 409` 判断。

### 3.6 响应版本校验函数

所有成功写接口必须校验：

```python
def assert_has_version(entity, expected=None):
    assert "version" in entity
    assert isinstance(entity["version"], int)
    if expected is not None:
        assert entity["version"] == expected
```


### 3.7 版本递增校验规则

除特别说明的幂等依赖操作外，成功修改同一实体的接口应满足：

- 需求本体修改成功：`requirements.version = base_version + 1`。
- 任务本体修改成功：`tasks.version = base_version + 1`。
- owner task 依赖集合实际发生变化：owner `tasks.version = base_version + 1`。
- 删除目标实体成功后目标行不存在，不要求返回实体 version。
- 任务新增 / 编辑 / 删除 / 状态 / 进度 / 依赖变化，不应导致父需求 `requirements.version` 递增。

校验最终状态时必须重新 `GET`，不要只相信写接口响应。

乐观 UI / 连续操作类测试必须使用上一次成功响应里的 `version` 发起下一次请求，不能重新 GET 后规避成功响应未返回最新版本的问题。

## 4. 可运行脚本骨架

建议脚本文件：`scripts/test_optimistic_lock_api.py`。

> 下面是可运行脚本结构要求。实现时不要修改业务逻辑；如果接口尚未完成乐观锁，应输出 FAIL / SKIP，而不是修业务代码。

```python
#!/usr/bin/env python3
import argparse
import concurrent.futures
import json
import os
import sys
import time
from dataclasses import dataclass
from typing import Any, Callable, Optional

import requests


@dataclass
class CaseResult:
    case_id: str
    name: str
    status: str
    reason: str
    endpoint: str
    details: Optional[dict[str, Any]] = None


class ApiClient:
    def __init__(self, base_url: str, token: str, timeout: int = 20):
        self.base_url = base_url.rstrip("/")
        self.timeout = timeout
        self.session = requests.Session()
        self.session.headers.update({
            "Authorization": f"Bearer {token}",
            "Content-Type": "application/json",
            "Accept": "application/json",
        })

    def request(self, method: str, path: str, body: Any = None, expected: Optional[int] = None):
        url = f"{self.base_url}{path}"
        resp = self.session.request(method, url, json=body, timeout=self.timeout)
        try:
            payload = resp.json() if resp.text else None
        except Exception:
            payload = resp.text
        if expected is not None and resp.status_code != expected:
            raise AssertionError({
                "method": method,
                "path": path,
                "body": body,
                "expected": expected,
                "status": resp.status_code,
                "response": payload,
            })
        return resp.status_code, payload


def login(base_url: str, employee_id: str, password: str, timeout: int) -> str:
    resp = requests.post(
        f"{base_url.rstrip('/')}/auth/login",
        json={"employee_id": employee_id, "password": password},
        timeout=timeout,
    )
    try:
        body = resp.json()
    except Exception:
        body = {}
    if resp.status_code >= 400:
        raise RuntimeError(f"login failed: {resp.status_code} {body}")
    token = body.get("token") or body.get("access_token") or body.get("data", {}).get("token")
    if not token:
        raise RuntimeError(f"login response has no token: {body}")
    return token


def is_edit_conflict(status: int, body: Any) -> bool:
    return status == 409 and isinstance(body, dict) and body.get("code") == "EDIT_CONFLICT"


def assert_version(entity: dict[str, Any], expected: Optional[int] = None):
    if "version" not in entity:
        raise AssertionError(f"missing version in response: {entity}")
    if not isinstance(entity["version"], int):
        raise AssertionError(f"version is not int: {entity.get('version')}")
    if expected is not None and entity["version"] != expected:
        raise AssertionError(f"version expected {expected}, got {entity['version']}")


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--cleanup", action="store_true")
    parser.add_argument("--include-regenerate-ac", action="store_true")
    args = parser.parse_args()

    base_url = os.getenv("API_BASE_URL", "http://localhost:8080/api/v1")
    timeout = int(os.getenv("TEST_TIMEOUT_SECONDS", "20"))
    prefix = os.getenv("TEST_PREFIX", f"OL_TEST_{int(time.time())}")
    token = os.getenv("TEST_TOKEN")
    if not token:
        employee_id = os.getenv("TEST_EMPLOYEE_ID", "liu_tl")
        password = os.getenv("TEST_PASSWORD", "123")
        token = login(base_url, employee_id, password, timeout)

    api = ApiClient(base_url, token, timeout)
    results: list[CaseResult] = []

    # 实现时按本文第 5 节 case 清单注册并执行。
    # 每个 case 内部自行创建唯一测试数据，默认不清理。

    print_report(results)


def print_report(results: list[CaseResult]):
    total = len(results)
    passed = sum(1 for r in results if r.status == "PASS")
    failed = sum(1 for r in results if r.status == "FAIL")
    skipped = sum(1 for r in results if r.status == "SKIP")
    print("| case_id | name | status | reason | endpoint |")
    print("| --- | --- | --- | --- | --- |")
    for r in results:
        print(f"| {r.case_id} | {r.name} | {r.status} | {r.reason} | {r.endpoint} |")
    print()
    print(f"Total: {total}, PASS: {passed}, FAIL: {failed}, SKIP: {skipped}")
    if failed:
        print("Failed cases:")
        for r in results:
            if r.status == "FAIL":
                print(json.dumps({
                    "case_id": r.case_id,
                    "name": r.name,
                    "reason": r.reason,
                    "details": r.details,
                }, ensure_ascii=False, indent=2))
        sys.exit(1)


if __name__ == "__main__":
    main()
```

## 5. 测试用例清单

执行顺序建议：

1. 每个 case 独立 setup，独立创建自己的 Requirement / Task。
2. 基础能力 A 组先跑；若 A1/A2/A3/A4 中 version 基础返回失败，后续依赖 version 的用例仍可继续执行，但失败原因应明确指向缺少 version / base_version 能力。
3. B/D/G/I/J/K 是 P0 核心回归，不能因为实现尚未完成而 SKIP。
4. C/E3/G8 是条件性用例，只有环境或公开 API 无法构造时才 SKIP。
5. 清理只在全部用例结束后执行；测试过程中不要删除其他用例还可能读取的资源。

如果开启 `--cleanup`，推荐按依赖反向清理：先删除任务依赖，再删除任务，再尝试删除无关联需求。遇到业务 409 或 404 只记录 cleanup 结果，不影响测试结论。

## B. 需求本体乐观锁

### B1. A/B 同时改同一需求字段

**Endpoint**：`PUT /requirements/{id}`

步骤：

1. A/B 都读取同一需求，得到 `version = v1`。
2. A 用 `base_version = v1` 修改 title。
3. A 成功，返回 `version = v2`。
4. B 仍用 `base_version = v1` 修改 title。
5. GET 最终需求。

期望：

- A 返回 `200`。
- B 返回 `409`。
- B body `code = EDIT_CONFLICT`。
- B body `current_version = v2`。
- 最终 title 是 A 的值。

### B2. A 改优先级，B 改标题但提交旧整组字段

**Endpoint**：`PUT /requirements/{id}`

步骤：

1. A/B 都读取同一需求，得到 `version = v1`。
2. B 保存旧快照：`priority`、`description`、`acceptance_criteria` 等。
3. A 修改 `priority`，成功到 `version = v2`。
4. B 用 `base_version = v1` 修改 `title`，请求体同时带旧 `priority`。
5. GET 最终需求。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 最终 `priority` 保持 A 的修改。
- B 的旧 `priority` 不得覆盖 A。

### B3. A 改 acceptance_criteria，B 改 description 但带旧 AC

3. B 用旧 `base_version = v1` 修改 `description`，并带旧 `acceptance_criteria`。
4. GET 最终需求。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 最终 `acceptance_criteria` 保持 A 的修改。

**Endpoint**：`PUT /requirements/{id}`

1. A/B 都读取需求 `version = v1`。

- B 返回 `409 EDIT_CONFLICT`。

### B5. cancelRequirement 旧版本冲突

**Endpoint**：`PUT /requirements/{id}`

步骤：

1. A/B 都读取需求 `version = v1`。
2. A 修改需求任意字段成功到 `v2`。
3. B 用 `base_version = v1` 提交 `{ "status": "cancelled" }`。
4. GET 最终需求。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 需求状态不被 B 取消。

### B6. restoreRequirement 旧版本冲突

**Endpoint**：`PUT /requirements/{id}/restore`

前置：创建一个 `cancelled` 状态需求。

步骤：

1. A/B 都读取该需求 `version = v1`。
2. A 修改该需求，例如更新 `priority`，成功到 `v2`，但保持需求仍为 `cancelled`。
3. B 用 `base_version = v1` 执行 restore。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- `current_version = v2`。
- 需求仍保持 A 修改后的状态。

补充业务 409 子场景：

1. 创建一个非 `cancelled` 需求，读取当前 `version`。
2. 用正确 `base_version` 执行 restore。
3. 期望返回业务 409，例如 `not_cancelled`，且不得被 `is_edit_conflict` 判定为并发冲突。

### B7. 删除需求旧版本冲突

**Endpoint**：`DELETE /requirements/{id}?base_version=...`

步骤：

1. 创建一个可删除测试需求，`version = v1`。
2. A/B 都读取 `version = v1`。
3. A 修改 title 成功到 `v2`。
4. B 用 `base_version = v1` 删除需求。
5. GET 需求。

期望：

- B 返回 `409 EDIT_CONFLICT`。
- 需求仍存在。

### B8. 删除需求业务 409 不应误判为 EDIT_CONFLICT

**Endpoint**：`DELETE /requirements/{id}?base_version=...`

步骤：

1. 创建需求。
2. 创建关联任务，使需求不可删除。
3. 用正确 `base_version` 删除需求。

期望：

- 返回业务 `409`，例如 `has_associations`。
- `code` 不能是 `EDIT_CONFLICT`。
- `is_edit_conflict` 判断结果必须为 false。

### B9. 需求空更新返回 400

**Endpoint**：`PUT /requirements/{id}`

步骤：

1. 创建需求，读取 `version = v1`。
2. PUT body 只有 `{ "base_version": v1 }`。

### B10. base_version 缺失 / 0 / 负数 / 非数字

**Endpoint**：`PUT /requirements/{id}`、`DELETE /requirements/{id}?base_version=abc`

分别测试：

1. 缺失 `base_version`。
2. `base_version = 0`。
3. `base_version = -1`。
4. DELETE query `base_version=abc`。

期望：

- 返回 `400`。
- 不修改数据。
- version 不递增。

## K. 真实并发压测型用例

### K1. 多个客户端同时更新同一需求

**Endpoint**：`PUT /requirements/{id}`

步骤：

1. 读取需求 `version = v1`。
2. 10 个并发请求都使用同一个 `base_version = v1` 修改同一需求 title。
3. GET 最终需求。

期望：

- 只有 1 个成功。
- 其余 9 个返回 `409 EDIT_CONFLICT`。
- 最终 version 只递增 1。
- 最终 title 等于成功请求的值。

### K2. 多个客户端同时更新同一任务状态

**Endpoint**：`PUT /tasks/{id}/status`

步骤：

1. 读取任务 `version = v1`。
2. 10 个并发请求都使用同一个 `base_version = v1` 修改同一任务 status。
3. GET 最终任务。

期望：

- 只有 1 个成功。
- 其余返回 `409 EDIT_CONFLICT`。
- version 只递增 1。

### K3. 多个客户端同时新增同一依赖

**Endpoint**：`POST /tasks/{id}/dependencies`

步骤：

1. T1 不依赖 T2。
2. 读取 T1 `version = v1`。
3. 10 个并发请求都使用同一个 `base_version = v1` 给 T1 新增依赖 T2。
4. GET T1。

期望：

- 只有 1 个请求实际新增依赖并 bump version。
- 其余请求应返回 `409 EDIT_CONFLICT`；如果实现选择在持锁后发现依赖已存在并返回幂等成功，也必须满足“不 bump version”，并在报告中记录该行为。
- 最终必须满足：
  - 依赖只存在一条。
  - owner task version 最多只递增 1 次。
  - 不出现 500。
