# Dashboard 风险列表测试方案 v2

> 本方案用于验证 `/dashboard/risks` 是否符合新的 P0 产品口径：
> **我的风险提示 = 我的需求超期 + 我的任务超期 + 我的依赖阻塞**。
> 接口应返回 Dashboard 风险组，而不是逐条 task risk item。

---

## 1. 测试目标

验证 `/dashboard/risks` 满足以下产品口径：

1. 不按角色默认扩大范围。

   * director 不默认全局；
   * pm 不默认全局；
   * team_leader 不默认团队；
   * employee 不看无关风险。

2. 同时支持需求级风险和任务级风险。

   * 需求级：`requirement_overdue`
   * 任务级：`deadline`
   * 任务级：`dependency_blocker`

3. 风险计算按真实对象生成。

   * 需求超期归属 requirement；
   * 任务超期归属 task；
   * 依赖阻塞归属被阻塞 task。

4. Dashboard 展示按 requirement 聚合。

   * 同一需求下多个可见任务风险，不应平铺成多张卡；
   * 同一任务同时命中 `deadline` + `dependency_blocker`，应在同一个任务摘要中合并展示；
   * 聚合前必须先按“我的风险事实”过滤，不能带出不可见任务风险。

5. `AttentionScore` 只影响排序和热度展示，不扩大风险来源。

---

## 2. 响应模型预期

`GET /dashboard/risks` 不再断言旧字段：

```text
taskId
riskType
RelatedObjectType = task
```

而应断言风险组结构。

建议目标字段：

```ts
type DashboardRiskGroup = {
  key: string
  displayType: "requirement_group" | "single_task"

  requirementId: string
  requirementTitle: string

  riskTypes: string[]

  requirementOverdue: boolean
  deadlineTaskCount: number
  dependencyBlockerCount: number

  representativeTask?: {
    taskId: string
    title: string
    riskTypes: string[]
    deadline?: string
    unfinishedDependencyCount?: number
  }

  summary: string
  attentionScore: number
  attentionLevel: string

  actionText: string
  targetUrl: string
}
```

### 2.1 displayType 判定

```text
包含 requirement_overdue
=> displayType = requirement_group

没有 requirement_overdue，但同一 requirement 下有多个“可见风险任务”
=> displayType = requirement_group

没有 requirement_overdue，且只有一个“可见风险任务”
=> displayType = single_task
```

注意：

```text
一个任务同时命中 deadline + dependency_blocker
仍然算 1 个可见风险任务
```

所以如果某需求下当前用户只可见 T1，且 T1 同时有两个风险：

```text
displayType = single_task
representativeTask.riskTypes = ["deadline", "dependency_blocker"]
```

## 4. 测试数据原则

1. 所有标题必须带唯一前缀：

```bash
PREFIX="RISK_V2_E2E_$(date +%Y%m%d_%H%M%S)"
```

2. 不直接写数据库。
3. 不删除数据，方便人工复查。
4. 每次执行使用新前缀。
5. 超期日期统一使用：

```text
2000-01-01
```

6. 今天到期使用服务端当天日期。
7. 未来日期统一使用：

```text
2099-01-01
```

8. 需求截止日期字段必须使用真实字段，例如：

```text
requirements.deadline / API deadline
```

9. 任务截止日期字段使用：

```text
tasks.due_date / API due_date
```

### 6.10 R11：排序用例

创建三个当前用户可见的风险组。

| 需求    | 风险    | deadline   | attention | 预期顺序      |
| ----- | ----- | ---------- | --------- | --------- |
| R11_A | 任务超期  | 2000-01-01 | 低         | 靠前        |
| R11_B | 任务超期  | 2001-01-01 | 高         | 在 R11_A 后 |
| R11_C | 仅依赖阻塞 | 无超期        | 高         | 在超期组后     |

排序预期：

```text
包含超期风险的组优先
超期日期更早的优先
AttentionScore 只在前两项无法区分时参与
仅依赖阻塞排在超期组之后
```

同时加一个不可见任务干扰：

```text
R11_A 下有一个 EMP_B 的任务，due_date = 1999-01-01
EMP_A 不可见
```

EMP_A 查询时：

```text
R11_A 的排序不能引用这个不可见任务的 1999-01-01
```

## 8. 断言规则

### 8.1 包含风险组

```js
function findGroup(groups, requirementId) {
  return groups.find(g => g.requirementId === requirementId);
}
```

### 8.2 不包含风险组

```js
expect(findGroup(groups, UNRELATED_REQUIREMENT_ID)).toBeFalsy();
```

### 8.3 不包含不可见任务

```js
const group = findGroup(groups, REQUIREMENT_ID);
expect(JSON.stringify(group)).not.toContain(UNRELATED_TASK_ID);
```

### 8.4 单任务风险断言

```js
expect(group.displayType).toBe("single_task");
expect(group.representativeTask.taskId).toBe(TASK_ID);
expect(group.targetUrl).toContain(`taskId=${TASK_ID}`);
```

### 8.5 需求风险组断言

```js
expect(group.displayType).toBe("requirement_group");
expect(group.targetUrl).toContain(`requirementId=${REQ_ID}`);
expect(group.targetUrl).not.toContain("taskId=");
```

### 8.6 双风险合并断言

```js
expect(group.representativeTask.taskId).toBe(TASK_BOTH_ID);
expect(group.representativeTask.riskTypes.sort()).toEqual([
  "deadline",
  "dependency_blocker"
].sort());
```

注意：
不要再断言旧逻辑：

```js
risks.filter(item => item.taskId === TASK_BOTH_ID).length === 2
```

新逻辑应断言：

```text
一个任务摘要里包含两个 riskTypes
```

### 8.7 聚合计数断言

```js
expect(group.requirementOverdue).toBe(true);
expect(group.deadlineTaskCount).toBe(2);
expect(group.dependencyBlockerCount).toBe(1);
```

### 8.8 排序断言

排序只对明确构造的排序数据断言，不要依赖所有返回结果全局顺序。

建议：

```js
const orderedIds = groups
  .filter(g => [R11_A, R11_B, R11_C].includes(g.requirementId))
  .map(g => g.requirementId);

expect(orderedIds).toEqual([R11_A, R11_B, R11_C]);
```

## 9. 最小冒烟用例

如果 Codex 额度不足，至少跑这 8 个。

1. DIRECTOR 不包含无关高风险 R5。
2. DIRECTOR 关注 R4 后包含 R4，取消关注后不包含。
3. PM 创建 R1 后，看到 R1 的 requirement_overdue + 任务风险聚合组。
4. TL 不默认看到团队成员任务风险。
5. EMP_A 只负责 R8_T1 时，不带出 R8_T2 / R8_T4。
6. EMP_A 关注 requirement 后，可以看到该 requirement 下任务风险。
7. R12_T1_BOTH 同时命中 deadline + dependency_blocker，但展示合并。
8. completed / cancelled requirement 下的需求风险和任务风险都不出现。

## 10. 通过标准

全部核心用例通过时，认为 `/dashboard/risks` 新口径验证通过：
