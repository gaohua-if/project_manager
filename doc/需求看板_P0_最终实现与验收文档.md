## 1. 业务边界

### 1.1 主数据域：`requirements`

负责以下事实状态的**唯一权威**：

- `Requirement`：标题、描述、需求验收标准、阶段、优先级、截止日期、参与团队、创建人、completed_at。
- `Task`：标题、负责人、任务验收标准、状态、进度、优先级、截止日期、completed_at。
- `TaskDependency`：同一需求内任务的上游依赖。
- `Requirement.progress`：由任务进度聚合产生（不存外部输入，只允许由后端写）。
- `TokenSource` 关联：通过 `sessions.requirement_id` / `sessions.task_id` 派生。

### 1.2 投影域：`dashboard`

只承担"当前用户视角"的查询编排：

- "我关注的事项"投影。
- "待处理风险"投影。
- 跳转参数（`requirementId` / `taskId`）。
- 复用旧服务输出（reports / sessions / tokens）做导航入口，不写主数据。

### 1.3 共享域

- `user_follows`：跨页面通用的"用户—对象"关注关系。Requirements 页与 Dashboard 共用同一张表，没有 Dashboard 私有副本。
- `service.DeriveTaskRisks` / `service.DisplayTaskStatus` / `service.AggregateRequirementProgress` / `service.SummarizeRequirementTasks`：唯一的风险与聚合派生函数。Requirements、Task、Dashboard 三处全部经过它，不重复实现。

### 1.4 严格禁止

- 在 Dashboard 域内落业务表（无 `dashboard_*` 表，无风险表，无 Dashboard 关注表）。
- 把 `blocked` 当成 `task.status` 存储。
- 前端浏览器自行计算 `blocked` / `overdue` 或 `requirement.progress`。
- 通过 Session / Token / LLM 自动改写任务进度或状态。
- 跨需求依赖、多级任务树、人工阻塞字段、archive 字段。

### 2.2 各层禁止事项

| 层 | 禁止 |
|---|---|
| L1 DB | 存 Dashboard 卡片文案、排序、计算后的风险；不放业务派生列（display_status / risk_types）|
| L2 Domain | 直接读写 DB；保持纯函数 |
| L3 Handler | 重复实现 Domain 规则；跨域写其它资源；让前端可写 `requirement.progress` 或 `task.status=blocked` |
| L4 DTO | 暴露 DB 行的内部命名；引入仅供调试的字段 |
| L5 Adapter | 在前端推导任务的最终状态（display_status / risk）|
| L6 ViewModel | 越过 adapter 直接调用任务/需求写接口；维护独立的关注或风险集合 |

## 3. 数据库模型（Layer 1）

迁移：`api/db/migrations/007_requirements_p0.sql`（在 001–006 基础上叠加）。

### 3.1 `requirements`

```
id              UUID PK
title           TEXT
description     TEXT
feishu_doc_url  TEXT NULL
acceptance_criteria TEXT[]      -- 需求验收标准，可选；为空不构成风险
creator_id      UUID FK users
creator_role    TEXT
status          TEXT  CHECK IN ('todo','review','active','completed','cancelled')   -- 默认 'todo'
priority        TEXT
progress        INT  0..100     -- 由后端聚合写入，前端禁止直接写
deadline        DATE NULL
completed_at    TIMESTAMPTZ NULL    -- 新增 (007)
created_at / updated_at TIMESTAMPTZ
```

迁移行为：
- `status` CHECK 收紧为五值，默认 `todo`。
- 历史 `status='completed'` 的行回填 `completed_at = updated_at`。

### 3.2 `requirement_teams`

```
requirement_id UUID FK requirements
team_id        UUID FK teams
PRIMARY KEY (requirement_id, team_id)
```

`team_leader` 的 Requirements / Tasks / Follows 可见性都通过这张表收敛。

### 3.3 `tasks`

```
id                    UUID PK
requirement_id        UUID FK requirements
title                 TEXT
acceptance_criteria TEXT[]                -- 任务验收标准，可选；不关联需求验收项
assignee_id           UUID NULL FK users
creator_tl_id         UUID FK users      -- 创建该任务的 TL/PM
status                TEXT CHECK IN ('todo','in_progress','done')  -- 由 007 收紧；blocked 不再存储
priority              TEXT
progress              INT NOT NULL DEFAULT 0 CHECK (0..100)   -- 新增 (007)
due_date              DATE NULL
completed_at          TIMESTAMPTZ NULL                          -- 新增 (007)
created_at / updated_at TIMESTAMPTZ
```

迁移行为：
- 历史 `status='blocked'` → `in_progress`。
- 历史 `status='done'` → `progress=100, completed_at=updated_at`。

索引：`idx_tasks_due_date ON tasks (due_date) WHERE status <> 'done'`（加速超期任务风险投影）。

### 3.4 `task_dependencies`

```
task_id        UUID FK tasks
depends_on_id  UUID FK tasks
PRIMARY KEY (task_id, depends_on_id)
```

应用层在写入前用递归 CTE 防环；同需求约束由 handler 校验。

### 3.5 `user_follows`（007 新增）

```
user_id     UUID FK users ON DELETE CASCADE
target_type TEXT CHECK IN ('requirement','task')
target_id   UUID
created_at  TIMESTAMPTZ DEFAULT now()
PRIMARY KEY (user_id, target_type, target_id)

INDEX idx_user_follows_target ON user_follows (target_type, target_id)
```

- 主键即幂等保证。
- 通过 `ON DELETE CASCADE` 在用户删除时自动清理。
- 反向索引便于"是否被关注"以及"谁关注了我"类查询。

### 3.6 复用未变更表

`sessions`（含 `requirement_id` / `task_id`）、`token_usage`、`daily_reports`、`team_reports`、`users` / `teams`。Token 来源汇总通过 `sessions JOIN token_usage` 派生，不新增链接表。

### 3.8 验收标准口径

- `requirements.acceptance_criteria` 是需求级验收标准，可选；为空不产生风险。
- `tasks.acceptance_criteria` 是任务级验收标准，可选；为空不产生风险。
- P0 不做任务关联/覆盖需求验收项，不计算覆盖率，不暴露 `acceptance_criteria_ids`。

## 4. Domain 层（Layer 2）

文件：`api/service/requirements.go`，单测：`api/service/requirements_test.go`。

### 4.1 风险派生 `DeriveTaskRisks(task model.Task, now time.Time) []string`

```
if task.Status == "done"  → []
risks := []
if 任一 dependency.Status != "done"  → append("blocked")  (只追加一次)
if task.DueDate 解析失败 / 为空        → return risks
today := UTC 今日零点
if due < today          → append("overdue")
return risks
```

- 支持两种日期格式：`2006-01-02`、`time.RFC3339`（SQL `NullString` 在含时区时返回 RFC3339）。
- 不派生 `due_soon` / 临期风险；未来截止日期不产生风险。
- `blocked` 可与 `overdue` 并存。
- 永远返回非 nil 切片（handler 会再做一次 nil → `[]` 归一化，保证 JSON 输出空数组）。

### 4.2 展示状态 `DisplayTaskStatus(task model.Task) string`

```
if task.Status == "done"       → "done"
if "blocked" ∈ task.RiskTypes  → "blocked"
otherwise                      → task.Status   // todo | in_progress
```

依赖：调用前 `task.RiskTypes` 必须已由 `DeriveTaskRisks` 填好。

### 4.3 进度聚合 `AggregateRequirementProgress(tasks) int`

```
if len(tasks) == 0  → 0
else                → sum(progress) / len(tasks)   // 整数除法
```

数据库侧 `TaskHandler.updateRequirementProgress` 用 `FLOOR(AVG(progress))::int` 实现同样语义，避免每次任务写完都让 handler 反查全部任务回算。

### 4.4 汇总 `SummarizeRequirementTasks(tasks)`

返回两个结构体：

```
RequirementTaskSummary { Total, Done, Blocked }
RequirementRiskSummary { Blocked, Overdue }
```

`Blocked` 在两边都计数；`Overdue` 只出现在风险汇总。

### 4.5 单元测试断言

`api/service/requirements_test.go`：

- `blocked + overdue` 共存；
- 数据库 RFC3339 日期被识别为 `overdue`；
- 未来截止日期不产生 `due_soon` / 临期风险；
- `status=done` 永远返回空数组（即便上游未完成）；
- `AggregateRequirementProgress([25,75,100]) == 66`、空切片返回 0。

## 5. HTTP Handler 层（Layer 3）

文件位置：`api/handler/`。路由集中在 `api/main.go` 中的 `/api/v1` 路由组下，使用 `AuthMiddleware`。

### 5.1 RequirementHandler（`requirement.go`）

| Method / Path | Handler | 说明 |
|---|---|---|
| GET `/requirements` | `List` | 支持 `status`、`team_id`；`team_leader` 自动按 `team_id` 过滤；`ORDER BY created_at DESC` |
| POST `/requirements` | `Create` | 必填 title/description/team_ids；priority 默认 medium；需求验收标准可选，空数组按空数组保存 |
| GET `/requirements/{id}` | `Get` | 单条 |
| PUT `/requirements/{id}` | `Update` | 支持 title/description/status/priority/deadline/feishu/AC；`status=completed` 时落 `completed_at = now()`，其它阶段清空 `completed_at` |
| GET `/requirements/{id}/ac` | `GetAC` | 需求验收标准列表；P0 不计算任务覆盖或完成度 |
| POST `/requirements/{id}/regenerate-ac` | `RegenerateAC` | `director/pm/team_leader` 权限；调用 `service.AIClient` 重新生成 |

每条 Requirement 响应都经过：

1. `loadTeams(req)`：从 `requirement_teams JOIN teams` 取 `team_ids` + `team_names`。
2. `loadProjection(req, userID)`：
   - 拉取该需求下全部任务（含任务验收标准、负责人、status/progress/due_date/completed_at）。
   - 每条任务通过 `TaskHandler.enrichTask` 注入依赖、风险、display_status、token_source_ids、is_followed。
   - 调用 `service.AggregateRequirementProgress` 写回 `req.Progress`。
   - 调用 `service.SummarizeRequirementTasks` 写回 `req.TaskSummary` / `req.RiskSummary`。
   - 查询 `user_follows` 写回 `req.IsFollowed`。
   - 查询 `sessions JOIN token_usage` 拼出 `req.TokenSourceIDs`（约束 `s.requirement_id=$1 AND s.task_id IS NULL`，即"直接挂在需求而非具体任务上"的 session）。

权限分支（仅列举关键差异）：
- `team_leader`（含 `team_id`）：列表中追加 `EXISTS requirement_teams` 条件。
- `admin / director / pm`：默认无范围限制。
- `employee`：列表上游未做行级过滤；可见性由其他接口（Task / Follow）保证。

### 5.2 TaskHandler（`task.go`）

| Method / Path | Handler | 说明 |
|---|---|---|
| GET `/tasks` | `List` | 支持 `requirement_id`、`assignee_id`、`status`；按角色过滤 |
| POST `/tasks` | `Create` | `admin/director/pm/team_leader`；`team_leader` 仅能给本团队员工创建且需求需挂本团队 |
| GET `/tasks/{id}` | `Get` | 单条 |
| PUT `/tasks/{id}` | `Update` | 拒绝 `status=blocked`；`progress` 0..100；`status=done` 自动 `progress=100, completed_at=now()` |
| PUT `/tasks/{id}/status` | `UpdateStatus` | 同上的单字段语义，原子 SQL 一次性处理 |
| PUT `/tasks/{id}/progress` | `UpdateProgress` | 单字段进度；不联动 status |
| POST `/tasks/{id}/dependencies` | `AddDependency` | `validateDependency` 三重校验 |
| DELETE `/tasks/{id}/dependencies/{dep_id}` | `RemoveDependency` | 物理删除 |

#### 5.2.1 enrichTask（读路径增强）

```
enrichTask(t *Task, userID string):
  loadDeps(t)                          // 填充 t.Dependencies + t.Blocking
  t.RiskTypes     = DeriveTaskRisks(*t, time.Now())
  t.DisplayStatus = DisplayTaskStatus(*t)
  if t.RiskTypes == nil { t.RiskTypes = []string{} }   // 输出空数组而非 null
  t.TokenSourceIDs = SELECT DISTINCT s.id FROM sessions JOIN token_usage WHERE s.task_id=$1
  t.IsFollowed = EXISTS user_follows WHERE user_id=userID AND target_type='task' AND target_id=t.id
```

#### 5.2.2 写路径 → 进度回写

`Update` / `UpdateStatus` / `UpdateProgress` / `Create` / `AddDependency` 完成后调用 `updateRequirementProgress(taskID)`：

```sql
UPDATE requirements
SET progress = COALESCE((SELECT FLOOR(AVG(progress))::int
                         FROM tasks WHERE requirement_id = $1), 0),
    updated_at = now()
WHERE id = $1
```

确保读路径不需要再次聚合即能拿到一致的 `Requirement.progress`。

#### 5.2.3 依赖校验 `validateDependency(requirementID, taskID, dependsOnID)`

1. `dependsOnID` 非空。
2. 非自依赖。
3. 上游任务必须存在且 `requirement_id` 与当前一致。
4. 通过 CTE 递归遍历上游集合，禁止当前任务出现在上游链中（防环）。

```sql
WITH RECURSIVE upstream(id) AS (
  SELECT depends_on_id FROM task_dependencies WHERE task_id = $dependsOn
  UNION
  SELECT td.depends_on_id FROM task_dependencies td JOIN upstream u ON td.task_id = u.id
)
SELECT EXISTS(SELECT 1 FROM upstream WHERE id = $taskID)
```

#### 5.2.4 状态合法性

`isStoredTaskStatus(status)` 仅允许 `todo | in_progress | done`。任何尝试写入 `blocked` 的请求返回：

```
400  {"error":"invalid status; blocked is derived from dependencies"}
```

### 5.3 FollowHandler（`follow.go`）

| Method / Path | Handler | 说明 |
|---|---|---|
| GET `/follows` | `List` | 列出当前用户全部关注（含 `created_at`） |
| POST `/follows` | `Follow` | 校验 `target_type ∈ {requirement, task}` + 目标可见性；`ON CONFLICT DO NOTHING` 实现幂等 |
| DELETE `/follows/{target_type}/{target_id}` | `Unfollow` | 物理删除（无需可见性校验，允许取消已不可见的关注） |

可见性 `targetVisible`：

- `requirement` + `team_leader` → `EXISTS requirement_teams` AND `team_id = u.TeamID`
- `requirement` + 其它 → 目标存在即可
- `task` + `team_leader` → assignee 在本团队 OR `creator_tl_id = u.ID`
- `task` + 其它 → 目标存在即可

返回体统一形如：

```json
{ "favorited": true, "target_type": "requirement", "target_id": "..." }
```

便于前端按布尔统一切换 UI 状态。

### 5.4 DashboardHandler（`dashboard.go`）

| Method / Path | Handler | 说明 |
|---|---|---|
| GET `/dashboard/follows` | `Follows` | 把当前用户的 `user_follows` 投影成 `DashboardFollowItem[]` |
| GET `/dashboard/risks` | `Risks` | 计算当前用户作用域内未完成任务的风险条目 |

#### 5.4.1 `/dashboard/follows` 编排

```
SELECT target_type,target_id FROM user_follows WHERE user_id=$1 ORDER BY created_at DESC

for each (target_type, target_id):
  if requirement:
     requirementFollowItem:
        load requirement basic fields
        skip if status IN (completed, cancelled)
        RequirementHandler.loadProjection(req, userID)        // 拿到 risk_summary
        item.Status   = requirementStatusLabel(req.Status)   // 待开始/评审/进行中/已完成/已取消
        item.Risk     = requirementRiskLabel(req.RiskSummary)// 优先级：超期 > 阻塞 > 正常推进
        item.URL      = /requirements?requirementId=...
  if task:
     taskFollowItem:
        load task + requirement title
        skip if task.Status == done
        TaskHandler.enrichTask(&task, userID)
        item.Status     = taskStatusLabel(task.DisplayStatus)
        item.Risk       = taskRiskLabel(task.RiskTypes)     // 优先级：超期 > 阻塞
        item.Dependency = "" if not blocked else "未完成上游：A、B"
        item.URL        = /requirements?requirementId=...&taskId=...
```

`recentUpdateLabel` 将 `updated_at` 渲染为中文相对时间（今天 / 昨天 / N 天前），由后端统一计算，避免每个前端用户重做时区运算。

#### 5.4.2 `/dashboard/risks` 编排

SQL 基础：

```sql
SELECT ...  FROM tasks t
JOIN requirements r ON r.id = t.requirement_id
LEFT JOIN users a ON a.id = t.assignee_id
WHERE t.status <> 'done' AND r.status NOT IN ('completed','cancelled')
```

按角色叠加：

- `employee`：
  ```
  AND ( t.assignee_id = $self
        OR EXISTS user_follows WHERE target=task t.id
        OR EXISTS user_follows WHERE target=requirement r.id )
  ```
- `team_leader`（含 `team_id`）：
  ```
  AND ( a.team_id = $team
        OR t.creator_tl_id = $self
        OR EXISTS user_follows ... (同上) )
  ```
- 其它角色（admin/director/pm）：无额外过滤。

排序：`ORDER BY t.due_date NULLS LAST, t.updated_at DESC`。

对每条任务调用 `enrichTask`，然后按 `risk_types` 笛卡尔展开：一个任务可产出多条风险条目。

| risk_type | riskType / source / level / tone | reason |
|---|---|---|
| `blocked`   | `dependency_blocker` / 依赖阻塞 / 高 / red    | "等待上游任务完成：A、B" |

## 9. 数据流：写路径（端到端）

### 9.1 创建需求

```
用户表单（RequirementCreatePage / RequirementsListPage 抽屉）
   └─► requirementsBoardApi.createRequirement(input)
         └─► api/client.ts: createRequirement(...)
               └─► POST /api/v1/requirements
                     └─► RequirementHandler.Create
                           ├─ 校验 title/description/team_ids
                           ├─ 需求验收标准可选，空数组不触发风险
                           ├─ INSERT requirements
                           ├─ INSERT requirement_teams 多条
                           └─ 重新 SELECT + loadTeams + loadProjection
                                 └─ AggregateRequirementProgress / SummarizeRequirementTasks
            返回 Requirement DTO
   失效缓存：requirements-board / dashboard.follows / dashboard.risks
```

### 9.2 更新任务进度

```
Slider/InputNumber
  └─► requirementsBoardApi.updateTaskProgress(id, n)
        └─► PUT /api/v1/tasks/{id}/progress  { progress: n }
              └─► TaskHandler.UpdateProgress
                    ├─ 范围校验 0..100
                    ├─ UPDATE tasks SET progress=$1, updated_at=now()
                    ├─ updateRequirementProgress(id)
                    │     └─ UPDATE requirements SET progress = FLOOR(AVG(progress))::int
                    └─ Get(w, r)  → enrichTask → DeriveTaskRisks + DisplayTaskStatus
           返回 Task DTO（含最新 display_status、risk_types）
  失效：requirements-board / dashboard.follows / dashboard.risks
```

### 9.3 添加上游依赖

```
任务详情：选择上游
  └─► addTaskDependency(taskId, depId)
        └─► POST /api/v1/tasks/{id}/dependencies
              └─► TaskHandler.AddDependency
                    ├─ 取 task.requirement_id
                    ├─ validateDependency(reqId, taskId, depId)
                    │     ├─ 非空 / 非自依赖
                    │     ├─ 同 requirement_id
                    │     └─ CTE 防环
                    ├─ INSERT task_dependencies ON CONFLICT DO NOTHING
                    └─ Get(w, r) → enrichTask
           返回 Task（含新的依赖列表与可能新增的 blocked 风险）
  失效：requirements-board / dashboard.risks
```

### 9.4 关注 / 取消关注

```
点击关注按钮
  └─► requirementsBoardApi.toggleFavorite(type, id)
        ├─ 已关注：DELETE /api/v1/follows/{type}/{id}
        └─ 未关注：POST   /api/v1/follows  { target_type, target_id }
              └─► FollowHandler.Follow
                    ├─ 校验 target_type / target_id
                    ├─ targetVisible(role, type, id) 拒绝越权
                    └─ INSERT user_follows ON CONFLICT DO NOTHING
  返回：{ favorited, target_type, target_id }
  失效：requirements-board.favorites / dashboard.follows / dashboard.risks
        + 需求/任务详情（更新 is_followed）
```

### 9.5 阶段流转 → 完成

```
需求拖动到"完成"列
  └─► requirementsBoardApi.updateRequirementStage(id, 'completed')
        └─► PUT /api/v1/requirements/{id}  { status: 'completed' }
              └─► RequirementHandler.Update
                    ├─ isRequirementStatus('completed') 校验
                    ├─ SET status='completed',
                    │       completed_at = COALESCE(completed_at, now())
                    └─ Get(w, r) → loadTeams + loadProjection
           完成列展示：前端按 completed_at 排序，截取最近 7 天且 ≤ 10 条
```

> 任务级 `done` 走 `PUT /tasks/{id}/status`：自动 `progress=100, completed_at=now()`，并触发需求进度回写；若全部任务完成而需求阶段仍为 `active`，由用户在 UI 显式拖动到完成列（P0 不强制自动完成）。

## 11. 一致性与刷新策略

### 11.1 后端一致性

- 每次任务级写操作（status / progress / dependency / create）后，**同请求内**完成 `updateRequirementProgress`，保证 `Requirement.progress` 与任务进度一致。
- `display_status` / `risk_types` **不落库**，仅在读路径派生：删除依赖、上游变 done、修改 due_date 后下一次查询自然消失。
- `status='completed'` 与 `completed_at` 在 Update 内联：`COALESCE(completed_at, now())`；恢复到非 completed 阶段时清 `completed_at`。

### 11.2 前端缓存失效

写操作成功回调中需要同时 invalidate：

- `["requirements-board"]`（覆盖列表/详情/任务树）
- `["dashboard", "follows"]`
- `["dashboard", "risks"]`
- 必要时 `["requirements-board", "favorites"]`

模块独立失败：Dashboard 三栏（关注 / 风险 / 报告 / Token）的 query 失败彼此隔离，不会导致整页崩溃。

### 11.3 不做客户端推导

前端拿到 DTO 后不再重做：
- ❌ 自己根据 dependencies 推 `blocked`
- ❌ 自己根据 due_date 推 `overdue`
- ❌ 自己根据任务列表推 `requirement.progress`

任何此类逻辑出现都视为违反分层契约。

## 12. 权限与角色作用域

## 15. 验收清单（按层）

### 15.1 数据库 (L1)

- [ ] `requirements.status` CHECK 五值；`completed_at` 列存在；`status='completed'` 历史行有 `completed_at`。
- [ ] `tasks.status` CHECK 仅三值；`progress` NOT NULL DEFAULT 0；`completed_at` 存在；`status='done'` 历史行 `progress=100, completed_at` 非空。
- [ ] `user_follows` 表与 `idx_user_follows_target` 存在。
- [ ] `idx_tasks_due_date` 部分索引存在。

### 15.2 Domain (L2)

- [ ] `DeriveTaskRisks` 对 `done` 任务返回空数组。
- [ ] `due < today` → `overdue`；未来截止日期不产生 `due_soon` / 临期风险。
- [ ] `blocked` 与 `overdue` 可共存。
- [ ] `AggregateRequirementProgress([25,75,100]) == 66`。
- [ ] `requirements_test.go` 全绿。

### 15.3 Handler (L3)
