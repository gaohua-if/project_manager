# AIcoding 管理平台风险通知设计说明

版本：v0.1
模块：控制台 / 风险项
状态：架构设计基准稿
目标：统一“风险项”产品口径，作为后续前端原型、需求文档、后端架构设计的基准。

---

## 1. 背景

AIcoding 管理平台当前核心页面为：

```text
控制台
需求看板
```

其中：

* 控制台负责提醒、聚合、跳转；
* 需求看板负责真实处理，包括需求、任务、deadline、任务依赖、验收标准等。

当前平台暂未设计独立风险 DB，也没有风险生命周期能力。因此第一版风险项不能设计成完整风险管理系统。

本设计将风险项定义为：

> 风险项是系统基于需求、任务、deadline、强依赖、关注关系临时聚合出来的提醒视图。

---

## 2. 设计原则

### 2.1 风险项不是独立业务对象

P0 阶段不设计独立 risk 表，也不设计风险工单。

风险项不支持：

```text
风险编号
风险负责人
风险关闭
风险状态流转
风险评论
风险处理记录
风险归档
风险已读 / 未读
风险通知历史
风险规则配置
```

风险项只作为控制台上的提醒和跳转入口。

---

### 2.2 风险通知不持久化

P0 阶段风险通知不持久化。

风险项在用户打开控制台时，基于当前已有对象状态实时计算或由聚合接口临时返回。

也就是说：

```text
需求 / 任务 / deadline / 依赖 / 关注关系
        ↓
临时计算风险信号
        ↓
按角色过滤、聚合、排序
        ↓
控制台展示风险项
```

不保存：

```text
风险通知记录
风险阅读状态
风险处理状态
风险关闭状态
```

---

### 2.3 处理风险等于处理原始对象

用户不是去“处理风险”，而是回到原始业务对象处理问题。

例如：

```text
任务超期风险
→ 去需求看板定位任务
→ 修改任务状态 / deadline / 任务拆解

依赖阻塞风险
→ 去需求看板定位当前任务和上游任务
→ 协调上游任务推进
```

控制台不提供直接编辑任务、修改 deadline、调整依赖的能力。

---

### 2.4 不做过度智能判断

P0 不判断：

```text
任务是否真的无法推进
任务是否关键路径
任务是否影响交付
跨组依赖是否需要升级
是否应该通知总监
是否应该自动关闭风险
```

只基于明确字段做可解释判断。

---

## 3. 当前确认的 P0 边界

### 3.1 第一版没有手动阻塞字段

P0 暂不做：

```text
task.blocked
手动标记阻塞
阻塞原因
阻塞解除
阻塞记录
```

因此第一版风险类型里不包含“手动阻塞风险”。

---

### 3.2 任务依赖只有强依赖

P0 只支持强依赖。

含义：

```text
任务 B 依赖任务 A
=
任务 A 未完成时，任务 B 的推进会受到直接影响。
```

不支持：

```text
弱关联
参考关系
相关任务
可选依赖
非阻塞性关联
```

如果只是普通关联，不应该建成上游任务。

---

### 3.3 上游未完成但未超期，不通知

P0 不把“上游未完成”直接视为风险。

规则：

```text
上游任务未完成，但未超期
→ 不进入控制台风险通知

上游任务未完成，并且已超期
→ 下游任务出现依赖阻塞风险
```

原因：

* 上游任务未完成可能是正常推进状态；
* 若未到 deadline 就提醒，会产生大量噪音；
* P0 不做提前预测，只做明确风险。

---

### 3.4 总监只看手动关注需求的风险

总监可能有权限看到很多需求，但“可见”不等于“关注”。

P0 总监风险通知范围只包括：

```text
总监手动关注的需求
```

不因为总监可见全部需求，就把全部临期、超期、依赖阻塞都推给总监。

---

### 3.5 风险通知不持久化

P0 风险通知仅作为当前状态聚合结果，不保存通知历史。

不做：

```text
已读 / 未读
通知中心
历史风险列表
风险关闭记录
风险处理记录
```

---

## 4. 核心概念

### 4.1 风险信号

风险信号是系统根据已有对象字段推导出的事实。

例如：

```text
任务 deadline 已过，但任务未完成
→ 任务超期信号

任务 deadline 在 48 小时内，且任务未完成
→ 任务临期信号

任务 B 依赖任务 A，任务 A 已超期未完成
→ 任务 B 的依赖阻塞信号
```

风险信号本身不区分角色。

---

### 4.2 风险通知

风险通知是风险信号经过角色范围过滤、聚合、排序后，在控制台展示出来的提醒。

同一个底层风险信号，对不同角色可以有不同展示粒度。

例如：

```text
任务 A 已超期，任务 B 依赖任务 A
```

个人看到：

```text
你的任务 B 受上游任务 A 影响。
```

PM 看到：

```text
需求 X 存在依赖阻塞，可能影响推进。
```

TL 看到：

```text
本组任务 B 受上游任务 A 影响。
```

总监只有在需求 X 是自己关注需求时才看到：

```text
关注需求 X 存在依赖阻塞。
```

---

### 4.3 风险通知范围

风险通知范围由角色和关注关系决定。

P0 不做复杂订阅，不做风险规则配置。

---

## 5. P0 风险类型

第一版只保留两类风险：

```text
deadline 风险
依赖阻塞风险
```

暂不包含：

```text
手动阻塞风险
缺 AC 风险
跨组依赖风险
风险升级
关键路径风险
资源风险
质量风险
日报风险
```

其中“日报发送失败”等报告异常可以在报告模块内部处理，不纳入本风险通知体系。

---

## 6. 风险类型一：deadline 风险

### 6.1 任务临期

判断规则：

```text
任务状态 != 已完成
+
任务 deadline 在未来 48 小时内
=
任务临期风险
```

风险等级：

```text
中
```

展示文案示例：

```text
中 · 临期
验收标准模板将在明天到期
AI 日报生成 / 补充验收标准模板 · 负责人：韩梅梅
查看任务
```

点击跳转：

```text
/requirements?requirementId=xxx&taskId=xxx&focus=deadline
```

---

### 6.2 任务超期

判断规则：

```text
任务状态 != 已完成
+
任务 deadline 已过
=
任务超期风险
```

风险等级：

```text
高
```

展示文案示例：

```text
高 · 已超期
飞书发送联调已超过 deadline
日报发送 / 飞书发送联调 · 负责人：李雷
查看任务
```

点击跳转：

```text
/requirements?requirementId=xxx&taskId=xxx&focus=deadline
```

---

### 6.3 需求临期 / 超期

如果需求自身有 deadline，也可以按同样规则生成需求级 deadline 风险。

判断规则：

```text
需求状态 != 已完成
+
需求 deadline 在未来 48 小时内
=
需求临期风险
```

```text
需求状态 != 已完成
+
需求 deadline 已过
=
需求超期风险
```

展示文案示例：

```text
高 · 需求超期
AI 日报生成需求已超过计划时间
负责人：周芷若
查看需求
```

点击跳转：

```text
/requirements?requirementId=xxx&focus=deadline
```

---

## 7. 风险类型二：依赖阻塞风险

### 7.1 基础定义

任务之间存在强依赖关系。

```text
任务 B 依赖任务 A
```

含义：

```text
任务 A 是任务 B 的上游任务。
任务 A 未完成时，任务 B 的推进可能受影响。
```

但只有在上游任务已超期未完成时，才进入控制台风险通知。

---

### 7.2 判断规则

```text
当前任务 B 依赖上游任务 A
+
任务 A 状态 != 已完成
+
任务 A deadline 已过
=
任务 B 出现依赖阻塞风险
```

注意：

```text
任务 A 未完成，但未超期
→ 不通知
```

---

### 7.3 风险等级

P0 统一为：

```text
高
```

原因：

* 上游任务已超期；
* 当前任务存在明确依赖；
* 对下游任务推进有直接影响。

---

### 7.4 展示文案示例

```text
高 · 依赖阻塞
飞书发送联调等待上游任务完成
日报发送 / 飞书发送联调 · 负责人：李雷
上游任务「发送目标确认」已超期
查看依赖
```

点击跳转：

```text
/requirements?requirementId=xxx&taskId=xxx&focus=dependency
```

需求看板应定位：

```text
当前任务 B
上游任务 A
依赖关系
```

---

## 8. 风险等级规则

P0 使用简单等级。

| 风险类型       | 条件              | 等级 |
| ---------- | --------------- | -- |
| 任务 / 需求已超期 | deadline 已过且未完成 | 高  |
| 依赖阻塞       | 上游任务已超期且当前任务依赖它 | 高  |
| 任务 / 需求临期  | 48 小时内到期且未完成    | 中  |

不做：

```text
复杂评分
多因子权重
角色化等级规则
风险等级配置
```

---

## 9. 角色视图设计

P0 不做“每个角色一套风险规则”。

统一规则是：

```text
底层风险信号统一；
不同角色只做范围过滤、粒度聚合、排序展示。
```

---

### 9.1 个人视图

个人关注的是自己需要处理的任务。

默认范围：

```text
我负责的任务
我关注的任务
```

展示粒度：

```text
任务级
```

展示内容：

```text
我负责 / 关注的任务临期
我负责 / 关注的任务超期
我负责 / 关注的任务被上游超期任务阻塞
```

示例：

```text
高 · 已超期
飞书发送联调已超过 deadline
日报发送 / 飞书发送联调 · 负责人：我
查看任务
```

```text
高 · 依赖阻塞
飞书发送联调等待上游任务完成
上游任务「发送目标确认」已超期
查看依赖
```

---

### 9.2 PM 视图

PM 关注需求推进，不应被所有普通任务淹没。

默认范围：

```text
我负责的需求
我关注的需求
```

展示粒度：

```text
需求级 + 关键任务提示
```

展示内容：

```text
我关注 / 负责的需求临期
我关注 / 负责的需求超期
我关注 / 负责的需求下存在超期任务
我关注 / 负责的需求下存在依赖阻塞任务
```

示例：

```text
高 · 需求推进风险
AI 日报生成存在 2 个任务超期
影响：日报发送联调、验收标准补充
查看需求
```

点击跳转：

```text
/requirements?requirementId=xxx&view=risks
```

---

### 9.3 TL 视图

TL 负责推进组内任务，因此可以看到更接近任务粒度的风险。

默认范围：

```text
本组需求
本组任务
TL 关注的需求 / 任务
```

展示粒度：

```text
组内任务级 + 需求聚合
```

展示内容：

```text
本组任务临期 / 超期
本组任务依赖上游超期任务
本组关注需求下的任务风险
```

示例：

```text
高 · 组内任务超期
本组有 2 个任务已超过 deadline
影响需求：日报发送、需求看板原型
查看组内任务
```

示例：

```text
高 · 依赖阻塞
本组任务「飞书发送联调」等待上游任务完成
上游任务「发送目标确认」已超期
查看依赖
```

点击跳转：

```text
/requirements?view=risks&scope=team
```

---

### 9.4 总监视图

总监不看全部可见需求的风险，只看自己手动关注需求下的风险。

默认范围：

```text
总监手动关注的需求
```

展示粒度：

```text
需求级
```

展示内容：

```text
关注需求临期 / 超期
关注需求下存在超期任务
关注需求下存在依赖阻塞任务
```

不展示：

```text
普通未关注需求风险
普通任务临期
普通任务超期
普通任务依赖阻塞
```

示例：

```text
高 · 关注需求风险
AI 日报生成存在延期风险
影响：2 个任务已超期
查看需求
```

示例：

```text
高 · 依赖阻塞
关注需求「日报发送」存在上游超期任务
影响任务：飞书发送联调
查看需求
```

点击跳转：

```text
/requirements?requirementId=xxx&view=risks
```

## 11. 点击跳转规则

控制台不承载处理动作，所有风险项点击后跳转需求看板。

### 11.1 任务 deadline 风险

```text
/requirements?requirementId=xxx&taskId=xxx&focus=deadline
```

需求看板应：

```text
打开对应需求
定位对应任务
聚焦 deadline 信息
```

---

### 11.2 需求 deadline 风险

```text
/requirements?requirementId=xxx&focus=deadline
```

需求看板应：

```text
打开对应需求
聚焦需求 deadline 信息
```

---

### 11.3 依赖阻塞风险

```text
/requirements?requirementId=xxx&taskId=xxx&focus=dependency
```

需求看板应：

```text
打开对应需求
定位当前任务
展示上游任务
展示依赖关系
```

---

### 11.4 查看全部

```text
/requirements?view=risks
```

对于角色范围，可追加参数：

```text
/requirements?view=risks&scope=mine
/requirements?view=risks&scope=team
/requirements?view=risks&scope=followed
```

P0 如路由未实现，可先模拟跳转参数。

## 12. 空状态

如果当前角色没有风险项，展示轻量空状态：

```text
暂无需要关注的风险
```

或更精确：

```text
暂无临期、超期或依赖阻塞的关注对象
```

提供入口：

```text
查看需求看板
```

不要展示空图表、占位大卡片或无意义统计。

## 15. 当前决策汇总

当前版本已确认：

```text
第一版没有手动阻塞字段。
任务依赖只支持强依赖。
弱关联不进入本期设计。
上游任务未完成但未超期，不进入控制台风险通知。
依赖阻塞只在上游任务已超期未完成时产生。
总监只看自己手动关注需求的风险。
风险通知不持久化。
控制台只做风险提醒和跳转。
真正处理发生在需求看板。
```

## 16. 一句话基线

风险通知 P0 的核心基线是：

```text
基于需求、任务、deadline、强依赖、关注关系，临时计算当前角色需要关注的临期、超期和依赖阻塞提醒。
```

它不是风险管理系统。

控制台只回答：

```text
现在有什么风险？
影响哪个需求 / 任务？
为什么有风险？
点哪里去需求看板处理？
```
