## 核心操作梳理

1. 看板总览：查看有效需求、待评审、进行中、依赖阻塞等统计。
2. 视图切换：在“需求看板”和“任务树”之间切换。
3. 搜索和筛选：按关键词、阶段、优先级、风险过滤需求。
4. 关注：关注或取消关注需求、任务。
5. 刷新：刷新需求、任务、Token 来源。
6. 新建需求：进入新建需求页。
7. 阶段推进：在看板中拖拽需求卡片修改阶段。
8. 展开需求详情：点击需求卡片或树中需求行打开需求详情抽屉。
9. 查看任务结构：在任务视图展开需求，查看子任务、负责人、状态、进度、依赖、Token。
10. 添加任务：从需求详情或任务视图为空需求添加任务。
11. 查看任务详情：点击任务行或“详情”按钮打开任务详情抽屉。
12. 维护任务：更新进度、状态、上游依赖、Token 来源。
13. Token 关联：给需求或任务关联/移除工作记录来源。

## 主要 UX 风险

1. 当前页面像“demo”的原因不只是数据质量，还是信息架构偏散：统计卡、视图标题、筛选、看板列、详情抽屉都在讲不同粒度的事，缺少一个明确的“我现在该处理什么”的主线。
2. “任务树”命名不准确。当前视图不是严格的 tree，也不是任务维度主导，而是“需求 -> 任务”的执行明细列表。建议命名为“执行拆解”或“需求明细”，如果要强调依赖关系，可用“任务依赖”。
3. 看板卡片点击区域可打开详情，但缺少明确 affordance。用户需要猜测卡片可点；星标又在同一区域，容易误触。
4. 需求详情抽屉信息过长且同级堆叠。基础信息、验收标准、任务拆解、Token 来源都是完整模块，抽屉应先给决策摘要，再逐步展开细节。
5. Token 信息太技术化，且在需求页里权重过高。对 PM/主管而言，应先表达“工作记录证据/投入来源”，Token 数只是辅助。
6. 风险表达弱。看板有“依赖阻塞”，详情有“1 个依赖阻塞”，但没有把“阻塞原因、影响任务、下一步动作”前置。
7. 列表视图横向信息密度较高，进度、依赖、更新、操作混在同一行，对窄屏和低熟悉度用户不友好。

## 文案建议

页面标题：
- 现状：需求看板
- 建议：需求推进
- 原因：页面不只是 kanban，还包含拆解、风险、依赖和证据来源。

默认视图：
- 现状：需求看板
- 建议：阶段看板
- 原因：和页面标题区分，说明它是按阶段推进的视图。

任务树：
- 现状：任务树
- 首选建议：执行拆解
- 备选：需求明细、任务拆解、依赖视图
- 原因：当前结构只有两层，不是复杂树；核心价值是看需求如何拆成任务、谁负责、卡在哪里。

“按阶段推进需求”：
- 建议：按阶段推进
- 原因：标题更短，和页面标题不重复。

“需求与任务树”：
- 建议：执行拆解
- 描述建议：查看每个需求的任务、负责人、进度和阻塞。

“我关注的”：
- 建议：只看关注
- 原因：这是筛选动作，不是对象集合入口。

“全部风险”：
- 建议：风险类型
- 原因：Select placeholder 应表达筛选维度。

“展开 / 收起全部”：
- 建议：全部展开 / 全部收起，按当前状态拆成两个明确按钮或动态文案。

“验收标准”：
- 可保留；如果面向业务，可补充为“验收标准（AC）”。

“任务拆解”：
- 建议保留在详情抽屉中；它比“任务树”更符合用户认知。

“Token 摘要 / Token 来源”：
- 建议：工作记录来源
- 次级信息：显示 Token 总量和工具来源。

## 改版建议

1. 把页面定位改成“需求推进工作台”：顶部只保留关键健康指标、风险入口和主操作，避免过多装饰卡片。
2. 视图切换改为“阶段看板 / 执行拆解”。看板解决“处在哪个阶段”，拆解视图解决“谁在做什么、哪里阻塞”。
3. 需求卡片重新设计为三段：标题和优先级、进度和任务完成度、风险与截止日期。把“打开详情”做成明确的 hover 行为或右侧 icon 按钮。
4. 需求详情抽屉顶部改成固定摘要区：阶段、优先级、进度、截止、阻塞数、主 CTA。下面用 Tabs 分为“概览 / 任务 / 验收 / 工作记录”。
5. 执行拆解视图从纯表格改为分组列表：需求行作为 group header，展开后任务使用 compact row；阻塞任务置顶并给出依赖对象。
6. 风险语言从“有风险”改成“需要处理什么”：例如“1 个任务被上游阻塞”“2 天后截止，仍有 2 个任务未完成”。
7. 弱化 Token 数值的视觉权重，把它改成证据来源模块，只在 hover/展开时显示记录明细。

## 可访问性与可用性风险

1. 星标按钮只有图标，虽然有 aria-label，但视觉上缺少 tooltip，鼠标用户难以理解。
2. 需求行和任务行整行可点击，但没有明显 focus/keyboard 操作提示，需要确认键盘可达。
3. 抽屉里的卡片式模块太多，阅读顺序长；屏幕阅读器用户会遇到大量同级标题。
4. 颜色承担了部分状态信息，建议状态标签始终保留文字，不只依赖色彩。
